= 予測フレームループとタイミング数学

従来のデスクトップアプリケーションでは、エンジンのメインループはしばしば単純な垂直同期（**VSync**）に支配されます。可能な限り速く描画し、スキャンラインが許すときにモニタが結果を表示します。しかし空間コンピューティングでは、この「描画して表示」モデルでは不十分です。描画開始からフォトンが網膜に届くまでの間にユーザーの頭が少しでも動くと、シーンが揺れて見えたり遅れたりし、即座の不快感や乗り物酔いを引き起こします。

これを解決するため、OpenXR は **予測フレームループ** を使います。世界の *現在* の状態を描くのではなく、ディスプレイが実際にストローブするときに世界が *どう見えるか* を描きます。

== 空間コンピューティングのリズム

OpenXR フレームループは、エンジンと XR ランタイムの間の対話です。いつ描くかだけでなく、未来を予測することです。この対話は 3 本柱の上に構築されます。

1.  **厳格なペーシング**: ランタイムは `xrWaitFrame` 経由でアプリケーションの正確な心拍を指示します。単なるフレーム制限ではなく、エンジンをディスプレイの実際のリフレッシュサイクルに揃える同期点です。
2.  **予測表示時刻**: 処理するすべてのフレームには `predictedDisplayTime` が付きます。これはエンジンで最も重要な数値です。現在構築中のフレームをユーザーがいつ見るかを正確に伝えます。
3.  **シミュレーション整合**: この予測時刻を使い、物理、アニメーション、そして——最も重要な——ヘッドトラッキングを進めなければなりません。頭が *どこにあるか* ではなく、`predictedDisplayTime` で頭が *どこにいるか* を尋ねます。

== タイミング数学が重要な理由

この章では、エンジンのメインループをレガシーな `glfwWindowShouldClose` ロジックから OpenXR フレームライフサイクルへ移行します。次を学びます。

-   XR ランタイムの心拍と CPU／GPU 作業を同期する。
-   予測タイミングデータを使い、正確なビューと投影行列を計算する。
-   「スウィム」効果と、予測数学がそれをどう排除するかを理解する。

この章の終わりには、エンジンはユーザーの動きに「反応」するのではなく「先取り」し、高品質な空間コンピューティングを定義する滑らかで低レイテンシな体験を提供します。
