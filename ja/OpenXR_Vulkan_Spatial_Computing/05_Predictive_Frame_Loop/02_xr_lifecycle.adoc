= XR ライフサイクル

エンジンのメインループを OpenXR ライフサイクルへ移行するには、3 つの主要な関数——`xrWaitFrame`、`xrBeginFrame`、`xrEndFrame`——が関わります。これらの関数は空間アプリケーションの「心拍」として働き、標準の無制限ループやレガシーな `glfwPollEvents` ベースのペーシングを置き換えます。

== エコシステムの視点

この章は次のカテゴリに属します: **OpenXR ランタイムと Vulkan を使う**。

エンジンのメインループを OpenXR ライフサイクルと同期することは、フォトンからモーションへの整合に必須です。滑らかで安定した体験を保証するため、ランタイムが提供する予測表示時刻を使わなければなりません。

== 時間のパラドックス: 未来を描く

XR ライフサイクルを理解するには、まず根本的な真実を受け入れる必要があります。**あなたは常に、まだ起きていない未来を描いている。**

デスクトップゲームでは「今」起きていることを描きます。XR では、コマンド記録、GPU への投入、ディスプレイ上のピクセル点灯に時間がかかるため、ユーザーが見るころには「今」はすでに過去です。「現在」の頭のポーズを描くと、世界はユーザーの頭の動きに対して「スウィム」したり遅れたりして見えます。

XR ライフサイクルは、**予測表示時刻** を提供することでこれを解決するよう設計されています。

== 1. `xrWaitFrame`: ペーシングの心拍

`xrWaitFrame` 関数はエンジンの主要な同期点です。これを呼ぶと、ランタイムはヘッドセットのリフレッシュレート（例: 90Hz）に同期させるため実行をブロックする場合があります。

[source,cpp]
----
// Sync with the runtime heartbeat and retrieve the predicted display time
XrFrameState frameState{XR_TYPE_FRAME_STATE};
XrFrameWaitInfo frameWaitInfo{
    .type = XR_TYPE_FRAME_WAIT_INFO
};
xrWaitFrame(xrSession, &frameWaitInfo, &frameState);
----

*   **予測**: `frameState` は `predictedDisplayTime` を含みます。これは、これから描くピクセルがヘッドセットのディスプレイに実際に表示される正確なナノ秒です。
*   **描画すべきか？**: `frameState.shouldRender` は、ランタイムが実際にフレームを欲しがっているかを伝えます。ヘッドセットがオフ、またはユーザーがシステムメニューにいる場合、電力節約のため「ウェイト」はしても「描画」作業はスキップすることがあります。

== 2. `xrBeginFrame`: ウィンドウを開く

ウェイトが満たされると、`xrBeginFrame` を呼びます。これはフレームの GPU 作業の開始をシグナルします。

[source,cpp]
----
XrFrameBeginInfo frameBeginInfo{
    .type = XR_TYPE_FRAME_BEGIN_INFO
};
xrBeginFrame(xrSession, &frameBeginInfo);
----

シミュレーション更新とビュー予測を行うのはこの間なので、`xrWaitFrame` と `xrBeginFrame` の間の時間はできるだけ短く保つことが重要です。

== 3. `xrEndFrame`: コンポジタへの投入

最後のステップは `xrEndFrame`——`vkQueuePresentKHR` の空間版です。

[source,cpp]
----
XrFrameEndInfo frameEndInfo{
    .type = XR_TYPE_FRAME_END_INFO,
    .displayTime = frameState.predictedDisplayTime,
    .environmentBlendMode = chosenBlendMode,
    .layerCount = 1,
    .layers = &layerPtr
};
xrEndFrame(xrSession, &frameEndInfo);
----

単一イメージを「プレゼント」するだけでなく、`xrEndFrame` は **Composition Layers** の配列を受け取ります。ランタイムのコンポジタは最後のマイクロ秒でこれらを重ね、描画完了後にユーザーの頭が動いた場合は **Late-Stage Reprojection（LSR）** で最終イメージを「微調整」することさえできます。

== 発展: サブフレームワークロードとレイテンシ監視

OpenXR がフレームペーシングを扱う一方、Vulkan ではより粒度の細かい制御を実装できます。

*   **サブフレームワークロードの調整**: Vulkan の **タイムラインセマフォ** を使い、メインの XR ライフサイクルから非同期に動くワークロードを実装できます。たとえば、`xrWaitFrame` を呼ぶ前に複雑な可視性またはライトベイキングのコンピュートパスをキックオフし、CPU が心拍を待っている間も GPU を忙しく保てます。
*   **GPU 側レイテンシの予測**: ランタイムが表示時刻を提供しますが、Vulkan の **Query Pool** で自前の GPU 側レイテンシを測定できます。このデータによりエンジン内部の予測オフセットを動的に調整し、汎用ランタイム予測よりタイトな整合を提供できます。

== エンジンスループのまとめ

エンジンのメインループは、今や次の正確なリズムに従います。

1.  **イベントポール**: システムメッセージ（「ヘッドセット接続」など）をチェックする。
2.  **Wait**: ランタイムの心拍と同期し、未来の `displayTime` を取得する。
3.  **Begin**: フレーム作業を開始する。
4.  **Render**: 未来の時刻でポーズを予測し、Vulkan コマンドを記録して投入する。
5.  **End**: 完成した（またはキューされた）レイヤーを表示のためランタイムに返す。
