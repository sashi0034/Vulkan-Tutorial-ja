= マルチレイヤー合成

**Quad-View** を描画するだけでは戦いの半分です。4 ビュー（ベースの眼と高ディテールのインセット）を得た後、最終表示のため **OpenXR コンポジタ** へ渡す必要があります。ここで **マルチレイヤー合成** が登場します。

== エコシステムの視点

この章は次のカテゴリに属します: **OpenXR ランタイムと Vulkan を使う**。

複数の合成レイヤーを投入することは OpenXR の標準プロセスです。各投影レイヤーをコンポジタへ正しく受け渡すため、ランタイムのレイヤー API と Vulkan の同期プリミティブを使わなければなりません。

== レイヤースタック: モジュラーな画像

OpenXR では単一イメージを投入するだけではありません。**Composition Layers** の配列を投入します。コンポジタはディスプレイに届く前に、最終の高優先度パスでこれらの画像を重ねます。

1.  **ベースレイヤー**: 標準の 3D 世界（クアッドビューのインデックス 0 と 1）。
2.  **インセットレイヤー**: 高ディテールの中心窩データ（インデックス 2 と 3）。
3.  **オーバーレイレイヤー**: 完璧にシャープなままの平面 HUD またはメニュー。

[source,cpp]
----
// 1. Defining Composition Layers in OpenXR using designated initializers
XrCompositionLayerProjection baseLayer{
    .type = XR_TYPE_COMPOSITION_LAYER_PROJECTION,
    .layerFlags = XR_COMPOSITION_LAYER_CORRECT_CHROMATIC_ABERRATION_BIT,
    .space = worldSpace,
    .viewCount = 2,
    .views = baseViewsPtr
};

XrCompositionLayerProjection insetLayer{
    .type = XR_TYPE_COMPOSITION_LAYER_PROJECTION,
    .layerFlags = XR_COMPOSITION_LAYER_CORRECT_CHROMATIC_ABERRATION_BIT,
    .space = worldSpace,
    .viewCount = 2,
    .views = insetViewsPtr
};

// 2. Submitting layers to xrEndFrame
std::vector<XrCompositionLayerBaseHeader*> layers = {
    reinterpret_cast<XrCompositionLayerBaseHeader*>(&baseLayer),
    reinterpret_cast<XrCompositionLayerBaseHeader*>(&insetLayer)
};

XrFrameEndInfo frameEndInfo{
    .type = XR_TYPE_FRAME_END_INFO,
    .displayTime = frameState.predictedDisplayTime,
    .layerCount = static_cast<uint32_t>(layers.size()),
    .layers = layers.data()
};
xrEndFrame(session, &frameEndInfo);
----

順序が重要です。リストの最後のレイヤーが最前面に描かれます。

== マスキングとブレンディング

コンポジタは高ディテールのインセットをどこに表示すべきかをどう知るのでしょうか？各レイヤーは **Projection View** を指定します。クアッドビューでは、インセットレイヤーのビューはより小さい FOV を持ちますが、カメラポーズは同じです。

コンポジタは FOV タンジェントを使い、高ディテール画像を低ディテールの上に数学的に「配置」します。その後 **ソフトエッジブレンド** を行い、20 PPD と 60 PPD 領域の間の遷移がユーザーに見えないことを保証します。

== 発展: 物理ベース合成とアルファブレンディング

Vulkan では、ランタイムのデフォルトを超えて独自の深度対応レイヤーマージを行えます。

*   **カスタムレイヤーマージ**: ランタイムはしばしば数レイヤーに制限されますが、Vulkan の **サブグループ演算** と **Input Attachments** を使い、受け渡し前に自前のレイヤーを単一の統合イメージへマージし、ランタイム制限を迂回できます。
*   **共有深度バッファ**: 特定の拡張を使わない限り、ランタイムはレイヤーの深度値を知りません。Vulkan で全合成レイヤーにまたがる **共有深度バッファ** を作成し、HUD、世界、インセット間の完璧なピクセル単位オクルージョンを有効にできます。

== リソース同期: マルチスワップチェーンペーシング

エンジンの視点では、これは今や毎フレーム **複数のスワップチェーン** を管理することを意味します。

*   **ペーシング**: *各* スワップチェーン（ベースとインセット）について `xrWaitSwapchainImage` と `xrAcquireSwapchainImage` を呼ばなければなりません。
*   **Vulkan 同期**: コマンドバッファ記録は、`xrReleaseSwapchainImage` を呼ぶ前に 4 ビューすべてが描画・遷移済みであることを保証しなければなりません。

このマルチレイヤーアプローチにより、各レイヤーが独自の解像度とビット深度を持て、コンポジタがシーン各部の特定のニーズに基づき最終画像を最適化できます。
