= 空間ビュー向けダイナミックレンダリング

OpenXR を Vulkan 1.3 と統合する旅路で、堅牢なハンドシェイクを確立し、実際には XR ランタイムが所有するイメージの管理方法を学びました。しかし、イメージを持つだけでは戦いの半分です。次にステレオビューをそこに描画する必要があります。以前のエンジン構築シリーズでは、**ダイナミックレンダリング**（`VK_KHR_dynamic_rendering` 経由）を主要なレンダリングパスとして採用しました。ここでは、その決定が空間コンピューティングにとってなぜ極めて重要だったかを見ます。

== XR でダイナミックレンダリングを活用する理由

空間コンピューティングでは、すでに構築したダイナミックレンダリングシステムが、レガシーなレンダーパスパイプラインでは苦戦する 3 つの大きな利点を提供します。

1.  **投入レイテンシの削減**: 複雑なレンダーパスとフレームバッファオブジェクトの必要をすでに迂回しているため、ビュー行列（次章で予測する）をより速くコマンドバッファ投入へ変換できます。
2.  **適応的ビューポート**: XR ヘッドセットはしばしば非対称投影と非平行ビューを使います。ダイナミックレンダリングパスでは、事前定義された静的構造に合わせる必要なく、眼ごとにビューポートとシザー状態を再構成するのが容易です。
3.  **マルチビュー統合**: シングルパスステレオレンダリング向けに `VK_KHR_multiview` へ移るとき、既存のダイナミックレンダリングロジックはシームレスに統合し、`vk::RenderingInfo` で直接 `viewMask` を指定できます。

この章では、エンジンのレンダリングロジックを空間ディスプレイの固有要件に適用します。まず XR スワップチェーンイメージをレンダリングセッションに直接バインドする方法を見、次に現代ヘッドセットが要求する複雑なビューポートの管理へ進みます。
