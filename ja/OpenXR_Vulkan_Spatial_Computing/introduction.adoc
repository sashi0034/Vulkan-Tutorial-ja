= OpenXR と Vulkan 1.3 による空間コンピューティング: はじめに

空間コンピューティングの最前線へようこそ。このチュートリアルシリーズでは、従来の 2D エンジンアーキテクチャと、Virtual Reality (VR)、Augmented Reality (AR)、Mixed Reality (MR)——総称して XR——に求められる没入感と低レイテンシ要件の間のギャップを埋めていきます。

== 前提条件

このチュートリアルを始める前に、Vulkan API とエンジンアーキテクチャの両方についてしっかりした基礎があることが望ましいです。具体的には、次を完了していることを前提とします。

1.  **xref:00_Introduction.adoc[Core Vulkan Tutorial]**: インスタンス作成、物理デバイス、論理デバイス、基本的なグラフィックスパイプラインなど、Vulkan のコア概念に慣れていること。
2.  **xref:Building_a_Simple_Engine/introduction.adoc[Building a Simple Engine]**: 本シリーズはエンジンチュートリアルで確立したコードとアーキテクチャパターンの上に直接構築されます。出発点として `Simple Engine` のコードベースを使用します。
3.  **モダンな C{pp} と RAII**: シリーズ全体で `vulkan-hpp` と RAII (Resource Acquisition Is Initialization) を利用します。C{pp} 統合のセクションを追うには、これらのパターンに慣れている必要があります。

== OpenXR エコシステム: ランタイムとローダー

コードに入る前に、OpenXR を動かす 2 つの中核コンポーネント——**Loader（ローダー）** と **Runtime（ランタイム）**——を理解しておくことが重要です。

1.  **OpenXR Loader**: アプリケーションがリンクする、ベンダー中立の小さなライブラリです。主な役割は交通整理役です。エンジン起動時、ローダーはシステムのレジストリや環境変数を見て、現在アクティブな XR ランタイムを見つけ、すべての API 呼び出しをそこにリダイレクトします。
2.  **OpenXR Runtime**: Meta、Valve、Microsoft、あるいはオープンソースの Monado プロジェクトなど、ハードウェアメーカーが提供する本丸です。ランタイムには OpenXR 仕様の実実装が含まれ、カメラ、ディスプレイ、トラッキングセンサー向けのドライバもここにあります。

このチュートリアルに従うには、開発マシン上に OpenXR 互換の環境が必要です。通常は SDK をインストールし、ランタイムがアクティブであることを確認します。

[TIP]
====
最新のインストール手順、必要なヘッダとローダーライブラリの入手については、公式の **link:https://github.com/KhronosGroup/OpenXR-SDK-Source[Khronos OpenXR-SDK-Source]** リポジトリを参照し、**link:https://www.khronos.org/registry/OpenXR/specs/1.1/html/xrspec.html[OpenXR Specification]**、またはプラットフォーム固有のセットアップガイドに優れた **link:https://www.openxr-tutorial.com/[OpenXR Tutorial]** に従うことを強く推奨します。

== なぜ OpenXR か？

XR の黎明期、開発者は市場のヘッドセットごとにカスタムバックエンドを書かなければなりませんでした。OpenXR はクロスプラットフォームで高性能な API を提供し、Meta Quest から Valve Index、Microsoft HoloLens まで、単一のアプリケーションを幅広いデバイスで動かせるようにしました。

しかし OpenXR は単なるラッパーではありません。ハードウェアのポーズ、予測フレームタイミング、そして XR ランタイム自身が所有する専用スワップチェーンを管理する、洗練された状態機械です。効果的に使うには、エンジンのメインループ、メモリ、同期の扱い方を見直し必要があります。

== Vulkan 1.3 の利点

Vulkan 1.3 は、空間コンピューティングで特に強力ないくつかの重要な機能をもたらします。

* **タイムラインセマフォ**: エンジンと XR コンポジタ間に必要な、複雑なプロセス間同期に不可欠です。
* **ダイナミックレンダリング**: ステレオビュー向けの柔軟で軽量なレンダリングパスを提供し、レガシーな Render Pass の硬直した状態を避けます。
* **Synchronization 2**: 低レイテンシなレイトラッチングに必要な所有権転送とバリアを簡素化します。
* **Maintenance 拡張**: ハードウェア境界をまたぐメモリとリソース可視性の制御を改善します。

== 何を構築するか

本シリーズ全体で、XR フレームのライフサイクル全体を扱います。

1.  **ハンドシェイク**: LUID マッチングと必須拡張を使い、OpenXR を Vulkan コンテキストに接続する。
2.  **リソース管理**: ランタイム所有のイメージを、RAII ベースのエンジン抽象にラップする。
3.  **予測ループ**: フレームタイミングを習得し、ユーザーが見るものが、表示瞬間の頭の位置と正確に一致するようにする。
4.  **空間シェーダー**: Slang を使い、効率的なマルチビューおよびフォビエイテッドレンダリングシェーダーを記述する。
5.  **高度なセンシング**: 環境メッシュを取り込み、ML 推論で空間データを洗練する。

このチュートリアルの終わりには、動作する XR アプリケーションだけでなく、現代の Vulkan エコシステムにおける高性能な空間コンピューティングに必要なアーキテクチャパターンへの深い理解が得られます。
