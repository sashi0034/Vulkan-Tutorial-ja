= 空間デバッグ: ブラックボックスを覗く

空間エンジンのデバッグは、標準の 2D アプリケーションとは根本的に異なります。XR ランタイムとその内部コンポジタはしばしばブラックボックスであるため、レンダリングエラーがエンジンか、OpenXR ハンドシェイクか、コンポジタ自身の再投影ロジックかを判断するのが難しいことがあります。

== エコシステムの視点

この章は次のカテゴリに属します: **OpenXR ランタイムと Vulkan の併用**。

デバッグラベルで空間エンジンを計装し、OpenXR の診断拡張を利用することは、プロ開発の標準要件です。ランタイムのエコシステム内でアプリケーション挙動の統一ビューを作るには、これらのツールを使う必要があります。

== 概念: オブジェクトタグ付け

**XR_EXT_debug_utils** 拡張は、Vulkan の `VK_EXT_debug_utils` と機能的に同一で、OpenXR リソースに「ラベル」を付けられます。**Action Sets**、**Spaces**、**スワップチェーン** に名前を付けることで、RenderDoc などの外部ツールでこれらの名前を見られます。

[source,cpp]
----
// Tagging an OpenXR swapchain using designated initializers
XrDebugUtilsObjectNameInfoEXT nameInfo{
    .type = XR_TYPE_DEBUG_UTILS_OBJECT_NAME_INFO_EXT,
    .objectType = XR_OBJECT_TYPE_SWAPCHAIN,
    .objectHandle = reinterpret_cast<uint64_t>(xrSwapchain),
    .objectName = "Main World Spatial Swapchain"
};

// Now, tools like RenderDoc will display this name for the resource
xrSetDebugUtilsObjectNameEXT(instance, &nameInfo);
----

デバッグが有効なとき、ランタイムはしばしば内部検証レイヤを活性化し、`predictedDisplayTime` などのパラメータが有効範囲内であることを保証します。

== 空間エンジン向け RenderDoc

**RenderDoc** は Vulkan デバッグのゴールドスタンダードで、空間パイプラインに優れたサポートを提供します。

1.  **マルチビュー検査**: マルチビュースワップチェーンの各レイヤを個別に検査し、一方の目だけが正しく描けない理由を見つけられる。
2.  **リソース状態検査**: `vk::RenderingInfo` が正しい `viewMask` を持ち、**Synchronization 2** バリアが正しいステージを待っていることを検証できる。
3.  **コンポジタの洞察**: 一部の高度なヘッドセットでは、RenderDoc が最終のワープ・ブレンド済みフレームをキャプチャでき、レイヤがどう操作されているかを正確に見られる。

== 発展: シェーダー内デバッグと同期検証

Vulkan は、標準検証を超える診断の深さを可能にします。

*   **シェーダー内デバッグ**: Vulkan の **Debug Printf** を使い、計算された FOV タンジェントなど、シェーダーから直接リアルタイムデータを出力でき、OpenXR レイヤが提供しない空間パイプラインへの可視性を得られる。
*   **クロス API レース条件の追跡**: Vulkan の **Synchronization Validation** を使い、スワップチェーンイメージがエンジンとコンポジタの両方から同時にアクセスされている瞬間を正確に検出し、フリッカやジャダーのデバッグを助けられる。

== フレーム分析: スタッタを見つける

1 フレームのドロップでも、没入感を壊す「スタッタ」を引き起こし得ます。プロファイリングツールを使い、**Late Latching** 待機が長すぎないかを確認します。ほとんどのランタイムは **Compositor Pacing** を示すトレーシングツールを提供し、GPU 時間が低いのに同期レース条件でフレームミスが多い場合の特定を助けます。
