= 空間診断のエンジンへの組み込み

**空間診断と CI/CD** を統合すると、`simple_game_engine` は自動テストと深いリソース検査を通じて絶対的な安定性を維持できます。`Renderer` にテレメトリフックを追加し、`engine.cpp` で「ヘッドレス」モードを構成して実現します。*現時点では、実装は読者への演習として残されています。*

== 診断ラベルの追加

`renderer_core.cpp` では、リソース作成ロジックを拡張し、OpenXR と Vulkan のデバッグラベルを含めます。これにより RenderDoc でフレームをキャプチャしたとき、各リソースが何を表すかを正確に見られます。

[source,cpp]
----
// renderer_core.cpp
void Renderer::setupSpatialResourceLabels(const XrInstance& xrInstance) {
    if (!xrMode || !hasDebugUtilsSupport) return;

    // 1. Tag the spatial swapchain for RenderDoc identification
    XrDebugUtilsObjectNameInfoEXT nameInfo{
        .type = XR_TYPE_DEBUG_UTILS_OBJECT_NAME_INFO_EXT,
        .objectType = XR_OBJECT_TYPE_SWAPCHAIN,
        .objectHandle = reinterpret_cast<uint64_t>(spatialSwapchain),
        .objectName = "Spatial Swapchain"
    };

    xrSetDebugUtilsObjectNameEXT(xrInstance, &nameInfo);

    // 2. Add a diagnostic hook to our main command buffer
    // This allows us to track the frame's progress in a GPU capture
    vk::DebugUtilsLabelEXT frameLabel{ .pLabelName = "Spatial Frame Rendering" };
    commandBuffer.beginDebugUtilsLabelEXT(frameLabel);
}
----

== 自動 CI/CD ヘッドレスモードの有効化

自動テストのため、`engine.cpp` 内でエンジンを「ヘッドレス」モードで動くよう構成します。この状態では、実 OpenXR ランタイムをシミュレータに置き換え、AI ビジョン分析用にスクリーンショットを保存します。

[source,cpp]
----
// engine.cpp
void Engine::runSpatialCITest(uint32_t totalFrames) {
    // 1. Configure the XR Context for the Headless Simulator
    xrContext.enableSimulatorMode("Baseline_Tests.json");

    for (uint32_t i = 0; i < totalFrames; ++i) {
        // 2. Perform the standard update and render
        update(fixedTimeStep);
        render();

        // 3. Capture every 100th frame for AI-driven visual validation
        if (i % 100 == 0) {
            renderer->captureFrameToDisk("CI_Spatial_Capture_" + std::to_string(i) + ".png");
        }
    }
}
----

== なぜこれらの変更が必要か？

診断フックと自動テストをエンジンに直接組み込むことで、新しい空間機能（**ライトフィールド** 合成や **Late Latching** など）が視覚回帰や性能回帰を導入しないことを保証します。`simple_game_engine` は内部 **Watchdog** 系でこれらのテレメトリポイントを利用し、**モーション・ツー・フォトン** ギャップが目標閾値を超えた場合に開発者へ警告します。

この「空間ファースト」CI/CD アプローチこそが、プロチームが多様なハードウェアターゲットにわたって高忠実度体験を維持しつつ、コードベースを安定かつ信頼できる状態に保つことを可能にします。
