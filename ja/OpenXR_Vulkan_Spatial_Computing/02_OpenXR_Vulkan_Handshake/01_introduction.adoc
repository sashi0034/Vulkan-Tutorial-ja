= OpenXR-Vulkan 1.3 ハンドシェイク: はじめに

XR ヘッドセットに 1 ピクセルでも描画する前に、アプリケーション、OpenXR ランタイム、Vulkan グラフィックスドライバの間で「ハンドシェイク」を確立しなければなりません。標準の Vulkan インスタンスを作ってライブラリに渡すほど単純ではなく、拡張の交渉とハードウェア検証の協調した手順が必要です。

この章では、成功する空間ハンドシェイクの 3 本柱を見ていきます。

1.  **システム統合**: 必須の OpenXR 拡張、特に `XR_KHR_vulkan_enable2` をサポートするよう、エンジンの `VulkanContext` を拡張する方法。
2.  **ハードウェア整合**: Locally Unique Identifier（**LUID**）を使い、OpenXR と Vulkan がまったく同じ物理 GPU と話していることを保証する。プロセス間のメモリ可視性と性能にとって極めて重要です。
3.  **Vulkan 1.3 要件**: タイムラインセマフォ、ダイナミックレンダリング、Synchronization 2 など、空間パイプラインを最小レイテンシで動かす現代的機能を有効化する。

== ハンドシェイクの概念

標準のデスクトップアプリケーションでは、エンジンがボスです。インスタンスを作り、デバイスを選び、スワップチェーンを所有します。OpenXR では関係はパートナーシップに近いです。**XR Runtime**（SteamVR や Oculus サービスなど、ヘッドセットを駆動するソフトウェア）は、合成レイヤーをレンダリングストリームに安全に注入できるよう、Vulkan インスタンスの構成を正確に知る必要があります。

このハンドシェイクを正しく行わないと、XR セッションを初期化できなかったり、さらに悪い場合、異なる GPU メモリコンテキスト間でイメージのコピーを強いられ、大幅な性能低下が起きたりします。

== OpenXR アプリケーションにおける Vulkan 1.3 の利点

*   CPU シミュレーションと GPU コンポジタ間の **低レイテンシ同期**。
*   マルチビューによる **シングルパスレンダリング**。
*   レガシーなレンダーパス状態のオーバーヘッドなしの **ダイレクトトゥディスプレイ投入**。

この章の終わりには、エンジンの初期化コードが完全に空間対応になり、続く予測フレームループとランタイム所有スワップチェーンの土台が整います。
