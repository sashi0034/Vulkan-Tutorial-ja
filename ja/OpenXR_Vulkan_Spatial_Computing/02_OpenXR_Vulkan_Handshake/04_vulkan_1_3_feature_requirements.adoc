= Vulkan 1.3 機能要件

空間ハンドシェイクの最後のステップは、`VkDevice` が正しい機能セットで初期化されていることを保証することです。OpenXR は技術的には古い Vulkan バージョンでも動作しますが、**Vulkan 1.3** 仕様は現代の空間コンピューティングの「ゴールドスタンダード」です。以前は任意だったいくつかの拡張をコア API に取り込み、高性能ツールの保証されたベースラインを提供します。

== エコシステムの視点

この章は次のカテゴリに属します: **OpenXR ランタイムと Vulkan を使う**。

タイムラインセマフォやダイナミックレンダリングなどの現代的 Vulkan 機能を有効化することは、OpenXR のベストプラクティスに沿った低レイテンシな空間パイプラインを構築するために不可欠です。

== 1. タイムラインセマフォ: 非同期の心拍

標準アプリケーションでは、GPU の同期にバイナリセマフォを使います。バイナリセマフォは使い捨て「進め」信号のようなものです。CPU シミュレーション、GPU レンダー、（別プロセスかもしれない）XR コンポジタ間に複雑な依存がある XR では、バイナリセマフォは硬すぎます。

**タイムラインセマフォ** は単調増加する 64 ビット値を提供します。GPU 作業の「グローバルクロック」と考えてください。XR ランタイムはこれらを使い、フレームが合成準備完了になった正確な瞬間をシグナルし、GPU 作業の「グローバルクロック」を管理して、リプロジェクションが可能な限り最後の瞬間に起きるようにします。

*   **ペーシング**: 単一タイムラインで値 `500` に達するだけで、「フレーム 500 は合成準備完了」とシグナルできます。
*   **レイトラッチング**: 最終行列更新を始める前に、「CPU が値 501 をシグナルするまで待て」と GPU に伝えられます。
*   **プロセス間同期**: OS ハンドルへエクスポートできるため、タイムラインセマフォは高価な CPU リードバックなしに、エンジンと XR ランタイムを完璧に同期させ続けます。

== 2. Synchronization 2: キュー所有権の習得

**Synchronization 2**（`VK_KHR_synchronization2`）は、パイプラインステージとアクセスマスクをより統一された構造にまとめ、同期を簡素化します。XR では、**キュー所有権転送** の扱いに極めて重要です。

XR ランタイムがスワップチェーンイメージを所有するため、描画完了時に自分のキューで「Release」操作を行い、ランタイム側で「Acquire」を行わなければなりません。Synchronization 2 を使うと、ランタイムはこれらの高性能な受け渡しを明示的に行い、エンジンとコンポジタが同じメモリに同時アクセスしないことを保証します。

== 3. ダイナミックレンダリング: 究極の柔軟性

XR スワップチェーンイメージに対し直接 `vkCmdBeginRendering` を使うことで、レガシーな `VkRenderPass` オブジェクトの重いオーバーヘッドと硬直した状態を避けます。

*   **非対称ビュー**: ほとんどのヘッドセットは左右のアイフラスタムが平行でない「キャンテッドディスプレイ」を使います。これはしばしば各眼で異なるビューポートとシザー設定を必要とします。
*   **動的解像度**: 安定した 90Hz を維持するため、エンジンは周辺視野の解像度を即座に下げる必要があるかもしれません。
*   **ランタイムによるビューポート管理**: ランタイムは、眼ごとに高価なフレームバッファ状態を再作成せずに多様なヘッドセットアーキテクチャを扱うため、**ダイナミックレンダリング** の柔軟性に依存します。

== RAII での Vulkan 1.3 機能の有効化

Vulkan 1.3 は機能の有効化を簡素化します。深い入れ子の `pNext` チェーンの代わりに、統一された `VkPhysicalDeviceVulkan13Features`（とその前身）を使い、指定初期化子でコア要件を切り替えられます。

[source,cpp]
----
// Enabling mandatory features for our spatial pipeline using designated initializers
vk::PhysicalDeviceVulkan12Features features12{
    .timelineSemaphore = VK_TRUE
};

vk::PhysicalDeviceVulkan13Features features13{
    .pNext = &features12,
    .synchronization2 = VK_TRUE,
    .dynamicRendering = VK_TRUE
};

vk::DeviceCreateInfo createInfo{
    .pNext = &features13,
    .queueCreateInfoCount = static_cast<uint32_t>(queueCreateInfos.size()),
    .pQueueCreateInfos = queueCreateInfos.data(),
    .enabledExtensionCount = static_cast<uint32_t>(deviceExtensions.size()),
    .ppEnabledExtensionNames = deviceExtensions.data()
};

vk::raii::Device device(physicalDevice, createInfo);
----

== 発展: OpenXR 必須セットを超えて

ハンドシェイクはベースラインを保証しますが、Vulkan 1.3 では OpenXR 仕様がまだ必須としていない「次世代」機能も有効化できます。

*   **メッシュシェーディングとレイトレーシング**: Vulkan の機能発見を使い、リアルタイムの空間反射や複雑なジオメトリ処理など、高忠実度の空間効果向けにこれらの強力なパイプラインを有効化でき、OpenXR 標準の追いつきを待つ必要はありません。
*   **カスタムパイプラインキャッシュ**: 独自の **VkPipelineCache** を管理することで、ランタイムの汎用シェーダー管理を迂回し、異なるヘッドセット構成でも空間シェーダーが即座にロードされることを保証できます。

TIP: Vulkan 1.3 とコア機能の詳細は、公式の link:https://registry.khronos.org/vulkan/specs/1.3-extensions/html/vkspec.html[Vulkan Specification]、link:https://docs.vulkan.org/guide/latest/index.html[Vulkan Guide]、および link:00_introduction.adoc[メインチュートリアルシリーズ] を参照してください。

== ハンドシェイクのまとめ

これで次を成功させました。
1.  OpenXR と拡張を交渉するよう `VulkanContext` を拡張した。
2.  ハードウェア LUID を使い `VkPhysicalDevice` 選択を整合した。
3.  低レイテンシな空間レンダリングに必要な現代的 Vulkan 1.3 機能を有効化した。

ハンドシェイクが完了したので、XR エンジンにおける最も大きなアーキテクチャ変更——エンジン所有スワップチェーンから **ランタイム所有スワップチェーン** への移行——に取り組む準備が整いました。
