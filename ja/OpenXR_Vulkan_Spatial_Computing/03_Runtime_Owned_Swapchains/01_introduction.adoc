= ランタイム所有スワップチェーンとイメージラッピング: はじめに

標準の Vulkan アプリケーションでは、エンジンがスワップチェーンを完全に制御します。`VkSwapchainKHR` を作成し、イメージ数とフォーマットを決め、メモリを所有します。空間コンピューティングでは、このモデルは逆転します。

快適な XR 体験に必要なほぼゼロのレイテンシを達成するため、**XR Runtime**（コンポジタ）が最終的なイメージセットを所有しなければなりません。これによりランタイムは、ピクセルがディスプレイにストローブされる正確な瞬間のユーザーの頭のポーズに合わせてイメージを後期ワープする **非同期リプロジェクション** を、データを自前のメモリ空間へコピーせずに行えます。

この章では、エンジン所有からランタイム所有リソースへのアーキテクチャ移行を探ります。

1.  **外部イメージ交渉**: OpenXR からスワップチェーンイメージを要求し、ランタイムが提供するフォーマットを扱う方法。
2.  **RAII リソース統合**: ランタイムが提供する生の `VkImage` ハンドルを、エンジンの `vk::raii::Image` および `vk::raii::ImageView` 抽象にラップする。
3.  **メモリ所有権ライフサイクル**: 標準の `vkAcquireNextImageKHR` ループに代わる **Wait-Acquire-Release** サイクルを習得する。

== 「Wait-Acquire-Release」のリズム

最も大きな変更は、レンダーターゲットへのアクセス方法です。デスクトップアプリケーションではスワップチェーンにイメージインデックスを求め、そこに描画します。OpenXR では厳密な 3 ステッププロトコルに従わなければなりません。

*   **Wait**: ランタイムがスワップチェーンイメージを渡せる準備ができるまでブロックする。
*   **Acquire**: 特定のイメージインデックスの所有権を正式に取得する。
*   **Release**: コマンドバッファ投入後、イメージをコンポジタに返す。

このリズムにより、コンポジタが表示中またはワープ中のイメージに描画することがなくなり、ティアリングを防ぎ、可能な限り高い視覚的安定性を保証します。

この章の終わりには、既存の RAII ラッパーを使いながらランタイムの所有権を尊重し、空間ビューをヘッドセットのコンポジタバッファへ直接描画できるようになります。
