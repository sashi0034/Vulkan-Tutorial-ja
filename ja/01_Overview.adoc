= 概要

この章では、まず Vulkan とその解決する問題について紹介します。その後、
最初の三角形に必要な要素を見ていきます。これにより、続く各章を位置づけるための
全体像が得られます。最後に、Vulkan API の構造と一般的な使い方のパターンを
扱います。

== Vulkan の起源

これまでのグラフィックス API と同様に、Vulkan は
https://en.wikipedia.org/wiki/Graphics_processing_unit[GPU]
を抽象化するクロスプラットフォームの抽象化として設計されています。
これらの API の多くに共通する問題は、設計された当時のグラフィックスハードウェアが、
主に設定可能な固定機能に限られていたことです。プログラマは標準形式で頂点データを
提供する必要があり、ライティングやシェーディングの選択肢については GPU
メーカーの裁量に任されていました。

グラフィックスカードのアーキテクチャが成熟するにつれ、ますます多くのプログラマブルな
機能が提供されるようになりました。この新しい機能はすべて、何らかの形で既存の API に
統合されなければなりませんでした。その結果、理想的とは言えない抽象化と、
プログラマの意図を現代のグラフィックスアーキテクチャに対応させるための、
グラフィックスドライバ側での多くの推測が生まれました。だからこそ、ゲームの
パフォーマンスを改善するためのドライバ更新が、時には大幅な改善をもたらすほど
多いのです。これらのドライバの複雑さのため、アプリケーション開発者も、
https://en.wikipedia.org/wiki/Shader[シェーダ] で受け入れられる構文のように、
ベンダー間の不整合に対処する必要があります。
これらの新機能に加えて、過去10年では強力なグラフィックスハードウェアを持つ
モバイルおよび組み込みデバイスの流入もありました。これらのモバイル GPU は、
エネルギーとスペースの要件に基づく異なるアーキテクチャを持っています。
その一例が https://en.wikipedia.org/wiki/Tiled_rendering[タイルレンダリング]
であり、プログラマがこの機能をより細かく制御できれば、パフォーマンス向上の
恩恵を受けられます。
これらの API の時代から続くもう一つの制限は、マルチスレッドサポートの不足であり、
これが CPU 側のボトルネックになり得ます。

Vulkan は、現代のグラフィックスアーキテクチャ向けにゼロから設計されることで、
これらの問題を解決します。機能は豊富だが冗長な API を使ってプログラマが意図を
明確に指定できるようにすることでドライバのオーバーヘッドを削減し、
複数のスレッドが並列にコマンドを作成・投入できるようにします。
単一のコンパイラによる標準化されたバイトコード形式に切り替えることで、
シェーダコンパイルの不整合を減らします。最後に、現代のグラフィックスカードの
汎用処理能力を認め、グラフィックスとコンピュートの機能を単一の API に
統合しています。

=== コーディング規約

すべての Vulkan 関数、列挙型、構造体は `vulkan.h` ヘッダで定義されており、
これは LunarG が開発した https://lunarg.com/vulkan-sdk/[Vulkan SDK]
に含まれています。次の章でこの SDK のインストールを見ていきます。
本チュートリアルでは、公式の Vulkan SDK に同梱されている vulkan.hpp ヘッダが
提供する C++ Vulkan API を使います。このヘッダは、生の C Vulkan API に対して
型安全で RAII フレンドリー、やや扱いやすいインターフェースを提供しつつ、
基盤となる Vulkan 関数や構造体への非常に近い低レベルの対応関係を維持しています。

== 三角形を描くために必要なこと

ここからは、きちんと動作する Vulkan プログラムで三角形を描画するために
必要なすべてのステップの概要を見ていきます。
ここで紹介する概念はすべて、続く章で詳しく説明します。
これは、個々のコンポーネントを関連づけるための全体像を与えるためだけのものです。

=== ステップ 1 - インスタンスと物理デバイスの選択

Vulkan アプリケーションは、`vk::Instance` を通じて Vulkan API をセットアップすることから始まります。
インスタンスは、アプリケーションと使用する API 拡張を記述して作成します。
インスタンスを作成した後、Vulkan 対応ハードウェアを問い合わせ、操作に使う
1 つ以上の ``vk::PhysicalDevice`` を選択できます。
VRAM サイズやデバイスの能力といったプロパティを問い合わせて、
例えば専用グラフィックスカードを優先するなど、望ましいデバイスを選べます。

=== ステップ 2 - 論理デバイスとキューファミリー

使用する適切なハードウェアデバイスを選択したら、`vk::Device`（論理デバイス）を
作成する必要があります。ここでは、マルチビューポートレンダリングや 64 ビット浮動小数点など、
使用する物理デバイスの機能をより具体的に記述します。
また、使用したいキューファミリーも指定する必要があります。
描画コマンドやメモリ操作など、Vulkan で行うほとんどの操作は、`vk::Queue` に
投入することで非同期に実行されます。
キューはキューファミリーから割り当てられ、各キューファミリーはそのキューで
特定の操作セットをサポートします。
たとえば、グラフィックス、コンピュート、メモリ転送操作用に別々のキューファミリーが
ある場合があります。
キューファミリーの可用性は、物理デバイス選択における区別要因としても使えます。
Vulkan 対応デバイスがグラフィックス機能を一切提供しないこともあり得ますが、
今日の Vulkan 対応グラフィックスカードは一般に、私たちが関心のあるすべての
キュー操作をサポートしています。

=== ステップ 3 - ウィンドウサーフェスとスワップチェーン

オフスクリーンレンダリングだけに関心があるのでなければ、描画した画像を
提示するためのウィンドウを作成する必要があります。
ウィンドウはネイティブのプラットフォーム API、または
https://www.glfw.org/[GLFW] や https://www.libsdl.org/[SDL]
のようなライブラリで作成できます。
本チュートリアルでは GLFW を使いますが、詳細は次の章で扱います。

実際にウィンドウへ描画するには、さらに 2 つの部品が必要です。ウィンドウサーフェス
（`vk::SurfaceKHR`）とスワップチェーン（`vk::SwapchainKHR`）です。
`KHR` 接尾辞に注意してください。これはこれらのオブジェクトが Vulkan 拡張の一部で
あることを意味します。Vulkan API 自体は完全にプラットフォーム非依存であり、
そのためウィンドウマネージャとやり取りするには標準化された WSI
（Window System Interface）拡張を使う必要があります。
サーフェスは描画先ウィンドウのクロスプラットフォーム抽象化であり、一般に
Windows の `HWND` のようなネイティブウィンドウハンドルへの参照を提供して
インスタンス化されます。
幸い、GLFW ライブラリにはこのプラットフォーム固有の詳細を扱う組み込み関数があります。

スワップチェーンはレンダーターゲットの集合です。
その基本的な目的は、現在描画中の画像が、現在画面に表示されているものとは
異なることを保証することです。
これにより、完成した画像だけが表示されることが保証されます。
フレームを描画するたびに、スワップチェーンに描画用の画像を要求する必要があります。
フレームの描画が終わると、画像はスワップチェーンに返され、いずれかの時点で
画面に提示されます。
レンダーターゲットの数と、完成した画像を画面に提示する条件は、プレゼンテーションモードに
依存します。
一般的なプレゼンテーションモードには、ダブルバッファリング（vsync）と
トリプルバッファリングがあります。
これらについてはスワップチェーン作成の章で見ていきます。

一部のプラットフォームでは、`VK_KHR_display` および
`VK_KHR_display_swapchain` 拡張を通じて、ウィンドウマネージャとやり取りせずに
ディスプレイに直接描画できます。
これらにより画面全体を表すサーフェスを作成でき、たとえば独自のウィンドウマネージャを
実装するのに使えます。

=== ステップ 4 - スワップチェーンのイメージビュー

スワップチェーンから取得した画像に描画するには、通常それを `vk::ImageView` で
ラップします。イメージビューは、例えば色書き込みに使う画像の特定部分を参照します。
スワップチェーンには異なる画像があり得るため、それぞれについて事前にイメージビューを
作成し、描画時に適切なものを選択します。

=== ステップ 5 - ダイナミックレンダリングの概要

Vulkan の初期バージョンでは、レンダーパスがフレームバッファを使ったレンダリング操作の
行われ方を定義し、使用する画像の種類（例: 色、深度）と、その内容の扱い
（例: クリア、ロード、ストア）を指定していました。
`vk::RenderPass` がサブパスとアタッチメントの使い方を定義し、`vk::Framebuffer`
が特定のイメージビューをこれらのアタッチメントにバインドしていました。

しかし、ダイナミックレンダリング（Vulkan 1.3 で導入）では、
もはや `vk::Framebuffer` を作成する必要がありません。
ダイナミックレンダリングは、事前定義されたレンダーパスとフレームバッファの必要性を
排除し、コマンド記録中にレンダリングアタッチメントを直接指定できるようにします。
フレームバッファ管理のオーバーヘッドを気にせず、その場でレンダリングターゲットを
定義できるため、API がはるかにシンプルになります。

ダイナミックレンダリングでは、もはや `vk::RenderPass` や `vk::Framebuffer` を
事前定義する必要はありません。
代わりに、コマンド記録の開始時に `vk::beginRendering` と
`vk::RenderingInfo` のような構造体を使い、必要なアタッチメント情報をすべて
動的に指定します。

最初の三角形描画アプリケーションでは、
ダイナミックレンダリングを使って単一の画像をカラーターゲットとして指定し、
描画直前にソリッドカラーでクリアするよう Vulkan に指示します。

=== ステップ 6 - グラフィックスパイプライン

Vulkan のグラフィックスパイプラインは、`VkPipeline` オブジェクトを作成して設定します。
ビューポートサイズや深度バッファ操作といったグラフィックスカードの設定可能な状態と、
`vk::ShaderModule` オブジェクトによるプログラマブルな状態を記述します。
`vk::ShaderModule` オブジェクトはシェーダバイトコードから作成されます。
ドライバはパイプラインでどのレンダーターゲットが使われるかも知る必要があり、
これはレンダーパスを参照して指定します。

既存の API と比べた Vulkan の最も特徴的な点の一つは、
グラフィックスパイプラインのほぼすべての設定を事前に行う必要があることです。
つまり、別のシェーダに切り替えたり、頂点レイアウトを少し変更したりしたい場合は、
グラフィックスパイプライン全体を再作成する必要があります。
レンダリング操作に必要なすべての組み合わせについて、多くの `vk::Pipeline`
オブジェクトを事前に作成しなければなりません。
ビューポートサイズやクリアカラーといった基本的な設定だけが動的に変更できます。
すべての状態も明示的に記述する必要があり、たとえばデフォルトのカラーブレンド状態は
ありません。

良い知らせは、ジャストインタイムコンパイルではなく事前コンパイルに相当することを
行っているため、ドライバにとってより多くの最適化の機会があることです。
別のグラフィックスパイプラインへの切り替えのような大きな状態変更が非常に明示的に
行われるため、ランタイムのパフォーマンスはより予測可能になります。

=== ステップ 7 - コマンドプールとコマンドバッファ

前述のとおり、描画操作など Vulkan で実行したい多くの操作は、キューに投入する
必要があります。
これらの操作は、投入する前にまず `vk::CommandBuffer` に記録する必要があります。
これらのコマンドバッファは、特定のキューファミリーに関連付けられた
`vk::CommandPool` から割り当てられます。
ダイナミックレンダリング以前に三角形を描くには、次の操作を持つコマンドバッファを
記録する必要がありました。

* レンダーパスを開始する
* グラフィックスパイプラインをバインドする
* 3 頂点を描画する
* レンダーパスを終了する

しかし、ダイナミックレンダリングでは事情が変わります。
レンダーパスの「開始」と「終了」の代わりに、
`vk::BeginRendering` でレンダリングを開始するときにレンダリングアタッチメントを
直接定義します。

これにより、必要なアタッチメントをその場で指定できるようになり、プロセスが
簡素化され、スワップチェーン画像が動的に選択されるシナリオにより適応しやすくなります。
したがって、スワップチェーンの各画像ごとにコマンドバッファを記録したり、
毎フレーム同じコマンドバッファを繰り返し記録したりする必要はありません。
操作はより合理化され効率的になり、Vulkan がレンダリングシナリオをより柔軟に
扱えるようになります。

=== ステップ 8 - メインループ

描画コマンドがコマンドバッファにラップされたので、メインループはかなり素直です。
まず `device.acquireNextImageKHR` でスワップチェーンから画像を取得します。
次にその画像に対応する適切なコマンドバッファを選択し、`graphicsQueue.submit()`
で実行します。
最後に、`presentQueue.presentKHR(presentInfo)` で画像をスワップチェーンに返し、
画面に提示します。

キューに投入された操作は非同期に実行されます。
したがって、正しい実行順序を保証するためにセマフォのような同期オブジェクトを
使う必要があります。
描画コマンドバッファの実行は、画像取得の完了を待つように設定しなければなりません。
さもないと、まだ画面提示のために読み取り中の画像への描画を開始してしまう可能性が
あります。
`presentQueue.presentKHR(presentInfoKHR)` 呼び出しは、逆に描画の完了を待つ必要が
あり、そのために描画完了後にシグナルされる 2 つ目のセマフォを使います。

=== まとめ

この駆け足のツアーで、最初の三角形を描くために必要な作業の基本的な理解が
得られたはずです。
実世界のプログラムには、頂点バッファの割り当て、ユニフォームバッファの作成、
テクスチャ画像のアップロードなど、さらに多くのステップがあり、それらは後の章で
扱います。ただし、Vulkan 自体の学習曲線がすでに十分急なので、まずはシンプルに
始めます。
最初は頂点バッファを使わず、頂点座標を頂点シェーダに埋め込むことで少しズルを
することに注意してください。
頂点バッファの管理には、まずコマンドバッファへの慣れが必要なためです。

つまり、最初の三角形を描くには次が必要です。

* インスタンスを作成する
* 対応するグラフィックスカード（`PhysicalDevice`）を選択する
* 描画と提示用の `Device` と `Queue` を作成する
* ウィンドウ、ウィンドウサーフェス、スワップチェーンを作成する
* スワップチェーン画像を `VkImageView` でラップする
* ダイナミックレンダリングを設定する
* グラフィックスパイプラインを設定する
* 可能なすべてのスワップチェーン画像について、描画コマンドを持つコマンドバッファを
割り当てて記録する
* 画像を取得し、適切な描画コマンドバッファを投入し、画像をスワップチェーンに返すことで
フレームを描画する

ステップは多いですが、各ステップの目的は続く章で基本的かつ明確になります。
個々のステップとプログラム全体の関係で混乱した場合は、この章に戻って参照してください。

== API の概念

この章の最後に、Vulkan API が低レベルでどのように構造化されているかの短い概要で
締めくくります。

たとえば、オブジェクト作成は一般に次のパターンに従います。

[,c++]
----
vk::XXXCreateInfo createInfo{};
createInfo.sType = vk::StructureType::eXXXCreateInfo;
createInfo.pNext = nullptr;
createInfo.foo = ...;
createInfo.bar = ...;

vk::XXX object;


try {
    object = device.createXXX(createInfo);
} catch (vk::SystemError& err) {
    std::cerr << "Failed to create object: " << err.what() << std::endl;
    return false;
}
----

Vulkan の多くの構造体では、`sType` メンバに構造体の型を明示的に指定する必要があります。
`pNext` メンバは拡張構造体を指すことができ、本チュートリアルでは常に `nullptr` です。
オブジェクトを作成または破棄する関数には、ドライバメモリ用のカスタムアロケータを
使える `VkAllocationCallbacks` パラメータがあり、本チュートリアルではこれも
`nullptr` のままにします。

ほとんどすべての関数は、`vk::result::eSuccess` またはエラーコードである
`vk::Result` を返します。
仕様書には、各関数が返す可能性のあるエラーコードとその意味が記述されています。

そのような呼び出しの失敗は C++ 例外で報告されます。例外は `vk::Result` を含む
エラーに関するより多くの情報で応答します。
これにより、1 回の呼び出しから複数のコマンドをチェックでき、コマンド構文を
クリーンに保てます。

=== バリデーションレイヤ

前述のとおり、Vulkan は高性能と低いドライバオーバーヘッドのために設計されています。
したがって、デフォルトでは非常に限られたエラーチェックとデバッグ機能しか
含まれません。
何か間違ったことをすると、ドライバはエラーコードを返す代わりにクラッシュすることが
多く、あるいはさらに悪いことに、自分のグラフィックスカードでは動作するように見えて
他では完全に失敗します。

Vulkan では、_バリデーションレイヤ_ と呼ばれる機能を通じて広範なチェックを
有効にできます。
バリデーションレイヤは、API とグラフィックスドライバの間に挿入できるコード片で、
関数パラメータの追加チェックやメモリ管理問題の追跡などを行います。
重要な利点は、開発中に有効にし、アプリケーションをリリースするときにはオーバーヘッド
ゼロで完全に無効にできることです。
誰でも独自のバリデーションレイヤを書けますが、LunarG の Vulkan SDK は
本チュートリアルで使う標準セットのバリデーションレイヤを提供しています。
レイヤからデバッグメッセージを受け取るには、コールバック関数も登録する必要があります。

Vulkan はすべての操作について非常に明示的であり、バリデーションレイヤも非常に
広範であるため、画面が真っ黒になる理由を突き止めるのは、OpenGL や Direct3D と
比べて実際にははるかに容易になり得ます！

コードを書き始める前に、あと 1 ステップだけあります。それは
xref:02_Development_environment.adoc[開発環境のセットアップ] です。
