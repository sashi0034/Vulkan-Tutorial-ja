= サブシステム: はじめに

== エンジンサブシステム入門

前章までで、レンダリングパイプライン、カメラシステム、モデル読み込みといったコアコンポーネントを実装し、シンプルなエンジンの基盤を築きました。ここでは、2 つの重要なサブシステム — オーディオと物理 — を追加してエンジンの能力を拡張します。

これらのサブシステムは、モダンなゲームやシミュレーションで没入感とインタラクティブ性を作るうえで不可欠です。これまで焦点を当ててきたグラフィックスパイプラインとは別物に見えるかもしれませんが、モダンなエンジンは Vulkan の計算能力を活用して、オーディオ処理と物理シミュレーションの両方を強化できます。

=== この章で扱う内容

この章では、ゲームやシミュレーションを生き生きとさせる 2 つの重要なエンジンサブシステムの実装を進めます。まずオーディオサブシステムから始め、サウンドと音楽の再生の基礎から、説得力のある 3D 空間オーディオのための Head-Related Transfer Function（HRTF）処理といった高度な技法へと進みます。この進行は、Vulkan コンピュートシェーダーが基本的なオーディオ再生を、3D 世界に応答する没入感のあるサウンドスケープへどう変えるかを示します。

物理サブシステムも同様の道をたどり、オブジェクトが believable に相互作用する必須の衝突検出と応答メカニズムから始めます。これらの基盤を発展させるにつれ、Vulkan の並列処理能力が物理計算を劇的に加速し、従来の CPU ベース手法では圧倒される多数の相互作用オブジェクトを持つシミュレーションを可能にする様子を示します。

この章全体を通じて、前章からのモダン C++ アプローチを継続します。

=== なぜオーディオと物理に Vulkan か？

オーディオ処理と物理シミュレーションに Vulkan を使う判断は最初は型破りに見えるかもしれませんが、モダンなハードウェア能力を活かす前向きなエンジン開発アプローチを表します。

モダンな GPU は、同時計算向けに設計された数千のコアを通じて巨大な並列処理能力を提供します。Vulkan のコンピュートシェーダーを通じて、グラフィックスレンダリングを超えたタスクにこの計算力を活用できます。オーディオ処理は並列操作から大きく恩恵を受けます。リアルタイム空間効果を持つ数十の同時音源の処理や、数千の相互作用オブジェクトを持つ複雑な物理シミュレーションを想像してください。

Vulkan の統合メモリモデルは、従来の分離アプローチでは得られない効率の機会を作ります。グラフィックス、オーディオ、物理処理がメモリスペースを共有すると、CPU と GPU の間で情報を繰り返し往復させるコストの高いデータ転送を排除できます。この共有メモリアーキテクチャは洗練された相互作用を可能にします。物理シミュレーションがパーティクルシステムに直接影響し、オーディオ処理が視覚効果に応答し、すべてのシステムがシームレスに連携できます。

クロスプラットフォームの一貫性は、複数デバイスを対象とするプロジェクトでますます価値が高まります。これらのサブシステムを Vulkan 経由で実装することで、Windows、Linux、モバイル、新興デバイスにわたって同一の振る舞いを保てます。この一貫性はデバッグ時間を減らし、デプロイ先に関係なくオーディオと物理の振る舞いが予測可能であることを保証します。

性能上の利点は生の計算能力を超えます。集中的な計算を GPU にオフロードすると、ゲームロジック、スクリプティング、AI 処理、逐次処理や複雑な分岐を要する他のタスクのために CPU リソースが解放されます。この分離により、各プロセッサ種別が最も効率的に扱えるタスクに集中できます。

加えて、ここでの意図は、アプリケーションでグラフィックス以外にも Vulkan を使う視点を提供することです。このチュートリアルの目標は本番品質のゲームエンジンを提供することではありません。あらゆる Vulkan アプリケーション開発に取り組むために必要な道具を提供し、アプリケーションが GPU からどう恩恵を受けられるかを批判的に考えることです。

=== 実務上の考慮: すべてを GPU にオフロードしない

Vulkan コンピュートは印象的な高速化をもたらし得ますが、すべてのサブシステムを GPU にオフロードするのが常に望ましいわけではありません。特にモバイルでは次に注意します。

* モバイルの電力と熱: 多くのスマートフォンやタブレットは big.LITTLE CPU クラスタと電力・熱制約のあるモバイル GPU を使います。持続的な重い GPU コンピュートは熱スロットリングを招きやすく、フレームレート低下と不安定なレイテンシにつながります。
* スケジューリングとレイテンシ: GPU はスループットに優れますが、タイトな制御ループや小さなポインタ多用の更新など、起動オーバーヘッドとスケジューリングレイテンシのため CPU 実行を好むタスクもあります。
* 決定性とデバッグ: ゲームプレイ上重要な物理では、CPU 上での決定性とステップ実行デバッグが有利なことがあります。モバイルではブロードフェーズや物理全体を CPU に残すか、ハイブリッド（例: CPU ブロードフェーズ + GPU ナローフェーズ）を検討してください。
* メモリ帯域: 統合アーキテクチャでは GPU/CPU がメモリ帯域を共有します。すべてを GPU に積極的に移すとグラフィックスと競合し、フレーム時間とバッテリー寿命を損なうことがあります。

オーディオ固有の指針:

* 可能ならプラットフォームのオーディオ API と利用可能な専用オーディオハードウェア / DSP（ミキシング、リサンプリング、エフェクト）を優先してください。この経路はしばしば低レイテンシ、より良い電力特性、デバイス横断での予測可能な振る舞いを提供します。
* 専用ハードウェアでの HRTF サポートは広く普及していません。多くのコンシューマデバイスは OS またはアプリケーションのソフトウェア HRTF に依存します。ニーズを評価してください。ソフトウェア HRTF パイプラインで十分なことが多く、多数の音源や複雑な効果がない限り、空間オーディオのためだけに GPU コンピュートを予約する必要は稀です。

実務上の推奨:

* まずプロファイル: 作業を移す前に CPU と GPU のベースラインを確立する。
* ハイブリッド設計を好む: 明らかに並列で重いカーネル（例: バッチ制約ソルブ、FFT/IR 畳み込み）をオフロードし、制御・調整は CPU に残す。
* モバイルを計画: デバイスクラス、熱状態、電力モードに基づき CPU/GPU 経路を切り替えるランタイムトグルを提供する。

=== 前提条件

この章は、前章で確立したエンジンアーキテクチャと Vulkan の基盤の上に大きく築かれます。実装してきたモジュラーデザインパターンは、既存のレンダリング、カメラ、リソース管理システムときれいに統合する必要があるサブシステムを追加するときに重要になります。

Vulkan コンピュートシェーダーの経験は不可欠です。オーディオ処理と物理計算の両方を加速するためにコンピュート能力を活用するためです。本編チュートリアルのコンピュートシェーダー節をまだ終えていない場合は、進む前に復習してください。並列処理の概念と GPU メモリ管理技法は、サブシステムの実装に直接つながります。

ゲーム開発におけるオーディオと物理の基本概念の理解は、実装全体での設計判断を評価する助けになります。各システムを構築する際に基礎は説明しますが、音の減衰、衝突検出、剛体ダイナミクスといった概念への慣れは、これらのサブシステムがインタラクティブアプリケーションのより広い目標にどう奉仕するかを深く理解させます。
