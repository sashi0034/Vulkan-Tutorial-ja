= モバイル開発: パフォーマンスの最適化

== モバイル向けのパフォーマンスの最適化

モバイル デバイスには、デスクトップ システムと比べて大幅に異なるハードウェア制約があります。このセクションでは、モバイル プラットフォームで優れたパフォーマンスを達成するために不可欠な主要なパフォーマンスの最適化について説明します。

[NOTE]
====
この章では、一般的なモバイルのパフォーマンスについて説明します。 GPU がタイルベース (TBR) であるために特に発生する実践については、「xref:Building_a_Simple_Engine/Mobile_Development/04_rendering_approaches.adoc[レンダリング手法: タイルベースのレンダリング]」を参照してください。
====

=== テクスチャの最適化

[NOTE]
====
ここではモバイル固有の決定に焦点を当てます。一般的な Vulkan イメージの作成、アップロードのステージング、およびディスクリプタのセットアップについては、エンジン シリーズの以前の章 (xref:Building_a_Simple_Engine/Engine_Architecture/04_resource_management.adoc[リソース管理]、xref:Building_a_Simple_Engine/Engine_Architecture/05_rendering_pipeline.adoc[レンダリングパイプライン])、または Vulkan ガイド (https://docs.vulkan.org/guide/latest/).) を参照してください。
====

多くの場合、テクスチャはグラフィックス アプリケーションでメモリを最も多く消費します。これらを最適化することは、モバイルとデスクトップの両方でのパフォーマンスにとって非常に重要です。

==== 効率的なテクスチャフォーマット

適切なテクスチャ形式を選択することは、プラットフォームを問わず重要です。異なるのは、特定のデバイス/ドライバーでどの形式がネイティブにサポートされているかです。

1. *圧縮形式*: 可能な限り、ハードウェアでサポートされている圧縮形式を使用します。
   - *ASTC* (Adaptive Scalable Texture Compression): 最新のモバイル GPU で広くサポートされており、デスクトップでも利用できるようになってきています。柔軟なブロック サイズによる優れた品質対サイズ比。
   - *ETC2/EAC*: OpenGL ES 3.0 に必要で、ほとんどの Android デバイスでサポートされています。 Vulkan スタックでも一般的に利用可能です。
   - *PVRTC*: 主に PowerVR GPU を搭載した iOS デバイスでサポートされます。
   - *BC* (ブロック圧縮、別名 DXT/BCn): デスクトップ上でユビキタス。一部のモバイル GPU でサポートされています。

2. *コンテンツとサポートに基づく形式の選択*: テクスチャのタイプとデバイスが報告する内容に基づいて形式を選択します。
   - 高詳細 (法線、粗さ) の場合: サポートされている場合は ASTC 4x4 または 6x6 を優先します。デスクトップでは、BC5/BC7 が一般的な代替手段です。
   - アルベド/ベースカラーの場合: ASTC 6x6–8x8 が利用可能な場合は適切に機能します。デスクトップでは、BC1/BC7 が一般的です。
   - シングル チャネル データの場合: R8 または圧縮されたシングル チャネルの代替が利用可能な場合は検討してください。

[NOTE]
====
このガイダンスはモバイル専用ではありません。ブロック圧縮により、すべてのプラットフォームでメモリ フットプリントと帯域幅が削減されます。携帯電話やタブレットでは帯域幅と電力の制約がより厳しいため、モバイルの章ではこれを強調しています。デスクトップでも同じ利点が適用されます。主な違いは、どの形式が一般的に利用できるかです (デスクトップでは BC、多くのモバイル デバイスでは ASTC/ETC2 など)。
====

Vulkan で圧縮形式を確認して使用する方法は次のとおりです。

[source,cpp]
----
bool is_format_supported(vk::PhysicalDevice physical_device, vk::Format format, vk::ImageTiling tiling,
                        vk::FormatFeatureFlags features) {
    vk::FormatProperties props = physical_device.getFormatProperties(format);

    if (tiling == vk::ImageTiling::eLinear) {
        return (props.linearTilingFeatures & features) == features;
    } else if (tiling == vk::ImageTiling::eOptimal) {
        return (props.optimalTilingFeatures & features) == features;
    }

    return false;
}

vk::Format find_supported_format(vk::PhysicalDevice physical_device,
                               const std::vector<vk::Format>& candidates,
                               vk::ImageTiling tiling,
                               vk::FormatFeatureFlags features) {
    for (vk::Format format : candidates) {
        if (is_format_supported(physical_device, format, tiling, features)) {
            return format;
        }
    }

    throw std::runtime_error("Failed to find supported format");
}
----

=== メモリの最適化

メモリはすべてのプラットフォームにおいて貴重なリソースです。モバイルでは、帯域幅、電力、熱バジェットが厳しくなるため、パフォーマンスがより重要になる傾向があります。以下に主な最適化をいくつか示します。

==== メモリ割り当てを最小限に抑える

1. *プール割り当て*: メモリ プールを使用して、頻繁な割り当てと割り当て解除のオーバーヘッドを削減します。

2. *大きなブロックからのサブ割り当て*: 多数の小さな Vulkan メモリ割り当てを作成する代わりに、より大きなブロックを割り当て、そこからサブ割り当てを行います。

[source,cpp]
----
class VulkanMemoryPool {
public:
    VulkanMemoryPool(vk::Device device, vk::PhysicalDevice physical_device,
                    vk::DeviceSize block_size, uint32_t memory_type_index)
        : device(device), block_size(block_size), memory_type_index(memory_type_index) {
        allocate_new_block();
    }

    ~VulkanMemoryPool() {
        for (auto& block : memory_blocks) {
            device.freeMemory(block.memory);
        }
    }

    struct Allocation {
        vk::DeviceMemory memory;
        vk::DeviceSize offset;
        vk::DeviceSize size;
    };

    Allocation allocate(vk::DeviceSize size, vk::DeviceSize alignment) {
        // Find a block with enough space
        for (auto& block : memory_blocks) {
            vk::DeviceSize aligned_offset = align(block.next_offset, alignment);
            if (aligned_offset + size <= block_size) {
                Allocation alloc;
                alloc.memory = block.memory;
                alloc.offset = aligned_offset;
                alloc.size = size;

                block.next_offset = aligned_offset + size;
                return alloc;
            }
        }

        // No block has enough space, allocate a new one
        allocate_new_block();
        return allocate(size, alignment);  // Try again with the new block
    }

private:
    struct MemoryBlock {
        vk::DeviceMemory memory;
        vk::DeviceSize next_offset = 0;
    };

    void allocate_new_block() {
        vk::MemoryAllocateInfo alloc_info;
        alloc_info.setAllocationSize(block_size);
        alloc_info.setMemoryTypeIndex(memory_type_index);

        MemoryBlock block;
        block.memory = device.allocateMemory(alloc_info);
        block.next_offset = 0;

        memory_blocks.push_back(block);
    }

    vk::DeviceSize align(vk::DeviceSize offset, vk::DeviceSize alignment) {
        return (offset + alignment - 1) & ~(alignment - 1);
    }

    vk::Device device;
    vk::DeviceSize block_size;
    uint32_t memory_type_index;
    std::vector<MemoryBlock> memory_blocks;
};
----

==== 帯域幅の使用量を削減する

1. *状態の変化を最小限に抑える*: 描画呼び出しをマテリアルごとにグループ化して、状態の変化を減らします。

2. *より小さいデータ型を使用*: 必要に応じて、16 ビットのインデックスと半精度浮動小数点を使用します。

3. *頂点フォーマットの最適化*: パックされた頂点フォーマットを使用してメモリ帯域幅を削減します。

[source,cpp]
----
// Traditional vertex format (48 bytes per vertex)
struct Vertex {
    glm::vec3 position;   // 12 bytes
    glm::vec3 normal;     // 12 bytes
    glm::vec2 texCoord;   // 8 bytes
    glm::vec4 color;      // 16 bytes
};

// Optimized vertex format (16 bytes per vertex)
struct OptimizedVertex {
    // Position: 3 components, 16-bit float each
    uint16_t position[3]; // 6 bytes

    // Normal: 2 components (can reconstruct Z), 8-bit signed normalized
    int8_t normal[2];     // 2 bytes

    // TexCoord: 2 components, 16-bit float each
    uint16_t texCoord[2]; // 4 bytes

    // Color: 4 components, 8-bit unsigned normalized
    uint8_t color[4];     // 4 bytes
};
----

[NOTE]
====
タイルベースの GPU (TBR) をターゲットにしている場合、帯域幅はアタッチメントのロード/ストア動作とタイル フラッシュによって大きな影響を受ける可能性があります。具体的なガイダンスについては、xref:Building_a_Simple_Engine/Mobile_Development/04_rendering_approaches.adoc[レンダリングのアプローチ] — セクション「タイラーでのアタッチメントのロード/ストア操作」および「タイラーでのパイプライン: サブパスの依存関係と BY_REGION」を参照してください。
====

=== 描画呼び出しの最適化

モバイル GPU は、描画呼び出しのオーバーヘッドの影響を特に受けやすくなります。

1. *インスタンス化*: インスタンス化を使用して、繰り返されるオブジェクトの描画呼び出しを減らします。

2. *バッチ処理*: 可能な場合は、複数のオブジェクトを 1 つのメッシュに結合します。

3. *詳細レベル (LOD)*: LOD システムを実装して、遠くにあるオブジェクトのジオメトリの複雑さを軽減します。

[NOTE]
====
タイルベースの GPU では、CPU オーバーヘッドを削減することが重要ですが、適切なパイプラインとサブパスを介して作業とデータをオンチップに維持すると、多くの場合、より大きな利益が得られます。バリア/サブパス パターンについては、xref:Building_a_Simple_Engine/Mobile_Development/04_rendering_approaches.adoc[レンダリングのアプローチ] — 「タイラーでのパイプライン: サブパスの依存関係と BY_REGION」を参照し、外部メモリ トラフィックを回避するloadOp/storeOp ガイダンスについては「タイラーでの添付ファイルのロード/ストア操作」を参照してください。
====

=== ベンダー固有の最適化

さまざまなモバイル GPU ベンダーが、対象を絞った最適化から恩恵を受ける可能性のある特定のアーキテクチャを持っています。

==== ベンダー固有の GPU の最適化

さまざまなモバイル GPU ベンダーが、ターゲットを絞った最適化から恩恵を受ける特定のアーキテクチャを備えています。

* *メモリ管理*: 多くのモバイル SoC は統合メモリ アーキテクチャを備えています。
   - 可能な場合は、`VK_MEMORY_PROPERTY_DEVICE_LOCAL_BIT | VK_MEMORY_PROPERTY_HOST_VISIBLE_BIT` メモリを使用してください
   - ユニファイド メモリ アーキテクチャで高速な CPU-GPU メモリ転送を活用する

* *テクスチャ圧縮*: デバイスによってサポートされるテクスチャが異なります。
圧縮形式:

[source,cpp]
----
// Check for texture compression format support
bool supports_texture_format(vk::PhysicalDevice physical_device, vk::Format format) {
    vk::FormatProperties props = physical_device.getFormatProperties(format);
    return (props.optimalTilingFeatures & vk::FormatFeatureFlagBits::eSampledImage);
}

// Get optimal texture format based on device capabilities
vk::Format get_optimal_texture_format(vk::PhysicalDevice physical_device) {
    vk::PhysicalDeviceProperties props = physical_device.getProperties();
    vk::PhysicalDeviceFeatures features = physical_device.getFeatures();

    // Check for ASTC support (widely supported on modern mobile GPUs)
    // Most games are written with knowledge of what the assets were compressed with so it's standard practice to only ensure the required format is supported.
    if (features.textureCompressionASTC_LDR) {
        return vk::Format::eAstc8x8SrgbBlock;
    }
}
----

* *パフォーマンス監視*: ほとんどのベンダーがパフォーマンス監視ツールを提供しています
 これは、ハードウェア固有のボトルネックを特定するのに役立ちます。

=== モバイルパフォーマンスのベストプラクティス

1. *ターゲット デバイスのプロファイル*: パフォーマンス特性はモバイル デバイスによって大きく異なります。さまざまなメーカーのさまざまな GPU アーキテクチャを備えたさまざまなハードウェアでテストします。

2. *モニター温度*: モバイル デバイスは、高温になるとパフォーマンスを低下させます。サーマル スロットルに適応するようにエンジンを設計します。

3. *品質とパフォーマンスのバランス*: ユーザーがデバイスの機能に基づいて品質とパフォーマンスのバランスを取れるようにするグラフィック設定を提供します。

4. *適応解像度の実装*: パフォーマンス メトリックに基づいてレンダリング解像度を動的に調整します。
