== アセットパイプラインを理解する

エンジンアーキテクチャとカメラシステムを調べた後は、レンダリング エンジンで 3D アセットがどのように管理されるかを理解することが重要です。適切に設計されたアセットパイプラインは、あらゆる実稼働環境でモデル、テクスチャ、その他のリソースを効率的に処理するために不可欠です。

=== アセット編成の概念

アセット組織システムを設計するときは、次の重要な原則を考慮してください。

1. *分類* - 類似したアセットをグループ化します
2. *階層* - ネストされた構造を使用して複雑さを管理します
3. *発見性* - アセットを簡単に見つけて参照できるようにします。
4. *スケーラビリティ* - プロジェクトの拡大に​​伴う成長を考慮した設計

以下は、最終製品でアセットがどのように編成されるかを示す例であり、4 つの原則すべてを示しています。

[source]
----
assets/
  ├── models/                  // 3D model files (Categorization)
  │   ├── characters/          // Character models (Hierarchy)
  │   │   ├── player/          // Player character models (Hierarchy)
  │   │   └── npc/             // Non-player character models (Hierarchy)
  │   ├── environments/        // Environment models
  │   │   ├── indoor/          // Indoor environment models
  │   │   └── outdoor/         // Outdoor environment models
  │   └── props/               // Prop models
  ├── textures/                // Texture files (Categorization)
  │   ├── common/              // Shared textures (Discoverability)
  │   └── high_resolution/     // High-res textures for close-up views (Scalability)
  ├── shaders/                 // Shader files
  │   ├── core/                // Essential shaders (Discoverability)
  │   ├── effects/             // Special effect shaders
  │   └── mobile/              // Mobile-optimized shaders (Scalability)
  └── config/                  // Configuration files
      └── quality_presets/     // Different quality settings (Scalability)
----

この例では、次の 4 つの原則をすべて示します。

- *分類*: アセットはタイプ (モデル、テクスチャ、シェーダー、構成) ごとにグループ化されています。
- *階層*: アセットは入れ子構造で編成されます (例: モデル > キャラクター > プレイヤー)
- *見つけやすさ*: 共通のアセットは専用のフォルダー (例: 共通のテクスチャ、コア シェーダー) に配置されているため、見つけやすくなります。
- *スケーラビリティ*: 構造は、さまざまな品質レベルとプラットフォーム固有のアセット (高解像度テクスチャ、モバイル シェーダー、品質プリセットなど) に対応します。

特定の組織はプロジェクトのニーズに合わせて調整する必要がありますが、基礎となる原則は異なるエンジン間でも一貫しています。

=== アセットパイプラインの概念

プロフェッショナル アセットパイプラインには通常、特定のエンジン実装に関係なく、いくつかの段階が含まれます。

1. *作成* - アーティストが 3D モデリング ソフトウェアでモデルを作成します。
2. *エクスポート* - モデルはゲーム エンジンに適した交換形式にエクスポートされます
3. *検証* - モデルに問題がないかチェックされます (スケールが正しくない、テクスチャが欠落しているなど)。
4. *最適化* - モデルは実行時のパフォーマンスに合わせて最適化されます
5. *変換* - 開発アセットは本番環境に対応した形式に変換されます
6. *統合* - アセットはエンジンにインポートされます
7. *ランタイム読み込み* - エンジンは実行中にアセットを効率的に読み込みます。

アセットパイプラインを設計するときは、次の重要な要素を考慮してください。

==== ファイル形式の選択

ファイル形式が異なれば、トレードオフも異なります。

1. *交換フォーマット* (glTF、FBX、Collada など)
   - 長所: モデリング ツールで広くサポートされており、ほとんどのデータが保存されます。
   - 短所: 不要なデータが含まれる可能性があり、実行時に最適化されていない

2. *ランタイム形式* (glb、エンジン固有のバイナリ形式など)
   - 長所: 読み込み速度とメモリ使用量が最適化されています。
   - 短所: エンジンの外部では編集できない場合があります

==== テクスチャ圧縮

テクスチャ圧縮はパフォーマンスにとって非常に重要です。

1. *開発形式* (例: PNG、JPEG)
   - 長所: ロスレスまたは高品質、編集ツールで広くサポートされている
   - 短所: ファイルサイズが大きく、GPU に最適化されていない

2. *ランタイム形式* (例: ktx、圧縮 GPU 形式)
   - 長所: ファイル サイズが小さく、GPU で直接使用できる
   - 短所: 品質が低下する可能性があること、プラットフォーム固有の考慮事項

==== アセットバンドル

アセットがどのようにパッケージ化されるかを検討してください。

1. *個別のファイル*
   - 長所: 個々のアセットの更新が簡単、バージョン管理が簡単
   - 短所: ファイル操作が増える、依存関係が失われる可能性がある

2. *バンドルされたアセット*
   - 長所: ファイル操作が少なく、依存関係が保証される
   - 短所: 大規模なアトミック更新、より複雑なバージョン管理

=== アーティストとエンジンのコラボレーションのコンセプト

アート アセットをレンダリング エンジンに正常に統合するには、アーティストとプログラマー間の明確なコミュニケーションと確立されたワークフローが必要です。考慮すべき重要な概念は次のとおりです。

==== 技術仕様

特定のエンジンに関係なく、以下を定義する必要があります。

1. *座標系* - アプリケーションごとに異なる座標系が使用されます (例: Y アップと Z アップ)
2. *スケール* - 一貫したスケールを確立します (例: 1 単位 = 1 メートル、または 1 単位 = 1 フィート)
3. *原点の配置* - さまざまなアセット タイプの原点の位置を定義します。
4. *詳細レベル* - さまざまなアセット タイプと使用シナリオに合わせてポリゴン数の範囲を指定します。

==== ワークフローのドキュメント

以下に対処するドキュメントを作成します。

1. *命名規則* - 一貫した命名は、整理と自動化に役立ちます
2. *材料標準* - 材料がどのように構造化されるべきかを定義します (例: PBR パラメータ)
3. *エクスポート設定* - 選択した交換形式の正しいエクスポート設定を文書化します。
4. *品質チェックリスト* - 提出前にアセットを検証するための基準を提供します

==== テクニカルアートブリッジ

次のようなテクニカル アートの役割を確立することを検討してください。

1. アートからエンジンへのパイプラインを合理化するツールを作成します
2. アセットがエンジンに入る前に検証します。
3. 技術的な要件についてアーティストにフィードバックを提供します
4. アセットがエンジン内で正しく表示されない場合の問題のトラブルシューティングに役立ちます

=== 開発から生産までのコンセプト

アーティスト向けの開発アセットから最適化されたプロダクション アセットへの移行には、いくつかの重要な概念が含まれます。

==== 開発資産と本番資産

各段階でのさまざまなニーズを理解する:

1. *開発資産*
   - 編集しやすさと反復速度を優先する
   - コンテンツ作成ツールで広くサポートされている形式を使用する
   - サイズが大きくなり、実行時のパフォーマンスがあまり最適化されていない可能性があります
   - 最大限の品質と情報を維持することに重点を置く

2. *プロダクションアセット*
   - 実行時のパフォーマンスとメモリ効率を優先する
   - ターゲット プラットフォームに最適化された形式を使用する
   - 適切な圧縮および最適化技術を適用する
   - 品質とパフォーマンス要件のバランスを取る

==== 資産の検証

パイプラインの主要なポイントで検証を実装します。

1. *提出前検証*
   - 技術仕様への準拠を確認する
   - 必要なテクスチャとマテリアルがすべて存在することを確認します
   - 適切なスケール、方向、原点の配置を確認する

2. *変換前の検証*
   - アセットが変換ツールで正常に処理できることを確認する
   - 変換中に問題を引き起こす可能性のある問題を確認する

3. *変換後の検証*
   - 変換されたアセットが視覚的な忠実性を維持していることを確認する
   - パフォーマンスの問題またはメモリ消費の問題を確認する
   - ターゲット プラットフォームとの互換性を確保する

==== 自動化に関する考慮事項

プロジェクトが成長するにつれて、自動化の重要性がますます高まります。

1. *バッチ処理*
   - 複数のアセットを一度に処理するためのスクリプトまたはツールを開発する
   - 自動検証チェックを実装する

2. *継続的インテグレーション*
   - アセット処理を CI/CD パイプラインに統合することを検討してください。
   - アセットがコミットされたときに自動的に検証して変換する

3. *バージョン管理*
   - アセットとその処理されたバージョンへの変更を追跡する
   - 依存関係の追跡を実装して、必要なものだけを再構築する

=== 実装に関する考慮事項

レンダリング エンジンにモデル読み込みシステムを実装する場合は、アプローチの指針となるいくつかの重要な考慮事項を考慮する必要があります。

==== 抽象化レイヤー

適切な抽象化レイヤーを使用してモデル読み込みシステムを設計します。

1. *ファイル形式レイヤー*
   - 特定のファイル形式 (glTF、FBX など) の解析を処理します。
   - フォーマット固有のコードを分離して、複数のフォーマットのサポートを容易にします。
   - ファイル形式の構造からエンジンの内部構造に変換します。

2. *リソース管理層*
   - メモリとGPUリソ​​ースを管理します
   - キャッシュと参照カウントを処理します
   - 基礎となるファイル形式に関係なく、一貫したインターフェイスを提供します

3. *シーングラフレイヤー*
   - モデルを階層構造で整理します
   - 変換と親子関係を管理します
   - カリングやシーンのトラバーサルなどの操作を容易にします。

==== パフォーマンスに関する考慮事項

柔軟性とパフォーマンスのバランスをとる:

1. *非同期読み込み*
   - メインスレッドのブロックを避けるために、バックグラウンドスレッドでモデルをロードすることを検討してください。
   - 部分的にロードされたモデルを処理するシステムを実装する

2. *メモリ管理*
   - 大規模なモデルを処理するための戦略を開発する
   - 複雑なシーンの詳細レベル (LOD) システムを検討する
   - 非常に大規模な環境にストリーミングを実装する

3. *バッチ処理とインスタンス化*
   - 効率的なレンダリングのために類似したモデルをグループ化する
   - 繰り返される要素にはインスタンス化を使用する

==== 拡張性

将来の拡張を考慮した設計:

1. *マテリアルシステム*
   - さまざまなシェーディング モデルを表現できる柔軟なマテリアル システムを作成する
   - 単純なマテリアルと複雑なマテリアルの両方をサポート

2. *アニメーションシステム*
   - さまざまなアニメーション タイプ (スケルトン、モーフ ターゲットなど) に合わせたデザイン
   - アニメーションが物理学やゲームプレイ システムとどのように相互作用するかを検討する

3. *カスタムデータ*
   - エンジン固有のメタデータをモデルに関連付けることができるようにする
   - ゲームプレイまたはレンダリング目的のカスタム プロパティをサポートする

これらの概念を理解すると、あらゆるレンダリング エンジンでモデル読み込みシステムを設計および実装するための強固な基盤が得られます。最初から抽象化、パフォーマンス、拡張性を慎重に考慮することで、プロジェクトのニーズに合わせて拡張し、変化する要件に適応する堅牢なシステムを作成できます。

== 私たちのプロジェクトの実施

アセットパイプラインの一般的な概念を説明したので、特定のプロジェクトでこれらの概念をどのように実装するかについて説明します。

=== ファイル形式とディレクトリ構造

このエンジンでは、次のファイル形式とディレクトリ構造を使用します。

1. *モデル形式*: KTX2 テクスチャが埋め込まれた glTF 2.0 バイナリ形式 (.glb) を使用します。この形式にはいくつかの利点があります。
   - 効率的なストレージとロードのためのコンパクトなバイナリ表現
   - テクスチャを埋め込む機能により、ファイル操作が軽減されます。
   - アニメーション、スキニング、PBR マテリアルのサポート
   - 幅広いツールをサポートする業界標準

2. *テクスチャ形式*: テクスチャには、Basis Universal 圧縮を備えた KTX2 を使用します。これにより、次のことが可能になります。
   - PNG/JPEGと比較してサイズが大幅に縮小
   - 直接アップロードできる GPU 対応フォーマット
   - トランスコーディングによるクロスプラットフォーム互換性
   - ミップマップとさまざまな圧縮形式のサポート

3. *ディレクトリ構造*:
[source]
----
assets/
  ├── models/                  // 3D model files
  │   ├── characters/          // Character models
  │   │   └── viking.glb       // Example character model
  │   ├── environments/        // Environment models
  │   │   └── room.glb         // Example environment model
  │   └── props/               // Prop models
  │       └── furniture.glb    // Example prop model
  └── shaders/                 // Shader files
      └── pbr.slang            // PBR shader
----

=== ツールとライブラリ

アセットパイプラインを実装するには、次のツールとライブラリを使用します。

1. *モデルの読み込み*: tinygltf ライブラリを使用して glTF ファイルを解析します。このライブラリは以下を提供します。
   - glTF 2.0仕様の包括的なサポート
   - バイナリ glTF ファイルの効率的な解析
   - すべての glTF コンポーネント (メッシュ、マテリアル、アニメーションなど) へのアクセス

2. *テクスチャの読み込み*: KTX-Software ライブラリを使用して KTX2 テクスチャを読み込みます。これにより、次のことが可能になります。
   - Basis Universal 圧縮テクスチャのロードとトランスコーディングのサポート
   - 効率的なミップマップ処理
   - Vulkan テクスチャ形式との統合

3. *アセット変換*: 開発アセットを本番アセットに変換するには、以下を使用します。
   - KTX-テクスチャ変換用ツール (PNG/JPEG から KTX2)
   - モデルの処理と最適化のための glTF-Transform
   - 変換プロセスを自動化するためのカスタム スクリプト

=== エンジンアーキテクチャとの統合

モデル読み込みシステムは、前の章のエンジンアーキテクチャと統合されます。

1. *リソース管理*: エンジンアーキテクチャの章のリソース管理システムを活用して、次のことを行います。
   - ロードされたモデルとテクスチャをキャッシュする
   - 効率的なメモリ管理のために参照カウントを実装する
   - モデルの非同期読み込みをサポート

2. *コンポーネント システム*: 次のコンポーネントを作成します。
   - ModelComponent: モデルのレンダリングとアニメーションを管理します。
   - MaterialComponent: マテリアルのプロパティとテクスチャを処理します
   - これらのコンポーネントは、「カメラ変換」の章の TransformComponent と連携して動作します。

3. *レンダリングパイプライン*: モデル読み込みシステムは、次の方法でレンダリングパイプラインと統合されます。
   - ジオメトリ パスのメッシュ データの提供
   - ライティング パスの PBR マテリアルのサポート
   - 繰り返されるモデルのインスタンス化レンダリングの有効化

=== アーティストのワークフロー

アーティスト向けのワークフローは次のようになります。

1. *開発段階*:
   - アーティストは Blender や Maya などのツールでモデルを作成します
   - 個別の PNG/JPEG テクスチャを使用して標準 glTF (.gltf) にエクスポート
   - glTF ビューアを使用してテストして、正しく表示されることを確認します。

2. *技術的要件*:
   - Y アップの右手座標系
   - 1ユニット = 1メートルスケール
   - 金属粗さワークフローを使用した PBR マテリアル
   - 2 のべき乗次元のテクスチャ

3. *変換プロセス*:
   - 技術要件に照らしてモデルを検証する
   - Basis Universal 圧縮を使用してテクスチャを KTX2 に変換します
   - glb ファイルにテクスチャを埋め込む
   - モデルの最適化 (未使用の頂点の削除、メッシュの圧縮など)

4. *統合*：
   - 変換されたアセットを適切なディレクトリに配置します
   - 資産管理システムへの資産の登録
   - 適切なコンポーネントを使用してエンティティを作成する

=== ランタイムロード

実行時に、エンジンは次のことを行います。

1. *モデルをロード*:
   - tinygltf を使用して glb ファイルを解析する
   - メッシュデータ、マテリアル、アニメーションを抽出する
   - 頂点とインデックスの Vulkan バッファを作成する

2. *プロセス材料*:
   - 埋め込まれた KTX2 テクスチャをロードする
   - Vulkan イメージ ビューとサンプラーを作成する
   - PBR レンダリング用のディスクリプタセットをセットアップする

3. *ハンドルアニメーション*:
   - glTF からのアニメーション データを解析する
   - スケルトンアニメーションシステムの実装
   - アニメーションのブレンドとトランジションをサポート

4. *レンダリングモデル*:
   - シーン グラフを使用してモデルを階層的に編成する
   - 変換コンポーネントから変換を適用する
   - 適切なマテリアルとシェーダを使用してレンダリングする

これらの特定のアプローチを実装することにより、私たちのエンジンは、この章で前述した一般的な概念と一致する、堅牢で効率的なアセットパイプラインを備えます。
