= シャドウ、光、そして葉の厄介さ

== 闇の幾何学

シャドウは不在です。表面と光源の間に何かが立ったときに残るものです。それでもその不在は膨大な知覚的重みを持ちます — 影は形について、時刻について、奥行きについて、シーン内のオブジェクト間の関係について教えてくれます。アルゴリズムが関与するはるか以前から、画家は何世紀もかけて影を説得力を持って描く方法を学んできました。人間の視覚系は影に極めて敏感だからです。

最も単純に考えられる影は **ハードシャドウ** — 晴れた日に室内で、窓が床に完璧にシャープな矩形の光を落とすようなものです。このシャープさは幾何学から生まれます。理想的な点光源は、見えるすべての表面を照らし、それ以外を完全な闇に残します。境界に曖昧さはありません。表面上の点は、光源への視線を持つか持たないかのどちらかです。影の縁は数学的な曲線 — 光源から見た遮蔽物のシルエットの投影 — です。

image::images/omm_hard_shadow.svg[点光源によるハードシャドウとシャープな境界の図, 600, 380]

レイトレースシャドウシステムでは、これを計算するのは概念的にきれいです。表面上の点から光に向かってレイを飛ばします。そのレイが光に到達する前に何らかのジオメトリと交差すれば、その点は影の中にあります。遮られずに光に到達すれば、その点は照らされています。ピクセルあたりちょうど 1 本のレイを飛ばし、答えが二値だから、影はくっきりします。

== 現実の光の柔らかさ

自然は点光源を提供しません。太陽は、どれほど遠くにあっても、空に小さな非ゼロの角度 — およそ半度の弧 — を張り、事実上ディスクとして振る舞います。室内照明はエリアライトを使います。管、パネル、球、ディフューザーです。ろうそくの炎でさえ物理的な広がりを持ちます。

光源に面積があると、影の幾何学はより興味深くなります。太陽に向かって見ている表面上の点を考えてください。遮蔽物（たとえば屋根の張り出しの縁）が太陽ディスク全体を視界から遮れば、その点は **アンブラ（本影）** — 完全に影の中にあり、直射日光を一切受けません。遮蔽物が太陽ディスクの一部だけを遮れば、その点は **ペナンブラ（半影）** — 遮られていないディスクの割合に応じた直射日光は受けますが、すべてではありません。遮られた面積と遮られていない面積の比が、その点の明るさを決めます。

image::images/omm_soft_shadow.svg[エリアライトによるソフトシャドウ。アンブラとペナンブラ、グラデーション境界の図, 600, 420]

**ペナンブラ** は、影の縁にあるあの美しいグラデーション — 完全な影から完全な光へのぼやけた遷移 — です。日常経験からあまりに馴染み深いため、レンダリング画像でそれが欠けるとすぐに人工的に感じます。私たちの目は柔らかい縁を期待するよう学習しています。大きく明るい光源から完璧にハードな影の縁を見ると、なぜか言語化できなくても何かがおかしく感じます。

私たちのエンジンでは、ソフトシャドウはピクセルあたり複数のシャドウレイを飛ばし、光源表面の異なる点をサンプリングし、結果を平均することで近似します。各レイは二値の答え（遮られたかそうでないか）を与え、平均が滑らかなグラデーションを生みます。ソフトシャドウの品質はサンプル数とともに向上し、パフォーマンスコストもそれに比例して増えます。

== 木の登場

ここで木のあるシーンを考えてみましょう。様式化されたカートゥーンの木ではなく、何千枚もの葉を持つリアルな木です。この木を作ったアーティストは、各葉を厚みとすべての面のジオメトリを持つ三次元の固体としてモデリングしませんでした。それは管理不能な数のポリゴンを必要とします。代わりに、ゲームやビジュアライゼーションで何十年も標準だった技法を使います。葉の各クラスタは、少数の大きな平坦な三角形で表現され、それぞれが詳細な葉の形を示すテクスチャを持ちます。

このテクスチャには二つの成分があります。葉の色（緑、葉脈パターンと光散乱の繊細さ）と、**アルファチャンネル** — テクスチャ矩形のどのピクセルが「葉」で、どれが「空」かを正確に定義するマスクです。アルファ値は葉があるところで 1.0、ないところで 0.0 で、厳しいピクセル単位のエイリアシングを避けるため縁に狭いグラデーションがあります。

遠くから見ると、これは美しく機能します。三角形はほとんど見えず、アルファが定義する葉の形だけが見えます。下にあるジオメトリは平坦な四角形のまばらな集まりであるにもかかわらず、シルエットは有機的で複雑に見えます。これが **アルファマスク** されたジオメトリであり、リアルタイムコンテンツでは至る所にあります。木、茂み、草、金網フェンス、錬鉄の手すり、カーテン、パーティクルスプライトなどです。

== シャドウレイが葉に出会うとき

ここで問題が始まります。シャドウレイがシーンを進み木に遭遇すると、BVH トラバーサルエンジンはレイが交差するバウンディングボックスを持つ候補三角形を特定します。固体の壁なら、ここで話は終わりです。レイが三角形に当たり、ヒットがコミットされ、表面上の点は影の中です。完了です。

葉の三角形では、話は終わりません。幾何学的には三角形はそこにあり — レイは平坦な矩形と交差しました。しかし交差点は本当に葉の上なのか、それとも葉の間の透明な隙間なのか？ GPU は三角形だけを見て知ることはできません。レイの交差点に対応する特定の UV 座標でアルファテクスチャを参照しなければなりません。その参照の後になって初めて、ヒットをコミットするか（点が固体の葉の上）、無視するか（点が透明空間にあり、レイは進み続ける）を決められます。

この参照は **any-hit shader** — ジオメトリがアルファテストとしてマークされているとき、すべての候補ヒットで発火する小さなプログラマブルシェーダー — によって行われます。そして重要なことに、レイは最初の候補で止まりません。木の樹冠を進むレイは、何十もの葉三角形を通過し得ます。それぞれが any-hit shader を発火させます。それぞれがテクスチャサンプルを必要とします。それぞれがトラバーサルを遅らせます。

== 問題の規模

これを体感するために想像してください。手が木陰にあるかどうかを判定しようとしているとします。しかし木が固体の不透明オブジェクトである代わりに、一枚一枚の葉がスイスチーズでできており — チーズの穴には何も詰まっておらず、ただ空いているとします。特定の穴が手に届く光を遮るかどうかを知るには、その葉を物理的に取り上げ、かざして、穴が光源と揃っているかを見なければなりません。次の葉も取り上げます。また次も。一本の木全体を 4K 解像度で、ピクセルあたり 16 本のソフトシャドウサンプルレイ、毎秒六十回の更新でレンダリングすると — 「チーズ検査」の数は毎秒数十億に達します。

image::images/omm_foliage_problem.svg[アルファテクスチャ付き葉メッシュと、各葉三角形を個別にテストする必要がある複数のシャドウレイの図, 600, 400]

それらの検査の一つ一つが、シェーダー実行、テクスチャサンプリング、判断を必要とします。ほとんどの場合、答えは「透明、続行」です。しかし質問するコストは、答えにかかわらず支払われます。

== より良い問い

ここで、より良い問いを立てる時です。シーン読み込み時に、葉のテクスチャが正確にどう見えるかは分かっています。各三角形のどの部分が不透明で、どの部分が透明かも分かっています。その情報はフレーム間で変わりません — 葉はテクセルレベルではアニメーションしません。ではなぜ、すべてのシャドウレイで、すべてのフレームで、それを再び発見しているのでしょうか？

もしロード時に一度だけ、各三角形のどの部分が確実に固体で、どの部分が確実に空かをハードウェアに伝えられたらどうでしょうか？ GPU がその事前ベイクされた知識をトラバーサル中に参照し、高価なシェーダーを本当に曖昧な縁領域に対してだけ実行できたらどうでしょうか？ それがまさに Opacity Micromaps の約束です。
