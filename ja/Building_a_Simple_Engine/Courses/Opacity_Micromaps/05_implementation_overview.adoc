= Simple Engine での Opacity Micromaps の構築

== 概念的アーキテクチャ

ハードウェアを理解することは一のこと、その理解がコードにどう翻訳されるかを見ることは別のことです。この章では、エンジンの Opacity Micromap 実装を概念レベルで歩きます。完全なソースは本カリキュラムと同じ `Courses/` フォルダにあり、特定のファイルを随所で指し示します。ここでの目標は、実装の形を理解することです。何がいつ起きるか、なぜ各ステップが必要か、そして部品がどうつながるかです。

実装は単一のクラス `OpacityMicromapBuilder` を中心に構成され、`opacity_micromap_builder.h` で定義され、`opacity_micromap_builder.cpp` で実装されています。このクラスは micromap ライフサイクル全体を担当します。ハードウェアサポートの確認、メッシュとテクスチャデータからの micromap 構築、GPU メモリの管理、BLAS ビルドプロセスへの提供です。

== フェーズゼロ: サポートの確認

いかなる micromap 作業も始める前に、エンジンは現在の GPU とドライバが `VK_KHR_opacity_micromap` をサポートしていることを検証しなければなりません。この拡張は比較的新しく、普遍的に利用可能ではありません。`OpacityMicromapBuilder` はデバイス初期化中に拡張を確認し、内部フラグを設定します。拡張がなければ、ビルダーへのその後のすべての呼び出しは no-op です — エンジンは通常のアルファテストで進み、micromap が導入される前とまったく同じです。

この優雅なフォールバックは任意ではありません。出荷アプリケーションにとっての厳しい要件です。コンテンツは OMM がアクティブかどうかにかかわらず正しく見えなければなりません。OMM は視覚結果を変えない（any-hit shader は Unknown のマイクロ三角形のフォールバックのまま残り、シェーダーの挙動は変わらない）ため、OMM を無効にしてもレンダリングは視覚的に同一のままで、単にフレームあたりのコストが高い状態に戻るだけです。フォールバックパスはシェーダーに特別なコードパスを必要としません。

拡張はまた、`VK_KHR_acceleration_structure` が有効であること（レイトレースレンダリングにはすでに必須）と、ビルドコマンドのための `VK_KHR_synchronization2` を要求します。これらの依存関係はメイン拡張と並べて確認されます。デバイス作成時に使うフィーチャ構造体は `VkPhysicalDeviceOpacityMicromapFeaturesKHR` で、コア micromap 機能とオプションの lossy 圧縮機能の可用性を公開します。ビルダーは両方を問い合わせ、どの能力がアクティブかを記録します。

== フェーズ一: 解析

メッシュがエンジンに読み込みのために提出されると、`OpacityMicromapBuilder` はメッシュのマテリアルリストを検査します。各サブメッシュについて、マテリアルプロパティを調べ、こう尋ねます。これはアルファテストマテリアルか？ 具体的には、アルファマスクテクスチャを持ち、完全な透明度ブレンディングではなくアルファカットオフを使うようフラグされているか？

答えがはいなら、そのサブメッシュは micromap 生成の候補です。ビルダーはどのサブメッシュが micromap を必要とするかを記録し、それぞれに適切な分割レベルを決定します。分割レベルの選択は現在、単純なヒューリスティックに基づきます。ビルダーはテクスチャ寸法と、メッシュの三角形数と表面積の比を調べます。小さな三角形上の細かく詳細なテクスチャは高い分割から恩恵を受け、単純なアルファパターンを持つ大きな三角形は低いレベルを使えます。エンジンはまた、分割レベル選択時にデバイスが報告する `maxOpacityLossy4StateSubdivisionLevel` と `maxMicromapTriangles` 制限を尊重し、選択されたパラメータがハードウェアが表現できる範囲内に留まることを保証します。`Courses/` フォルダ内の設定ファイルがヒューリスティックと、マテリアルごとの上書き方法を文書化しています。

解析フェーズは高速で、通常のメッシュ読み込みの一部として CPU 上で起きます。その出力は、次のフェーズが消費するサブメッシュごとの記述子です。

== フェーズ二: 分類

分類が実質的な作業が起きる場所です。各アルファテストサブメッシュについて、またそのサブメッシュ内の各三角形について、ビルダーは選択した分割レベルでのマイクロ三角形を反復します。各マイクロ三角形について、元の三角形内でのマイクロ三角形の重心の重心座標を計算します。それらの座標から、三角形の UV 座標を補間して、重心のテクスチャ空間での位置を見つけます。次にその UV 位置で CPU 側のアルファテクスチャをサンプリングします。

ここでのサンプリングは意図的に単純です。重心でのバイリニアサンプル、オプションで境界でのエイリアシングアーティファクトを減らすため、近くのいくつかのジッターサンプルと平均します。サンプリングされたアルファが上側閾値（典型的には 0.9）より上なら、マイクロ三角形は Opaque に分類されます。下側閾値（典型的には 0.1）より下なら、Transparent です。閾値の間なら、Unknown です。

分類結果はコンパクトな配列 — マイクロ三角形あたり 2 ビット — にパックされ、GPU micromap ビルドの生入力になります。大きなメッシュでは、この分類ステップに目立つ時間がかかることがあり、そのため出荷コンテンツではアプリケーション起動時ではなくオフライン前処理の一部として行われるのが典型です。エンジンは両方のモードをサポートします。ロード時のオンライン分類（開発用）と、バイナリファイルからの事前分類 micromap データの読み込み（本番用）です。

ここで非対称性を味わう価値があります。分類ステップは一度走ります。利益はアプリケーションがレンダリングするすべてのフレームにわたって蓄積します。出荷するゲームやビジュアライゼーションアプリケーションでは、見返り比は莫大です。

== フェーズ三: GPU 構築

分類データが CPU 上で準備できたら、ステージングバッファにアップロードされ、GPU ローカルメモリへ転送されます。ビルダーは次に `VkAccelerationStructureBuildGeometryInfoKHR` 構造体を埋め、`type = VK_ACCELERATION_STRUCTURE_TYPE_OPACITY_MICROMAP_KHR` とジオメトリ型 `VK_GEOMETRY_TYPE_MICROMAP_KHR`、使用フラグ、三角形ごとのフォーマット記述子、入力データバッファアドレス、ビルド操作用のスクラッチバッファを指定します。宛先のアクセラレーション構造ハンドルは、まず `vkCreateAccelerationStructureKHR` を呼び出すことで得られます — KHR 拡張は専用の micromap 作成コマンドを導入する代わりに、標準のアクセラレーション構造作成パスを再利用します。

ビルダーは次に `vkCmdBuildAccelerationStructuresKHR` の呼び出しをコマンドバッファに記録し、デバイス上で micromap の実際のビルドを行います。ホスト側のビルドパスはありません — すべての micromap 構築はキューにサブミットされなければなりません。これによりアクセラレーション構造の作業はデバイス側にしっかり留まり、ハードウェアの内部データレイアウトをホストメモリを往復せずにドライバが決定・適用できます。

結果の `VkAccelerationStructureKHR` ハンドルは、サブメッシュの既存 BLAS ジオメトリ記述子と並べて格納されます。`VK_KHR_opacity_micromap` では、micromap は別のオブジェクト型ではありません。拡張は意図的に標準のアクセラレーション構造ハンドルを micromap に再利用し、作成型によってのみ区別します。これにより寿命管理はかなり単純になります — BVH アクセラレーション構造に適用されるのと同じ作成、バリア、破棄パターンが micromap アクセラレーション構造にも等しく適用され、プログラマは並行するオブジェクト管理規則のセットを学ぶ必要がありません。ハンドルはそれを参照する BLAS がビルドされる前に作成されなければならず、BLAS が生きている間は破棄してはなりません。

注目すべき詳細: micromap のための `vkCmdBuildAccelerationStructuresKHR` 呼び出しは、BLAS ビルド自体と同様に、コマンドバッファに記録され GPU キューにサブミットされます。micromap データは CPU 上で計算されてプレーンバッファとして GPU に押し込まれるのではありません — ビルドステップは独自の内部コンパクションとレイアウト変換を行い、データをトラバーサルハードウェアのネイティブ形式にパックします。エンジンは単一の転送およびコンピュートキューサブミッションを使って、新しく読み込まれたメッシュのすべての micromap をビルドし、BLAS ビルドに進む前にセマフォでシグナルします。

== Micromap を BLAS に接続する

micromap の `VkAccelerationStructureKHR` オブジェクト自体は何もしません。BLAS がビルドまたは更新される前に BLAS ジオメトリ記述に添付されたときだけ有効になります。この添付は `VkAccelerationStructureTrianglesOpacityMicromapKHR` 構造体を通じて起き、関連サブメッシュの `VkAccelerationStructureGeometryTrianglesDataKHR` の `pNext` チェーンに追加されます。この構造体の `micromap` フィールドは上記で作成した `VkAccelerationStructureKHR` ハンドルを持ち、各三角形にどの micromap エントリが対応するかを識別する `indexBuffer` フィールドはプレーンな `VkDeviceAddress` です。ここでもデバイス側アドレスのみが受け付けられ、パイプライン全体で使われるデバイスのみの構築モデルと一貫しています。

このチェーンが、人間が読める概念（「この葉メッシュは micromap を持つ」）とハードウェアに見える結果（「この BLAS は内部表現に micromap データを含む」）の間の文字どおりのリンクです。このチェーンを置いて BLAS がビルドされれば、その BLAS に対するその後のすべてのトラバーサルクエリは自動的に micromap を参照します — クエリするシェーダーが必要な実行モードを宣言している場合に限ります。前章で議論したとおりです。

micromap が添付された後、BLAS は再ビルドされなければなりません。メッシュが静的 — 植生は通常そう — なら、これは一度きりのコストです。メッシュが変形する場合（micromap のユースケースでは珍しいが可能）、micromap を再ビルドし、BLAS もそれに応じて更新しなければなりません。`OpacityMicromapBuilder` はどの BLAS に micromap が添付されているかを追跡し、micromap データが変われば再ビルド用にフラグを立てます。

== シャドウシェーダーには小さな追加が一つ必要

ここがほぼ優雅な部分です。このインフラ作業すべて — 解析、分類、GPU ビルド、BLAS 添付 — の後、シャドウシェーダーが必要とするのは単一の追加だけで、論理の書き換えではありません。

`VK_KHR_opacity_micromap` では、トラバーサルハードウェアが micromap 最適化を適用するために、レイクエリシェーダーは `OpacityMicromapKHR` SPIR-V 実行モードを宣言しなければなりません。GLSL では、`GLSL_EXT_opacity_micromap_ray_query_mode` 拡張経由で `gl_EnableOpacityMicromapExt` を有効にすることで表現されます — シェーダーファイルの先頭の一行です。それなしでは — アクセラレーション構造に opacity micromap が含まれるとき — SPIR-V 仕様は **未定義動作** を義務付け、any-hit パスへの優雅なフォールバックではありません。シェーダーの残りは触れられません。`rayQueryInitializeEXT`、`rayQueryProceedEXT`、標準クエリ関数への呼び出しは同一のままです。micromap は依然としてシェーダーの論理に対して透過的です — トラバーサルハードウェアが Opaque と Transparent のマイクロ三角形をシェーダー層が関与する前に処理し、Unknown のマイクロ三角形は以前と同じメカニズムで any-hit shader を呼び出します。シェーダーは、与えられた交差が micromap で解決されたか any-hit パスで解決されたかを決して知りません。単一の宣言は論理の変更ではありません。micromap 参加の微妙なトラバーサル順序の含意をシェーダー著者が考慮したことを保証するため、KHR 仕様が要求するオプトインです。

学生は `Courses/` フォルダ内の `opacity_micromap_builder.cpp` を開き、三つのフェーズの完全な実装を見て、エンジンのメイン初期化シーケンスを `simple_engine.cpp` で見て、`OpacityMicromapBuilder::buildForScene()` がどこで呼ばれるかを確認すべきです。統合は意図的に最小です — ビルダーは既存のシーン読み込みパイプラインを再構成せずに差し込む、自己完結したユーティリティです。

== メモリとオブジェクトの寿命

ビルダーが作成するすべての micromap `VkAccelerationStructureKHR` は、バッキングデバイスメモリアロケーションと並べて、所有ハンドルのベクタに格納されます。ビルダーは、すべての micromap アクセラレーション構造を破棄しデバイスメモリを解放する `releaseAll()` メソッドを提供します。これはシーンがアンロードされるとき、またはエンジンがシャットダウンするときに呼ばれます。

順序要件は厳格です。BLAS は、それらが参照する micromap アクセラレーション構造より先に破棄されなければなりません。`VK_KHR_opacity_micromap` の micromap は標準の `VkAccelerationStructureKHR` オブジェクトであるため、学ぶべき別の破棄コマンドはありません — BLAS を解体するのと同じ `vkDestroyAccelerationStructureKHR` 呼び出しが micromap も解体します。BLAS は常にその micromap へのライブ参照を持ちます。KHR モデルには破棄可能またはオプションの micromap 添付という概念はありません。エンジンの解体シーケンスはこの順序を尊重し、すべての BLAS アクセラレーション構造を破棄してから micromap に対して `releaseAll()` を呼びます。エンジンのシャットダウンシーケンスを変更する場合は、この順序を保ってください。

多くの異なるメッシュ型を持つ非常に大きなシーンでは、micromap メモリが積み上がることがあります。`OpacityMicromapBuilder` は総 micromap メモリフットプリントを問い合わせるメソッドを含み、予算ベースのポリシーを実装するために使えます。micromap メモリが閾値を超えたら、優先度の低いサブメッシュ（ビュー内のシャドウキャスト三角形が少ないもの）をより低い分割レベルでビルドするか、完全にスキップできます。デバイスが報告する `maxMicromapTriangles` プロパティも、単一の micromap アクセラレーション構造にビルドできるものの上限を知らせ、ビルダーはこの制限を尊重して、大きなメッシュを複数の micromap オブジェクトに分割するかどうかを決めます。
