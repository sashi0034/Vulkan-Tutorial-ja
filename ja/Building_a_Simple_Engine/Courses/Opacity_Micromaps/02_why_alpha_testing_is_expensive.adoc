= GPU が実際にしていること

== どこまでも続くファイリングキャビネット

アルファテストが高い理由を理解するには、GPU がレイが当たる可能性のある三角形をどう見つけるかについての心的モデルが必要です。すべてのレイについてシーン内のすべての三角形を調べるのは壊滅的に遅いでしょう — モダンなシーンには数百万の三角形があり、4K フレームには数百万のピクセルがあり、それぞれがシャドウレイを飛ばします。総当たりのコストはフレームあたり数兆回の演算に達します。

GPU はこれを、**Bounding Volume Hierarchy**、略して **BVH** と呼ばれる空間データ構造で回避します。ファイリングキャビネットを想像してください。最上位の引き出しには、シーン全体を囲む一つの大きな箱があります。その箱の中には、シーンの主要な領域 — たとえば左半分と右半分 — をそれぞれ囲む、いくつかの小さな箱があります。それらの中にはさらに小さな箱があり、さらに細分されます。階層の底では、各葉ノードが少数の実際の三角形を含みます。

レイが BVH に入ると、まず最上位のバウンディングボックスをテストします。外れれば完了 — レイはシーン全体の何にも当たりません。当たれば、その箱を開けて子をテストします。当たったものを辿り、外れたものを無視します。トラバーサルは幾何学的に狭まります。百万の三角形をテストする代わりに、レイはおそらく数十のバウンディングボックスをテストし、候補三角形は一握りに達します。これがリアルタイムでレイトラバーサルが現実的である理由です。

完全に **不透明** なジオメトリでは、話は三角形レベルで終わります。トラバーサルがレイが三角形と交差することを見つけると、ハードウェアは **committed hit（コミットされたヒット）** と呼ばれるものを発行します — ヒット距離と表面データを記録し、トラバーサルは終了できます（またはモードに応じて、より近いヒットを探し続けます）。シェーダーコードは関与しません。固定機能ハードウェアがすべてを処理します。

== Any-Hit Shader: 必要な割り込み

アルファテストされたジオメトリは、このきれいな話を壊します。BVH トラバーサルがアルファテストメッシュ上の候補三角形を見つけたとき、交差点が実際に固体かどうかを確認せずにヒットをコミットできません。そのためには、プログラマブルシェーダー — **any-hit shader** — を呼び出さなければなりません。

any-hit shader は交差データを受け取ります。具体的には、三角形内のヒット点の **重心座標（barycentric coordinates）** です。重心座標は、レイが三角形内の正確にどこに着地したかを、三頂点の重み付き結合として表します。重心座標から、シェーダーはヒット点の UV テクスチャ座標を補間できます。次にそれらの UV でアルファテクスチャをサンプリングし、結果を調べます。アルファ値がマテリアルの閾値より上なら、ヒットは固体 — シェーダーはデフォルトのコミットを進ませます。アルファが閾値未満なら、シェーダーは ignore 関数を呼び出し、トラバーサルは次の候補を探し続けます。

これはいくつかの逐次操作 — 重心補間、UV 補間、テクスチャサンプル、比較、判断 — であり、トラバーサルが進む前に完了しなければなりません。しかも、すべての候補ヒットについて完了しなければなりません。

== 並列性の問題: ダイバージェンス

GPU は逐次マシンではありません。何百、何千もの並列実行ユニット上で同時にシェーダーコードを実行します。これらのユニットは **ワープ**（NVIDIA の用語）または **ウェーブフロント**（AMD の用語）と呼ばれるグループで動作します — 典型的には 32 または 64 のシェーダースレッドがロックステップで実行し、異なるデータに対して同時に同じ命令を走らせます。この大規模並列実行モデルが、GPU がフレームあたり数百万ピクセルをシェーディングできる理由です。

ロックステップモデルには致命的な弱点があります。**ダイバージェンス** です。ワープ内の異なるスレッドが異なるコードパスを取る必要があるとき — あるスレッドが不透明ヒットを見つけ、別のスレッドが透明ヒットを見つけるとき — ワープはロックステップを保てません。両方のコードパスを実行し、各パスでアクティブでないスレッドをマスクオフしなければなりません。ワープの半分が待ち、もう半分がその分岐を実行し、その逆です。実効並列性は半分、あるいはそれ以下になります。

アルファテストはダイバージェンスの機械です。ワープ内の 32 本のシャドウレイのうち、あるものは葉三角形の不透明領域に当たり、あるものは透明領域に当たり、あるものは葉に全く当たらないかもしれません。any-hit shader はスレッドごとに異なる三角形で発火します。あるスレッドはアルファチェックを早く終え、別のスレッドはキャッシュ挙動により長くかかります。ワープはシリアライズされます。実行ユニットは遅れたものを待ってアイドルになります。GPU の並列スループットの利点は、最も痛いところで蒸発します。

== テクスチャキャッシュの惨事

最初のコストを悪化させる、関連する第二のコストがあります。**テクスチャキャッシュミス** です。GPU にはテクスチャデータ専用のキャッシュがあり、テクスチャサンプルを効率的に供給するよう設計されています。ラスタライズでは、近くのスクリーンピクセルがテクスチャ空間の近い点をサンプリングする傾向があるため、テクスチャキャッシュは美しく機能します — 空間的コヒーレンスにより隣接スレッドが同じキャッシュラインから読み、キャッシュは温かく保たれます。

BVH トラバーサルでは、レイは任意のスクリーン位置から来て、任意の UV 座標で三角形に当たり得ます。同じワープ内の 17 本のシャドウレイが、完全に異なる 17 枚の葉三角形をテストし、それぞれ異なる UV 座標で、テクスチャアトラス全体に散らばっているかもしれません。各サンプルがテクスチャメモリの異なる領域に着地するかもしれません。キャッシュはコールドミスにコールドミスを重ねます。コヒーレントなラスタライズアクセス向けに作られたテクスチャユニットは、ビデオメモリからデータがフェッチされる間、繰り返しストールします。

テクスチャメモリ帯域は有限で、GPU 上で同時に起きている他の操作と共有されます。any-hit shader でのキャッシュミスは、その帯域を奪い合います。森のシーンでは、any-hit shader 呼び出しによるテクスチャキャッシュのスラッシングが、総フレーム時間のかなりの割合を占めることがあります。

== コストを数える

直感を築くために、具体的な数字を置きましょう。森のシーンに、シャドウビューフラスタム内で見える葉三角形が 50,000 枚あるとします。シャドウマップ方式なら、これらを低解像度でフレームあたり一度処理します。私たちのレイトレースシステムでは、1080p でピクセルあたり 16 本のシャドウレイを飛ばし、およそ 3300 万ピクセル × 16 レイ — 約 5 億 3000 万本のシャドウレイになります。各シャドウレイは、不透明ヒットを見つけるか樹冠を出るまでに、平均して 3 から 5 枚の葉三角形を通過するかもしれません。つまりフレームあたり 15 億から 25 億回の any-hit shader 呼び出しであり、それぞれがテクスチャサンプルを必要とし、すべてがキャッシュを奪い合い、すべてがダイバージェンスを導入します。

モダンな GPU はおそらく毎秒 500 億から 1000 億回の単純な演算を実行できます。しかし any-hit shader は単純な演算ではありません — テクスチャサンプルとメモリ依存の判断を持つ分岐シェーダーです。現実的には、各呼び出しは数十ナノ秒のコストがかかります。計算は居心地が悪いです。

== 根本的な洞察

ここでの無駄は、アルゴリズム的な意味での計算の無駄ではありません — any-hit shader は、ハードウェアが知っていることに照らして、まさに必要なことをしています。無駄は情報的なものです。GPU は実行時に、シーンロード時に利用可能だった情報を発見しています。アルファテクスチャはフレーム間で変わりません。各三角形とそのアルファマスクの関係も変わりません。すべての any-hit shader 呼び出しは、あらかじめ知り得たことを再発見しています。

これが問題の根であり、解決策を直接指し示します。GPU がすでに、アクセラレーション構造に直接符号化された形で、各三角形のどの部分領域が不透明でどれが透明かを知っていれば、本当に不確かな縁領域を除き、シェーダーを一切実行せずにトラバーサル判断ができます。any-hit shader は普遍的ではなく稀になります。

それがまさに Opacity Micromaps が提供するものです。
