= 違いを見て、いつ使うかを知る

== シャドウがどう見えるか

まず実際に見えるものから始めましょう。シンプルエンジンで Opacity Micromaps を有効にし、密な植生のあるシーンをレンダリングすると、視覚結果は非 micromap 版と区別できないはずです。葉の影は同じ詳細なシルエットで落ちるべきです。樹冠下のペナンブラグラデーションは同じ柔らかさを持つべきです。個々の葉の形は、下の地面の影パターンで識別できるべきです。

これは制限ではなく機能です。Opacity Micromaps は近似ではなく、パフォーマンス最適化です。完全にシェーダー駆動のパスとまったく同じ視覚出力を出すよう設計されており、例外は分類境界（Unknown のマイクロ三角形のあの細いリング）が以前と同じ any-hit shader に頼ることだけです。出力の視覚的同一性は偶然ではありません — 拡張の設計要件です。

変わるのはプロファイラが示すものです。植生が支配的なシーン — 森の空き地、広葉植物のある庭のテラス、密な頭上樹冠のある公園 — では、any-hit shader に費やされる時間が劇的に下がるはずです。削減量はコンテンツ、分割レベル、アルファテストジオメトリと交差するシャドウレイの割合に依存しますが、よく合うシーンでは any-hit 呼び出し回数の 60 から 90 パーセントの削減が一般的です。高サンプル数のソフトシャドウでは、これはフレーム時間の節約に直接つながり、GPU プロファイラはレイトラバーサル段階の時間短縮として示します。

ビストロ風のシーン — 籐の椅子のあるテラス、頭上の葉のつる、半透明のカフェ傘 — では、OMM によるフレーム時間の改善が、フレーム予算を後退させずに追加のソフトシャドウサンプル段を解放できるほど大きくなり得ます。そういう無料の品質向上こそ、よく狙われた GPU 最適化の感触です。

== OMM が元を取るとき

Opacity Micromaps は、特定の条件の組み合わせで最も効果的です。第一は **シャドウパス内のアルファテストジオメトリ** です。アルファマスクされたオブジェクトは、実際にレンダリングされているピクセルに届く影を落としていなければなりません。シーンの向こう側にある美しい木で、シャドウカメラフラスタムの外や光から背を向けているものは、シャドウレイ交通に何も寄与せず、OMM からも何も得ません。

第二の条件は **高いシャドウレイ数** です。OMM からの節約はレイあたりなので、ピクセルあたりのシャドウレイ数とともにスケールします。ピクセルあたり 8、16、または 32 サンプルのソフトシャドウ構成は、利益を比例して増幅します。ピクセルあたり 1 本のハードシャドウをレンダリングしている場合、OMM は依然として助けますが、絶対的なフレーム時間の節約は小さくなります。

第三の条件は **シャドウパス内の幾何学的複雑さ** です。少数の葉三角形のまばらな配置は、OMM のオーバーヘッドを正当化するほどの any-hit shader 圧力を生みません。何千もの重なる葉層を持つ密な樹冠で、多くのレイが固体ヒットを見つけるか出るまでに通過しなければならないものが、まさに OMM が対処するよう設計されたケースです。

屋内植生、密な森の回廊、公園、ジャングル環境、リアルな都市緑化のあるシーンはすべて強い候補です。金網フェンス、工業用グレーチング、穴あきパネル、パーティクルベースの効果（パーティクルクアッドがアルファマスクを使うもの）のあるシーンも大きく恩恵を受けます。

== OMM が助けないとき

完全に不透明なジオメトリ — 固体の壁、床、車両、アルファテストされた髪や服のないキャラクター — では、ハードウェアはすでに高速な committed-hit パスを使っています。排除すべき any-hit shader 呼び出しはありません。OMM は不透明ジオメトリのパフォーマンスに影響せず、不透明メッシュに micromap を添付することもありません。

三角形数が少ない単純なシーンは良い候補ではありません。micromap をビルドし、GPU メモリに格納し、BLAS に添付するオーバーヘッドは、小さくても実在します。十数枚の葉三角形しかないシーンでは、このオーバーヘッドは節約より高くつきます。

アルファテストジオメトリがシャドウキャスト構成に稀にしかないシーンも、限定的な利益しか見ません。太陽がカメラの後ろにあり植生が完全に逆光なら、植生のアルファテスト三角形のほとんどはどの表面上の点と光の間にもなく — シャドウレイはそれらに到達しません。そのジオメトリの any-hit shader 呼び出し回数はすでに低いです。

最後に、OMM は伝統的な意味での **部分的透明度** — 煙、ガラス、半透明の植生シェーダーのような中間アルファ値を持つジオメトリ — には利益を提供しないことに注意してください。それらのマテリアルは物理的に正確な部分遮蔽を計算するために any-hit shader を必要とするため、OMM を使えません。OMM は具体的に、正しい答えが常に完全に不透明か完全に透明であり、決して中間値ではない二値アルファマスキング向けです。

== 分割レベルの選択

分割レベルは、精度とメモリ消費のバランスを支配します。ほとんどの植生テクスチャでは、レベル 2（元の三角形あたり 16 マイクロ三角形）から始めるのが良い最初の選択です。このレベルでは、葉の縁が十分な解像度で捉えられ Unknown の割合を小さく保ち、メモリオーバーヘッドは控え目です。

プロファイリングが any-hit shader が依然として大きな時間を消費していること — 高い Unknown 割合を示す — を明らかにしたら、レベル 3（64 マイクロ三角形）に上げると、通常、意味のある差を生むほど分類精度が向上します。レベル 4（256 マイクロ三角形）に行くことは稀にしか必要なく、大きなシーンではメモリ予算を超える micromap データを生み得ます。

エンジンの `OpacityMicromapBuilder` は、分類後に Unknown 割合を問い合わせるメソッドを公開します。マイクロ三角形の何パーセントが Unknown 状態になったかです。この割合が 15 または 20 パーセントを超えていれば、分割レベルを上げる価値がある可能性が高いです。すでに 5 パーセント未満なら、現在のレベルはおそらく最適です。

`VK_KHR_opacity_micromap` は **lossy ビルドモード** を導入し、micromap をビルドするときに `VK_BUILD_ACCELERATION_STRUCTURE_MICROMAP_LOSSY_BIT_KHR` フラグを設定することで有効になります。このフラグが設定されると、ドライバは内部で不可逆圧縮を適用することが許可され、引き換えに標準の `maxOpacity4StateSubdivisionLevel` 上限を超える分割レベル — `maxOpacityLossy4StateSubdivisionLevel` で報告される値まで — をサポートできることがあります。境界線上のマイクロ三角形の偶発的な再分類が許容できるコンテンツでは、これは標準ハードウェア制限が許すより先まで分類精度を押し上げる有用なレバーであり、ドライバが格納状態に対してより大きな自由を取るコストを伴います。

ハードウェアはまた、単一の micromap オブジェクトに含まれ得るマイクロ三角形の総数に絶対上限 — `maxMicromapTriangles` — を課します。非常に高密度のメッシュ、たとえば大きな地形パッチや、高い分割レベルで何千ものベース三角形を持つ極めて詳細な植生クラスタでは、拡張されたマイクロ三角形数がこの上限を超えることがあります。その場合、メッシュを複数のサブメッシュに分割し、それぞれに独自の micromap を持たせ、どの個々の micromap も報告された最大値を超えないようにしなければなりません。ビルド時の計画段階でこの上限に対して確認することで、実行時の驚きを避けます。

== ハードウェア要件と優雅なフォールバック

機能の可用性は、デバイス初期化時に `VkPhysicalDeviceOpacityMicromapFeaturesKHR` を問い合わせてテストされます — その構造体の `micromap` フィールドが `VK_TRUE` のとき、拡張は存在し利用可能と見なされます。`OpacityMicromapBuilder` は機能が欠けると優雅にフォールバックします。

フォールバック挙動は単なる安全網ではありません — 第一級の構成です。多くのユーザーは OMM サポートのないハードウェアでアプリケーションを実行し、体験は正しく受け入れ可能でなければなりません。OMM はレンダリング画像を変えないため、フォールバックパスは視覚的に同一です。唯一の違いはパフォーマンスであり、OMM 以前のベースラインへ優雅に劣化します。

常にフォールバックを設計してください。OMM の可用性を決して仮定しないでください。プロファイラを使って両方のパスを測定し、コンテンツのパフォーマンス差を文書化してください。優雅なフォールバック姿勢は、よく設計された GPU 機能の特徴です。
