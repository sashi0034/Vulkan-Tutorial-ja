= Micromaps: ハードウェアがすでに知るべきことを教える

== 中心となるアイデア

ここまで議論してきたすべては、一つの洞察に収束します。GPU は、あらかじめ完全に知り得る情報を発見するために、高価な実行時作業をしているということです。葉のアルファテクスチャはアニメーションしません。フェンスメッシュの不透明度分布はフレームごとに変わりません。この情報をアクセラレーション構造そのもの — ハードウェアが BVH トラバーサルに使う同じデータ構造 — にベイクできれば、ハードウェアはシェーダーを一切呼び出さずに不透明度判断ができます。

これが **Opacity Micromap** の完全な説明です。GPU のアクセラレーション構造内の三角形メッシュに付随するデータ構造で、各三角形を小さな三角形の規則的なグリッドに分割し、それぞれに事前計算された不透明度状態を割り当てます。「マイクロ」という言葉は正確に使われています。これらはレイトレーサーがトラバースしなければならない追加の幾何三角形ではありません。トラバーサルハードウェアがメタデータとして読み取り、与えられたレイと三角形の交差が実際のヒットを表すかどうかの判断を加速するために使う、サブ三角形の分類です。

== 二つのフォーマット: 2-State と 4-State

`VK_KHR_opacity_micromap` は二つの opacity micromap フォーマットを定義します。どちらを選ぶかは、境界でアルファコンテンツがどれだけの精度を必要とするかによります。

**2-state フォーマット**（`VK_OPACITY_MICROMAP_FORMAT_2_STATE_EXT`）は、マイクロ三角形あたり 1 ビットを格納します。各マイクロ三角形は Transparent（0）または Opaque（1）のどちらかです。この二値分類は可能な限り最も単純なモデルです。シェーダーは決して呼び出されず — ハードウェアは格納されたビットに基づいてすべてのヒットを即座にコミットまたは破棄します。2-state フォーマットは、アルファコンテンツを完全に制御でき、どのマイクロ三角形も本当にアルファ境界をまたがないことを保証できる場合、あるいは強制的な二値分類による小さな視覚誤差を受け入れられる場合に理想的です。

**4-state フォーマット**（`VK_OPACITY_MICROMAP_FORMAT_4_STATE_EXT`）は、マイクロ三角形あたり 2 ビットを格納し、四つの異なる状態をサポートします。

* **Transparent (0)**: シェーダーを一切実行せずにヒットを即座に破棄。コストゼロ。
* **Unknown-Transparent (1)**: any-hit shader が呼び出される。シェーダーが明示的に判断しなければ、トラバーサルはデフォルトでヒットを透明として扱う。
* **Unknown-Opaque (2)**: any-hit shader が呼び出される。シェーダーが明示的に判断しなければ、トラバーサルはデフォルトでヒットを不透明として扱う。
* **Opaque (3)**: シェーダーを一切実行せずにヒットを即座にコミット。コストゼロ。

二つの Unknown 状態が、4-state 方式を正しくかつ実用的にする鍵です。あらゆる三角形のあらゆるピクセルを事前に完璧に分類しようとする — それには無限の分割が必要 — 代わりに、保守的でいられます。アルファテクスチャがマイクロ三角形全体にわたってはっきり固体なら、Opaque (3) を符号化します。はっきり空なら、Transparent (0) を符号化します。境界領域でのみ — 一つのマイクロ三角形がアルファカットオフの縁をまたぐところ — 不確実性を認め、二つの Unknown 状態の一方を使って any-hit shader にフォールバックします。Unknown-Transparent は植生の典型的な選択です。ヒントがドライバを透明側に偏らせ、アルファ境界での統計的期待に合います。

よく設計された葉テクスチャでは、Opaque と Transparent 領域が支配的です。その境界は細い周囲です。マイクロ三角形グリッドが十分細かければ、その周囲領域は全マイクロ三角形のおよそ 5 パーセントか 10 パーセントに相当します。残りの 90 から 95 パーセントは確定的に分類され、それらに対して any-hit shader は決して発火しません。コスト削減はそれに比例します。

== 分割の可視化

image::images/omm_triangle_subdivision.svg[葉形の三角形がマイクロ三角形の規則的グリッドに分割された図。中央は不透明で緑、外側は透明で赤、葉の縁の細いリングは不明で黄, 600, 500]

一枚の葉三角形 — アルファテクスチャがカエデの葉を示す平坦な四角形 — を考えてください。micromap の分割は、この三角形に規則的なグリッドを重ねます。葉の中心では、アルファはしっかり 1.0 です。その領域のすべてのマイクロ三角形がサンプリングされ、完全に不透明と判定されます。心的モデルでは緑に色付けされます — ハードウェアはここに落ちるヒットを即座にコミットします。

葉の形から遠いクアッドの縁では、アルファはしっかり 0.0 です。その領域のすべてのマイクロ三角形がサンプリングされ、完全に透明と判定されます。赤に色付けされます — ハードウェアはここに落ちるヒットを即座にスキップします。

葉のシルエットの正確な境界では、テクスチャ設計者が厳しい縁を避けるためにアルファを 1.0 から 0.0 へブレンドしたところで、一部のマイクロ三角形が両方の値をまたぎます。これらは黄 — unknown — に色付けされます。トラバーサル中に any-hit shader をトリガーする唯一のものです。この黄色いリングは細く、おそらく幅が 1 か 2 マイクロ三角形で、三角形の面積の小さな割合を表します。

== 分割レベル: 精度とコスト

**分割レベル（subdivision level）** は、マイクロ三角形グリッドの細かさを制御します。レベル 0 では、三角形全体が単一の分類を受け取ります — 全体に一つの状態です。これは本当に一様なジオメトリ（全体が不透明または全体が透明）にのみ有用で、アルファテストコンテンツでは珍しいです。

レベル 1 では、各元の三角形は 4 個のマイクロ三角形に分割されます。レベル 2 では 16 個、レベル 3 では 64 個、レベル 4 では 256 個になります。レベルが上がるごとにマイクロ三角形の数はおおよそ 4 倍になり、分類の精度が向上する一方で、より多くのストレージとより多くの事前計算が必要になります。

典型的な植生テクスチャでは、レベル 2 またはレベル 3 が良いバランスを提供します。一枚の葉三角形に対するレベル 3 の micromap は 64 個のマイクロ三角形を含みます。この解像度で葉の形は十分よく近似されます — 葉の本体のほとんどは Opaque、空き空間のほとんどは Transparent、unknown のリングは狭いです。レベル 4 に進むと精度はわずかに向上しますが、メモリフットプリントは 4 倍になり、実行時の利益はおそらく限定的です。

正しい選択は、テクスチャの内容とアプリケーションのパフォーマンス目標に依存します。シャープで高コントラストのアルファマスク（ほぼ二値アルファの金網フェンスなど）を持つコンテンツは、境界がはっきりしているため、低い分割レベルでも大きく恩恵を受けます。柔らかい段階的なアルファ遷移（ヘアカードや羽毛など）を持つコンテンツは、unknown の割合を小さく保つために、より高い分割レベルが必要になることがあります。

== Micromap の構築: 一度きりの投資

micromap のビルドプロセスには、時間的に分離できる二つの概念的に異なるフェーズがあります。

**フェーズ 1 — 分類（CPU、アセットあたり一度）:** アルファテストメッシュ内の各三角形について、また選択した分割レベルでの各マイクロ三角形について、ビルダーはマイクロ三角形内のいくつかの点 — 典型的には重心、または複数のジッター位置 — でアルファテクスチャをサンプリングします。それらのサンプルを平均し、閾値と比較します。平均アルファが上側閾値（1.0 に近い）より上なら、マイクロ三角形は Opaque に分類されます。下側閾値（0.0 に近い）より下なら、Transparent に分類されます。閾値の間に落ちれば、Unknown（具体的にはアプリケーション方針に応じて Unknown-Transparent または Unknown-Opaque）に分類されます。出力はコンパクトなマイクロ三角形あたり 2 ビットの配列 — 典型的なメッシュで数百キロバイト — です。

**フェーズ 2 — GPU ビルド（デバイスのみ、ロード時に実行）:** 分類された 2 ビットデータが GPU メモリにアップロードされます。`vkCmdBuildAccelerationStructuresKHR` コマンドが記録され、デバイスキューにサブミットされます。ドライバは内部レイアウト変換とコンパクションを行い、ハードウェアのネイティブなトラバーサル形式で micromap の `VkAccelerationStructureKHR` を生成します。`VK_KHR_opacity_micromap` はホスト側のビルドパスを提供しません — この GPU サブミットステップは常に必要です。

二つのフェーズはロード時に結合（**オンラインモード**）することも、分離（**オフラインモード**）することもできます。オンラインモードでは、分類と GPU ビルドの両方がシーンの初回読み込み時に実行されます — 開発中には有用ですが、メッシュとテクスチャの複雑さに比例した起動コストが加わります。オフラインモードでは、フェーズ 1 はコンテンツ制作時の前処理として実行され、結果の 2 ビット配列がディスクに保存され、アプリケーションとともに配布されます。実行時にはフェーズ 2 のみが実行され、起動コストを最小に保ちます。GPU ビルド自体は事前ベイクできません。ドライバはネイティブな内部表現を生成するために、ターゲットハードウェア上で実行する必要があります。

これはメッシュごと、テクスチャごと、分割レベルごとの一度きりの操作です。出荷するゲームやビジュアライゼーションアプリケーションでは、見返り比は莫大です — 分類コストは制作時に一度だけ支払われ、利益はアプリケーションがレンダリングするすべてのフレームにわたって蓄積します。

このプロセスの出力は、コンパクトな 2 ビット値の配列（4-state フォーマットの場合）です。マイクロ三角形あたり一つ。Opaque は 3、Transparent は 0、Unknown-Transparent は 1、Unknown-Opaque は 2 として符号化されます。ほとんどの植生では、境界のマイクロ三角形に Unknown-Transparent (1) が使われます。any-hit shader に最終判断を委ねつつ、結果が透明である可能性が高いことをヒントします。分割レベル 3 で 10,000 枚の葉三角形を持つメッシュでは、micromap データは 10,000 × 64 マイクロ三角形 × 2 ビット、つまり約 160 キロバイトです。2-state フォーマットを使うと、any-hit フォールバックを完全に捨てる代わりに、およそ 80 キロバイトに半減します。このメモリフットプリントは、メッシュデータ自体と比べて無視できるほどです。

== データが住む場所

分類されたマイクロ三角形データは GPU にアップロードされ、型が `VK_ACCELERATION_STRUCTURE_TYPE_OPACITY_MICROMAP_KHR` に設定された **`VkAccelerationStructureKHR`** の構築に使われます。これは KHR API における意図的な統一です。micromap データ用に別のオブジェクト型を導入する代わりに、`VK_KHR_opacity_micromap` は micromap を既存のアクセラレーション構造抽象に直接折り込みます。BLAS や TLAS に使うのと同じ `VkAccelerationStructureKHR` ハンドルが、micromap のハンドルでもあります — 作成時の `type` フィールドがそれらを区別します。micromap アクセラレーション構造は `vkCreateAccelerationStructureKHR`（`VK_KHR_acceleration_structure` から）経由で割り当てられ、型として `VK_ACCELERATION_STRUCTURE_TYPE_OPACITY_MICROMAP_KHR` を渡します。micromap のビルドは、`geometryType` を `VK_GEOMETRY_TYPE_MICROMAP_KHR` に設定した `vkCmdBuildAccelerationStructuresKHR` を使います — すべてのアクセラレーション構造ビルドに使われるのと同じデバイス側コマンドです。KHR API には micromap 用の別ホスト側ビルドパスはありません。すべての micromap 構築は GPU 駆動です。

micromap アクセラレーション構造がビルドされると、BLAS のビルドまたは再ビルド時に対応する BLAS に添付されます。添付は `VkAccelerationStructureTrianglesOpacityMicromapKHR` 構造体を通じて指定され、アクセラレーション構造ビルドに渡すジオメトリ記述にチェーンされます。この構造体の `micromap` フィールドは、いまビルドした micromap の `VkAccelerationStructureKHR` ハンドルです。元の三角形を micromap エントリにマップする `indexBuffer` フィールドは、プレーンな `VkDeviceAddress` です — KHR API にはホストアドレス変種がなく、デバイスのみの設計思想を反映しています。その時点で、micromap データはアクセラレーション構造そのものの一部になります — トラバーサル用のハードウェアネイティブ形式で GPU メモリに格納されます。

トラバーサルハードウェアは、通常の動作の一部として BVH トラバーサル中に micromap データを読み取ります。追加の API 呼び出しも、シェーダー変更も、フレームごとの同期プリミティブもありません。micromap が添付されて BLAS がビルドされれば、ハードウェアは単にそれを使い、すべてのフレームで、メッシュに触れるすべてのレイに対して — 自動的に、透過的に、ハードウェアスピードで — 使います。レイクエリシェーダーは、opacity micromap データがトラバーサル結果に影響し得ることをコンパイラに伝えるため、`SPV_KHR_opacity_micromap` 拡張経由で宣言される `OpacityMicromapKHR` SPIR-V 実行モードを必要とします。これは GLSL または HLSL ソースでの一行の宣言であり、実行時コストはありません。

== Lossy ビルド: 完璧さを高精密と交換する

`VK_KHR_opacity_micromap` API は、もう一つの自由度を提供します。**lossy ビルドフラグ**、`VK_BUILD_ACCELERATION_STRUCTURE_MICROMAP_LOSSY_BIT_KHR` です。micromap ビルド時にこのフラグが設定されると、ドライバは micromap データに不可逆圧縮を適用することが許可されます。その自由と引き換えに、ドライバはより高い実効分割レベルをサポートできることがあります — 利用可能なメモリ予算に本来収まらないほどマイクロ三角形グリッドを細かくできます。標準の `maxOpacity4StateSubdivisionLevel` 上限を超える分割レベルが、`maxOpacityLossy4StateSubdivisionLevel` で報告される値まで利用可能になることがあります。

実用的な含意は微妙です。lossy ビルドは、圧縮された状態に基づいて、マイクロ三角形を Unknown から Opaque または Transparent のいずれかに昇格させることがあり、それは非圧縮の分類が曖昧だったにもかかわらず、そのマイクロ三角形に対して any-hit shader が発火しないことを意味します。ほとんどの植生とほとんどの視距離では、これは視覚的に知覚できません。アルファ境界での精度が重要なコンテンツ — 細い髪、羽毛、またはクローズアップの葉ジオメトリ — では、lossy フラグを避け、やや粗い分割を受け入れる方がよい場合があります。選択はアプリケーションに属します。

== 深遠な単純さ

一歩下がって、何が起きたかを味わってください。アルファテスト付きレイトラバーサルの最も高価な部分 — 不透明度発見のためのヒットごと・フレームごとのシェーダー呼び出し — をホットパスから完全に外しました。固定機能スピードでハードウェアが読む事前ベイクされたデータ構造に置き換えました。視覚結果は変えていません。Unknown のマイクロ三角形に対しては any-hit shader が依然として発火し、境界での正しさを保証します。アプリケーションの API フットプリントも大きく変えていません。ビルドステップ一つと添付構造体一つを追加しただけです。

ほとんどの植生では、これは葉ジオメトリとのシャドウレイ相互作用のおよそ 95% が、シェーダーの観点では今やコストゼロであることを意味します — 不透明ジオメトリのトラバーサルと同じサイクル予算で、ハードウェアがすべてを処理します。縁ピクセルのあの細いリング — 本当にアルファ境界をまたぐ 5% — だけが、依然として any-hit shader のコストを払います。改善は丸め誤差ではありません。ワークロードの性質における質的な変化です。
