= トラバーサルの内部: GPU が Micromap をどう使うか

== 一本のシャドウレイのための二つの旅

Opacity Micromaps が実行時に何をするかを理解する最良の方法は、一本のシャドウレイを両方の世界 — まず OMM なしの世界、次に OMM ありの世界 — を通して追い、作業が正確にどこで起き、どこで消えるかを観察することです。

image::images/omm_traversal_comparison.svg[OMM なしのシャドウレイトラバーサル（多数の any-hit shader 呼び出し）と OMM あり（ほとんどのヒットがハードウェアで解決、シェーダー呼び出しは少ない）の対比図, 700, 420]

=== OMM なしの旅

私たちのシャドウレイは、木の樹冠の真下にある地面上の点から飛ばされます。目標は、上にあるエリアライト上のサンプル位置です。レイは BVH に入ります。

まず、シーン全体の最上位バウンディングボックスをテストし、交差することを見つけます。シーンのサブノードに降り、各レベルでバウンディングボックスをテストします。非葉ジオメトリ — 幹、地面、近くの壁 — は BVH によってすばやく除外されます。レイの経路は、葉の樹冠を含むバウンディングボックス領域を通ります。この領域には数百枚の葉三角形があります。

BVH の葉レベルで、トラバーサルは個々の三角形のテストを始めます。レイは最初の候補葉クラスタのバウンディングボックスと交差します。いくつかの葉三角形が候補です。各候補について、トラバーサルハードウェアはジオメトリがアルファテストとしてフラグされていると判断します。ヒットを自動的にコミットできません。any-hit shader を呼び出します。

any-hit shader が走ります。交差の重心座標を受け取ります。UV 座標を補間します。テクスチャサンプル要求を発行します。テクスチャユニットが要求を処理します — 最初のフレームでキャッシュが冷えているため、おそらくキャッシュミスです。サンプルが返ります。シェーダーはアルファ値をマテリアルのカットオフ閾値と比較します。アルファは 0.02 — 葉の間の空き空間です。トラバーサルは続きます。

次の候補: 別の葉三角形。any-hit shader が再び発火します。UV 補間、テクスチャサンプル（またミス）、比較。アルファは 0.87 — 固体の葉です。ヒットがコミットされます。レイは遮られます。シャドウクエリは「影の中」を返します。

しかしこれは、この一本のレイだけで any-hit shader 呼び出しが 2 回であり、森のシーンのほとんどのシャドウレイはもっと多く遭遇します。樹冠を浅い角度で進むレイは、固体ヒットを見つけるか木を出るまでに、8、12、あるいはそれ以上の葉三角形をテストするかもしれません。各テスト: シェーダー呼び出し、テクスチャサンプル、判断。

ピクセルあたり 16 本のシャドウレイを掛けます。数百万ピクセルを掛けます。any-hit shader は、フレーム全体で最も頻繁に呼び出されるシェーダーです。

=== OMM ありの旅

同じシャドウレイが飛ばされます。BVH トラバーサルは同一に進みます — micromap は BVH 構造やバウンディングボックス階層を変えません。レイは葉ジオメトリ領域に達し、候補三角形のテストを始めます。

最初の候補葉三角形はアルファテストメッシュですが、今度は micromap が添付されています。トラバーサルハードウェアは以前と同様に交差点の重心座標を計算します。次に — 依然として固定機能ハードウェア内で、シェーダーの関与なしに — この交差点を含むマイクロ三角形をルックアップします。マイクロ三角形は Transparent に分類されています。ハードウェアはヒットを即座に破棄します。シェーダーは発火しません。テクスチャサンプルもありません。トラバーサルは、以前の何分の一かの時間で続きます。

二番目の候補: ハードウェアがマイクロ三角形を確認します。Opaque。ヒットは即座にコミットされます。シェーダーは発火しません。テクスチャサンプルもありません。レイは遮られます。シャドウクエリは「影の中」を返します。

このレイの any-hit shader 呼び出し合計: ゼロ。ハードウェアがすべてを解決しました。

たまたま葉の縁領域 — Unknown のマイクロ三角形 — に正確に着地した別のレイでは、any-hit shader は発火します。しかしこれは例外であり、規則ではありません。よく分割された葉テクスチャでは、これはおそらく二十回のレイと葉の交差のうち一回であり、二十回のうち十九回ではありません。

== ハードウェアが Micromap をどう読むか

micromap データは、ベースポインタ、ストライド、分割レベルが与えられればトラバーサルハードウェアが直接アドレッシングできる形式で、密にパックされた 2 ビット値の配列として GPU メモリに格納されます。micromap が添付されて BLAS がビルドされると、アクセラレーション構造はこのデータへの参照をジオメトリと並べて格納します。

トラバーサル中、ハードウェアが候補三角形の交差を特定すると、本質的にインデックス計算であるマイクロ三角形ルックアップを行います。ヒット点の重心座標と分割レベルが与えられたら、点がどのマイクロ三角形に落ちるかを計算し、配列から 2 ビット状態を読み取ります。この計算は単純な整数演算です — 汎用シェーダーも、別管理リソースからのテクスチャフェッチもありません。バウンディングボックスをテストし交差距離を計算するのと同じ回路である、固定機能トラバーサルユニットの一部です。

これが、ハードウェアが **ハードウェア加速の不透明度評価** を提供するということの意味です。micromap はシェーダーが読むコンピュートバッファではありません。アクセラレーション構造に埋め込まれたメタデータであり、トラバーサルハードウェア自体が読めます。

`VK_KHR_opacity_micromap` では、micromap は別のオブジェクト型としてではなく、型 `VK_ACCELERATION_STRUCTURE_TYPE_OPACITY_MICROMAP_KHR` で作成された標準の `VkAccelerationStructureKHR` として表現されます。標準アクセラレーション構造オブジェクトとのこの統一は、KHR 拡張における意図的な設計選択です。micromap がシーン内の他のすべてのアクセラレーション構造と同じメモリモデル、バリアルール、寿命管理に参加することを意味し、並行するオブジェクト階層を導入しません。

== OpacityMicromapKHR 実行モード

`VK_KHR_opacity_micromap` では、レイクエリのためのハードウェア micromap ファストパスは **自動ではありません**。シャドウシェーダーは、トラバーサルハードウェアが micromap データを参照すべきであることを示すため、`SPV_KHR_opacity_micromap` が定義する `OpacityMicromapKHR` SPIR-V 実行モードを明示的に宣言しなければなりません。GLSL ではこの宣言は次のようになります。

[source,glsl]
----
layout(constant_id = N) gl_EnableOpacityMicromapExt;
----

ここで `N` はスペシャライゼーション定数 ID です。この宣言は `SPV_KHR_opacity_micromap` SPIR-V 拡張によって要求されます。レイクエリシェーダーが opacity micromap を含むアクセラレーション構造をトラバースするのに `OpacityMicromapKHR` を `true` の値で宣言しない場合、SPIR-V 仕様は **未定義動作** を義務付けます — any-hit パスへの優雅なフォールバックではありません。宣言を省略した特定の結果に頼ってはいけません。宣言はハードウェアの意図を明確にし、ドライバとコンパイラがトラバーサル挙動を正しく推論できるようにします。

== ワープ効率へのカスケード効果

前章から、ダイバージェンスの問題を思い出してください。ワープ内の異なるスレッドが異なるコードパスを取ると、ワープはシリアライズされます。アルファテストされたジオメトリは、any-hit shader が異なるレイに対して異なる結果で発火するため、慢性的なダイバージェンスの源でした。

OMM があると、このダイバージェンスは劇的に減ります。ほとんどのレイと葉の交差はシェーダーに到達しません。「固体に当たったか透明か？」で分岐していたスレッドは、今やシェーダーコンテキストでその判断に到達しません — ハードウェアが、候補交差が存在したことすらシェーダーが知る前に、統一された固定機能パスで静かに判断します。透明なマイクロ三角形を通過するレイを持つスレッドは、何にも当たらないレイを持つスレッドと同期したままです。なぜなら両方のパスがコストゼロのハードウェア操作だからです。

any-hit shader は Unknown のマイクロ三角形に対して依然として発火し、そこではダイバージェンスがまだ起き得ます。しかしこのパスをトリガーする呼び出しの割合は十分小さく、ワープ効率は大幅に改善します。典型的な植生の多いシーンでは、OMM なしからよく分類された OMM へ移ることで、any-hit shader 呼び出しを 80 から 95 パーセント削減できます。ワープ効率の利得は比例して大きいです。

== ソフトシャドウへの乗数効果

OMM のパフォーマンス利益は、ピクセルあたりのシャドウレイ数とともにスケールします。ハードシャドウ — ピクセルあたり 1 本 — では、節約は実在しますが上限があります。ピクセルあたり 16 サンプルのソフトシャドウでは、節約は 16 倍になります。これが技法の最も重要な実用的含意の一つです。

ソフトシャドウは、レイトレースレンダリングの最も視覚的に魅力的な機能の一つです。また最も高価なものの一つでもあります。コストがサンプル数に直接比例するからです。シャドウクエリのレイあたりコストを減らすものはすべて、ソフトシャドウ性能に乗数的な影響を持ちます。したがって OMM は、視覚品質にとって最も重要なレンダリングモード — 植生下での高サンプル数ソフトシャドウ — でまさに最も効果的です。

より微妙なキャッシュの利点も働いています。近くのピクセルからの 16 本のシャドウレイがすべて同じ葉三角形をテストするとき（近くのピクセルは近くのジオメトリを見るため、そうなりがちです）、それらの三角形の micromap データは最初の数本のレイからすでに GPU の L1 と L2 キャッシュで温かくなっています。後続レイの micromap ルックアップは、メモリの観点ではほぼ無料です。対照的にアルファテクスチャは毎回フェッチが必要で、異なるレイの UV 座標が変わり得るため、温かく保たれにくいです。

== シャドウレイのライフサイクル、完全版

image::images/omm_shadow_ray_lifecycle.svg[OMM 付きシャドウレイのライフサイクルのフローチャート: レイを飛ばす、BVH トラバーサル、候補ヒット発見、micromap ルックアップ、Opaque / Transparent / Unknown への分岐。Opaque と Transparent はハードウェアで解決、Unknown は any-hit shader にフォールバック, 600, 520]

OMM 有効シーンにおけるシャドウレイの完全なライフサイクルは、決定木として要約できます。レイは BVH トラバーサルに入ります。各候補三角形交差について、ハードウェアは添付された micromap があるかを確認します。micromap がなければ（非アルファテストジオメトリ）、標準の不透明ジオメトリパスを使います。ヒットを即座にコミット。micromap があれば、マイクロ三角形の状態をルックアップします。Opaque: コミット、シェーダーなし。Transparent: 破棄、シェーダーなし。Unknown: any-hit shader を呼び出し、シェーダーに最終判断を委ねます。

この決定木は、Opaque と Transparent の分岐では完全に固定機能ハードウェア内に住みます。Unknown 分岐だけがプログラマブル実行に入ります。その結果、GPU のシェーダー実行ユニットはアルファテスト作業の圧倒的大部分から解放され、本当にシェーダーを必要とするケースに集中できます。`VK_KHR_opacity_micromap` では、この最適化されたパスに到達するには、シェーダーが `OpacityMicromapKHR` 実行モードを宣言していることも必要です。アクセラレーション構造に opacity micromap が含まれるのにその宣言を省略すると、SPIR-V 仕様により未定義動作になります — 正しい結果のためには、ハードウェアとシェーダーの両方が合意していなければなりません。
