= レンダリングパイプライン概要（このサンプル）

このエンジンは Vulkan Dynamic Rendering と、小さく読みやすいパス列を使います。順序は意図的です。トーンマッピングを明示し、透過を扱いやすくし、平面反射や Ray Query コンピュート経路などのオプション機能を差し込みやすくします。

== パスの順序

1. オプションの反射パス（オフスクリーン）:
   * 平面（例: 床や窓）でカメラをミラーする。
   * 不透明ジオメトリを反射レンダーターゲット（カラー + デプス）へ描く。
   * 次フレーム用に反射イメージを `SHADER_READ_ONLY_OPTIMAL` へ遷移する。

2. オフスクリーンカラーへの不透明:
   * すべての不透明オブジェクトをオフスクリーンカラーイメージ（`opaqueSceneColor`）へ描く。
   * デプスは通常どおり読み書き（または depth pre-pass を走らせた場合は読み取り専用）。

3. スワップチェーンへのコンポジット:
   * 不透明レンダリングを終了する。
   * `opaqueSceneColor` を `SHADER_READ_ONLY_OPTIMAL` へ遷移する。
   * スワップチェーンイメージをターゲットに新しいレンダリングインスタンスを始め、オフスクリーンカラーをサンプルするフルスクリーンパスを描く（トーンマッピング含む）。

4. その上に透過:
   * スワップチェーンイメージをカラーアタッチメントのままにし、シーンデプスをバインドする。
   * 透過オブジェクト（ガラス／液体）を奥から手前へ描く。
   * 有効時、ガラスは前フレームの反射テクスチャをサンプルする。

5. UI:
   * UI を最前面に描く。
   * レンダリング終了後、スワップチェーンイメージを `PRESENT_SRC_KHR` へ遷移する。

== コードの参照先

* メインレンダーループ + パス順序:
** `renderer_rendering.cpp`
* パイプラインセットアップ（dynamic rendering のアタッチメント、レイアウト、フォーマット）:
** `renderer_pipelines.cpp`
* コンポジットパスシェーダー（トーンマッピング + 提示）:
** `shaders/composite.slang`
* 複数パイプラインで共有する PBR シェーディングユーティリティ:
** `shaders/pbr.slang`
** `shaders/pbr_utils.slang`
** `shaders/lighting_utils.slang`

== この形にする理由

* 単一のオフスクリーンバッファでトーンマッピングを明示し、ガンマ不正なコピー経路を避ける。
* そのパスではスワップチェーンが現在のカラーアタッチメントなので、透過の順序付けが単純に保たれる。
* 反射パスはオプションで自己完結している。

== 今後のアイデア

このパイプラインをさらに進めたい場合:

* 重いシーン向けに depth pre-pass を追加する（early-z に効き、より正確な Forward+ クラスタリングも可能に）。
* オフスクリーン不透明パスとコンポジットのあいだに軽いブルームチェーンを追加する。
* 重なり合うガラスが多い場合向けに、専用の透過 resolve 経路（weighted blended OIT）を追加する。
* ラスタシェーダーから ray queries を呼ぶハイブリッド描画を示す（リフレクションプローブ、シャドウテスト、ガラス限定の反射など）。
