= 私たちのエンジンにおける平面反射

実世界のシーンでは、光る床や窓をよく目にするでしょう。ゲームでは、その見た目をしばしば疑似します。このエンジンでは実用的で信頼できる技法を選びました: 平面反射です。このページでは、それが何か、なぜ使うか、どう実装されているか、いつ別の手法を選ぶかを説明します。

== 平面反射とは？

平面反射は、単一の平面（例: 平らな床や窓）にわたってシーンの鏡像を描画します。「ミラーカメラ」が反射面の向こう側からシーンを見ていると考えてください。そのミラービューをテクスチャへ描き、ガラス（または任意の反射する平面）を描くときにそのテクスチャをサンプルします。

平面反射が得意なのは:

* 平らな鏡、穏やかな水面、磨かれた床、ガラス板。
* 強いノイズや時間的不安定なしに、安定した高品質の反射が欲しいシーン。

向いていないのは:

* どこでも光沢のある視線依存のぼかしが必要な曲面／粗い表面。
* 任意の反射方向（例: 複雑なマイクロジオメトリを持つ金属）。

== サンプルで平面反射を選んだ理由

次のような反射手法が欲しいからです:

* 理解しやすい（追加パス 1 つ、追加テクスチャ 1 つ）。
* 決定的で安定（「きらめき」や時間蓄積の頭痛がない）。
* Forward レンダラーで単一の主要な反射物（ガラス、床）に実用的。

平面反射はこれら 3 つを満たします。レイトレーシングハードウェアも不要で、GPU 間のスケールも良好です。

== エンジンでの仕組み

メインパスに小さなパス 1 つと小さなブレンド 1 つを足します:

1. ミラーパス（オフスクリーン）
** 平面（例: 地面なら Y=0）でカメラを反射し、ミラーされたビュー行列を計算する。
** フェースカリングを無効化（または調整）して、不透明シーンを反射カラー＋デプスターゲットへ描く。
** 次のパスでのサンプリング用に反射イメージを同期する。

2. メインパス（通常カメラ）
** 不透明＋透過オブジェクトを通常どおり描く。
** ガラスを描くとき、反射テクスチャをサンプルし、Fresnel＋roughness＋ユーザー制御の「反射強度」でガラスシェーディングとブレンドする。

以上です。特別なレンダーグラフマジックも、ray queries も、時間蓄積もありません。

== ミラーの数学（短く要点だけ）

ワールド空間で平面を定義します: `ax + by + cz + d = 0`。

その平面から反射行列 `R` を組み立てます。通常のカメラビューに `R` を適用してミラービューを得ます。実務では、反射でワインディング順が変わるため、ミラーパスではカリングを反転するか `cullMode = none` にします。

オプションのクリッピング用に平面をシェーダーへ渡すこともあります:

* ワールド位置との単純なドット積で、ミラーパスで平面の「後ろ」にあるフラグメントを捨てられます。

== レンダリング手順の詳細

ミラーパス:

* 反射カラーイメージ（メインパスのニーズに合うフォーマット; コンポジット／ガラスパスがサンプルしやすいカラーフォーマットを選ぶ）と反射デプスイメージを作る。
* 描画前: Synchronization 2（`vkCmdPipelineBarrier2`）で反射カラーイメージを `SHADER_READ_ONLY_OPTIMAL` から `COLOR_ATTACHMENT_OPTIMAL` へ遷移する。デプスも同様に `DEPTH_ATTACHMENT_OPTIMAL` へ。
* Dynamic rendering を開始し、不透明用 PBR パイプラインをバインドし、カリングを無効化（または前面を反転）。
* 不透明メッシュを描く。必要なら平面に対するクリップテストを追加できる。
* レンダリングを終了。メインパスでのサンプリング用に反射カラーイメージを `SHADER_READ_ONLY_OPTIMAL` へ遷移する。

メインパス:

* 不透明を通常どおり描く（後でトーンマップ付きコンポジットするためオフスクリーンバッファを使う）。
* 透過パス: ガラスを描くとき、反射テクスチャをサンプルしてブレンドする:
** Fresnel（グレージング角で強い）を使い、roughness で弱める。
** UI で調整できる小さな「反射強度」を掛ける。

== 同期とバリア（重要な点）

Vulkan Synchronization 2 でシンプルに保ちます:

* アクティブな dynamic render pass の内側でイメージレイアウトを変えない。先に終了する。
* `vkCmdPipelineBarrier2` を使い、正しいソース／デスティネーションの stage マスク、access マスク、old／new レイアウトを指定する。
* 反射カラー: ミラーパス前は `SHADER_READ_ONLY_OPTIMAL → COLOR_ATTACHMENT_OPTIMAL`; 後は `SHADER_READ_ONLY_OPTIMAL` へ戻す。
* スワップチェーンイメージ: コンポジット／透過向けに `COLOR_ATTACHMENT_OPTIMAL` へ遷移; 最後のレンダリングパス終了後にだけ `PRESENT_SRC_KHR` へ遷移する。

== ディスクリプタ: 反射はどこにバインドされるか？

* PBR セットの binding 10 を反射サンプラー用に予約する。
* フレームごとの「安全な時点」（前フレームの作業が終わったとき）で、現在フレームの binding 10 を前フレームの反射イメージを指すよう更新する。
* ガラスシェーダーは UBO フラグ（`reflectionEnabled`）を確認し、有効な反射イメージがあるときだけサンプルする。

== ガラスのブレンド: 親しみやすいモデル

ガラスは主に透過ですが、鮮やかで妥当な反射が欲しいです。使うのは:

* Fresnel 項（Schlick）: グレージング角で強い反射。
* Roughness 係数: roughness が大きいほど弱くぼやけた反射（ここでは単純に強度を落とすだけ）。
* 反射強度スライダー: UI に出し、数秒で見え方を調整できる。

完全な物理的スペクトルモデルではありませんが、それで問題ありません。読みやすく、説得力のある結果になります。

== 代替手段と選び方

スクリーン空間反射（SSR）::
* 追加パスなしで動く; フレームの既存カラー／デプスを使う。
* 水たまりや局所効果に向くが、画面外オブジェクトを見逃し、時間的不安定にも苦しむ。
* どこでも手早い反射が欲しく、時折のアーティファクトを許容できるなら SSR を選ぶ。

環境マップ／キューブマップ／リフレクションプローブ::
* 非常に速い; 事前計算。
* 近傍オブジェクトには視線正確ではない; 遠い光沢反射に最適。
* 一般的な環境反射向け、または表面が完全な鏡でないときにプローブを選ぶ。

レイトレーシング（ハードウェア）／ハイブリッド手法::
* 非常に正確; 複雑な反射をサポート。
* ハードウェアと高度なデノイズが必要; コードとパフォーマンスコストが増える。
* ハイエンド GPU を対象に「見た目が正しい」反射をどこでも欲しいなら RT を選ぶ。

平面反射（このサンプル）::
* 追加パス 1 つ、決定的で安定。
* 1 つまたは少数の大きな平面反射物（床、窓、穏やかな水面）に最適。
* レイトレーシングを採用せず、特定表面向けの高品質ミラーが欲しいときに選ぶ。

== パフォーマンスのヒント

* ミラーパスを低解像度で描く（解像度スケールスライダーを用意している）。
* 積極的にカリングする（CPU 視錐台カリングは通常カメラとミラーカメラの両方に効く）。
* 反射面が見えていないときはミラーパスを無効化する。
* 柔らかい見た目が欲しいなら、粗い表面向けに反射サンプルのぼかしを検討する。

== トラブルシューティング

「反射が弱すぎる」::
* Reflection intensity スライダーを上げる、および／または roughness を下げる。

== 今後のアイデア

平面反射をさらに進めたい場合:

* 反射テクスチャの roughness 対応ぼかし（ミップチェーンまたは分離ぼかし）を追加する。
* 複数の反射平面を追加する（複数階のシーンに便利）。
* スクリーン空間フォールバック（SSR）を追加し、有効なところで平面反射とブレンドする。
* 非平面表面向けに選択的な ray query 反射を追加する（ハイブリッド手法）。
