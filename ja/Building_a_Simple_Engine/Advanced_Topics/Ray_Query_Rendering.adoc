== Ray Query レンダリング

このエンジンには、Vulkan の *ray queries* に基づくレイトレース描画モードがあります。完全なレイトレーシングパイプライン（raygen／miss／hit シェーダー）を組む代わりに、ray queries は通常のシェーダーから直接交差テストを行えます。

このサンプルでは、フレーム全体を描くために *コンピュートシェーダー* から ray queries を使います:

* *BLAS*（メッシュごと）と *TLAS*（シーンインスタンス）を構築する。
* 次を行うコンピュートシェーダーをディスパッチする:
** カメラからピクセルごとに 1 本のプライマリレイを生成する。
** `TraceRayInline()` で TLAS 内の交差を探す。
** ラスタ経路と同じ PBR ユーティリティでヒットをシェーディングする。
* 結果をストレージイメージに書き、スワップチェーンへコンポジットする。

=== なぜ ray queries か？

Ray queries は「ハイブリッド」レンダラーに適しています:

* コンピュート、フラグメント、その他のシェーダーステージから呼べる。
* 標準のディスクリプタシステムを再利用する。
* 制御フローがシェーダーコードに残る: どう辿るか、いつヒットを受け入れるか、どうシェーディングするかを自分で決める。

=== 高レベルアーキテクチャ

高レベルでは、Ray Query モードは 3 つの領域に触れます:

* *アクセラレーション構造*: シーンの頂点・インデックスバッファから構築。
* *ディスクリプタ*: TLAS、出力ストレージイメージ、シェーディングに必要なシーンデータをバインド。
* *シェーダー*: レイを生成し、クエリし、ヒットをシェーディング。

重要な考え方は、ray query シェーダーがシーンを「所有」しないことです。ラスタライズと同じシーンアセット（メッシュ、マテリアル、テクスチャ）を読みますが、コンピュート経路向けに設計された別のディスクリプタセットを通します。

=== アクセラレーション構造の構築（BLAS／TLAS）

シーンの準備ができたらアクセラレーション構造を構築します:

* 一意なメッシュごとに BLAS を作成する。
* 各シーンインスタンスをトランスフォーム付きで TLAS に追加する。
* 各 TLAS インスタンスはカスタムインスタンスインデックスを符号化し、シェーダーが対応する `GeometryInfo` テーブルへインデックスできるようにする。

Ray Query シェーダーはそのインスタンスごとのインデックスで次を参照します:

* 頂点・インデックスバッファのデバイスアドレス
* マテリアルインデックス
* 正しいワールド空間法線のためのインスタンスごとの法線トランスフォーム

=== ディスクリプタレイアウト

Ray Query モードは専用のディスクリプタセットレイアウトを使います。正確なバインディング番号はシェーダーと一致させる必要があるため重要です。

このエンジンでの典型的なバインディングは:

* Binding 0: 小さな Ray Query 専用 UBO（カメラ行列、exposure／gamma、トグル）
* Binding 1: TLAS
* Binding 2: 出力ストレージイメージ
* Binding 3: ライトバッファ
* Binding 4: `GeometryInfo` バッファ
* Binding 5: マテリアルバッファ
* Binding 6: テクスチャテーブルとして使う大きな combined image sampler 配列

=== ストリーミング安全なテクスチャアクセス

このエンジンはテクスチャを非同期にストリームします。Ray Query モードの重要な設計選択は、シェーダーが固定サイズ配列（「テクスチャテーブル」）経由でテクスチャをインデックスすることです。

実行時:

* マテリアルはテーブルへのテクスチャ *インデックス* を保持する（baseColor、normal、metallic-roughness、occlusion、emissive）。
* レンダラーは *現在の* テクスチャハンドルでテーブルを更新する。
* スロット `0..4` は共有デフォルトテクスチャ用に予約（サンプリングが常に有効なフォールバックを持つように）。

このやり方でシェーダー側のシェーディングは単純に保てます。サンプリングは `NonUniformResourceIndex()` と `SampleLevel(..., 0.0)` を使います（コンピュートでは明示 LOD が重要）。

=== ディスパッチと結果の提示

Ray Query コンピュートシェーダーはストレージイメージ（通常 HDR 対応）へ書き込みます。

ディスパッチ後:

* バリアで Ray Query 出力イメージを `GENERAL`（書き込み）から `SHADER_READ_ONLY_OPTIMAL`（読み取り）へ遷移する。
* フルスクリーンコンポジットパスが出力イメージをサンプルし、スワップチェーンへ書き込む。
* 最終遷移でスワップチェーンを present 用に準備する。

これにより、ラスタと ray query の両経路で同じポストプロセス制御（exposure／gamma）を再利用できます。
