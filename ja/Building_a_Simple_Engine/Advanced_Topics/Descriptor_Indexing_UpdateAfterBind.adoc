= Descriptor Indexing と安定したディスクリプタ更新

Vulkan のディスクリプタは強力ですが、「フレームインフライト」の寿命ルールをうっかり破るのもいちばん簡単です。

このエンジンでは、単純なルールを一つ使います:

*既知の安全な時点でのみディスクリプタを更新する。*

このルールでストリーミングは安定し、バリデーションはきれいに保たれ、（いちばん重要なことに）コードが読みやすく保たれます。

== 安全な時点

各フレームインフライトには fence があります。新しいフレームの開始時にその fence を待ちます。シグナルしたら、このフレームのディスクリプタセットを参照していた作業は GPU が完了しています。これが、このフレームのセットを更新する安全な瞬間です。

なぜ重要か: 使用中のセットを更新すると、無効な書き込みや、いわゆる「update-after-bind」違反につながります。それらを意図的に選び、パイプラインをそれに合わせて組まない限りです。安全な時点パターンは移植性が高く明確です。

== 更新するもの

* ストリーミングが完了したマテリアルテクスチャ。
* 平面反射用の反射テクスチャバインディング（binding 10）。
* リサイズ時の Forward+ 用フレームごとのバッファ（タイルヘッダ／インデックス、ライト SSBO）。

Ray Query モードでは、大きなテクスチャテーブル（固定サイズのサンプラー配列）も更新し、パイプラインを作り直さずに新しくストリームされたテクスチャが見えるようにします。

安全な時点では現在フレームのセットだけを更新し、他フレームは自分の番で更新します。これにより、交互フレームでテクスチャが違って見えるクロスフレームの「フリップフロップ」を防ぎます。

== Descriptor Indexing: いつ使うか

Descriptor Indexing は、可変サイズ配列や update-after-bind などの機能を開きます。強力ですが、複雑さを同期と寿命ルールへ移します。このサンプルでは明確さを重視します:

* ディスクリプタレイアウトは単純で安定に保つ。
* コマンドバッファがまだ保留されているかもしれないあいだではなく、安全な時点で更新する。

Descriptor Indexing 機能を使うときは、特定の理由だけです: 大きく非一様にインデックスされるディスクリプタ配列（例: Ray Query のテクスチャテーブル）。その場合、正しさは次に依存します:

* GPU が要求する Descriptor Indexing のフィーチャビットを有効化する
* （サポート時）正しいバインディングフラグでバインディングをマークする
* 非同期ストリーミングをまたいで古い Vulkan image／sampler ハンドルをキャッシュしない

プロジェクトが本当に動的なディスクリプタ配列や頻繁なフレーム途中更新を必要とするなら、Descriptor Indexing は正しい道具になり得ます — 新しい不変条件を丁寧に文書化してください。

== 実践的なヒント

* ディスクリプタ更新を集中させる; フレーム全体に書き込みを散らさない。
* プレースホルダーにはデフォルトテクスチャを使い、一度だけ差し替える — 本物とデフォルトのあいだで跳ね返さない。
* 教育向けサンプルでは combined image sampler を優先する; 柔軟性が必要なときだけ image／sampler を分ける。

== 今後のアイデア

より高度なディスクリプタパターンを探りたい場合:

* （ターゲット向けにデバイスサポートが十分なら）テクスチャテーブルを可変ディスクリプタ数へ移す。
* 多くのテクスチャでサンプラーを共有するため、分離 image／sampler ディスクリプタを使う。
* 寿命ルールを検証するため、テクスチャを急速にストリームする「ディスクリプタストレステスト」モード（開発専用）を追加する。
