= 同期とストリーミング

現代の Vulkan は、アセットがストリームインしているあいだも GPU を忙しく保つ強力な手段を提供します。このエンジンは、ストールとフリッカーを避けるために、バックグラウンドアップローダー、専用の transfer キュー、Synchronization 2 を使います。動いている部品と、それらがどう噛み合うかを見ていきましょう。

== 考え方

- ファイル I/O と staging はレンダースレッドの外で行う。
- GPU コピーとレイアウト遷移は graphics キューではなく transfer キューで動かす。
- timeline semaphore により、リソースごとの fence を細かく管理せずに、graphics が「最新の完了したアップロード」を待てる。
- ディスクリプタは安全な時点（インフライトフレームの fence を待った直後）だけ更新し、GPU がまだ使っているセットへ書き込まない。

これで、大きなテクスチャがストリームインしているあいだも、フレームループはシンプルで応答性を保てます。

== バックグラウンドアップローダー

テクスチャジョブ（トランスコード／IO → staging バッファ → デバイスイメージ）をキューに入れます。専用スレッドが:

1. 保留中のコピーを transfer キュー上のコマンドバッファにバッチする。
2. Synchronization 2 を使い、`TRANSFER_DST_OPTIMAL` から `SHADER_READ_ONLY_OPTIMAL` へのレイアウト遷移を記録する。
3. 一度だけサブミットし、単調増加する timeline 値をシグナルする。
4. どのテクスチャが「サンプル可能」になったかをレンダラーに通知する。

render のサブミットは最新のアップロード timeline 値への wait を含むので、描画時点ではテクスチャが利用可能です。

== ディスクリプタ更新の安全な時点

Vulkan は、使用中のディスクリプタセットの変更を許しません。エンジンは代わりに次のようにします:

* 各フレームの開始時、そのフレームインフライトに対応する fence を待つ。
* これでこのフレームのディスクリプタセットを更新しても安全（使用中ではない）。
* この時点で、アップロード済みテクスチャの view／sampler でイメージバインディングを更新する。

結果として、テクスチャの「フリップフロップ」やフリッカーは起きません。本物のテクスチャがプレースホルダーを置き換えたら、そのままです。

== 実践での Synchronization 2

アップロード経路は、明確で最小のスコープの `vkCmdPipelineBarrier2` を使います:

* staging → イメージコピー: 宛先イメージを `TRANSFER_DST_OPTIMAL` にする。
* 最終コピー後: `SHADER_READ_ONLY_OPTIMAL` へ遷移（src stage = `eTransfer`、dst stage = `eFragmentShader`）。
* ownership transfer は、transfer と graphics が異なるファミリを使う場合のみ（ほとんどのデスクトップドライバはファミリを共有）。

graphics 側では、アタッチメントのレイアウト遷移を dynamic render pass インスタンスの外に保ち、可読性のため同様に `vkCmdPipelineBarrier2` を使います。

== 典型的なテクスチャの旅路

1. ファイルパス付きでジョブをエンキュー。
2. バックグラウンドスレッド: staging へロード／トランスコード、デバイスイメージを割り当て、コピーを記録。
3. transfer キューへサブミット; timeline をシグナル。
4. レンダラーの次フレームが始まり、フレームごとの fence が解除される。
5. このフレームのディスクリプタがアップロード済みイメージを指すよう更新（安全な時点）。
6. 描画: フラグメントシェーダーがストールなしで新しいテクスチャをサンプル。

== ヒントと落とし穴

* ディスクリプタ更新は安全な時点に保つ。使用中セットの更新を避ける。
* 一括コピーには transfer キューを使い、graphics キューは描画に集中させる。
* 明確さのために Synchronization 2 を優先する（stage／access の組が明示的で、遷移が目立つ）。
* アップロードをバッチする: サブミットが少ないほど CPU オーバーヘッドは低い。

これだけで、ストリーミングはプレイヤーにとって「見えない」ものになり、保守もしやすくなります。

== 今後のアイデア

ストリーミングをさらに進めたい場合:

* ミップレベル単位でストリームする（低いミップを先に）、その後バックグラウンドで精細化する。
* 開発ビルドフラグの裏に、小さなストリーミング HUD（キュー中バイト、アップロード済みバイト、準備完了テクスチャ）を追加する。
* リソースごとの優先度（カメラ距離、重要度タグ）を追加し、いちばん目立つアセットが先に届くようにする。
* 急速な入れ替わりの下でディスクリプタ寿命ルールを検証するため、テクスチャの「ホットリロード」を追加する。
