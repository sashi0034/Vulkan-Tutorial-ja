= 視錐台カリングと距離ベース LOD

カリングは、GPU をカメラが見えるものに集中させるいちばん単純な方法です。このエンジンでは意図的に実用的に保っています: CPU の視錐台テストに加え、数ピクセルしか寄与しないオブジェクトを飛ばす小さな「距離／サイズ LOD」です。

* メッシュごとの AABB に対する CPU 視錐台カリング
* 非常に小さなオブジェクトを飛ばす小さな距離／サイズ LOD（投影サイズしきい値）

== やっていること

1. 各フレーム一度、`proj * view` からカメラの視錐台平面を抽出する。
2. 各メッシュインスタンスについて、ローカル AABB をワールド空間へ変換し、平面に対してテストする。
3. 有効なら投影ピクセルサイズを推定し、しきい値未満のオブジェクトを飛ばす（不透明と透過で別しきい値）。

== コードの参照先

* 平面抽出と AABB テスト:
** `renderer_rendering.cpp`（ファイル先頭付近のヘルパー）
* フレームごとのカリング適用:
** `renderer_rendering.cpp`（レンダーリスト構築とパスごとのフィルタリング）
* UI コントロール:
** `renderer_rendering.cpp` の ImGui パネル — 「Frustum culling」「Distance LOD」、パスごとのしきい値

== こう組んでいる理由

* AABB の変換とテストは安い; CPU で行えば明らかに見えないドローを送らずに済む。
* 投影サイズのカットオフは、大きなシーン向けフル LOD 系の実用的な代替になる。

== チューニングのヒント

* 保守的（小さめのしきい値）から始め、移動中にポップインが気にならなくなるまで上げる。
* 透過オブジェクトは、極小サイズでのブレンディングアーティファクトのため、やや高めのしきい値が必要なことが多い。

== 今後のアイデア

さらに進めたい場合:

* マテリアルごと／レイヤーごとのカリングルールを追加する（例: 看板を長く読めるように保つ）。
* 非常に大きなシーン向けに階層カリング（AABB の BVH）を追加する。
* パイプラインが「読みやすいサンプル」規模を超えたら、GPU オクルージョンカリング（HZB）を追加する。
* 投影サイズヒューリスティックを本物のメッシュ LOD（またはメッシュレット）に置き換える。
