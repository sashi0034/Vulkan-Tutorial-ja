= パフォーマンス監査と最適化

低レベルアーキテクチャとメモリモデルから、GPU 駆動パイプラインと非同期コンピュートまで、高度な Vulkan コンピュートトピックを幅広くカバーしてきました。しかしすべての開発者が最終的に直面する問いが一つあります。「これは可能な限り速いか？」この問いに答えるのは、推測や直感ではなく、厳格で方法的な **パフォーマンス監査** のアプローチです。

GPU コンピュートの世界では、「速い」カーネルも多くのものに足を引っ張られ得ます。メモリ待ち（**メモリバウンド**）かもしれないし、複雑な算術に圧倒されている（**コンピュートバウンド**）かもしれないし、「発散」——**サブグループ**（または **ワープ／ウェーブフロント**）内の異なる呼び出しが異なる実行パスを取ることを強いられ、ハードウェアが作業を直列化する——に苦しんでいるかもしれません。

最適化は「賢い」コードを書くことだけではありません。**ボトルネック**を理解することです。カーネルがメモリバウンドなら、算術演算を足しても遅くなりませんが、速くもなりません。逆にコンピュートバウンドなら、メモリアクセスパターンを最適化しても目立った利益は得られないかもしれません。

== 素朴な最適化を超えて

Vulkan のような大規模並列環境での最適化を語るとき、導くための標準的なメトリクスとモデルのセットが必要です。本章では次を紹介します。

* **Roofline モデル**: カーネルが **VRAM**（Video Random Access Memory）のピーク帯域に制限されているか、**ALU**（Arithmetic Logic Unit）のピークスループットに制限されているかを可視化できる基本的な分析ツール。
* **命令スループット分析**: 個々の **ISA**（Instruction Set Architecture）コマンドのコストを理解し、倍精度 float や複雑な超越関数のような「重い」演算を特定する方法。
* **発散監査**: **SIMD**（Single Instruction, Multiple Data）実行が崩れ、他が働いている間レーンがアイドルする場所を特定する手法。

高レベルな Slang や GLSL コードを見ることを超え、ハードウェアが実際に何を見ているかを考え始めます。これには **CU**（Compute Unit）や **SM**（Streaming Multiprocessor）の **占有率** の理解と、メモリレイテンシによる **パイプラインストール** の最小化が含まれます。

本章の終わりまでに、「動くようにする」から「飛ばすようにする」へ進む手法を身につけます。

== 章のロードマップ

1. **命令スループット分析**: Roofline モデルを使い、コンピュートバウンド対メモリバウンドなカーネルを特定することを学ぶ。
2. **発散監査**: 発散する分岐論理を可視化しリファクタリングする技法。
