= 発散監査: 分岐発散の特定とリファクタリング

GPU コンピュートで最も陰湿な性能キラーの一つが **分岐発散（Branch Divergence）** です。理由を理解するには、GPU がロックステップで同じ命令を実行する呼び出しのグループ（ウェーブフロントまたはワープ）上で動作することを思い出す必要があります。コードに分岐——`if-else` 文など——があり、サブグループ内の一部の呼び出しが `if` パスを、他が `else` パスを取ると、ハードウェアはそれらのパスを **直列化** せざるを得ません。

ハードウェアは関連するすべての呼び出し向けに `if` パスを実行し（他をマスクアウトしながら）、その後残りの呼び出し向けに `else` パスを実行します（最初のグループをマスクアウトしながら）。この間、非アクティブ呼び出しの ALU は実質的にアイドルし、GPU のスループットを実質的に半分に切っています。

== 発散の特定

**発散監査** は、これらの「発散」分岐がどこで起きているかを特定し、影響を最小化するためにコードをリファクタリングする体系的なプロセスです。

=== ツールベースの特定
プロファイラで **Active Lane Ratio** や **Instruction Execution Efficiency** のようなメトリクスを探してください。低い比率は、サブグループ内の多くのレーンが発散制御フローによってアイドルにされていることを示します。

たとえば NVIDIA Nsight では、「Warp Execution Efficiency」メトリクスを見るかもしれません。一貫して 50% 未満なら、重大な発散問題がある可能性が高いです。モバイル作業向けに導入したモバイルプロファイラ（Arm Performance Studio、Qualcomm Snapdragon Profiler）は、「warp/wave occupancy」や「thread divergence」のような名前で同じ考えを公開します——そして発散はモバイルで*特に*高くつきます。アイドル中でも無駄なレーンが電力を消費するためです。デバイス側プロファイリングワークフローは xref:Advanced_Vulkan_Compute/12_Mobile_and_Embedded_Compute/03_mobile_optimization.adoc[モバイル GPU 向け最適化] を参照してください。

=== シェーダ内可視化
**サブグループ演算**（第 4 章）を使い、シェーダ内で直接発散を手動検査することもできます。`WaveActiveBallot()` を使うことで、特定のパスを取っている呼び出しのビットマスクを生成できます。

[source,slang]
----
// シェーダ内で発散を可視化
bool local_test = data[globalID.x] > threshold;

// Ballot はサブグループ内のどのレーンが 'true' かを正確に教える
uint4 lane_mask = WaveActiveBallot(local_test);

// 一部のレーンだけが true なら、発散している！
uint active_lanes = countbits(lane_mask.x) + countbits(lane_mask.y) +
                    countbits(lane_mask.z) + countbits(lane_mask.w);
----

== リファクタリング戦略

発散分岐を特定したら、リファクタリングする方法はいくつかあります。

=== 戦略 1: サブグループレベル分岐
分岐がサブグループ内のすべての呼び出しで同一に評価できるなら、ハードウェアはペナルティなしで実行できます。これはしばしば「ユニフォーム分岐」と呼ばれます。

[source,slang]
----
// リファクタリングされた、サブグループ意識の分岐
bool local_test = data[globalID.x] > threshold;

if (WaveActiveAllTrue(local_test)) {
    // 高速パス: 全員が同じ仕事をしている！
    do_complex_work_fast();
} else if (WaveActiveAnyTrue(local_test)) {
    // 低速パス: 一部だけが仕事をしているが、必要なときだけ入る
    do_complex_work_slow();
}
----

=== 戦略 2: 制御フローをデータフローで置き換える
より高度な技法は **制御フローをデータフローで置き換える** ことです。2 つの計算の間を選ぶために `if` を使う代わりに、両方を実行し、数学的トリックで結果を選べます。これにより実行パイプラインが「飽和」状態に保たれ、分岐の直列化ペナルティを避けられます。

`lerp()`、`clamp()`、`step()` のような関数がここでの最良の友です。多くの場合、数回の余分な算術演算を行う方が、発散分岐のコストより速いです。

=== 戦略 3: ワークソート
発散が異なる種類のデータの処理によって起きている場合（例: 一部のレイが複雑なマテリアルに当たり、他が単純なものに当たるレイトレーサ）、**ソートパス** を使って類似のワークロードをグループ化できます。サブグループ内のすべての呼び出しが同じ種類のデータを処理することを保証することで、ソートのコストで発散を完全に排除できます。

定期的な発散監査を行うことで、コンピュートカーネル内の「隠れた」コストを特定し、より効率的で SIMD フレンドリーなパターンへリファクタリングできます。これが「ただ動く」コードと、GPU アーキテクチャを真にマスターするコードの違いです。
