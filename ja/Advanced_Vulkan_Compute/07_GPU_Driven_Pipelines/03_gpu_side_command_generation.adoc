= GPU 側コマンド生成: 自律実行

== GPU 上でコマンドチェーンを構築する

典型的な Vulkan アプリケーションでは、CPU が一連のコマンドを記録します。`vkCmdBindPipeline`、`vkCmdBindDescriptorSets`、`vkCmdDispatch` などです。これらのコマンドはコマンドバッファへ「焼き付け」られます。

**GPU 側コマンド生成**（しばしばエンジン内蔵のインダイレクトバッファと BDA を活用）では、さらに一歩進められます。CPU がコマンドの全シーケンスを決める代わりに、GPU が実行したい「コマンド」のリストを構築できます。

これは **GPU 駆動パイプライン** の重要な構成要素です。GPU は自身の状態を分析し、次を決定できます。

1.  物理セットを更新するコンピュートシェーダをディスパッチする。
2.  BVH を構築する別のコンピュートシェーダをディスパッチする。
3.  更新されたシーンをレンダリングする一連のドローコールを生成する。

== 仕組み: バッファファーストアプローチ

GPU 駆動パイプラインでは、CPU は典型的に「マスター」コンピュートディスパッチを記録します。このディスパッチがオブジェクトやタスクを反復し、一連の **インダイレクトコマンドバッファ** へ書き込みます。

*   **ディスパッチインダイレクトバッファ**: 将来のコンピュートシェーダ向けの `x, y, z` パラメータを格納。
*   **ドローインダイレクトバッファ**: 将来のレンダリング向けの `vertexCount`、`instanceCount` などを格納。
*   **リソースバッファ**: それらのシェーダとドローが必要とする生ポインタ（Buffer Device Address）を格納。

== なぜ性能に重要か

GPU が自身のコマンドを生成すると、CPU は完全にループから外れます。次はもはや不要です。

*   **コマンドバッファの再記録**: 毎フレームの CPU オーバーヘッドなし。
*   **CPU 側カリング**: 「往復」データ交換なし。
*   **同期オーバーヘッド**: 同期は GPU のコマンドストリーム内だけで起きる。

== 自律の力

この自律モデルにより **シングルパスレンダリング**（単一 GPU パスでカリングとドローを行う）が可能になります。何を描くかを知るために CPU が GPU のカリング結果を待つ代わりに、GPU は同じコマンドストリーム内でオブジェクトをカリングし、描画できます。

これは多くの現代のハイエンドレンダリングエンジンの標準アーキテクチャです。「カリングとドロー」のコストが CPU 性能に依存しないため、何百万のオブジェクトへスケールします。
