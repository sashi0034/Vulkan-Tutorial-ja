= Multi-Draw Indirect（MDI）: コンピュートからグラフィックスへ橋渡し

== ギャップを埋める

本章全体で、コンピュートシェーダをより自律的にする方法に焦点を当ててきました。しかしほとんどのグラフィックスアプリケーションの最終目標は……そう、グラフィックスです。コンピュートベースのカリングや分析の結果をドローコールへ変える方法が必要です。

**Multi-Draw Indirect（MDI）** は、コンピュートとグラフィックスパイプラインの間の究極の橋です。単一の Vulkan コマンドが任意数のドローコールを実行でき、各ドローコールのパラメータは GPU 側バッファから来ます。

== 仕組み: MDI パイプライン

1.  **カリングフェーズ（コンピュート）**: コンピュートシェーダがシーン（例: 何百万のオブジェクト）を分析し、どれが見えるかを決める。各可視オブジェクトの `vertexCount`、`instanceCount` などを大きな **インダイレクトバッファ** へ書く。
2.  **カウントバッファ**: コンピュートシェーダは何個のオブジェクトが見えたかのアトミックカウンタも保ち、このカウントを別の **ドローカウントバッファ** へ書く。
3.  **ドロー（グラフィックス）**: CPU が `vkCmdDrawIndexedIndirectCount` を呼ぶ。この単一コマンドが、GPU に自身のカウントとパラメータを読みオブジェクトを描くよう伝える。

[source,cpp]
----
// MDI バッファ内のデータのレイアウト（vk::DrawIndexedIndirectCommand に一致）
struct IndirectDrawCommand {
    uint32_t indexCount;
    uint32_t instanceCount;
    uint32_t firstIndex;
    int32_t  vertexOffset;
    uint32_t firstInstance;
};
----

== なぜ MDI が GPU 駆動レンダリングに不可欠か

MDI がなければ、CPU は可視性カウントを GPU からリードバックし、可視オブジェクトごとに別の `vkCmdDraw` を記録しなければなりません。10,000 可視オブジェクトのシーンでは、毎フレーム 10,000 回の CPU 呼び出しと 10,000 回のコマンド記録になります。

MDI があれば、それらの 10,000 オブジェクトは**1 コマンド**でレンダリングされます。これが現代エンジンが巨大な「カリングファースト」アーキテクチャを扱える方法です。

== MDI のベストプラクティス

*   **最大ドローカウント**: エラー時に GPU がバッファを過剰読みしないよう、`vkCmdDrawIndexedIndirectCount` 呼び出しで常に妥当な最大ドローカウントを指定する。
*   **バッファ整列**: インダイレクトバッファがハードウェアの正しい整列とストライド要件に従うことを保証する。
*   **BDA と組み合わせる**: **Buffer Device Address**（第 7 章）を使い、オブジェクト固有データ（マテリアルや変換など）をシェーダへ渡し、伝統的なディスクリプタセットを迂回する。

== まとめ: 未来は GPU 駆動

**インダイレクトディスパッチ**、**GPU 側コマンド生成**、**Multi-Draw Indirect** を習得することで、伝統的な「CPU 主導」パイプラインから現代の「GPU 駆動」アーキテクチャへ移りました。アプリケーションはよりスケーラブルで、低レイテンシで、より効率的になりました。

次の章では、**非同期コンピュートのオーケストレーション** を使い、これらの重いコンピュートワークロードをグラフィックスレンダリングとどう協調させるかを見ます。
