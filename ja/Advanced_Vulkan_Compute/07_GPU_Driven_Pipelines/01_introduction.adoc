= インダイレクトディスパッチと GPU 駆動パイプライン

== はじめに

伝統的な Vulkan アプリケーションでは、CPU がオーケストラの「指揮者」です。何を描くか、何本のスレッドをディスパッチするか、どのリソースをバインドするかを決めます。GPU は単に、CPU が与えたコマンドを実行する「演奏者」です。

しかしシーンが複雑になり——何百万もの動的オブジェクトと複雑な物理があると——CPU は追いつけなくなります。どのオブジェクトが見えるかを CPU が計算し、何千ものコマンドバッファを記録するオーバーヘッドが主ボトルネックになります。

本章では、GPU が指揮者の役割を引き継ぐ **GPU 駆動パイプライン** を探ります。

== 静的ディスパッチを超えて

静的ディスパッチ（`vkCmdDispatch`）は、CPU が実行するワークグループ数を正確に知る必要があります。オブジェクトカリングのようなことをしている場合、GPU が作業を終えるまで、CPU は何個のオブジェクトがカリングを通過するかを知りません。

**インダイレクトディスパッチ**（`vkCmdDispatchIndirect`）では、CPU がディスパッチサイズを提供しません。代わりに、ディスパッチパラメータを含む **Vulkan バッファ** を提供します。GPU 自身がこのバッファに書き込み、実質的にどれだけの仕事が必要かを決められます。

== 自律的な GPU

GPU 駆動パイプラインは、次のような機能でこれをさらに進めます。

1.  **GPU 側コマンド生成**: 現代エンジン機能を使い、GPU 上でコマンドチェーン全体を構築し、自身の実行パスを「決定」できるようにする。
2.  **Multi-Draw Indirect（MDI）**: 単一コンピュートディスパッチが何千ものドローコールを生成し、CPU 側ループなしでシーン全体をレンダリングできる機能。
3.  **可変サイズワークロード**: パーティクルシステムから高忠実度カリングまで、CPU 側介入なしですべてを扱う。

== なぜ重要か

「意思決定」を GPU へ移すことで、次ができます。

*   **CPU ボトルネックの排除**: AI、ゲームロジック、その他のタスクのために CPU を解放する。
*   **レイテンシの最小化**: GPU のコンピュート分析とその後のレンダリングの間の往復遅延を排除する。
*   **数百万へのスケール**: 伝統的な CPU バウンドパイプラインでは不可能なシーン複雑度を扱う。

本章では、これらの自律パイプラインの構築を学びます。まずは基本構成要素——**インダイレクトディスパッチ**——から。
