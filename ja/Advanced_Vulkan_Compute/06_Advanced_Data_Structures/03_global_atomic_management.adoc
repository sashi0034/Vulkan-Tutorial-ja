= グローバルアトミック管理: ロックフリーリストとキュー

== なぜ 64 ビットアトミックか？

初期の Vulkan では、アトミックは 32 ビット整数に制限されていました。単純なカウンタには有用でしたが、ポインタや複雑なデータ構造を扱うには足りませんでした。**Vulkan 1.4** では 64 ビットアトミックがコア機能となり、真にロックフリーなデータ構造を構築する扉が開きます。

64 ビットアトミックは、値と「タグ」（**ABA 問題**——値が A から B へ変わり再び A に戻り、スレッドが変化しなかったと勘違いすること——を避けるため）の両方、または完全な 64 ビット **Buffer Device Address**（ポインタ）を格納できます。

== 「ロックフリー」は実際に何を意味するか？

伝統的な CPU プログラミングでは、2 スレッドが同じデータを更新したいとき、**ミューテックス**（相互排他）でデータを「ロック」し、更新し、「アンロック」します。GPU ではこれは災難です。何千ものスレッドがロックステップ（**SIMT**——Single Instruction, Multiple Threads）で動くため、1 スレッドがロックを取り他が待つと、GPU 全体が停止し得ます——**デッドロック** と呼ばれる状況です。

**ロックフリー** アルゴリズムは、システム内の少なくとも 1 スレッドが有限ステップで進捗することを保証するものです。ロックの代わりに **アトミック演算** を使います。これらは「Read-Modify-Write」シーケンスを単一の割り込み不能ステップで行う特別なハードウェア命令です。

リンクリストの例では、`InterlockedExchange`（GLSL では `atomicExchange`）を使います。この命令は言います。「この新しい値を取り、メモリに入れ、以前あったものを返せ——その間に他のスレッドがそのメモリ位置に触れないように。」

すべてのスレッドが「交換」を成功裏に完了し一意な `oldHead` を得るため、どのスレッドも他を待つ必要がありません。すべてが同時に進捗します。これが GPU 上でロックフリーであることの本質です。

== ロックフリーリンクリストの構築

リンクリストは多くの GPU アルゴリズム、特に **Order-Independent Transparency（OIT）** の基盤です。ピクセルごとのリンクリストでは、フレームバッファ内のすべてのピクセルがそのピクセルに当たった透明フラグメントのリストへの「ヘッド」ポインタを格納します。

=== GPU リンクリストの解剖

GPU 常駐リンクリストは主に 3 つの構成要素からなります。

1.  **ヘッドバッファ**: 各ピクセルの最初のノードのインデックスを格納する 2D テクスチャまたはバッファ（画面解像度に一致）。「null」値（例: `0xFFFFFFFF`）で初期化されます。
2.  **ノードプール**: すべてのフラグメントの実際のデータを格納する大きな線形バッファ。
3.  **アトミックカウンタ**: プールからノードを「割り当て」る単一整数。

[source,slang]
----
struct Node {
    float4 color;      // フラグメント色
    float  depth;      // フラグメント深度
    uint   nextIdx;    // プール内の次のノードのインデックス
};

RWStructuredBuffer<uint> headBuffer; // サイズ: width * height
RWStructuredBuffer<Node> nodePool;   // サイズ: 最大総フラグメント数
RWStructuredBuffer<uint> counter;    // サイズ: 1
----

フラグメントが処理されると:

1.  スレッドが `counter` をアトミックにインクリメントし、一意な `newNodeIdx` を得る。
2.  スレッドがピクセル位置の `headBuffer` に対して `InterlockedExchange` を使う。`newNodeIdx` を書き、`oldHead` を受け取る。
3.  スレッドがデータと `oldHead`（`nextIdx` として）を `nodePool[newNodeIdx]` へ書く。

この構造により、グローバルロックを必要とせずに、何千ものフラグメントを何百万もの異なるリストへ同時に追加できます。

=== Exchange を超えて: Compare-and-Swap（CAS）

`InterlockedExchange` は単純なリストに優れますが、より複雑な構造（スレッドセーフキューなど）はしばしば **Compare-and-Swap（CAS）**——Slang では `InterlockedCompareExchange`——を必要とします。

CAS はこう働きます。「現在の値が私の『期待』値と一致する場合だけ、このメモリを更新せよ。」一致しない場合、別のスレッドが先にデータを変えたことを意味します。その場合、スレッドは新しい値で操作を「リトライ」しなければなりません。この「成功するまでループ」パターンは高度なロックフリープログラミングで一般的で、実際の競合があるときだけ待つため伝統的なロックよりはるかに効率的で、ハードウェアスケジューラを離れることもありません。

=== GLSL: atomicAdd と 64 ビットアトミック

GLSL では `atomicAdd` と `atomicExchange` 関数を使います。64 ビットアトミックには `GL_EXT_shader_atomic_int64` 拡張を有効にしなければなりません。

[source,glsl]
----
#extension GL_EXT_shader_atomic_int64 : enable

layout(binding = 0) buffer HeadBuffer { uint64_t heads[]; };
layout(binding = 1) buffer Counter    { uint64_t count; };

void addNode(uint pixelIdx, Node newNode) {
    // グローバルカウンタへの 64 ビットアトミック加算
    uint64_t newNodeIdx = atomicAdd(count, 1UL);

    // ヘッドポインタ更新のための 64 ビットアトミック交換
    uint64_t oldHead = atomicExchange(heads[pixelIdx], newNodeIdx);

    // ... ノードと次ポインタを更新
}
----

Slang が異なるビット幅で動くより統一された `InterlockedAdd` を提供するのに対し、GLSL は拡張と型（例: 64 ビットリテラルに `1UL`）について明示的である必要があります。

上の例は単純さのため 32 ビットインデックスを使っていますが、64 ビットアトミックにより、生ポインタを使って異なるバッファや異なるメモリ型をまたいでも同じことができます。

== ワークキューの構築

**ワークキュー** は、GPU が実行する必要のあるタスクのリストです。**GPU 駆動パイプライン** では、あるコンピュートディスパッチがカリングすべきオブジェクトのリストを生成し、別のディスパッチがそのリストを処理するかもしれません。

=== ワークキューの解剖

ワークキューは本質的に **プロデューサ–コンシューマ** 構造です。GPU 上では、典型的にはアトミックカウンタ付きの **線形バッファ**、または持続的ワークロード向けの **リングバッファ** として実装されます。

[source,slang]
----
struct Task {
    uint objectID;
    uint drawCommandIdx;
};

struct WorkQueue {
    RWStructuredBuffer<Task> data;    // 保留タスクのストレージ
    RWStructuredBuffer<uint> counter; // 現在キュー内のタスク数
};

void pushTask(WorkQueue queue, Task myTask) {
    uint slot;
    // 一意なスロットを請求するためのアトミックインクリメント
    InterlockedAdd(queue.counter[0], 1, slot);

    // バッファオーバーフローをチェック！
    if (slot < MAX_QUEUE_SIZE) {
        queue.data[slot] = myTask;
    }
}
----

グローバルワークキューを使うことで、CPU に戻ることなく可変サイズのワークロードを扱えます。

== サブグループによるアトミックの最適化

アトミックは GPU 全体で調整が必要なため、比較的高価です。何千ものスレッドが同じカウンタに加算しようとすると、ハードウェアがそれらを直列化し、巨大な性能低下につながります。

第 4 章で議論したように、**サブグループ演算** を使ってこれらのアトミックを **合体**（複数操作を 1 つにまとめる）できます。すべてのスレッドが `InterlockedAdd` を呼ぶ代わりに、サブグループ内のスレッドが **サブグループリダクション** で加算すべき総量を計算し、サブグループ全体のために単一アトミック加算を行う 1 つの「リーダー」スレッドを選び、結果のベースインデックスを他のスレッドへ配布できます。

この単純な最適化はグローバルアトミックのスループットを 32 倍または 64 倍改善でき、最も要求の厳しいリアルタイムアプリケーションでも複雑なデータ構造を実用的にします。
