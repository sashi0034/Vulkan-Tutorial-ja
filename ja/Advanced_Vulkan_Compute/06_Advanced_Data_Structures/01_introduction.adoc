= GPU 上の高度なデータ構造

== はじめに

本チュートリアルの前半では、コンピュートディスパッチの実行方法とメモリ管理に焦点を当てました。主に線形配列（バッファ）や 2D/3D グリッド（テクスチャ）のような単純なデータ構造を扱いました。これらは GPU プログラミングの定番ですが、多くの実世界の問題はより複雑な組織化を必要とします。

本章では、「データ並列」配列から「GPU 常駐」データ構造へ移ります。**木（BVH/Octree）**（BVH はバウンディングボックス、Octree は 3D 空間分割）、**リンクリスト**、**ワークキュー** のような複雑な構造を、デバイス上だけで構築・走査する方法を探ります。

== データ構造を GPU へ移す

伝統的に、複雑なデータ構造は CPU で構築され、GPU が読むために配列へ「平坦化」されました。これは動きますが、巨大なボトルネックを生みます。構造への更新のたびに CPU-GPU 往復が必要になるのです。

現代の Vulkan コンピュートでは、これらの構造の*構築*と*管理*を GPU へ移すことで、このボトルネックを排除できます。これを可能にするのは 3 つの主要技術です。

1.  **64 ビットアトミック**: GPU 全体にわたるグローバルカウンタとポインタへのスレッドセーフな更新を可能にします。詳細を探る **ロックフリー** データ構造に不可欠です。
2.  **Buffer Device Address**: 複雑なディスクリプタセットから離れ、グラフ状構造を構築するための生のポインタのような柔軟性へ移る。
3.  **サブグループ演算**: 第 4 章で学んだウェーブ意識の論理を使い、複数の操作を単一アトミック更新へ **合体（coalescing）** することで、これらの構造をより速く構築する。

== なぜ重要か

GPU 常駐データ構造は、現代の高性能レンダリングとシミュレーションの基盤です。

*   **レイトレーシング**: Bounding Volume Hierarchy（BVH）は、レイがどの三角形に当たる可能性があるかを素早く見つけるために使われます。
*   **物理とロボティクス**: Octree やグリッドベースハッシュのような空間分割構造が衝突検出に使われます。
*   **Order-Independent Transparency（OIT）**: 透明オブジェクトを CPU で事前ソートせずにレンダリングする技法。ピクセルごとのリンクリストが、GPU 上で透明フラグメントの格納とソートに使われます。
*   **GPU 駆動パイプライン**: ワークキューにより GPU が自身の仕事を生成でき、次の章で探ります。

本章の終わりまでに、GPU を「愚かな配列プロセッサ」として扱うのをやめ、自律的で複雑なデータ管理のプラットフォームとして扱い始められるようになります。
