= GPU 常駐の木: BVH と Octree

== なぜ木か？

木は空間分割の基本構造です。レイがどの三角形に当たるか、どのオブジェクトがパーティクルの近くにあるかを探すとき、すべてのオブジェクトを線形探索するのは遅すぎます。

木は CPU 上では再帰的に構築しやすい一方、SIMD 実行モデルと共有ヒープの欠如のため GPU 上での構築は悪名高く難しいです。しかし正しいアプローチなら、GPU は CPU よりはるかに速く木を構築できます。

== Bounding Volume Hierarchy（BVH）

BVH は、各ノードがすべての子を含むバウンディングボックスを表す木です。あらゆるレイトレーシングエンジンの心臓部です。

伝統的に BVH は線形配列へ「平坦化」され、子リンクは配列インデックスとして表されます。現代ハードウェアでは、**Radix Tree**（空間最適化された木）や **Morton 符号** を使ってこれらの木を構築できます。

=== BVH ノードの解剖

典型的な **内部ノード** は空間境界と子へのポインタを格納しなければなりません。**葉ノード** は境界と、含むプリミティブ（三角形など）のリストを格納します。

スペースを節約しキャッシュ局所性を改善するため、これらはしばしば単一構造体にパックされます。

[source,slang]
----
struct BVHNode {
    float3 min;           // AABB Min: バウンディングボックスの最小コーナー
    uint   childOrLeaf;   // 子へのインデックス、または葉内の最初の三角形
    float3 max;           // AABB Max: バウンディングボックスの最大コーナー
    uint   count;         // 子の数（内部）または三角形数（葉）
};
----

このレイアウトでは:

*   **AABB（Axis-Aligned Bounding Box）**: 2 点（`min` と `max`）で表されます。レイがボックスに当たるかの判定が GPU 上で極めて速いため、最も一般的な体積です。
*   **childOrLeaf**: 木の次のレベルを指す単一の 32 ビット整数。`count > 0` なら葉ノード。`count == 0` なら内部ノードで、`childOrLeaf` は `nodePool` バッファ内の最初の子のインデックスです。

=== 階層の構築: Morton 符号

1.  **Morton 符号化**: 3D 位置を 1D の **空間充填曲線**（**Z 曲線**）へマップします。X、Y、Z 座標のビットを**インターリーブ**することで行われます。たとえば X が `010`、Y が `101` なら、Morton 符号は `011001`（X[0] から 0、Y[0] から 1、X[1] から 1、Y[1] から 0 など）になります。このマッピングには、3D で近い点が 1D でも近くなるという魔法のような性質があり、3D 階層をソート済みリストへ効果的に「平坦化」します。
2.  **基数ソート**: Morton 符号を得た後、高性能 GPU **基数ソート（radix sort）** を使います。基数ソートはビット単位で数をソートする非比較ソートアルゴリズムです。複雑な比較を必要としないため、SIMD ハードウェアで極めて効率的です。Morton 符号のソートが、オブジェクトを空間的に「グループ化」する実際の作業です。
3.  **階層構築**: ソート後、コンピュートディスパッチ内の各スレッドが木の一部分を構築します。隣接する 2 つの Morton 符号が最初に異なるビットを見ることで、スレッドは木のノードをどこで分割すべきかを判定できます。これは **Linear BVH**（LBVH）として知られ、グローバルロックなしで木全体を並列に構築できます。

このプロセスは「恥ずかしがり屋な並列（embarrassingly parallel）」であり、数百万の三角形の BVH をわずか数ミリ秒で構築できます。

== Octree

Octree は各ノードがちょうど 8 つの子を持ち、空間を **オクタント**（立方体の角のような 3D 空間の自然な 8 分割）へ分割する木です。流体シミュレーションや、空間のどの部分が占有されているかを素早く知る必要がある **ボクセルベースレンダリング** に最適な構造です。

=== Octree ノードの解剖

柔軟な子の数を持つ BVH ノードと異なり、純粋な Octree ノードは常に、8 つの小さな立方体へ分割できる完全な立方体を表します。

[source,slang]
----
struct OctreeNode {
    uint childIndices[8]; // 8 つのオクタントへのインデックス
    float3 center;        // ノードの立方体の中心
    float  extent;        // ノードの立方体の半幅
    uint   payload;       // ユーザーデータ（例: 色、密度、マテリアル）
};
----

ただし、ノードあたり 8 つの 32 ビットインデックス（32 バイト）を格納するのはメモリ集約的になり得ます。実際には、単一の `firstChildIndex` だけを格納し、8 つの子が `nodePool` 内で連続していることを保証する **ポインタベース Octree** を使うことがよくあります。これによりノードサイズが単一インデックスと数ビットのメタデータに減ります。

=== 構築戦略: トップダウン対ボトムアップ

*   **ボトムアップ**: すべてのオブジェクトを葉として始め、`SubgroupMatch`（第 4 章）で同じ親ノードに属すべきオブジェクトを見つける。速いが複雑なソートが必要。
*   **トップダウン**: 1 つのルートノードから始め、必要に応じてグローバルアトミックでノードを「細分化」する。直感的だが高いメモリ競合につながり得る。

== 木の走査: スタックとビットフィールド

木が構築されたら、どう使うのでしょうか。C{pp} では再帰関数を使います。GPU では、各スレッドがコールスタック用の限られた量の **プライベートメモリ**（レジスタ）しか持たないため、再帰はしばしば禁止されるか性能が悪いです。

代わりに **スタックベース走査** または **スタックレス走査** を使います。

1.  **スタックベース**: 各スレッドが「訪れるノード」の小さな配列をレジスタまたはローカルメモリに維持します。速いですが貴重なレジスタを消費し、**占有率**（ハードウェアが同時に実行できるアクティブスレッド数）を下げる可能性があります。
2.  **スタックレス（スレッド付き木）**: 各ノードが **深さ優先探索（DFS）** 順の次のノードへの「スキップポインタ」を格納します。レイがノードを外すと、スキップポインタに従いそのノードのすべての子を迂回します。スタック空間はゼロですが、木構築がより複雑になります。

== 実装の課題: メモリボトルネック

GPU 上で木を構築するときの最大の課題は **メモリ管理** です。伝統的な C{pp} アプリケーションでは、必要に応じて `malloc` や `new` でノードを作ります。コンピュートシェーダにはこれらは存在しません。

なぜか？ 伝統的な CPU は `malloc` に **集中ヒープマネージャ** を使い、異なるスレッドが同じメモリを請求しないようグローバルロックに依存します。10,000 の GPU スレッドが同時に `malloc` を呼ぼうとすると、ハードウェアはロック待ちにすべての時間を費やし、巨大な性能崩壊につながります。

代わりに、**「ノードプール」**（大きなバッファ）を事前割り当てし、単一のアトミックカウンタでこのプールからノードを「割り当て」ます。ノードが必要な各スレッドはカウンタをインクリメントし、結果をプールへの一意なインデックスとして使います。

[source,slang]
----
// アトミックを使った単純なノード割り当て
struct Node { ... };
RWStructuredBuffer<Node> nodePool;
RWStructuredBuffer<uint> nodeCounter;

uint allocateNode() {
    uint index;
    InterlockedAdd(nodeCounter[0], 1, index);
    return index;
}
----
