= メモリモデルと一貫性: はじめに

== 概要

前の章では、高い占有率と密なデータレイアウトによって GPU を忙しく保つ方法を見ました。しかしコンピュートディスパッチを単純な独立タスクから複雑で協調的なアルゴリズムへスケールすると、はるかに難しい問題にすぐ直面します。**メモリ一貫性**（GPU のすべての部分が同じ時刻に同じデータを見ることを保証すること）です。

ある呼び出しが書いた値が別の呼び出しに見えることを、どう知るのでしょうか。2 つの呼び出しが同じ場所に同時に書こうとするとどうなるのでしょうか。競合状態に陥らずに、何千ものスレッド間でデータをどう効率的に共有するのでしょうか。

=== Vulkan の明示性

CPU では、キャッシュコヒーレンスハードウェアが、スレッド A がストアを完了すると他の CPU コアが最終的に新しい値を観測することを保証します。ただし CPU スレッドプログラミングでも明示的な同期は必要です——`std::atomic`、ミューテックス、メモリフェンスなしでは、コンパイラとアウトオブオーダーな CPU コアはデータ競合を生む形で操作を並べ替えたり遅らせたりできます。

GPU では、事情は**異なります**。複数のコンピュートユニットをまたいで何千ものスレッドが並行実行し、それぞれが独自の階層キャッシュ（**L1** - Level 1、**L2** - Level 2 など）を持つため、全員のメモリの見え方を同期に保つのは極めて高価な作業です。Vulkan の哲学は単純です。**同期は決して自動ではない**。書き込みを読み取りに可視にしたいなら、明示的にそう言わなければなりません。

== メモリ管理の 3 本柱

本章では、この複雑さを管理するために Vulkan が提供する仕組みを深く掘り下げます。

1.  **Vulkan Memory Model**: Availability、Visibility、Domain 操作を習得し、スレッド間に形式的な **「Happens-Before」** 関係（操作の厳格な順序）を作る。
2.  **共有メモリ（LDS）**: 高速なデータ交換と手動キャッシュのために、小さく超高速なワークグループローカルメモリを活用する。
3.  **メモリ一貫性**: Slang の `GroupMemoryBarrierWithGroupSync` と、Vulkan 1.4 の細粒度バリアを使い、パイプラインストールを最小化してスループットを最大化する。

== なぜ重要か

効率的なメモリ同期は、高性能シミュレーションと、壊れた非決定的な混乱の違いです。

*   **過剰同期**: すべてのスレッドが常に他のすべてのスレッドを待つため、カーネルが遅くなります。
*   **不足同期**: ちらつく結果、「ゴースト」データ、特定ハードウェアでだけ現れるデバッグ困難なクラッシュが起きます。

まずは理論的基盤である **Vulkan Memory Model** から始めます。最初は抽象的に感じるかもしれませんが、スマートフォンからハイエンドワークステーションまであらゆる GPU で動く、移植可能で堅牢なコンピュートコードを書く鍵です。
