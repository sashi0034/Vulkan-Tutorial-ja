= 共有メモリ（LDS）: 高速データ交換

GPU のメインメモリ（VRAM）は大きいですが比較的遅いです。多くのコンピュートタスクでは、VRAM からのデータフェッチが主ボトルネックです。これを解くため、GPU は各ワークグループに局所な、小さく超高速なメモリを提供します。

Vulkan ではこれを **共有メモリ（Shared Memory）** と呼びます。物理ハードウェアでは、しばしば **LDS（Local Data Store）** や **スクラッチパッドメモリ**（ローカルデータ用の高速一時メモリ）と呼ばれます。

== なぜ共有メモリか？

共有メモリは、メモリバウンドなカーネルを最適化する最も強力な道具です。典型的には次に使います。

*   **手動キャッシュ**: VRAM からデータブロックを一度読み、共有メモリに格納し、ワークグループ内の全スレッドがその高速なローカルコピーを複数回読む。
*   **データ交換**: 同じワークグループ内のスレッド間でデータを渡す（例: **プレフィックス和**——各要素がそれ以前の全要素の和——や **リダクション** の計算）。
*   **ワークグループレベルリダクション**: 各ワークグループ内でまずリダクション（複数値を 1 つにまとめる）し、大きなデータセットの最大値や最小値を求める。

== Slang での共有メモリの使い方

Slang（および HLSL）では、`groupshared` キーワードで共有メモリを宣言します。Compute Unit に物理的に局所なため、ワークグループ内の全スレッドで共有されますが、他のワークグループからは見えません。

[source,slang]
----
groupshared float sharedData[256];

[numthreads(256, 1, 1)]
void main(uint3 tid : SV_GroupThreadID) {
    // 各スレッドが共有メモリ内の自分のスロットを初期化
    sharedData[tid.x] = someBuffer[tid.x];

    // 重要: 全スレッドの書き込み完了を待ち、かつそれらの書き込みを可視にしなければならない！
    // GroupMemoryBarrier: 以前のメモリ書き込みが完了し可視であることを保証。
    // WithGroupSync: 実行バリアとして働き、グループ内の全スレッドが到着するのを待つ。
    GroupMemoryBarrierWithGroupSync();

    // これで隣が書いたデータを安全に読める
    float neighborValue = sharedData[(tid.x + 1) % 256];
}
----

=== 難解な名前の分解

`GroupMemoryBarrierWithGroupSync()` は長く感じるかもしれませんが、その名前が 2 種類の同期で何をしているかを正確に教えてくれます。

1.  **GroupMemoryBarrier**: これは**メモリバリア**です。スレッドが `groupshared` メモリに行った書き込みが「押し出され」、ワークグループ内の他の全スレッドに可視になることを保証します。これがないと、隣が共有メモリのあなたのスロットから古い、または未初期化の値を読むかもしれません。
2.  **WithGroupSync**: これは**実行バリア**です。ワークグループ内のすべてのスレッドをこの正確な行で止め、待たせます。グループ内の*すべての*スレッドがこの点に到達するまで、どのスレッドも次の命令へ進めません。

これらを組み合わせることで、スレッドがこの行を過ぎるとき、隣が作業を終え、そのデータが読み取り可能であることが保証されます。

=== GLSL: shared と barrier()

GLSL では、`shared` キーワードでワークグループローカルメモリを宣言します。同期は `barrier()` 関数が担い、実行バリアと `shared` メモリ向けのメモリバリアの両方として働きます。

[source,glsl]
----
shared float sharedData[256];

layout(local_size_x = 256) in;
void main() {
    uint tid = gl_LocalInvocationID.x;

    // 各スレッドが共有メモリ内の自分のスロットを初期化
    sharedData[tid] = someBuffer[tid];

    // 全スレッドがこの点に到達し、メモリが可視になるのを待つ
    barrier();

    // これで安全に読める
    float neighborValue = sharedData[(tid + 1) % 256];
}
----

ここでの主な違いは、Slang の `GroupMemoryBarrierWithGroupSync()` が、メモリバリアと実行同期を組み合わせる一般的なパターンに対してより説明的な名前であることです。

== バンク競合: 速度の罠

共有メモリは **バンク**（並列メモリモジュール、典型的には 32 バンク）に組織されています。各バンクはクロックサイクルあたり 1 リクエストを扱えます。スレッドが異なるバンクにマッピングされる形でメモリにアクセスすると、操作はフルスピードで並列に行われます。

しかし、束（サブグループ）内の 2 つ以上のスレッドが、**同じバンク**内の異なるアドレスにアクセスしようとすると、**バンク競合（Bank Conflict）** が起きます。ハードウェアはこれらのリクエストを直列化しなければならず、その命令の実行時間が 2 倍や 3 倍になることがあります。

[source,text]
----
// 競合なし（高速）
Thread 0 -> Bank 0
Thread 1 -> Bank 1
Thread 2 -> Bank 2

// バンク競合（低速）
Thread 0 -> Bank 0 (Address 0)
Thread 1 -> Bank 0 (Address 32)
----

バンク競合を避けるには、`thread_id` が `index` と一致する線形アクセスパターンを目指してください。**ストライド** 1（要素を順にアクセス）が、フルスピードを保証する通常最も安全な方法です。

== ライフサイクルとスコープ

共有メモリは単一ワークグループの寿命の間だけ有効です。ワークグループが完了すると、その共有メモリは破棄されます。

重要な点として、**共有メモリはワークグループ間でコヒーレントではありません**。ワークグループ A からワークグループ B へデータを送る必要がある場合、グローバル VRAM に書き戻し、前のセクションで述べた適切な Vulkan メモリバリアを使わなければなりません。
