= コオペラティブ行列の扱い

[NOTE]
====
本セクションには `VK_KHR_cooperative_matrix` 拡張と Vulkan 1.3 以上が必要です。Vulkan 1.4 では、この拡張はコア API へ昇格しています。
====

**コオペラティブ行列** がなぜこれほど強力かを理解するには、GPU 上での行列乗算へのアプローチを考え直す必要があります。伝統的な「素朴な」ループでは、各呼び出しが結果行列の 1 つ以上の要素の計算を担当します。これには多くの冗長なメモリ読み取りが伴い、本質的にハードウェアの標準浮動小数点スループットに縛られます。

コオペラティブ行列は、呼び出しのグループ（**サブグループ**）が単一の行列乗算と累積（**GEMM** 演算）で協力する新しい方法を導入することでゲームを変えます。個々の呼び出しが孤立して働く代わりに、サブグループ全体が演算を実行するために「協力」します。

== 概念: 行列フラグメントとサブグループスコープ

コオペラティブ行列の鍵は **フラグメント** の概念です。コオペラティブ行列型を宣言するとき、データは任意の呼び出しがアクセスできる単一の連続配列には格納されません。代わりに、サブグループ内の呼び出しに**分散**されます。

各呼び出しは行列の小さな断片だけを所有します。ハードウェアが行列をレジスタ間で「シャーディング」していると考えることができます。これにより GPU は、伝統的な ALU 命令のオーバーヘッドなしに、それらのレジスタ上で直接数学を行う専用ハードウェアユニット（NVIDIA の **Tensor Core** や AMD の **Matrix Core** など）を使えます。

重要な点として、演算は **サブグループスコープ** で起きます。つまりサブグループ内のすべての呼び出しが、ロード、乗算、ストア演算に同時に参加しなければなりません。サブグループの一部メンバが非アクティブな発散分岐の中でコオペラティブ行列関数を呼ぼうとすると、未定義動作や GPU ハングに遭遇する可能性が高いです。

これらのユニットが行う標準 GEMM 演算は次のとおりです。
[latexmath]
++++
D = A \times B + C
++++
ここで latexmath:[A] は latexmath:[M \times K] 行列、latexmath:[B] は latexmath:[K \times N] 行列、latexmath:[C, D] は latexmath:[M \times N] 行列です。

== メモリレイアウト: ストライドとメジャーネス

メモリからフラグメントをロードするとき、バッファ内で行列がどう配置されているかを指定しなければなりません。

1. **行メジャー対列メジャー**: ほとんどの Vulkan アプリケーションは **行メジャー**（行の要素が連続）を好みます。
2. **ストライド**: ある行の開始から次の行の開始までの距離（バイトではなく要素数）です。単純な密にパックされた行列では、ストライドは列数に等しいです。

バッファに大きな行列があり、小さな $16 \times 16$ タイルだけをロードする場合、ストライドは*全体*の大きな行列の幅になります。

== Slang: タイル化行列乗算

Slang はコオペラティブ行列を第一級型として扱い、表現力豊かなタイル化アルゴリズムを可能にします。より大きな行列乗算のブロックをどう実装するかの例です。

[source,slang]
----
import slang_vulkan_compute;

// 物理デバイスがサポートする行列次元
const int M = 16;
const int N = 16;
const int K = 16;

struct Params {
    uint64_t addrA, addrB, addrC;
    uint32_t strideA, strideB, strideC;
    uint32_t totalK;
};

ParameterBlock<Params> cb;

[numthreads(32, 1, 1)] // サブグループサイズはハードウェアの期待に一致しなければならない
void computeMain(uint3 threadId : SV_GroupThreadID, uint3 groupId : SV_GroupID) {
    // 各サブグループが出力行列 C の 1 つの (M x N) タイルを扱う
    CooperativeMatrix<float, M, N> acc = 0.0f;

    // K 次元を 'K' のブロックでループ
    for (uint32_t k = 0; k < cb.totalK; k += K) {
        CooperativeMatrix<float16, M, K> matA;
        CooperativeMatrix<float16, K, N> matB;

        // Buffer Device Address を使ってメモリからタイルをロード
        matA.load((float16*)cb.addrA, getOffsetA(groupId, k), cb.strideA);
        matB.load((float16*)cb.addrB, getOffsetB(groupId, k), cb.strideB);

        // 積を累積: acc = matA * matB + acc
        acc = mul(matA, matB) + acc;
    }

    // 最終累積タイルをストア
    acc.store((float*)cb.addrC, getOffsetC(groupId), cb.strideC);
}
----

== GLSL: 低レベルな勝利

Slang がコードを標準の行列数学のように見せる一方、`GL_KHR_cooperative_matrix` 拡張を通じて Vulkan 1.4 が提供する「勝利」を理解するには、GLSL 相当を見ると役立ちます。レジスタ割り当ての最適化をコンパイラにヒントする明示的な「Use」型に注意してください。

[source,glsl]
----
#extension GL_KHR_cooperative_matrix : enable
#extension GL_EXT_shader_explicit_arithmetic_types_float16 : enable

// 明示的なスコープと用途でフラグメントを定義
layout(constant_id = 0) const int M = 16;
layout(constant_id = 1) const int N = 16;
layout(constant_id = 2) const int K = 16;

// UseA と UseB は入力、UseAccumulator は C と D 用
coopmat<float16_t, gl_ScopeSubgroup, M, K, gl_MatrixUseA> matA;
coopmat<float16_t, gl_ScopeSubgroup, K, N, gl_MatrixUseB> matB;
coopmat<float, gl_ScopeSubgroup, M, N, gl_MatrixUseAccumulator> acc;

void main() {
    // 明示的ロードはバイトオフセットと行ストライドが必要
    coopMatLoad(matA, dataA, offsetA, strideA, gl_CooperativeMatrixLayoutRowMajor);
    coopMatLoad(matB, dataB, offsetB, strideB, gl_CooperativeMatrixLayoutRowMajor);

    // acc = matA * matB + acc
    acc = coopMatMulAdd(matA, matB, acc);

    coopMatStore(acc, dataC, offsetC, strideC, gl_CooperativeMatrixLayoutRowMajor);
}
----

== ハードウェア制約と能力

物理次元（latexmath:[M, N, K]）は任意ではありません。サポートされる組み合わせを見つけるため、`VkPhysicalDeviceCooperativeMatrixPropertiesKHR` を問い合わせなければなりません。

* **サブグループサイズ**: NVIDIA では、これらのユニットは典型的にサブグループサイズ 32 を期待します。AMD では 64 かもしれません。`[numthreads]` で間違ったサブグループサイズを使うと、コオペラティブ行列型の初期化に失敗します。これはモバイルでさらに重要です。増え続ける Arm と Qualcomm の部品がオンデバイス ML 向けに `VK_KHR_cooperative_matrix` を公開しますが、サブグループサイズはデスクトップと異なるため、デスクトップの値をコピーせず常にサポートされる構成を問い合わせてください（xref:Advanced_Vulkan_Compute/12_Mobile_and_Embedded_Compute/02_android_compute.adoc[Android 上のコンピュート] を参照）。
* **精度のトレードオフ**: スループット最大化と帯域節約のため入力行列（A と B）に `float16` を使い、アキュムレータ（C と D）に `float32` を使うのが標準的です。この「混合精度 GEMM」は速度と数値安定性の最良のバランスを提供します。
* **整列**: `.load()` と `.store()` に渡すメモリアドレスは通常特定の整列（例: 16 バイト）を必要とします。非整列アドレスからのロードはデバイスロストエラーにつながり得ます。

これらの専用ユニットを活用することで、標準浮動小数点ユニットで可能なものをしばしば桁違いに超えるスループットを達成できます。これにより、GPU 上のあらゆる性能クリティカルな線形代数にとってコオペラティブ行列は不可欠です。
