= 非同期コンピュートのオーケストレーション

これまでの章では、個々のカーネルをできるだけ速くすることに大きく焦点を当ててきました。メモリアクセスを最適化し、サブグループを活用し、GPU 上にデータ構造全体さえ構築しました。しかししばしば見落とされる、より高レベルの最適化があります。エンジンの残りのワークロードと並べて、これらのディスパッチをどうスケジュールするかです。

現代の GPU は単一のモノリシックプロセッサではありません。並列に動ける複数のハードウェアエンジンを持つ複雑なシステムです。非同期コンピュートを理解するには、まず物理ハードウェアを理解する必要があります。典型的な高性能 GPU にはいくつかの専用エンジンがあります。

* **グラフィックスエンジン**: 頂点処理、ラスタライズ、フラグメントシェーディング、および汎用コンピュートが可能な主エンジン。
* **非同期コンピュートエンジン（ACE）**: コンピュートディスパッチ専用のスケジューラとハードウェアパス。グラフィックスワークロードが完全には使っていないコンピュートユニット（CU）やストリーミングマルチプロセッサ（SM）を使い、グラフィックスエンジンと完全に並列に動けることが多い。
* **転送／コピーエンジン**: コンピュートリソースを消費せずにホストとデバイスメモリ間でデータを動かす専用 DMA（Direct Memory Access）エンジン。

Vulkan はこれらのハードウェアエンジンを **キューファミリー** 経由で公開します。各ファミリーは一連の **能力**（例: `VK_QUEUE_GRAPHICS_BIT`、`VK_QUEUE_COMPUTE_BIT`、`VK_QUEUE_TRANSFER_BIT`）を持ちます。メインのグラフィックスキューファミリーは通常すべてをサポートしますが、「Dedicated Compute」や「Async Compute」ファミリーはコンピュートと転送*のみ*をサポートすることがあります。

これらの専用ファミリーから別のコンピュートキューを使うことで、パストレースのデノイズ、物理シミュレーション、複雑な AI パスファインディングのような重いコンピュートディスパッチを、メインのグラフィックスレンダリングパスとオーバーラップできます。グラフィックスハードウェアがジオメトリ処理と三角形のラスタライズで忙しい間、コンピュートユニットは同時にシミュレーションの数値計算を回せます。

本章では、単純な「すべてに 1 キュー」モデルを超えます。Vulkan の **Synchronization 2**（`VK_KHR_synchronization2`）を使い、**パイプラインストール**（GPU がリソース待ちでアイドルする）を起こさずに複雑で並行なワークロードをオーケストレーションする方法を探ります。また、どのタスクが本当にレイテンシクリティカルかをハードウェアに伝える機能——**キュー優先度**——も見ます。背景シミュレーションが時間制約のある物理更新を遅らせないようにするためです。

これらのワークロードのオーケストレーションには、GPU のタイムラインの考え方の転換が必要です。もはや単なる線形なコマンド列ではなく、異なる種類のトラフィックが異なる速度で動き、全体スループットを最大化するために時折合流したり譲ったりする多車線高速道路です。
