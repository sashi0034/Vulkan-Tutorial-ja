= タイムラインセマフォ: 統一された同期

バイナリセマフォ（古典的な `VkSemaphore`）はキュー間の単純な「ウェイト／シグナル」関係に有用ですが、複雑な非同期パイプラインではすぐに管理の悪夢になります。各バイナリセマフォはウェイトされてリセットされるまで一度しかシグナルできず、追跡困難なセマフォオブジェクトの増殖につながります。

これが **タイムラインセマフォ**（Vulkan 1.2 および `VK_KHR_timeline_semaphore` で導入）が非同期コンピュートにとって画期的な理由です。単純なブールの「オン／オフ」状態の代わりに、タイムラインセマフォは単調増加する **64 ビット整数値** を含みます。

== 単一値の力
タイムラインセマフォでは、セマフォがシグナルされるのを待つだけでなく、**特定の値** に達するのを待ちます。これにより、単一オブジェクトで作業のタイムライン全体を表現できます。たとえば:

*   **値 10**: 物理シミュレーション完了。
*   **値 11**: デノイズパス完了。
*   **値 12**: UI 合成向けにフレーム準備完了。

異なるキューが同じセマフォを異なる値でシグナルでき、他のキューは必要な進捗レベルちょうどを待てます。

== Wait-Before-Signal（ホスト側）
タイムラインセマフォの最も強力な機能の一つは、**まだ到達していない**値を待つコマンドバッファを投入できることです。実際、ウェイトが投入される時点で、シグナル操作が GPU に投入されている必要すらありません。

これにより CPU は複雑な依存グラフを構築し、一度に異なるキューへすべて投入できます。カウンタが増えるにつれ、GPU ハードウェアがストールとウェイクアップを自動処理します。

[source,cpp]
----
// 特定のタイムライン値へのウェイトを定義
vk::TimelineSemaphoreSubmitInfo timelineInfo {
    .waitSemaphoreValueCount = 1,
    .pWaitSemaphoreValues = &requiredValue,
    .signalSemaphoreValueCount = 1,
    .pSignalSemaphoreValues = &newValue
};

vk::SubmitInfo submitInfo {
    .pNext = &timelineInfo,
    .waitSemaphoreCount = 1,
    .pWaitSemaphores = &*timelineSemaphore // vk::raii::Semaphore からハンドルを取り出す
};
// ...
----

== ホストの問い合わせと待機
タイムラインセマフォは GPU と CPU のギャップも埋めます。CPU は `vkGetSemaphoreCounterValue` でいつでもセマフォの現在値を問い合わせられます。さらに良いことに、CPU は `vkWaitSemaphores` でセマフォが特定の値に達するまでブロックできます。

これは多くのシナリオで `VkFence` の必要性を置き換えます。フェンスがするようにコマンドバッファ全体の完了を待つ代わりに、CPU は GPU タイムラインの特定の点を待てます。これは **パイプライン化されたリソース管理** に非常に有用です——たとえば、CPU は GPU が値 `N` に達するのを待てば、値 `N` をシグナルしたコマンドが使ったバッファを再利用しても安全だと分かります。

== 非同期コンピュートでなぜ重要か
非同期コンピュート構成では、しばしば交差依存を持つ複数の作業ストリームがあります。たとえば物理エンジン（コンピュートキュー）がパーティクルシステム（グラフィックスキュー）が必要とするデータを生み、それがさらにデノイザ（コンピュートキュー）が必要とするデータを生むかもしれません。

これにバイナリセマフォを使うと、「シグナル A → ウェイト A → シグナル B → ウェイト B」の複雑な網が必要になります。タイムラインセマフォでは、単一の「エンジンタイムライン」を持つだけです。すべてのタスクがカウンタをインクリメントして完了をシグナルし、すべての依存タスクが特定の前提値を待ちます。これによりオーケストレーション論理が大幅に簡素化され、セマフォ管理のオーバーヘッドが減ります。
