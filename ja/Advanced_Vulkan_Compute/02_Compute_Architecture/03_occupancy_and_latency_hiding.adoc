= 占有率とレイテンシ隠蔽: GPU を忙しく保つ

== はじめに

前のセクションでは、ワークグループが GPU の工場フロア（Compute Unit や SM）へどうマッピングされるかを学びました。しかしワークグループを CU に載せるだけでは戦いの半分です。設計が悪いとハードウェアの潜在能力のごく一部しか使えず、高価なシリコンが遊んでしまいます。

なぜこうなるかを理解するには、**レイテンシ**と**占有率**について話す必要があります。

== レイテンシのギャップ

GPU はメモリバウンドです。現代の GPU は毎秒数兆回の浮動小数点演算（**TFLOPS**）をこなせますが、**VRAM**（Video Random Access Memory）から 1 データの取得には数百、時には数千クロックサイクルかかることがあります。

呼び出しの束（ワープやウェーブフロント）がメモリから読む必要があると、待たねばなりません。その CU に実行できる束が 1 つしかないと、データが届くまで CU 全体が沈黙します。性能にとって災難であり、これを**メモリレイテンシ**と呼びます。

GPU の解決策は**並行性**です。1 つの束を待つのではなく、実行準備ができている別の束へ切り替えます。単一 CU 上で「インフライト」な束が多いほど、メモリフェッチのレイテンシをうまく隠せます。

== 占有率の定義

**占有率（Occupancy）** は、CU 上のアクティブな束が理論最大に対してどれだけあるかの尺度です。しばしばパーセンテージで表されます。

*   **100% 占有率**: CU が束で完全に埋まっています。1 つがメモリを待っても、ほぼ確実に別の準備ができている束があります。
*   **低い占有率**: アクティブな束が少数だけです。すべてが同時にメモリフェッチに当たると、CU はストールします。

=== リソース制限と占有率圧力

「なぜ常に何千ものスレッドをディスパッチしないのか？」と思うかもしれません。問題は、各 Compute Unit が固定の物理リソースプールを持つことです。スレッドを増やすたびに、そのプールの一部を消費します。

占有率を制限する主な 3 要素は次のとおりです。

1.  **レジスタ**: 各スレッドは変数を格納するレジスタセットを必要とします。シェーダが 128 レジスタを使うと、32 しか使わない場合より載せられるスレッド数が減ります。
2.  **共有メモリ（LDS）**: このメモリはワークグループ全体で共有されます。ワークグループが LDS を 32KB 使い、CU が 64KB しか持たない場合、スレッド数がいくらでも、その CU にはワークグループを 2 つしか載せられません。
3.  **スレッド／ワープスロット**: ハードウェアスケジューラが一度に追跡できるスレッド数にはハード上限があります（例: CU あたり 2048 スレッド）。

|===
| リソース使用 | 占有率への影響 | 結果

| 高いレジスタ数
| **負**
| CU あたりの束が減り、レイテンシ隠蔽が難しくなる。

| 高い LDS 使用
| **負**
| CU あたりのワークグループが減り、並行性が制限される。

| 小さいワークグループサイズ
| **中立／負**
| ワープスロットを埋め切れない可能性；スケジューリングオーバーヘッド。
|===

== 理論占有率の計算

ほとんどの GPU ベンダーは、占有率を計算してくれるツール（NVIDIA の Nsight や AMD の RGP など）を提供します。ただし、シェーダのリソース使用を見れば自分で見積もることもできます。

CU に共有メモリ 64KB があり、ワークグループが 32KB 使うなら、その CU は常に最大 2 ワークグループしかホストできません。ワークグループサイズが小さい場合（たとえば 64 スレッド）、CU あたり 128 スレッドになります。そのハードウェアが 2048 スレッドを追跡できるなら、占有率はおよそ 6% にすぎません。

これが、「太い」シェーダ（レジスタや共有メモリを大量に使うもの）が、慎重にチューニングしないと性能が落ちがちな理由です。

== 利用率の監視

実エンジンでは推測だけでは足りません。現代の Vulkan エンジンは、`VK_KHR_performance_query` 拡張を通じてパフォーマンスカウンタを使い、ハードウェア利用率をリアルタイムで監視します。

**ValuUtilization**（AMD）や **SM Active**（NVIDIA）などのメトリクスを追うことで、カーネルが実際にハードウェアを忙しく保てているかが分かります。メモリレイテンシが高く占有率が低い場合は、レジスタ使用や共有メモリフットプリントを最適化する必要があることが分かります。
