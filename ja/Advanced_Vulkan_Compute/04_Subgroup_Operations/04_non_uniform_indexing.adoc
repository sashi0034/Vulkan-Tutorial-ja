= 非ユニフォームインデックス付け: サブグループ内のリソース配列

== ディスクリプタの制限

伝統的な Vulkan では、**ディスクリプタセット**（テクスチャやバッファなどのリソースの集合）はドローコールやディスパッチ全体で「ユニフォーム」です。つまり、ワークグループ内のすべてのスレッドが、与えられたディスクリプタセットインデックスから同じリソースにアクセスしなければなりません。

しかしテクスチャの配列があり、スレッド A がテクスチャインデックス 5 に、同じサブグループのスレッド B がテクスチャインデックス 12 にアクセスしたい場合はどうでしょうか。初期の Vulkan では、これは未定義動作やデバイスクラッシュになりました。

そこで **非ユニフォームインデックス付け（Non-Uniform Indexing）** が登場します。

== 非ユニフォームインデックス付け（Descriptor Indexing）

Vulkan の **Descriptor Indexing**（1.2 から標準、1.4 で洗練）により、変数をディスクリプタ配列へのインデックスとして使えます。ただしサブグループ内のスレッドはロックステップで実行するため、インデックスがサブグループ間で異なる（「非ユニフォーム」）可能性があるとき、ハードウェアはそのことを知る必要があります。

Slang（HLSL を継承）では、`NonUniformResourceIndex` 関数を使い、コンパイラに「このインデックスはスレッドごとに異なるかもしれないので、ユニフォーム値として最適化しないで」と伝えます。

[source,slang]
----
// ディスクリプタセット内のテクスチャ配列
Texture2D textures[];

// 各スレッドがマテリアル ID に基づいて自分のテクスチャを選ぶ
uint materialID = getMaterialID();

// インデックスを非ユニフォームとして明示的にマークしなければならない
float4 color = textures[NonUniformResourceIndex(materialID)].Sample(sampler, uv);
----

=== GLSL: nonuniformEXT

GLSL では `GL_EXT_nonuniform_qualifier` 拡張が必要です。関数呼び出しの代わりに、特別なキーワード `nonuniformEXT` を使います。

[source,glsl]
----
#extension GL_EXT_nonuniform_qualifier : enable

layout(binding = 0) uniform sampler2D textures[];

// NonUniformResourceIndex の GLSL 相当
uint materialID = getMaterialID();
vec4 color = texture(textures[nonuniformEXT(materialID)], uv);
----

`nonuniformEXT` がないと、コンパイラは `materialID` がサブグループ内の全スレッドで同じだと想定してアクセスを最適化する可能性があり、結果が不正になります（全スレッドが同じテクスチャ値——おそらく最初のスレッドのインデックス——を得る）。

== なぜこれはサブグループ機能なのか？

なぜメモリ章ではなくサブグループ章にあるのか疑問に思うかもしれません。理由は、ハードウェアがこの命令をどう実行するかです。

サブグループが非ユニフォームインデックスに遭遇すると、GPU はアクセスを **スカラー化**（一意なインデックスごとにアクセスを直列化）しなければなりません。実質的に、サブグループ内に存在する一意なインデックスをループします。

1.  テクスチャ 5 を欲しがる全スレッドを見つける。
2.  それらのスレッド向けにロードを行う。
3.  テクスチャ 12 を欲しがる全スレッドを見つける。
4.  それらのスレッド向けにロードを行う。

この処理はハードウェアが担いますが、前のセクションで議論した同じサブグループパーティショニング論理に依存します。この「ループ」がサブグループレベルで起きると理解することで、発散したリソースアクセスの性能影響をよりよく予測できます。

== パフォーマンスのベストプラクティス

*   **発散を最小化**: サブグループ内の 32 スレッドすべてが同じテクスチャにアクセスするなら、ハードウェアはロードを 1 回だけ行えばよい。32 スレッドすべてが*異なる*テクスチャにアクセスするなら、ロード操作は最大 32 倍長くかかるかもしれない。
*   **サブグループソート**: 大きなワークロードがある場合、同じサブグループのスレッドが同じまたは近いリソースにアクセスしやすくなるようソートすることを検討する。


== まとめ

サブグループ演算は、GPU プログラミングにおけるパラダイムシフトを表します。「ワークグループ全体の同期」から「ウェーブ意識の通信」へ移ることで、現代 GPU アーキテクチャの潜在能力をフルに引き出せます。
