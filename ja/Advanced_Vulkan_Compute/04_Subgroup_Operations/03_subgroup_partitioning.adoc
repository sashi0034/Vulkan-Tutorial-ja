= サブグループパーティショニング: Ballot と Match

== ロックステップ実行を超えて

前のセクションでは、サブグループ内のスレッドがどうデータを共有できるかを見ました。しかし同じサブグループのスレッドが異なることをしたい場合はどうでしょうか。そこで **サブグループパーティショニング** が登場します。

GPU では、サブグループ（SIMD 束）内のすべてのスレッドが同じ時刻に同じ命令を実行します。`if` 文があるとき、一部のスレッドは「真」分岐を、他は「偽」分岐を取るかもしれません。ハードウェアは、現在の命令を実行すべきでないスレッドを「マスクアウト」することでこれを処理します。これを **分岐発散（Branch Divergence）** と呼び、第 2 章で議論したように大きな性能キラーになり得ます。

**Ballot** や **Match** のようなサブグループパーティショニングツールにより、これらのマスクを「見て」コード最適化に使えます。

== サブグループ Ballot

**Ballot** 演算は、サブグループ内のすべてのスレッドに真偽の質問をし、各ビットが 1 スレッドの答えを表す **ビットマスク**（各ビットがスレッドを表すビット列）を返します。

[source,slang]
----
// このスレッドは有効な結果を持つか？
bool hasResult = computeIsSuccessful();

// 有効な結果を持つサブグループ内の全スレッドのビットマスクを得る
uint4 activeMask = WaveActiveBallot(hasResult);
----

Slang（および Vulkan SPIR-V）では、最大 128 スレッド幅のサブグループをサポートするため ballot は `uint4`（128 ビット）を返しますが、32 または 64 がより一般的です。

このマスクがあれば、ビット演算で判断できます。

*   `WaveActiveCountBits(hasResult)`: アクティブなスレッドは何本か？（Slang が便利な短縮形を提供）
*   `countbits(activeMask)`: マスク上の低レベルなビットカウント。
*   `WavePrefixCountBits(hasResult)`: アクティブスレッドの中での自分の **ランク**（より低いインデックスを持つアクティブスレッド数）は何か？

これは **ストリーム圧縮** に非常に有用です。32 のうち 5 スレッドだけがバッファへ書くデータがある場合、アトミック演算なしで、出力バッファ内のどのインデックスに書くべきかをこれらの演算で正確に計算できます。

== サブグループ Match

`Ballot` が真偽値で動くのに対し、**Match** は値で動きます。与えられた変数について*同じ値*を持つサブグループ内のすべてのスレッドを見つけます。

[source,slang]
----
// 各スレッドが 'key' を持つ（例: マテリアル ID やハッシュ）
uint myKey = ...;

// 自分と同じキーを持つ全スレッドのマスクを得る
uint4 sameKeyMask = WaveMatch(myKey);
----

これは専門的な演算（しばしば Vulkan 1.1 または特定の拡張が必要）で、**グローバルアトミックリダクション**（複数スレッドからのアトミック演算を 1 つにまとめる）にとって画期的です。

32 スレッドがすべて同じグローバルカウンタに加算しようとしている場面を想像してください。通常、これは 32 回の直列化されたアトミック演算になります。`WaveMatch` があれば、スレッドはどれが同じアドレスに当たっているかを特定し、グループ全体のために単一のアトミック加算を行う 1 つの **「リーダー」** スレッド（グループの代理として動く 1 スレッド）を選び、結果を戻して配布できます。

== サブグループ Elect

パーティショニングの最も単純な形は `WaveIsFirstLane()` です。サブグループ内のちょうど 1 スレッド（または **レーン**）（通常は最も低いアクティブ ID を持つもの）に対して `true` を返し、他すべてには `false` を返します。

[source,slang]
----
if (WaveIsFirstLane()) {
    // サブグループ内で 1 スレッドだけがこの高価なタスクを実行
    performGlobalLogging();
}
----

デバッグメッセージの書き込みやグローバルタイムスタンプの更新など、ウェーブあたり一度だけ起きればよいタスクに最適です。

== フロー制御へのマスクの利用

これらの演算を組み合わせることで、スレッドがどう分岐しているかに適応する「ウェーブ意識」のコードを書けます。ハードウェアにマスクアウトさせるだけでなく、`activeMask` を明示的に検査し、関心のあるスレッドがいなければコードブロック全体をスキップしたり、発散を最小化するためにマスクを使って作業を並べ替えたりできます。
