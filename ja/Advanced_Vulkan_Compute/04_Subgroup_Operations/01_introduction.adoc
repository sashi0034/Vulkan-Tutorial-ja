= サブグループ演算: 隠れた力

== はじめに

これまでの章では、**共有メモリ（LDS）** と明示的なバリアを使い、ワークグループ内の数百〜数千スレッド間でデータを共有する方法を見ました。強力ですが、この手法には大きなコストがあります。バリアのたびに GPU が一時停止して待ち、共有メモリへのアクセスのたびに貴重な帯域を消費します。

もっと速くデータを共有できたらどうでしょう。VRAM にも LDS にも触れずに値を交換できたらどうでしょう。そこで登場するのが **サブグループ演算** です。今日存在する最も高度に最適化された GPU アルゴリズムの多くを支える「秘密のソース」です。

== なぜサブグループが重要か

**サブグループ** は、同じ SIMD ユニット上で完全なロックステップで実行される、ハードウェアレベルのスレッドの束（典型的には NVIDIA/Intel で 32、AMD で 32/64）です。ハードウェアがすでにこれらのスレッドを物理的に同期しているため、しばしば 1 クロックサイクル並みに速い専用命令で互いと通信できます。

これはデスクトップ専用の機能ではありません。Arm、Qualcomm、Imagination のモバイル GPU もサブグループ演算を公開しますが、幅ははるかに可変（4 から 128 レーン）で、*想定せず実行時に問い合わせる*必要があります——xref:Advanced_Vulkan_Compute/12_Mobile_and_Embedded_Compute/02_android_compute.adoc[Android 上のコンピュート] を参照。そうしたタイルベース部品では、データをメモリではなくサブグループレジスタに保つことは速度の勝利だけでなく、電力を直接節約し熱スロットリングを減らします。

本章では、サブグループの隠れた力を探ります。

1.  **呼び出し間通信**: サブグループシャッフル、ブロードキャスト、算術演算を使い、メモリを完全に迂回してレジスタ経由で直接データを交換する。
2.  **サブグループパーティショニング**: 「Ballot」と「Match」演算を実装し、束全体で複雑な分岐とデータフィルタリングを行う。
3.  **非ユニフォームインデックス付け**: サブグループ内のスレッドごとに異なる可能性のあるリソース配列に安全にアクセスするため、現代の Vulkan 機能を活用する。

== バリアを超えて

サブグループ演算により、小規模なデータ交換向けに「バリアフリー」なカーネルを書けます。単一の float を共有するためだけにワークグループ内の全スレッドをバリアで待たせる代わりに、サブグループシャッフルでその値を即座に渡せます。

これにより次が得られます。

*   **より高い性能**: 待機スレッドによるパイプラインストールがない。
*   **より低いレイテンシ**: データ交換がレジスタ速度で起きる。
*   **より大きな柔軟性**: アルゴリズムがより「ウェーブ意識」になり、ハードウェアのネイティブ実行幅に適応できる。
