= AI + RenderDoc 統合: 自動化された GPU 分析

== はじめに: 時間を凍らせる

**RenderDoc** は Vulkan デバッグの決定的ツールです。時間を凍らせ、GPU 上のあらゆるドローコール、ディスクリプタセット、パイプラインステートオブジェクトを検査できます。しかし複雑なシーンでは、単一フレームに何千ものイベントが含まれ得ます。「Shadow Acne」や「Missing Texture」を引き起こしている 1 つのドローコールを見つけるのは、しばしば気が遠くなる手動タスクです。

**協調エンジニアリング** では、RenderDoc の生の状態追跡と AI エージェントの推論力を組み合わせます。「アーティファクト狩り」を手動検索から **自動化された状態監査** へ変えます。

== コア概念: プログラマティック検査

AI を RenderDoc と統合するには、GUI を超え **RenderDoc コマンドラインツール（`renderdoccmd`）** を活用します。これにより自律エージェント（**Goose** など）が GPU キャプチャをプログラマティックに「閲覧」できます。

「ブラックスクリーン」に遭遇する典型的なシナリオを考えてください。ドローコールが起きていることは分かっていますが、50 のパイプラインステートのどれが誤設定か分かりません。伝統的ワークフローでは、RenderDoc UI をクリックして回るのに 1 時間かかります。

エージェンティックソリューションでは、`rdc-cli` を実行できるツールをエージェントに与え、単にキャプチャを分析するよう頼みます。例えば次のように言うかもしれません。*「最新の RenderDoc キャプチャを分析して。『MainPass』のドローコールを見つけ、ディスクリプタセットを監査して。なぜフラグメントシェーダがディフューズテクスチャにゼロ値を受け取っているのか？」*

エージェントは単に画面を「見る」だけではありません。**構造化パイプラインステート**（JSON または Markdown）をエクスポートし、C{pp} ソースコード（**コンテキストブリッジ** 経由）と比較し、不一致を特定します。

== 実世界体験: 「ゴースト」テクスチャ

欠けたテクスチャをデバッグしているところを想像してください。RenderDoc ではテクスチャはバインドされていますが、シェーダ出力は黒です。

=== ステップ 1: 自動化された状態検索
タブをクリックする代わりに、アシスタントに尋ねます。*「Goose、RenderDoc ツールを使い、『G-Buffer』パスの 3 番目のドローコールの `VkSampler` 状態を確認して。」*

=== ステップ 2: 発見
エージェントはサンプラ状態をエクスポートし、`minLod` が `10.0f` に設定されている一方、テクスチャは 5 ミップレベルしかないことを見つけます。
**AI 診断:** *「存在しないミップレベルからサンプリングするよう GPU を強制するサンプラをバインドしている。これがテクスチャが黒く見える理由だ。`VulkanContext.cpp` の `SamplerBuilder` を確認して。」*

=== ステップ 3: 自動化された修正
エージェントはソースコードにもアクセスできるため、問題を伝えるだけではなく、直すことも申し出ます。バインドされたイメージの実際のミップ数に `minLod` を自動クランプするよう、サンプラ作成論理をリファクタします。

== 実践チュートリアル: Goose を RenderDoc に接続する

GPU 分析を真に自動化するには、エージェントに **RenderDoc コマンドラインツール（`renderdoccmd`）** を使う能力を与えなければなりません。これにより AI は `.rdc` キャプチャファイルをテキストドキュメントのように「検査」できます。

「RenderDoc ブリッジ」を構築する次のステップに従ってください。

1.  **RenderDoc が PATH にあることを確認:**
    `renderdoccmd` バイナリ（Windows では `renderdoccmd.exe`）がターミナルからアクセス可能でなければならない。
+
[source,bash]
----
# On Windows (PowerShell)
$env:Path += ";C:\Program Files\RenderDoc"
# On Linux
export PATH=$PATH:/usr/bin/renderdoc
----

2.  **フレームをキャプチャ:**
    RenderDoc GUI を使い Vulkan アプリからフレームをキャプチャ。`debug_capture.rdc` として保存。

3.  **検査の指示:**
    Goose セッションで、アシスタントにキャプチャを分析するよう頼む。次のプロンプトはすべてのドローコールを列挙し、特定シェーダを使うものを見つけるよう設計されている。
+
[source,text]
----
Goose, list all draw calls in 'debug_capture.rdc' using 'renderdoccmd'.
Find the one using 'MainPass' and export its full
pipeline state to a structured file named 'state.txt'.
----

4.  **論理監査の実行:**
    状態がエクスポートされたら、GPU の現在構成をソースコードと相関させるよう AI にタスクを与える。
+
[source,text]
----
Compare 'state.txt' with the 'VkGraphicsPipelineCreateInfo'
logic in 'Renderer.cpp'. Are there any mismatches in the
depth test settings or cull mode that could explain
why our geometry is missing?
----

5.  **「修正からソース」ステップ:**
    AI はキャプチャで `depthTestEnable` が `VK_FALSE` である一方、コードは `VK_TRUE` に設定 *すべき* であることを特定する。次にプロジェクトを検索し、デプステストを誤って無効にしているコード行を見つけ、直すことを申し出る。

== なぜ構造化状態が重要か

AI は RenderDoc GUI を効果的に「見る」ことはできませんが、**構造化データ** の達人です。GPU 状態を AI がパースできる形式へエクスポートすることで、冗長な状態変更を特定する能力を与えます。例えば、介在するドローコールなしに同じパイプラインをコマンドバッファ内で 5 回リバインドしていることに気づけます。

また、頂点バッファオフセット（例: 12 バイト）がハードウェア要件（例: 16 バイトアラインメント）を満たすか確認することでメモリアラインメントを監査できます。さらに、バリデーションレイヤエラーを取り、そのエラーが発生した RenderDoc タイムライン上の正確な瞬間を見つけることで VUID をクロス参照できます。

== まとめ: 自動化された GPU 分析

AI を RenderDoc と統合することで、グラフィックスデバッグを高レベルな会話へ変えます。午後を GUI の入れ子メニューをクリックして過ごすのではなく、単に AI に状態不一致を **見つけて説明** するよう頼みます。これはグラフィックスエンジニアにとって大きな効率向上です。
