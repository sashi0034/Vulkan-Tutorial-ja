= フェーズ 3: 自動化された進化（レビューフェーズ）

== はじめに: エンジンの進化

Vulkan コードは静的ではありません。Vulkan 1.0 の「ベストプラクティス」（あらゆる小さなサブパスに `VkRenderPass` と `VkFramebuffer` を使うなど）は、**Dynamic Rendering** を優先する Vulkan 1.3 では「レガシー」と見なされます。

ワークフローのこの最終フェーズでは、「共コーディング」から **監督** へ移ります。**自律エージェント**（**Goose** など）を使い「深いリファクタ」を行います。単一関数に焦点を当てる IDE アシスタントと異なり、エージェントはエンジン全体を「見て」、初期化コードから最終ドローコールまで主要なアーキテクチャシフトが一貫して実装されることを保証できます。

== コア概念: 「エージェンティックシフト」

**フェーズ 3** では役割が逆転します。もはや AI の助けでコードを書いているのではなく、AI にエンジニアリング目標を与え、**テクニカルレビュア** として働きます。

*   **問題:** エンジンを GLSL から **Slang** へ移行したい。すべてのシェーダの更新、ディスクリプタバインディング論理の変更、シェーダコンパイルビルドスクリプトの修正が必要。人間なら退屈で誤りやすい 1 週間の作業。
*   **エージェンティックソリューション:** エージェントにターミナルとプロジェクトへのアクセスを与える。言う。*「`post_process/` フォルダを Slang へ移行して。`CMakeLists.txt` を Slang コンパイラを使うよう更新し、新しいリフレクションデータを使うよう C{pp} をリファクタし、コンパイルエラーを直して。」*

エージェントは変更を「提案」するだけではありません。ファイルを読み、リファクタを適用し、コンパイラを実行し、エラーを見て、タスクが完了するまで **反復** します。

== なぜ自律エージェントを使うか？

Vulkan コードのリファクタはしばしば「カードの家」タスクです。`VkRenderPass` を取り除くなら、次も更新しなければなりません。

1.  `VkPipelineRenderingCreateInfo` を使うよう `VkGraphicsPipelineCreateInfo`。
2.  `vkCmdBeginRenderPass` 呼び出しを `vkCmdBeginRendering` へ。
3.  影響を受けるすべてのメモリバリアのレイアウト遷移。

自律エージェントは複数ファイルにわたる **「エラーチェーン」** を辿り、自分が作ったコンパイルエラーを直してシステムが新しい安定状態に達するまで続けられるため、これに独自に適しています。

== 実世界体験: 「Dynamic Rendering」移行

エージェントが Bloom システムの Dynamic Rendering への移行をどう扱うかを見ます。

=== ステップ 1: ガードレールの設定
開始前に、クリーンな Git ブランチにいることを確認します。
**あなたの役割:** スコープを定義する。*「`BloomRenderer` クラスと関連シェーダのみ変更。コア `VulkanDevice` 初期化にはまだ触れない。」*

=== ステップ 2: エージェンティックループ
命令を与える。*「`BloomRenderer.cpp` を Dynamic Rendering を使うようリファクタして。すべての `VkRenderPass` メンバを `VkPipelineRenderingCreateInfo` に置き換えて。イメージバリアが `VkImageMemoryBarrier2` を使うよう更新されていることを保証して。」*

ターミナルを見ます。エージェントは:

*   コードを読み、依存チェーンを特定する。
*   パイプラインを変更する。
*   `ninja` を実行する。
*   synchronization2 向けの `vulkan_core.h` 更新をインクルードし忘れたため「変数が見つからない」エラーを見る。
*   インクルードを直し再ビルドする。

=== ステップ 3: テクニカルレビュー
ビルドは通っているが、コードは *良い* か？
**あなたの役割:** diff をレビューする。エージェントが「動かす」ためにすべてに `IMAGE_LAYOUT_GENERAL` を使っていることに気づく。訂正する。*「Bloom パスのレイアウトを、レンダーターゲットに `ATTACHMENT_OPTIMAL`、ソーステクスチャに `READ_ONLY_OPTIMAL` を使うよう最適化して。」*

== まとめ: 進化するエンジン

フェーズ 3 は **協調ループ** の集大成です。「重いリファクタ」に自律エージェントを使うことで、エンジンがレガシーパターンに囚われないことを保証します。グラフィックステクノロジーの **イノベーション** に集中し、AI が進化の複雑な詳細を管理します。

セクション 5 の完了により、**機能ライフサイクル**——クラウドベース設計からローカルベース実装、自律進化まで——をマスターしました。次のセクションでは、**AI と RenderDoc によるプロレベルデバッグ** の「暗黒の技」を探ります。
