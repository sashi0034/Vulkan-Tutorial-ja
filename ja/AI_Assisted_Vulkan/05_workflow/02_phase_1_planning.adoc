= フェーズ 1: システム設計（設計フェーズ）

== はじめに: 飛ばされがちなステップ

伝統的な Vulkan 開発では、「計画」フェーズはしばしば最も飛ばされるステップです。開発者はすぐに `VkGraphicsPipelineCreateInfo` ボイラープレートの記述へ飛び込み、途中でディスクリプタ管理が実装したい新しいバインドレステクスチャシステムをサポートしないことに気づきます。

**協調エンジニアリング** では、計画は任意ではありません——それは **基盤** です。高推論モデルでコードを書く *前に* 機能を設計することで、実装フェーズが一貫した仕様どおりのロードマップに根ざすことを保証します。

== コア概念: 「契約」

**協調エンジニアリング** では、計画フェーズを **技術契約** として扱います。これはコードが書かれる前にシステムの「法則」を定義する詳細なロードマップです。Vulkan 機能の成功するロードマップは、いくつかの重要な柱に取り組まなければなりません。

第一の柱は **リソースライフサイクル** です。どのバッファ、イメージ、サンプラが必要か、誰がそれらを「所有」するか、いつ作成・破棄されるかを正確に定義しなければなりません。これなしでは、AI 支援実装はメモリリークや use-after-free エラーに苦しみがちです。

第二の柱は **同期戦略** です。Vulkan 開発がしばしば失敗する場所です。ロードマップは機能のすべてのステージに必要な `VkImageMemoryBarrier2` 遷移を明示的に述べるべきです。計画フェーズでこれらの遷移を定義することで、後で正しい同期コードを生成するのに必要な「構文的真実」を AI に与えます。

最後に、**インタフェース境界** を定義しなければなりません。この新機能は既存の `VulkanDevice`、`Swapchain`、`CommandPool` ラッパとどう相互作用するか？ 既存の C{pp} クラスシグネチャを AI に与えることで、大規模リファクタなしに新コードがエンジンアーキテクチャへ滑らかに統合されることを保証します。

== なぜ高推論モデルを使うか？

アーキテクチャ計画には **システム推論**——複数の相互作用するシステムを同時に心に保つ能力——が必要です。小さなローカルモデルは単一関数の記述に優れますが、エンジン全体にわたる変更の高レベル含意に苦戦し得ます。

**Claude 4.6** や **GPT-5.3** のようなモデルはこのフェーズに優れます。エンジンの既存アーキテクチャの 500 行を分析し、既存の同期を壊したりターゲットハードウェアのメモリアラインメント規則に反したりせずに新機能を統合する方法を提案できます。

== 実世界体験: Bloom パイプラインの設計

協調ループを使い、現代的な **Bloom ポストプロセスパイプライン** を設計しましょう。

=== ステップ 1: 「深いコンテキスト」の提供
単に *「ブルームシステムを設計して。」* と頼まないでください。代わりに **コンテキストブリッジ** の結果を AI に与えます。`VulkanContext.hpp` ファイルとエンジンのメモリ管理ヘッダを提供します。
**プロンプト:** *「Bloom パスを実装している。エンジンは Dynamic Rendering とメモリに VMA を使う Vulkan 1.3 を使っている。これが `Renderer` クラスだ。13-tap ダウンサンプル/アップサンプルチェーンを使う Bloom パイプラインを設計して。クラス構造と各ミップレベルのイメージレイアウト遷移を定義するロードマップを提供して。」*

=== ステップ 2: ロードマップの監査
AI が計画を生成し、人間のリードとして **意味監査** を行います。例えば AI が Bloom パスに `STORAGE_IMAGE_OPTIMAL` を提案したら、ターゲットモバイルハードウェア（Mali GPU など）は代わりに入力アタッチメント付き `ATTACHMENT_OPTIMAL` の方が性能が良いと気づくかもしれません。その後、モバイルで帯域を節約するためアップサンプルパスにサブパス風の入力アタッチメントを使うようロードマップを調整するよう AI に訂正を伝えられます。

=== ステップ 3: 「契約」の確定
結果は検証済みの `bloom_design.md` ファイルです。このファイルは、次のフェーズで実際のコードを生成するために使う詳細な技術仕様を含みます。

== 実践チュートリアル: グラフィックス機能ロードマップの構築

**フレームグラフ** や **Clustered Renderer** のような複雑な機能を AI で計画するのは特定のスキルです。高レベル会話と一連のエンジニアリング青写真の違いです。

次の機能向けに高品質な「契約」を作る次のワークフローに従ってください。

1.  **コンテキスト読み込み:**
    **MCP ツール**（特に `mcp-Vulkan` サーバ）を使い、構築したい機能の最新仕様を照会する。これにより AI がレガシーパターンではなく現行の Vulkan 1.3 または 1.4 拡張を使うことを保証する。

2.  **「システム境界」を定義:**
    エンジンの制約を明確に述べる。
+
[source,text]
----
I am adding a Bloom pass.
Our engine uses a fixed 24-bit depth buffer.
We use VMA for all allocations.
We are using Dynamic Rendering (no manual RenderPasses).
----

3.  **多層ロードマップを要求:**
    アシスタントに計画を 3 つの特定レイヤへ分解するよう頼む。
+
[source,text]
----
Generate a 3-layer roadmap:
1. Architectural Design: (Class hierarchy, state management)
2. Data Flow: (Descriptor layout, push constants, VMA allocation flags)
3. Synchronization Map: (Pipeline barriers, layout transitions, and image memory dependencies)
----

4.  **洗練とコミット（「真実」ファイル）:**
    人間のリードとして AI の提案を監査する。計画が固まったら、AI にリポジトリ内の `DESIGN_DOC.md` ファイルへ書かせる。このファイルを実装フェーズの「地上真理」として使う。

== まとめ: システム設計のマスター

フェーズ 1 は **設計レベルでの人間-AI 協調** です。高エンド LLM の推論力を活かして堅実な「契約」を作ることで、Vulkan の圧倒的な複雑さを一連のよく定義された実装タスクへ変えます。C{pp} の 1 行も書かれる前に、実質的にアーキテクチャを「事前設計」しています。
