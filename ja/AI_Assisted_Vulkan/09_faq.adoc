= よくある質問とトラブルシューティング（FAQ）

== はじめに: 協調の障害物コース

AI 支援 Vulkan 開発の旅を始めると、必然的に障害に遭遇します。それらは単なる技術バグではなく、**協調エンジニアリング** パラダイムの「文化」と「プライバシー」のハードルです。本章では最もよくある懸念に答え、安定・安全で高い効果の AI 強化グラフィックスラボを構築するためのトラブルシューティングのヒントを提供します。

== プライバシーと IP セキュリティ

**Q: 「プロプライエタリなレンダリングエンジンで作業している。ソースコードが漏れたり、他モデルの学習に使われたりしないようにするには？」**

**A: 「ローカルゲーティング」戦略。**
プライバシーを最大化する唯一の方法は、日常の主要ドライバとして **ローカル推論エンジン**（Ollama、LM Studio）を使うことです。**Qwen 3-Coder** のようなモデルを自前ハードウェアで動かすことで、シェーダと C{pp} ファイルはマシンを離れません。よくある「ハイブリッドルール」は、**実装** にローカルモデルを使い、高レベルの **システム設計** には Claude や GPT のようなクラウドモデルを予約し、将来の学習にデータが使われないよう **API ベースアクセス** を使うことです。

== 精度と幻覚

**Q: 「AI が『発明した』拡張やレガシー Vulkan 1.0 パターンを提案し続ける。自分で書くより AI のコードを直す時間の方が長い。」**

**A: コンテキストが王。**
幻覚はほぼ常に **コンテキストギャップ** の症状です。解決するには、**Model Context Protocol（MCP）** でアシスタントを `vk.xml` レジストリに直接リンクしてください。この地上真理なしでは、AI は事実上推測しています。また、セッション開始時に次のような **ベースラインプロンプト** を明示してください。*「Vulkan 1.3 と Dynamic Rendering を使っている。レガシーサポートを明示的に求めない限り `VkRenderPass` や `VkFramebuffer` を提案しないこと。」* 複雑なプロジェクトで一貫した結果を得るには、エンジン固有のスタイルとメモリ割り当てパターンを「デフォルト」にする小さな **LoRA アダプタ** も検討してください。

== マルチモーダルとビジョンの限界

**Q: 「スカイボックスの微妙なバンディングのスクリーンショットを送ったが、AI は画像がクリアだと言う。なぜバグが見えないのか？」**

**A: 解像度圧縮。**
ほとんどのマルチモーダルモデルは画像を小さな「ビジョンバッファ」（しばしば 1024x1024 以下）へダウンスケールします。4K フレームの数ピクセルしかない微妙なアーティファクトは、この圧縮で完全に平滑化され得ます。解決策は **クロップとズーム**——AI に送る前にアーティファクトに完全に焦点を当ててスクリーンショットをクロップすることです。問題が AI のビジョンバッファの 2% ではなく 80% を占めれば、モデルははるかに確実に識別します。

== ハードウェアと VRAM 管理

**Q: 「AI がコード生成を始めると Vulkan アプリが『Device Lost』（TDR）エラーでクラッシュし続ける。新しい GPU が必要か？」**

**A: リソース競合の罠。**
おそらく AI とレンダラが同時に GPU の 100% を奪い合い、**Timeout Detection and Recovery（TDR）** イベントに当たっています。緩和するには、4-bit（Q4_K_M）量子化で AI の VRAM フットプリントを減らし、推論エンジン（Ollama や LM Studio など）で AI が使う GPU レイヤ数に上限を設け、デスクトップとレンダラ用のコンピュート余裕を残してください。プロ用途の究極解は **専用 AI GPU** です。セカンダリスロットの古いカードでも **Llama 4** のような 17B モデルをホストでき、プライマリのハイエンドカードを Vulkan 専用に残せます。

**Q: 「Ollama でモデルをダウンロードしたが、動かないかシステムがクラッシュする。何が起きた？」**

**A: VRAM/GPU RAM 限界。**
ダウンロードしたからといって、現在のハードウェアで動くとは限りません。利用可能 VRAM を超えるモデル（30B Qwen 3-Coder など）を読み込もうとすると、Ollama はレイヤをシステム RAM へ「オフロード」しようとし、著しく遅くなり、システム全体のフリーズや Vulkan アプリの「Device Lost」につながり得ます。実行前に常にモデルサイズを **VRAM 予算式**（xref:AI_Assisted_Vulkan/03_model_selection_specialization/03_hardware_vram.adoc[第 3.3 章]）と照合してください。

**Q: 「Ollama WebUI（localhost）と Windows ネイティブアプリのどちらを使うべきか？」**

**A: 性能はネイティブ、管理は WebUI。**
WebUI（`localhost` 上）はモデル管理に馴染みのあるブラウザ UI を提供しますが、グラフィックス開発者には一般に **Windows ネイティブアプリ**（または CLI）が好まれます。ネイティブアプリは Windows のハードウェアスケジューリングに直接アクセスでき、IDE 拡張や Goose のようなエージェントとの低レイテンシ通信をしばしば提供します。Vulkan レンダラ稼働中に AI 応答が遅れる場合は、ブラウザ抽象ではなくネイティブバックグラウンドサービスを使っているか確認してください。

**Q: 「Goose に拡張（Filesystem Access など）を追加したが動かない。どう有効化するか？」**

**A: 「レシピ」アクティベーション。**
一部の Goose バージョンでは、設定に拡張を追加するだけでは足りず、アクティブな **Recipe** で明示的に有効化する必要があります。「Filesystem Access」（HPI）拡張が応答しない場合、Goose 設定を開き、使っている特定タスクプロファイル内で拡張がチェックまたは「許可」されているか確認してください。この有効化なしでは、エージェントはツールの存在を「知って」いても、現セッションで使う権限を欠くことがあります。

== まとめ: 情報に基づくグラフィックスエンジニアリング

**協調エンジニアリング** への移行にはマインドセットの転換が必要です。もはや単なる「コーダ」ではなく、**インフラマネージャ** であり **監査リード** です。これらのトラブルシューティング技法をマスターすることで、AI アシスタントがフラストレーション源ではなく重要な資産であり続けることを保証します。
