= ハードウェアと VRAM 予算: リソース戦略

== はじめに: VRAM をめぐる戦い

Vulkan 開発はハイパフォーマンスの追求です。ローカル AI アシスタントを導入すると、もはや CPU サイクルをめぐる他アプリとの競争だけではなく、自分のエンジンと **VRAM** をめぐって競合します。

「協調エンジニアリング」の時代、GPU は共有リソースです。成功するには「プラグアンドプレイ」を超え、AI アシスタントが単一フレームを描くのに必要なメモリを Vulkan アプリから奪わない **リソース戦略** を開発しなければなりません。

== コア概念: VRAM 競合と「スワップの罠」

ほとんどの開発者は VRAM を静的なバケツだと考えます。16GB あり、10GB 使い、6GB 残る。しかし AI 支援環境では、GPU は 2 つの「重い」ワークロード間で常にコンテキストスイッチしています。

1.  **LLM 推論エンジン:** コード生成のため何十億もの重みをメモリに保持。
2.  **Vulkan アプリケーション:** グラフィックス表示のためテクスチャ、バッファ、レンダーターゲットを保持。

合計 VRAM 要件が物理 GPU メモリを超えると、システムは **スワップの罠** に当たります。OS は AI の重みや Vulkan テクスチャをはるかに遅いシステム RAM（DDR4/5）へ移し始めます。グラフィックス開発者にとってこれは壊滅的です。AI 生成は秒間 50 トークンから 2 へ落ち、Vulkan アプリのフレームレートは一桁になります。

== 技術式: 「真の予算」の計算

スワップの罠を避けるには、モデルサイズ以上を考慮する式で VRAM 予算を計算しなければなりません。**KV Cache**（AI の短期記憶）も考慮しなければなりません。

[source,text]
----
Total VRAM - (OS/Desktop + Vulkan Peak + (Model Size * Quantization) + KV Cache) = Safety Margin
----

*   **OS/Desktop:** Windows/Linux で通常 **1.5 - 2 GB**。
*   **Vulkan Peak:** 重いデバッグセッション中にエンジンが使う最大 VRAM。
*   **Model Size * Quantization:** **30B モデル**（**Qwen 3-Coder** など）の **4-bit（Q4_K_M）** 量子化は約 18GB、**12B モデル**（**Mistral-Nemo** など）は約 8GB。
*   **KV Cache Tax:** 最も見落とされがちな部分です。AI とチャットするにつれ、会話の前の部分を覚えるメモリが必要です。**16k コンテキストウィンドウ** では追加で **1–2 GB** の VRAM を消費し得ます。

**実世界体験: 24GB VRAM 予算**
**RTX 4090（24GB）** を使っているところを想像してください。

*   **OS + Vulkan アプリ:** 6GB。
*   **Qwen 3-Coder（30B）@ Q4_K_M:** 18GB。
*   **予算:** `24 - (6 + 18) = 0GB`。
*   **結果:** 安全マージンは **ゼロ**。長い質問をして KV Cache が成長するとすぐにスワップが始まります。直すには、より小さなモデル（**Mistral-Nemo 12B** や **Llama 4（17B）**）を使うか、AI のコンテキストウィンドウを **4,096 トークン** に減らし、その 1–2 GB の「息継ぎの余地」を取り戻す必要があります。

== 量子化: モデルフットプリントの最適化

モデルを完全な非圧縮精度（FP16）で動かす必要はありません。代わりに **量子化** で各重みに使うビット数を減らします。ニューラルネットワーク向けのテクスチャ圧縮（例: BC7 や ASTC）のように考えてください。わずかな数学精度をトレードし、メモリフットプリントを大幅に減らします。

**グラフィックス開発者向け「スイートスポット」:**
**4-bit（Q4_K_M）** または **5-bit（Q5_K_M）** 量子化を推奨します。

*   **なぜか？** 4-bit ではモデルサイズは 75% 減りますが、複雑な C{pp} コードについての推論能力はほぼ 98% 保たれます。4-bit より下（例: 2-bit）へ行くと、AI は「筋を失い」、重みに Vulkan 状態機械遷移の微妙な関係を表す精度が足りなくなるため同期論理で幻覚につながります。

== 実践チュートリアル: モデルの量子化方法

ほとんどの開発者は HuggingFace から事前量子化モデルをダウンロードしますが、プロのグラフィックスエンジニアはこの「圧縮」を自分で行う方法を知っているべきです。これにより研究ラボがリリースした「生」モデル（FP16）を取り、ローカル GPU 向けに即座に最適化できます。

主なツールは **llama.cpp** です。ステップバイステップのプロセスは次のとおりです。

1.  **ツールのクローンとビルド:**

    [source,bash]
    ----
    git clone https://github.com/ggerganov/llama.cpp
    cd llama.cpp
    make -j  # Builds the quantize and convert binaries
    ----

2.  **GGUF 形式へ変換:**
    ほとんどのモデルは HuggingFace の `SafeTensors` 形式でリリースされます。まずローカル推論向けに最適化された `GGUF` 形式へ変換する必要があります。

    [source,bash]
    ----
    python3 convert_hf_to_gguf.py models/my_vulkan_model/
    ----

3.  **量子化の実行:**
    ここで圧縮を適用します。速度と精度のバランスを取る「Medium」4-bit 量子化である `Q4_K_M` メソッドを使います。

    [source,bash]
    ----
    ./quantize models/my_vulkan_model/model-f16.gguf models/my_vulkan_model/model-Q4_K_M.gguf Q4_K_M
    ----

4.  **検証:**
    結果のファイルは大幅に小さくなります。**32GB** の FP16 モデルは約 **18GB** になります。このファイルを **Ollama** や **LM Studio** に直接読み込み、Vulkan スワップチェーン用の余分な VRAM「息継ぎの余地」を楽しめます。

== 競合の緩和: プロ戦略

ハードウェアが限られているなら、**リソースマネージャ** にならなければなりません。

1.  **コンテキスト制限:** 推論エンジン（Ollama/LM Studio）で、`context_length` を手動で **8192** に設定。これは通常数ファイルの Vulkan コードには十分で、デフォルト設定より 1.5GB の VRAM を節約できるほど小さいです。
2.  **レイヤオフロード（GGUF 戦略）:** **GGUF** 形式を使う場合、モデルを「分割」できます。モデルが 60 レイヤあり GPU がいっぱいなら、10 レイヤを CPU へ移せます。AI は 20% 遅くなりますが、Vulkan アプリは 3GB の VRAM を取り戻しスタッタが止まります。
3.  **「セカンダリ GPU」ハック:** 古い 8GB または 12GB カードがあれば、2 つ目の PCIe スロットに取り付けます。**プライマリ GPU** を Vulkan アプリに使い、**Ollama** を排他的に **セカンダリ GPU** に向けます。これがプロエンジン開発者向けの究極の「競合なし」セットアップです。

== まとめ: 戦略的ラボ

ハードウェア予算は、安定した AI 支援環境の「秘密のソース」です。VRAM フットプリントを理解し、KV Cache と量子化を積極的に管理することで、AI アシスタントがグラフィックスエンジンのリソース食いライバルではなく役立つ共著者であり続けることを保証します。
