= ベースモデルの選択: AI モデル戦略

== はじめに: グラフィックス開発者の制約

Vulkan 開発向けベースモデルの選択は、単に「最大」のモデルを選ぶことではありません。グラフィックスプログラミングの世界では独自の制約——**GPU リソース競合**——に直面します。AI モデルが消費する VRAM の 1 ギガバイトごとに、Vulkan テクスチャ、頂点バッファ、フレームバッファに使えないギガバイトが生じます。

成功するには、汎用モデルを超え、推論深度と管理可能なハードウェアフットプリントのバランスを取る特化モデルを選ばなければなりません。

== 概念: モデルの評価方法

急速に進化する AI の風景では、「最良」のモデルは数か月ごとに変わります。効果的であり続けるには、現在人気のモデル一覧を暗記するだけでは足りません。代わりに、グラフィックス開発向けの 3 つの重要な基準に基づき新しいモデルを評価する方法を学ばなければなりません。

=== 1. VRAM 税: パラメータ数対量子化

最初に問うべきは、*「このモデルは Vulkan アプリと並んで私のハードウェアに収まるか？」* 一般に、パラメータ数が多いほど「知能」と推論能力が高くなります。しかし巨大な 70B モデルは、Vulkan アプリから VRAM を奪いはるかに遅いシステム RAM を強制するなら逆効果です。

これらのモデルをコンシューマハードウェアに収めるため、通常 **GGUF** や **EXL2** のような形式で **量子化**（圧縮）に頼ります。4-bit（Q4）量子化はしばしば「ちょうどよい」ゾーン——モデルのメモリフットプリントを 75% 減らしつつ、推論精度の低下は無視できる程度です。

=== 2. 推論深度: 「構造的推論テスト」

モデルは Vulkan の複雑な状態機械を理解できるか？ 新しいモデルをテストするには、自明でないアーキテクチャ質問をします。例えば: *「コンピュートシェーダがストレージバッファへ書き、次のパスで頂点バッファとして使われるコンピュート対グラフィックス依存の同期要件を説明して。」*

このテストに失敗するモデルは、しばしば汎用の `vkDeviceWaitIdle` や、必要なアクセスマスクを定義しない単純な実行バリアを出します。対照的に「合格」モデルは、`VK_ACCESS_SHADER_WRITE_BIT` から `VK_ACCESS_VERTEX_ATTRIBUTE_READ_BIT` への遷移要件を正しく特定し、データの可視性と実行順序を保証する適切なパイプラインステージを提案します。

NB: AI モデルは常に改善しており、本チュートリアル執筆時点で真であることも現在は違う可能性があるため、このターゲットは時間とともに変わります。したがって、開発の過程で本チュートリアルが教えるツールと技法を通じて新しいモデルとその能力に適応できると想定してください。私たちは正確な版ではなくエコシステムを教えています。企業環境向けにローカルサーバを立てられるかもしれず、ここでの SLM ターゲティング情報によりニーズに合わせてさらに洗練できます。使うモデルと必要な追加学習を、開発ニーズに応じて多かれ少なかれ助けを与えるハイパーパラメータとして扱うことを推奨します。

=== 3. コンテキストウィンドウ: 「エンジンメモリ」

Vulkan エンジンは大きいです。小さなコンテキストウィンドウ（例: 4,000 トークン）のモデルは、パイプラインセットアップを読み終わる頃にはメモリアロケータを「忘れ」ます。2026 年の標準では、少なくとも **32k から 128k のコンテキストウィンドウ** を持つモデルを探してください。これによりアシスタントは特定実装に取り組みながらレンダラのヘッダファイル全体をアクティブメモリに保て、プロジェクト固有のアーキテクチャパターンを意識し続けられます。

== 2026 年のベストインクラス（例として）

将来のモデルには上記の評価基準を使うべきですが、これらの特性を体現する現在のリーダーは次のとおりです。

=== 1. Qwen 3-Coder（30B）: 高推論モデル

**Qwen 3-Coder（30B）** は現在、C{pp} 向けに世界で最も能力の高いローカル非クラウドコーディングモデルです。

**実世界体験: バインドレス移行**
エンジンを **バインドレステクスチャ** システムへリファクタしているところを想像してください。Descriptor Set Layout バインディング、`descriptorIndexing` 機能、GLSL アクセスが SPIR-V へどう翻訳されるかの機微の深い理解が必要です。

このリファクタを Qwen 3-Coder に頼むと、単なるボイラープレート `VkDescriptorSetLayoutCreateInfo` ではありません。シェーダコードを分析して `sampler2D textures[]` の使い方を見て、バリデーションエラーを防ぐ正確な `DESCRIPTOR_BINDING_PARTIALLY_BOUND_BIT` フラグを提案します。その推論深度により、より小さなモデルが見落とす同期問題を捕まえられます。複雑なアーキテクチャ計画と Vulkan 1.3/1.4 機能実装に理想的です。ただし重要なリソース——Q4_K_M 量子化で約 20GB の VRAM——を必要とし、24GB GPU の開発者やローカルサーバ環境で作業する人に最も適します。

=== 2. Llama 4（17B）: 高速アシスタント

入力中にほぼ即時のフィードバックが必要な開発の「インナーループ」では、速度がすべてです。**Llama 4（17B）** は現在の低レイテンシコーディングのチャンピオンです。

**体験:** 新しい `VkPipelineShaderStageCreateInfo` 配列を打っているところです。GPU 上でローカルに動く Llama 4 は、最初の構造体を書き終える前に次の 10 行を予測します。頂点とフラグメントシェーダを使っていることを正しく特定し、プロジェクトコンテキストから正しいエントリポイント名を自動で引き、このパスで specialization constants を使っていないと見て `pSpecializationInfo` を `nullptr` に設定します。

低い VRAM フットプリント——約 6GB のみ——により、中級ハードウェア（12GB–16GB GPU）上で重い Vulkan アプリと快適に並走でき、リアルタイムオートコンプリート、ユニットテスト、迅速なシェーダ反復に最適です。

=== 3. Mistral-Nemo（12B）: 「コンテキストの王」

NVIDIA と Mistral AI の協業である **Mistral-Nemo（12B）** は、巨大な **128k コンテキストウィンドウ** での高効率推論向けに特に調整された、ローカル開発の強力なエンジンです。

**実世界体験: エンジン全体リファクタ**
シーングラフから最終ポストプロセスパスまで何十ものファイルに触れる新しい **Global Illumination** システムを実装しているところを想像してください。現在のファイルだけを見るのではなく、アーキテクチャ階層全体のコンテキストを保つアシスタントが必要です。

Mistral-Nemo（12B）はこうした「ロングコンテキスト」シナリオに優れます。アクティブメモリに最大 128,000 トークンを保てるため、レンダラのヘッダディレクトリ全体を「読み」、コードベース全体でアーキテクチャ的に一貫した提案を提供できます。大規模リファクタと、より小さなモデルが糸を失うような複雑なクラス階層のナビゲートに特に効果的です。約 8GB–10GB（Q4）のスリムな VRAM フットプリントで、超高速 17B モデルとリソース重い 30B+ モデルの完璧な中間を提供し、複雑で複数ファイルの Vulkan システムで作業する開発者に理想的です。

== 対話スタイル: 意図対明示的指示

クラウドモデルとローカルモデルの重要な違いは、「話しかけ方」にあります。この区別を理解することが、開発ワークフロー中のフラストレーション回避の鍵です。

=== クラウド: 高レベル意図

**Claude 4.6** や **GPT-5.3** のようなクラウドモデルは、高い抽象レベルを扱うよう設計されています。人間のエンジニアに対するように対話でき、「行間を読む」ことと会話的リクエストを構造化タスクへ翻訳することに優れます。例えば次のようにプロンプトすると: *「遠い地形で Z-fighting が見える。標準の 24-bit デプスバッファを使っている。最初に何を見るべきか？」*、モデルは視覚アーティファクトを直す意図を理解します。デプスバッファ精度や近/遠平面比のようなありそうな犯人を特定し、投影行列セットアップの技術詳細を指定せずに優先チェックリストを提供します。

=== ローカルモデル: 明示的指示（指示追従モデル）

ローカルモデルは、**Qwen 3-Coder（30B）** のような先進的なものでも、一般により具体的で冗長な指示を必要とします。カジュアルな一文から意図を推論する能力は低く、詳細な技術仕様に従う高度な実装者として扱うときにより効果的です。ローカルモデル向けにより効果的なプロンプトは次です。*「Z-fighting が起きている。次の投影行列計算を分析して。近平面が近すぎないか（現在 0.01）、遠平面が遠すぎないか（10000.0）を確認して。比が 1:1,000,000 を超えるなら対数デプスバッファ実装を提案して。頂点シェーダ更新の GLSL を提供して。」* これらの明示的制約を与えることで、ローカルモデルは会話的抽象ではなく技術詳細で力を発揮するため、焦点と精度を保てます。

== 翻訳戦略: クラウドからローカルへのパイプライン

「協調エンジニアリング」パラダイムで最も強力なワークフローの一つは、高推論クラウドモデルを使い、特化ローカルモデル向けの技術仕様を準備することです。これにより会話的意図と精密実装のギャップが効果的に埋まります。

=== 「契約」の作成（LLM_NOTES.md）

**Claude 4.6** のようなモデルで計画フェーズにいるとき、単にコードを求めないでください。**Qwen 3-Coder** や **Mistral-Nemo** のようなローカルモデル向けに特に最適化された「契約」または指示セットを生成するよう頼んでください。

*   **ワークフロー:**
    1.  **クラウドと対話:** 高レベルアーキテクチャ目標を議論する（例: 「GPU 駆動カリングシステムを実装したい」）。
    2.  **仕様を生成:** クラウドモデルに頼む。*「次に、ローカルコーディングアシスタントに渡せる `LLM_NOTES.md` という技術仕様ファイルを書いて。今議論した正確な Vulkan 構造体、メモリレイアウト、同期要件を、冗長で明示的な指示で含めて。」*
    3.  **ローカルモデルに渡す:** IDE で `LLM_NOTES.md` を開き、ローカルアシスタント（Goose や Junie 経由）をそのファイルに向ける。ローカルモデルは高レベル意図を推測せずに高精度で従える「地上真理」ドキュメントを持つ。

=== なぜこれが効くか

この戦略は、**クラウド** で「人間から技術へ」の翻訳を扱い冗長な指示のページ書きを省き、**ローカルモデル** で「技術からコードへ」の実装を扱うことで、両方の世界の最善を活かします。プロプライエタリなエンジン論理をマシン上に保ち、低レイテンシ実行を提供します。

クラウド AI を **翻訳者**、ローカル AI を **実装者** として扱うことで、「指示ドリフト」のリスクを最小化し、構想からコミットまで Vulkan エンジンが技術的に健全であることを保証します。この戦略は「計画」次いで「実装」のワークフローで最もよく使われます。タスクが小さすぎると、実装ステップをローカル LLM にオフロードするよりクラウド LLM が自分で仕事を終えるのと同程度の時間がかかり、ローカル LLM に実装を任せる利点が消えるため、バランスが必要です。

NB: `LLM_NOTES.md` ファイルは、現在の AI セッション向けコンテキストを提供する一時的で開発者向けの「契約」です。最終ソースコードドキュメント向けではないプライベートなアーキテクチャ決定とプロジェクト固有指示を含むため、エンジンのソースプロジェクトの一部では **なく**、Git リポジトリにチェックインすべきではありません。AI 向けのローカル「地上真理」であり、恒久的なプロジェクトファイルではありません。

この戦略は、複数回の実装反復を要する大きなタスクに最適です。ローカル LLM が問題に「行き詰まる」ことがよくあり、短いクラウド AI セッションがこの戦略でローカル LLM を救い、正しい軌道に戻せます。

== 意思決定フレームワーク: バランスを見つける

これらの強力なモデル間でどう選ぶか？ 開発フェーズに基づく **ハイブリッド戦略** を推奨しますが、これらの役割は流動的であることを覚えてください。「最良」のモデルは、Vulkan アプリからリソースを奪わずに現在のタスクに合うものです。

**計画とシステム設計フェーズ** では、クロスプラットフォームフレームグラフやスレッドセーフなメモリアロケータの設計のようなアーキテクチャブレインストーミングに高推論 **クラウドモデル** または **Qwen 3-Coder（30B）** を使います。これらは（クラウドモデルの場合）ゼロ VRAM コストで最大の推論深度を提供します。**実装と検証フェーズ** では、厳密なバリデーション要件を満たすコードへアーキテクチャを翻訳するため **Mistral-Nemo（12B）** または **Qwen 3-Coder** へ切り替えます。その精度は fix ループに費やす時間を大幅に減らします。最後に **迅速反復フェーズ** では、**Llama 4（17B）** または **Gemini 3.1 Flash** のようなクラウドベース「Flash」モデルを使います。その高速度により、AI が実装詳細、ユニットテスト、単純リファクタを支援する間、Vulkan レンダラを高フレームレートで動かし続けられます。

== まとめ: 統合モデル戦略

ベースモデルの選択は、**「知能対 VRAM」比** の管理と、いつクラウドを活用するかを知ることです。「深い計画」にクラウドモデル、「精密リファクタ」に Nemo や Qwen のような特化ローカルモデル、「日々のインナーループ」に Llama を使うことで、深く知識がありつつ高い応答性の開発環境を作ります。

最も成功する開発者は、AI モデルを動的リソースとして扱い——グラフィックスタスクの複雑さ、予算、ハードウェア制約に応じて「知能」を上下させ、ローカルとクラウドプロバイダを切り替える——人々です。
