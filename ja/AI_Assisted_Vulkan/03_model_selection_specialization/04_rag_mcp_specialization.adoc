= 地上真理: MCP & RAG 特化

== はじめに: 「幻覚」ギャップ

**Qwen 3-Coder** のような最も強力なベースモデルでさえ、その **学習カットオフ** に囚われています。Vulkan Working Group が今日新しい拡張をリリースしても、AI モデルはそれを知りません。これは危険な「幻覚ギャップ」を生みます。AI が存在しない構造メンバやビットマスクを「発明」し、何時間もの無駄なデバッグにつながることがあります。

このギャップを埋めるため、「クローズドブック」AI から **「オープンブック」スペシャリスト** へ移ります。主に 2 つの方法——**MCP** と **RAG**——で行います。

== コア概念: グローバル真理対ローカル真理

真に効果的な AI 支援ワークフローを構築するには、2 種類の知識の違いを理解しなければなりません。

1.  **グローバル真理（Vulkan Spec）:** API の公式定義です。地球上のすべての開発者が共有します。この「グローバル真理」へは **MCP** 経由で AI をつなぎます。
2.  **ローカル真理（エンジンアーキテクチャ）:** エンジン固有のコーディングスタイル、メモリ管理、内部抽象化です。プライベートなため、世界のどの AI モデルも知りません。この「ローカル真理」へは **RAG** 経由で AI をつなぎます。

== 戦略 1: コンテキストブリッジ（MCP）

環境セクションで探ったように、**Model Context Protocol（MCP）** により AI はリアルタイムデータを「手を伸ばして」照会できます。

**実世界体験: `vk.xml` ルックアップ**
複雑なポストプロセスパスを実装し、Android デバイス上のストレージイメージ向けの有効な `VkImageUsageFlags` を知る必要があるところを想像してください。IDE を離れる代わりに、アシスタントに尋ねます。*「ローカル Vulkan SDK の `vk.xml` でストレージイメージの有効な usage flags を照会して。」*

AI は学習データからフラグを「推測」しません。**MCP ツール** を使い、Vulkan SDK 内の実際の XML ファイルを読みます。正確なビットマスク定義を見つけ、仕様と一貫した答えを提供します。これが「確率的推測」と「地上真理」の違いです。

*   **勝利:** AI を **Vulkan レジストリ** につなぐことで、昨日リリースされた拡張でもコードが常に構文的に正しいことを保証します。

== 戦略 2: Retrieval-Augmented Generation（RAG）

MCP が「構造化データ」（XML やドキュメントなど）向けである一方、**RAG** は「コンテキストデータ」——エンジン自身のソースコード、内部 wiki、独自のアーキテクチャパターン——向けです。

**体験: 「コンテキスト認識」**
新しい `Buffer` ラッパに取り組んでいます。既存の `Texture` クラスのコーディングスタイルに合わせたい。RAG 対応ツールを使います。尋ねます。*「`Texture` クラスを参照として使い、カスタム RAII ラッピングとメモリ割り当てパターンに従う新しい `Buffer` クラスを生成して。」*

汎用 C{pp} クラスを返す代わりに、システムは **RAG** でローカルプロジェクトから `Texture` クラスの実装を「取得（Retrieve）」します。次にエンジン固有の「声」で生成を「増強（Augment）」します。結果のコードは、自分で書いたものとまったく同じように見えます。

*   **勝利:** RAG は、基になるモデルを再学習せずに、AI の提案が特定コードベースと **アーキテクチャ的に一貫** することを保証します。

== 特化の仕組み: RAG パイプライン

グラフィックス開発者にとって RAG パイプラインの理解は不可欠です。ローカルファイルだけで汎用モデルを **Vulkan エキスパート** へ「特化」できるからです。実質的に、元の学習データよりはるかに大きく正確な「ワーキングメモリ」を AI に与えています。

エンジン向けの RAG 駆動特化を構築するには、パイプラインの 4 フェーズを理解しなければなりません。

=== フェーズ 1: 知識キュレーション（「ソース真理」）
エンジンの「地上真理」ドキュメントを特定しなければなりません。Vulkan プロジェクトでは次が含まれます。

*   **ヘッダファイル（`.h`, `.hpp` など）:** 内部 API を定義。
*   **エンジニアリングガイドライン（`GUIDELINES.md`）:** メモリ管理や同期の哲学を説明するドキュメント。
*   **外部技術ドキュメント:** サードパーティライブラリマニュアル（例: Slang、VMA、Volk、SDL3）のローカル Markdown または HTML 版、およびまだ MCP ツールに統合されていない特化 GPU ベンダー最適化ガイド。

NB: RAG インデックスの広さと取得速度の間にはトレードオフがあります。広いインデックスはより正確な答えを提供できますが、クエリ時間を遅くすることがあります。逆に狭いインデックスはクエリを速くできますが精度を犠牲にし得ます。加えて、トークンのコストにもなり得、実際の作業セッションに使えるトークンが減るため、ここでやりすぎないでください。

=== フェーズ 2: 埋め込みとインデックス（「数学マップ」）
コンピュータはコードを「読め」ません。数学を理解します。

1.  **チャンキング:** ファイルを小さく重なる断片（例: 単一関数やクラス定義）に分割。
2.  **埋め込み:** 小さな特化モデル（**埋め込みモデル**、`bge-small-en` や `nomic-embed-text` など）がこれらのテキストチャンクを高次元数学ベクトルへ変換。
3.  **ベクトルデータベース:** これらのベクトルはローカルデータベースに格納されます。「概念的に類似」するチャンク（例: `VkBuffer` 作成と `VkImage` 作成）はこの数学空間で互いに「近く」格納されます。

=== フェーズ 3: 取得（針を見つける）
質問すると、システムはプロンプトをベクトルに変え、最も関連する「近傍」をデータベースで検索します。「Pipeline Barriers」について尋ねると、同期を扱うエンジン内の特定ファイルを取得します。

=== フェーズ 4: 増強と生成（最終組み立て）
取得されたチャンクは隠れたコンテキストとして AI のプロンプトに「詰め込まれ」ます。LLM へ送られる最終プロンプトは次のように見えます。
[source,text]
----
Context: [Your engine's Buffer.cpp code]
Context: [The VMA Reference for allocation]
User Prompt: Create a new vertex buffer for our triangle.
----
モデルは今、正しく特化した答えを生成するのに必要なすべての「事実」を持ちます。

== 実践実装: 知識インデックスの構築

多くの IDE プラグインが RAG を自動化しますが、環境セクションですでにセットアップしたツールを使い、より強力で透明な「特化インデックス」を構築できます。これにより、特定ハードウェアとエンジン向けにカスタム構築されたエキスパートとして AI を扱えます。

=== ステップ 1: 「知識ボルト」構造
プロジェクト内に RAG 専用ディレクトリを作成します。ホームフォルダ全体をインデックスするのではなく、精密に。
[source,text]
----
/vulkan_project
  /docs
    /vma_library_reference.md
    /engine_architecture.md
    /naming_conventions.md
  /src
    (Your engine code)
----

=== ステップ 2: Goose による意味取得
隠れたプラグインに頼る代わりに、**Goose** で明示的 RAG を実行できます。新機能を実装する必要があるとき、次の複数ステップ取得ワークフローに従ってください。

1.  **コンテキスト読み込み:**
    コードを求める前に、エージェントにドキュメントをインデックスするよう伝える。
+
[source,text]
----
Goose, read 'docs/naming_conventions.md' and
'src/core/Buffer.hpp'. We are going to implement
a new Index Buffer class.
----

2.  **増強プロンプト:**
    取得したファイルを明示的に参照しながら実装を求める。
+
[source,text]
----
Using our naming conventions and the RAII pattern
seen in 'Buffer.hpp', generate 'IndexBuffer.hpp'
and 'IndexBuffer.cpp'. Ensure it uses 'vmaCreateBuffer'
for allocation.
----

**なぜこれが効くか:** 「取得」フェーズを手動で実行しています。Goose 経由の **Model Context Protocol（MCP）** を使っているため、エージェントはファイルを地上真理として「読め」、内部重みをプロジェクト固有の法則で効果的に増強します。

=== ステップ 3: ローカルベクトルインデックス（任意だが推奨）
プロジェクトが何千ものファイルに成長すると、手動取得は遅くなります。この段階では、**ChromaDB** や **Pinecone**（MCP 経由）のようなローカル **ベクトルデータベース** を使うべきです。これにより AI はミリ秒単位でエンジン全体にわたる「意味検索」を行い、正確なファイル名を知らなくても関連コードスニペットを見つけられます。

=== ステップ 4: 知識グラフ（構造コンテキスト）
ベクトルデータベースが「類似」テキストの発見に優れる一方、**知識グラフ** は「関連」構造の発見に優れます。Vulkan エンジンでは、所有階層の理解（例: どの `VkDevice` がどの `VkCommandPool` を作成したか）に不可欠です。

プロジェクトを知識グラフとしてインデックスすることで、AI は関係を辿れます。
*   **クエリ:** 「`GlobalDescriptorSet` に依存するすべてのシステムを見せて。」
*   **結果:** AI はセットを使うクラスだけでなく、論理的にリンクされたシェーダ、パイプラインレイアウト、同期バリアも特定する。

=== ステップ 5: 関数レジストリ（ツール能力）
**関数レジストリ** は、AI が呼び出できる能力の「電話帳」として働きます。単にコードを書く代わりに、AI はレジストリを使い、タスク解決に利用可能な既存エンジンツールや Vulkan 関数を特定します。

例えば「フレームをキャプチャして」とタスクを与えると、AI は関数レジストリを参照し、エンジンに `RenderDoc::CaptureFrame()` メソッドがあることを特定し、キャプチャシステムをゼロから実装しようとせずその既存ツールを呼ぶことを選びます。

**高度なヒント: 「クラウドからローカル」特化パイプライン**
プラグインなしで 17B SLM を効果的に「特化」するには、高推論クラウド LLM（**Claude 4.6** など）で **フェーズ 1: 知識キュレーション** を実行できます。クラウドモデルにレンダラを分析し、「アーキテクチャ法則」を単一 Markdown ファイルに要約するよう頼んでください。このキュレート済みドキュメントを RAG 経由でローカル SLM に与えることで、VRAM コストなしに大きなモデルの「知恵」を小さなモデルに与え、推論ギャップを埋めます。

=== SLM の利点: 「外部メモリ」としての RAG

Retrieval-Augmented Generation の最も強力な側面の一つは、**Small Language Models（SLM）**——通常 7B から 30B パラメータ範囲——が、ゲームエンジンの特定ドメインで 400B+ パラメータの「巨大」モデルを上回れることです。C{pp} コードベースの「暗記」を RAG インデックスへオフロードすることで、SLM はカスタムクラス名を覚えようと内部パラメータを浪費せず、提供した論理に推論の 100% を集中できます。

== まとめ: 情報に基づくスペシャリスト

ベースモデルの **事前学習済み重み** を MCP の **グローバル真理** と RAG の **ローカル真理** と組み合わせることで、AI を「予測テキスト生成器」から **グラフィックスエンジニアリングアシスタント** へ変えます。

*   **MCP** は「事実」（仕様）を提供。
*   **RAG** は「味」（エンジン）を提供。
*   **情報に基づくコンテキスト** により、モデルがローカルアーキテクチャ法則を尊重することを保証。

本セクションの最終章では、特化の究極形——エンジンの直感的法則をモデルの重みに直接「焼き込む」**ファインチューニング（LoRA）**——と、結果のスペシャリストを共有チームサーバへデプロイする方法を探ります。
