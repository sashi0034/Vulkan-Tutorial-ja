= Android & iOS: モバイルの最前線

== はじめに: 帯域の戦い

モバイル開発は Vulkan エンジンの究極のテストです。高性能レンダリングと厳しい電池制限、熱制約のある GPU、多言語「グルーコード」の複雑さのバランスを取らなければなりません。

**協調エンジニアリング** では、プラットフォーム統合の「構文ノイズ」を扱い、モバイル GPU アーキテクチャ向け **帯域分析アシスタント** として働くために AI を使います。本章では、JNI ボイラープレートの自動化と **Tile-Based Deferred Renderers（TBDR）** 向けレンダリングパイプラインの最適化方法を探ります。

== コア概念: 「グルー」エンジニア

モバイル開発では、最初の `vkCreateInstance` 呼び出しに到達する前に何百行ものコードが必要です。Android でも iOS でも、アプリケーションのライフサイクルと、プラットフォームのネイティブ言語を C{pp} コアへ橋渡しする「グルーコード」を管理しなければなりません。

== チュートリアル: モバイルエントリポイントの構築

Vulkan エンジンをクロスプラットフォームにするには、C{pp} コアにネイティブサーフェスを提供するプラットフォーム固有「グルーコード」を実装しなければなりません。このプロセスを自動化する次のステップに従ってください。

=== ステップ 1: JNI ブリッジの自動化（Android）
新しいファイル `VulkanEntry.cpp` を作成し、AI アシスタント（Goose または IDE Chat）で JNI ボイラープレートを生成します。

[source,cpp]
----
// Example JNI bridge generated by AI for surface management
#include <jni.h>
#include <android/native_window_jni.h>
#include "VulkanRenderer.hpp"

extern "C" JNIEXPORT void JNICALL
Java_com_example_vulkan_MainActivity_onSurfaceCreated(JNIEnv* env, jobject thiz, jobject surface) {
    ANativeWindow* window = ANativeWindow_fromSurface(env, surface);
    VulkanRenderer::getInstance().initSurface(window);
}
----

**AI への指示:** *「Goose、完全な `VulkanEntry.cpp` JNI ブリッジを生成して。`onSurfaceCreated`、`onSurfaceChanged`、`onSurfaceDestroyed` を扱う必要がある。`ANativeWindow_fromSurface` を使い、レンダラの `initSurface` メソッドに渡すことを保証して。」*

=== ステップ 2: iOS SwiftUI ブリッジ
iOS では、Vulkan サーフェスを SwiftUI 互換ビューにラップしなければなりません。`UIViewRepresentable` ブリッジを生成するよう AI にタスクを与えます。

[source,swift]
----
// Example SwiftUI wrapper for a Vulkan-ready CAMetalLayer
import SwiftUI
import MetalKit

struct VulkanView: UIViewRepresentable {
    func makeUIView(context: Context) -> UIView {
        let view = UIView()
        let metalLayer = CAMetalLayer()
        view.layer.addSublayer(metalLayer)
        // AI will add the logic to pass this layer to C++ here
        return view
    }

    func updateUIView(_ uiView: UIView, context: Context) {}
}
----

**AI への指示:** *「Vulkan エンジン向けに `CAMetalLayer` を提供する SwiftUI `UIViewRepresentable` を生成して。MoltenVK と KosmicKrisp の両方をサポートする必要があるので、異なるネイティブハンドル型を扱えるほどブリッジが柔軟であることを保証して。」*

== チュートリアル: タイルメモリ向け最適化（TBDR）

デスクトップ GPU と異なり、モバイル GPU（Adreno、Mali、Apple）は **タイル** 型です。性能を最大化するには、G-Buffer データをオンチップに保つ **サブパス** を使わなければなりません。

=== ステップ 1: サブパスリファクタ
エンジンが現在 G-Buffer と Lighting に別々のレンダーパスを使っているなら、サブパス付きの単一パスへリファクタするよう AI に頼んでください。

[source,text]
----
Goose, refactor our 'DeferredRenderer::createRenderPass' method.
Combine the G-Buffer and Lighting passes into a single
'VkRenderPass' with two subpasses. Use 'subpassLoad'
in the fragment shader to read the G-Buffer data directly
from tile memory.
----

=== ステップ 2: 依存論理
AI は、GPU がタイルデータを VRAM へ早すぎる段階で書き込まないことを保証するのに必要な `VkSubpassDependency` 構造を生成します。

[source,cpp]
----
// AI-generated subpass dependency for on-chip data flow
VkSubpassDependency dependency = {};
dependency.srcSubpass = 0; // G-Buffer
dependency.dstSubpass = 1; // Lighting
dependency.srcStageMask = VK_PIPELINE_STAGE_COLOR_ATTACHMENT_OUTPUT_BIT;
dependency.dstStageMask = VK_PIPELINE_STAGE_FRAGMENT_SHADER_BIT;
dependency.srcAccessMask = VK_ACCESS_COLOR_ATTACHMENT_WRITE_BIT;
dependency.dstAccessMask = VK_ACCESS_INPUT_ATTACHMENT_READ_BIT;
dependency.dependencyFlags = VK_DEPENDENCY_BY_REGION_BIT;
----

この「オンチップ」データフローを使うことで、VRAM 帯域を最大 70% 削減し、モバイルデバイスの電池寿命とフレームレートを大幅に向上させます。

== まとめ: 電池効率の良いグラフィックス

モバイル Vulkan は **帯域管理** です。AI で JNI「グルー」を自動化し、タイル GPU 向けにサブパス依存を最適化することで、エンジンが高速かつ電池効率的であり続けることを保証します。「C{pp} コーダ」から、プラットフォームのネイティブコードと GPU ハードウェアアーキテクチャの複雑な相互作用を管理する **モバイルグラフィックスエンジニア** へ移ります。
