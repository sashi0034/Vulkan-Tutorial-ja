= フレームインフライトアーキテクチャ: エンジンの心拍

== はじめに

グラフィックスプログラミングの初期には、レンダリングを線形なシーケンスと考えがちでした。CPU がコマンドを記録し、GPU が実行し、次のフレームを始める前に CPU が GPU の完了を待つ。これは単純ですが、信じられないほど遅いです。GPU がレンダリングしている間 CPU はアイドルで、CPU が次のフレームを記録している間 GPU は作業を待ちます。

高性能を得るには、これら二つのプロセスを重ねる必要があります。これが **フレームインフライトアーキテクチャ** です。複数のフレームを同時に処理したい——例えば、ディスプレイがフレーム 1 を表示している間に、GPU がまだフレーム 2 を描き、CPU がフレーム 3 を記録している、といった具合です。この概念はベースチュートリアルの xref:03_Drawing_a_triangle/03_Drawing/03_Frames_in_flight.adoc[フレームインフライト] の章で紹介されていますが、ここではタイムラインセマフォを使って次のレベルへ進めます。

== 同期の課題

複数の同時フレームの管理は、Vulkan エンジンでおそらく最も複雑な同期課題です。次を保証しなければなりません。

1.  **データの整合性**: GPU が前のフレーム用に読んでいるユニフォームバッファを上書きしないこと。
2.  **リソースの寿命**: GPU がその使用を完全に終えたと確信できるまで、テクスチャやコマンドバッファを破棄しないこと。
3.  **前進の保証**: 大きな入力遅延を招いたりメモリを使い果たしたりするほど多くのフレームをサブミットしないこと。

レガシーな Vulkan 1.0 の世界では、これはインフライトの各フレーム用のフェンスとバイナリセマフォの複雑な配列で扱われました。「同期だらけ」のコードになり、スケールしにくく壊れやすかったです。

== タイムラインの利点

基盤として **タイムラインセマフォ** を使うことで、このアーキテクチャを大幅に簡素化できます。すべてのフレームに別々のフェンスを管理する代わりに、「完了したフレームインデックス」を表す単一の単調カウンタを使います。

この章では、任意の数のフレームインフライトを扱えるようメインエンジンループを再構築します。タイムラインを使って CPU と GPU をどう協調させるか、そして Vulkan オブジェクトの破棄や再利用がいつ安全かをタイムラインで正確に判断する堅牢なリソース管理システムをどう実装するかを探ります。

エンジンの心拍であるメインレンダーループの再構築から始めましょう。
