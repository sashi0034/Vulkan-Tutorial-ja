= 非同期コンピュートと実行のオーバーラップ: GPU の並列化

== はじめに

多くのレンダリングアーキテクチャでは、作業はイベントの線形なシーケンスとしてサブミットされます。シャドウを描き、次にジオメトリ、次にコンピュートベースのポストプロセスパスを走らせます。この「直列」アプローチは理解しやすいですが、GPU ハードウェアのかなりの部分をアイドルにしがちです。現代の GPU は、グラフィックスパイプライン、コンピュートユニット、転送エンジンなど、同時に——そして同時に働くべき——独立した複数のユニットで構成されています。

**非同期コンピュート** は、メインのグラフィックスキューが自身の作業で忙しい間に、専用のコンピュートキューでコンピュートワークロード（物理、オクルージョンカリング、ポストプロセスなど）を走らせる実践です。正しく行えば、GPU の実行タイムラインの「穴」を実質的に埋めることで、大きな性能向上につながります。

== 「バブル」問題

高性能の主な敵は、しばしば「バブル」と呼ばれる **パイプラインストール** です。これは、GPU の一部が作業を終えたのに、まだ満たされていない依存を待っているために次のタスクを開始できないときに起きます。バリアが過度に保守的だと——例えばコンピュートパスを始める前に「すべてのコマンド」が終わるのを待つよう GPU に伝えると——コンピュート作業がはるかに早く始められたとしても、ハードウェアを実質的に直列モードに強制しています。

== オーバーラップのための設計

真の実行オーバーラップを得るには、単純な「top-of-pipe」から「bottom-of-pipe」への依存を超える必要があります。データが準備できた正確な瞬間を表現するよう、`vk::DependencyInfo` と **タイムラインセマフォ** を設計する必要があります。

この章では次を探ります。

1.  **スループットの最大化**: オーバーラップの良い候補となるワークロードの見極め方と、GPU 占有率をできるだけ高く保つサブミッションの構造化。
2.  **非同期ポストプロセス**: よくある実世界のパターン——コンピュートベースのブルームやトーンマッピングを、次フレームのシャドウやジオメトリパスと並行して走らせる——を実装します。
3.  **ストールの排除**: ハードウェアプロファイラと同期バリデーションを使い、捉えにくい「バブル」を見つけ、ステージマスクを洗練して排除する方法を学びます。

この章の終わりには、エンジンを直列シーケンスから並列実行モデルへ移し、どのハードウェアユニットもアイドルのままにしないようにできるはずです。
