= バブル問題: ストールの発見と修正

== バブルの特定

GPU タイムライン上の「バブル」は、依存が満たされるのを待っているために一部のユニットがアイドルになる期間です。コードを見るだけでは見つけにくいことがあります。オーバーラップを有効にしたつもりでも、ステージマスクが広すぎると GPU は依然としてストールしているかもしれません。

これらを見つけるには、**NVIDIA Nsight Graphics**、**AMD Radeon GPU Profiler**、あるいは **LunarG Synchronization Validation** レイヤなどのハードウェアプロファイラを使います。プロファイラでは、バブルは、一方が忙しい間に Graphics または Compute の行が空いているタイムライン上の隙間として見えます。

image::/images/vulkan_simplified_pipeline.svg[簡略化された Vulkan パイプライン, width=400, alt="バブルが起き得る場所を示すための簡略化された Vulkan パイプライン図"]

== バブルのよくある原因

1.  **過度に保守的なステージマスク**: すべてのバリアに `vk::PipelineStageFlagBits2::eAllCommands` を使うと、GPU はすべてをフラッシュし、次のタスクを始める前にアイドルになるのを待ちます。これがバブルの最も一般的な原因です。常に可能な限り最も具体的なステージマスクを使ってください。
2.  **逐次サブミッション**: 二つのキューがあっても、CPU コードが一方の完了を待ってから他方へサブミットすると、CPU 側にバブルを作っています。適切なところで **Wait-Before-Signal** パターンと複数のサブミッションスレッドを使います。
3.  **依存チェーン**: 小さな依存の連鎖は、わずかに広い一つのバリアより高くつくことがあります。互いに待つ五つのコンピュートパスがあると、それぞれが小さなストールを導入します。これらを単一のコンピュートサブミッションにバッチする方が良い場合もあります。

== ストールの修正

バブルを見つけたら、修正は通常 `vk::DependencyInfo` を洗練することです。

-   **ステージマスクの洗練**: `srcStageMask` をパイプラインのより後ろへ、または `dstStageMask` をより前へ移せないか確認します。例えば、コンピュート作業は `eFragmentShader` を待たずに、`eVertexShader` が終わり次第開始できますか？
-   **メモリバリアの賢い使用**: より多くの作業をより早く開始できるなら、いくつかのイメージバリアよりグローバルメモリバリアの方が良い場合があります。
-   **並行性の向上**: プロファイラがコンピュートユニットの利用率不足を示しているなら、より多くの作業（オクルージョンカリングなど）をグラフィックスからコンピュートへ移せますか？

これらのバブルを体系的に見つけて排除することで、「とりあえず動く」レンダラから真にプロ級のレンダラへ移ります。次の章では、同じ原則がモダンゲームで最も一般的なバックグラウンドタスクの一つ——アセットストリーミング——にどう適用されるかを見ます。
