= 依存関係の解剖: はじめに

== 概要

単純なカラークリアから複雑なレイトレース反射パスまで、あらゆる Vulkan 操作は、私たちが定義する依存関係によって成り立っています。この章では、データが実際に Vulkan パイプラインをどう流れるかの核心と、なぜ同期が単なる「ビットマスクの設定」以上のものなのかを深く掘り下げます。

image::/images/rendering_pipeline_flowchart.png[レンダリングパイプラインのフローチャート, width=600, alt="モダンな Vulkan レンダリングパイプラインの各ステージを示すフローチャート"]

同期を本当に極めるには、まず GPU がコマンドを処理するときに何が起きるかを分解する必要があります。GPU を「巨大な並列プロセッサ」と呼ぶことは多いですが、それがデータの整合性にとって何を意味するのでしょうか。まずは **実行依存**（GPU 作業の「いつ」）と **メモリ依存**（データの「どこ」と「可視性」）の根本的な違いから解体していきます。

=== この章で学べること

この章は、「動かす」から「なぜ動くかを知る」へ進むためのものです。次を扱います。

*   **ハードウェアの視点**: なぜ実行バリアだけでは、現代のマルチキャッシュ GPU でデータ破壊を防げないのか。
*   **実行依存とメモリ依存**: ステージを止めることと、そのデータが次のステージで実際に読めるようにすることの違い。
*   **Synchronization 2 の利点**: 新しい `vk::DependencyInfo` と `vk::CmdPipelineBarrier2` が単なる構文の整理ではなく、意図をドライバへ伝える方法の根本的な転換である理由。
*   **パイプラインステージによる精密指定**: `vk::PipelineStageFlagBits2` と `vk::AccessFlagBits2` で特定のハードウェアユニットを狙い、不要なパイプラインバブルを避けて GPU 占有率を最大化する。

この章の終わりには、任意の二つの GPU 作業の間で必要な「ハンドシェイク」が明確に理解できるはずです。この基礎は、単純なイメージレイアウト遷移から複雑な非同期コンピュートアーキテクチャまで、以降のすべてに不可欠です。
