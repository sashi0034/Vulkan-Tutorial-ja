= 実行依存とメモリ依存

== はじめに

同期がこれほど重要な理由を理解するには、まず GPU が作業を処理するときに内部で何が起きているかを見る必要があります。概ね線形で予測可能な順序で命令を実行する CPU とは異なり、GPU は特化したハードウェアユニットの巨大で高度に並列な配列です。コマンドバッファをサブミットしても、GPU は先頭から末尾まで順番に進むだけではありません。ジオメトリ、ラスタライズ、フラグメントシェーディングなどの各パイプラインステージにタスクを分散し、しばしば同時に処理します。

この並列性こそが Vulkan の強みですが、危険の源泉でもあります。フラグメントシェーダがコンピュートシェーダが書いたばかりのデータを読むには、その依存関係を正確に定義しなければなりません。Vulkan では、これは二つの異なる概念に分かれます。**実行依存** と **メモリ依存** です。

=== 「いつ」: 実行依存

**実行依存** は同期の最も単純な形です。「この作業はいつ開始できるか？」に答えます。

コマンド A とコマンド B があるとします。A から B への実行依存は、GPU にこう伝えます。「コマンド A の指定されたパイプラインステージが終わるまで、コマンド B の指定されたパイプラインステージを開始するな。」

これは単純に聞こえますが、落とし穴があります。現代のハードウェアでは、コマンド A の作業が終わったことと、そのデータがコマンド B に使える状態であることとは *同じではありません*。実行はトリガーに過ぎず、メモリこそが中身です。

=== アーキテクチャの現実: キャッシュとメモリタイプ

Vulkan のメモリは、テクスチャやバッファを入れる大きな一つのバケツではありません。ハードウェアによっては、物理的な場所とアクセス速度が異なる複雑な風景です。効果的に同期するには、何に対して同期しているかを知る必要があります。

**ディスクリート GPU** では、システムの RAM から物理的に分離された専用の Video RAM（VRAM）があります。この二つ間のデータ移動は **DMA（Direct Memory Access）** エンジンの仕事です。PCI Express バス越しにデータをコピーでき、メインのシェーダコアを煩わせません。テクスチャをアップロードするとき、しばしば DMA エンジンとグラフィックスパイプラインを同期させています。

一方、多くのノート PC やモバイルデバイスは **Unified Memory Architecture（UMA）** を使い、CPU と GPU が同じ物理 RAM を共有します。これなら簡単に思えますが、実際には隠れた複雑さ——**キャッシュ**——が加わります。RAM を共有していても、CPU には独自の L1/L2/L3 キャッシュがあり、GPU にも独自の L1/L2 キャッシュがあります。GPU が共有バッファに書き込んでも、そのデータは GPU の L2 キャッシュに留まり、物理 RAM に届かないことがあります。CPU が読もうとしても、RAM や自身のキャッシュにある古い値を見ます。

Vulkan では、これらの振る舞いを主に三つのメモリタイプに分類します。

*   **Device Local**: GPU がアクセスするのに「最も速い」メモリ。ディスクリートカードでは VRAM、UMA では共有 RAM の一部です。
*   **Host Visible**: c{pp} アプリケーションのアドレス空間に「マップ」でき、CPU が直接読み書きできるメモリ。
*   **Host Coherent**: Host Visible の特殊な種類で、キャッシュを手動でフラッシュしなくても CPU と GPU が同じデータを見ることをハードウェアが自動的に保証します（ただし、書き込みが *終わった* ことを保証するには、依然として実行依存が必要です）。

=== 「どこ」: メモリ依存

多くの Vulkan 開発者がつまずくのはここです。コマンド A が終わっていても、その出力は特定のシェーダコア上のローカル L1 キャッシュに残っていたり、メインプールへ書き戻されていない共有 L2 キャッシュにあったりします。コマンド B——GPU のまったく別の部分、あるいは CPU 上で動いているかもしれない——が「利用可能」になる前にメインメモリからそのデータを読もうとすると、古いデータを読みます。

だから実行だけでは足りないのです。「コンピュートシェーダが終わるまでフラグメントシェーダを開始するな」と伝えても、ハードウェアは喜んで従います。しかしフラグメントシェーダがテクスチャを読もうとすると、コンピュートシェーダの書き込みがどこかのローカルキャッシュに閉じ込められたままなので、古いデータが見つかります。

**メモリ依存** は、データをキャッシュとメインメモリの間で適切に移動し、安全に読めるようにします。これには二つの重要なステップがあります。

1.  **Availability（利用可能性）**: ソースのローカルキャッシュからデータを「フラッシュ」し、共有メモリプール（L2 キャッシュやメインメモリなど）から見えるようにする操作。
2.  **Visibility（可視性）**: 宛先ステージのローカルキャッシュを「無効化」し、既に持っているかもしれない古いビットではなく、共有メモリプールから新しいデータを読むよう強制する操作。

実行依存とメモリ依存の *両方* がなければ、**ハザード** の世界に生きることになります。最も一般的なのは「Read-After-Write」（RAW）ハザードで、コンピュートシェーダの書き込みが終わる前にフラグメントシェーダがテクスチャを読み、初期の Vulkan 実装でよく見られるちらつきや「シャドウアクネ」につながります。

=== 実践的なハンドシェイク

プロフェッショナルな握手と考えてください。実行依存は二人が会うことに合意すること。メモリ依存は、一方が実際に文書を渡し、他方が古いメモではなく新しい文書を見ていることを確かめることです。

Synchronization 2 では、このハンドシェイクを `vk::PipelineStageFlagBits2` と `vk::AccessFlagBits2` で定義します。ステージフラグが *いつ*（実行依存）を、アクセスフラグが *どのように*（メモリ依存）を定義します。正しく組み合わせることで、データが正しい順序で処理されるだけでなく、探しに行ったときに実際にそこにあることを保証できます。

== Simple Engine での実装: キャッシュと安全性

`Simple Engine` では、これらのアーキテクチャ上の現実を `MemoryPool` クラス（`memory_pool.cpp`）で扱います。バッファやイメージ用にメモリを確保するとき、役割を決めるために `vk::MemoryPropertyFlags` を指定します。例えば `UniformBuffer` オブジェクトは通常 `HostVisible | HostCoherent` として確保されます。つまり CPU が書き込め、手動の `flushMappedMemoryRanges` 呼び出しなしに GPU から自動的に見えます。

ただし **coherent** だからといって実行依存を無視してよいわけではありません。`Simple Engine` でも、GPU がフラグメントシェーダで読んでいる最中に CPU が `HostCoherent` ユニフォームバッファを更新すると、**データレース** が起きます。だからこそ、次のフレーム用に CPU がリソースを変更し始める前に GPU がそのフレームのリソースの使用を終えたことを保証するために、依然として `inFlightFences` とセマフォを使います。

テクスチャと頂点バッファには、最大性能のため `DeviceLocal` メモリを使います。これらはホストコヒーレントではないため、`vk::DependencyInfo` と `vk::ImageMemoryBarrier2` で「Availability」と「Visibility」のハンドシェイクを明示的に管理しなければなりません。これにより、`vkCmdCopyBufferToImage` の後、データが転送ユニットのキャッシュから適切にフラッシュされ、フラグメントシェーダのキャッシュ向けに無効化されることが保証されます。
