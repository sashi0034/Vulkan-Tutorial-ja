= タイムラインセマフォ: マスタークロック

== はじめに

長年、Vulkan 開発者は二つの非常に異なる同期プリミティブを扱わねばなりませんでした。**バイナリセマフォ** と **フェンス** です。バイナリセマフォは専ら GPU 間同期（例えばイメージの準備ができてからサンプリングする）に使われ、フェンスは GPU から CPU への同期（例えばコマンドバッファが終わってからリソースを再利用する）に使われました。

この分割は、本質的には同じ問題——「作業はもう終わったか？」——に対して二組の異なる論理を書かせました。また、デバッグが悪名高いほど難しい複雑な「セマフォチェーン」にもつながりました。

拡張として導入され、現在は Vulkan 1.2+ のコアである **タイムラインセマフォ** がすべてを変えます。単純な単調増加の `uint64_t` カウンタを使い、GPU 間同期と GPU から CPU への同期の両方を扱える、単一の統一プリミティブを提供します。

== 単調増加の世界

タイムラインベースのシステムでは、進捗は増える一方の値で測られます。GPU に作業をサブミットするとき、「これが終わったらセマフォの値を 10 にせよ」と伝えます。別の作業がそのデータを必要とするなら、「セマフォの値が少なくとも 10 になるまで開始するな」と伝えます。

この単純な変化が、エンジンの設計に深い影響を持ちます。

1.  **統一された論理**: 「待つ側」が CPU か GPU かを気にしなくなります。インターフェースは同じです。特定の値を待ちます。
2.  **Wait-Before-Signal**: タイムラインセマフォの最も強力な機能の一つは、まだ到達していない値を待つ作業を GPU にサブミットできることです。これにより、サブミッション論理と実行論理を分離できます。
3.  **より良いデバッグ**: 値が単純な整数なので、簡単にログし、デバッガで検査し、エンジンの進捗のビジュアルプロファイラを構築することさえできます。

[NOTE]
====
現時点で留意すべき制限の一つ: **Window System Integration（WSI）**。現行の Vulkan 仕様では、スワップチェーンのアクワイアとプレゼント操作は依然としてレガシーなバイナリセマフォを必要とし、タイムラインセマフォを直接待ったりシグナルしたりすることはまだできません。
====

この章では、タイムラインセマフォをレンダラの「マスタークロック」として実装する方法を探ります。まずレガシーなフェンスとバイナリセマフォの置き換えから始め、次に単調カウンタの実装と、非常に効率的な wait-before-signal サブミッションパターンに入ります。

同期プリミティブの統一から始めましょう。
