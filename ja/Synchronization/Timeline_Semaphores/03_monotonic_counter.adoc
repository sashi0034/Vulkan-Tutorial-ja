= 単調カウンタ: グローバル進捗の追跡

== カウンタの理解

すべてのタイムラインセマフォの中心には、単一の `uint64_t` 値があります。この値は単調で、増える一方です。この単純な性質により、GPU/CPU エンジンの実行全体を一つの統一されたタイムラインとして扱えます。

コマンドバッファをキューにサブミットするとき、タイムラインセマフォ上のシグナル操作と関連付けます。そのシグナルに特定の値——例えば `frame_index * 10 + pass_index`——を割り当てます。GPU が各パスを完了するにつれ、セマフォの値が増加します。

== 進捗の追跡

`device.getSemaphoreCounterValue` でいつでもこの値を CPU から問い合わせられるため、はるかに知的なエンジン論理を構築できます。例えば「レンダー完了」フェンスを待つ代わりに、タイムラインを問い合わせ、GPU が現在どのステージで作業しているかを正確に見られます。

[,cpp]
----
uint64_t currentValue = device.getSemaphoreCounterValue(*timelineSemaphore);
if (currentValue >= PassValues::eShadowPassComplete) {
    // We can start preparing the next pass that depends on shadows
}
----

これは非同期リソース管理に特に有用です。リソースに最後に使われたタイムライン値をタグ付けできます。リソースを再利用または破棄する必要があるとき、現在のセマフォ値がそのタグを超えたかを確認するだけです。これにより、GPU バブルと CPU ストールの主な原因になりがちな保守的な `deviceWaitIdle()` 呼び出しが不要になります。

== 戦略的な値の選択

タイムライン値をどう増やすかはアーキテクチャ上の決定です。よくあるパターンは、各フレームに大きな増分（例: 1000）を使い、そのフレーム内の各主要パスに小さなサブ増分を使うことです。

*   フレーム 1:
    *   開始: 1000
    *   シャドウパス: 1010
    *   G-Buffer パス: 1020
    *   ライティングパス: 1030
*   フレーム 2:
    *   開始: 2000
    *   シャドウパス: 2010
    *   ...

この番号付けは、エンジン内のすべての同期値を再計算せずに新しいパスやサブステップを追加するための十分な「余裕」を与えます。また、フレーム番号がタイムライン値に明確に符号化されるため、ログもはるかに読みやすくなります。

タイムラインを「マスタークロック」として扱うことで、個々の依存関係を細かく管理することから離れ、レンダラの全体状態と進捗を管理する方向へ移ります。次の節では、これが Vulkan で最も強力なサブミッションパターンの一つ——wait-before-signal——をどう可能にするかを見ます。

== Simple Engine: フレーム進捗の追跡

`Simple Engine` では、単調カウンタを使って各システムの進捗を追跡します。フレーム内の主要なマイルストーンを表す `TimelineValues` のセットを定義します。例えば `Renderer` は、特定のレンダリングステージが終わったことを示すために `currentFrameIndex * 10 + passOffset` のような値をシグナルできます。

これは `MemoryPool` と組み合わせると非常に強力になります。単純な「破棄からのフレーム数」カウンタ（現在 `pendingASDeletions` で使っているもの）の代わりに、各リソースに GPU が最後に使った正確な `TimelineValue` をタグ付けできます。`MemoryPool` がメモリを回収する必要があるとき、現在のセマフォ値を問い合わせるだけです。`currentValue >= resourceTagValue` なら、余分なストールや保守的な待ちなしに、そのリソースは破棄または再利用して安全であることが保証されます。
