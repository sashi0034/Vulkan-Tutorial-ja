= 同期バリデーションによるデバッグ: ハザードを見つける

== はじめに

Vulkan 同期は「信頼しつつ検証する」システムです。完璧に正しいと思う `vk::DependencyInfo` と `vk::SubmitInfo2` のコードを書けても、絶対に確かめる唯一の方法は、実際のハードウェア挙動に対してテストすることです。しかし同期バグは悪名高いほど見つけにくいです。微妙なちらつき、時折のクラッシュ、あるいは——最悪の場合——開発マシンでは完璧に動き顧客の GPU では完全に失敗する、といった形で現れます。

ここで **LunarG Synchronization Validation** レイヤの出番です。間違いなく、Vulkan デバッグ兵器庫で最も重要な道具です。API 使用エラーをチェックする標準バリデーションレイヤとは異なり、同期バリデーションレイヤはエンジン内のすべてのリソースの状態を追跡し、データ破壊につながる「Read-After-Write」（RAW）、「Write-After-Read」（WAR）、「Write-After-Write」（WAW）ハザードを特定します。

== 直面するハザード

同期は本質的に、これら三種類のハザードの管理です。

1.  **Read-After-Write（RAW）**: 前のステージが書き込みを終える前に、あるステージがリソースを読もうとする。「ゴミ」データの最も一般的な原因です。
2.  **Write-After-Read（WAR）**: 前のステージがまだ読んでいる間に、あるステージがリソースへ書き込もうとする。前のステージが「半分更新された」データを読むことにつながり得ます。
3.  **Write-After-Write（WAW）**: 二つのステージが同じリソースへ同時に書き込もうとする。結果は予測不能で、ほぼ常に破壊につながります。

== 探ること

この章では、エンジンを完璧に堅牢にするためにバリデーションレイヤをどう活用するかを学びます。次を探ります。

1.  **バリデーションレイヤ**: エンジンのデバッグビルド内で LunarG Synchronization Validation レイヤをどう設定し有効にするか。
2.  **VUID の解釈**: Vulkan Validation Unique Identifiers（VUID）は圧倒的に感じられます。これらの複雑なエラーメッセージを解読し、実行可能なコード修正に変える方法を学びます。
3.  **ハザードの特定**: 手動検査ではほぼ見つけられないハザードを、バリデーションレイヤがどう捕まえるかの実世界の例を見ます。

この章の終わりには、同期コードが「だいたい正しい」だけでなく「Vulkan バリデーション済み」正しいことを保証するための道具と知識を手にしているはずです。
