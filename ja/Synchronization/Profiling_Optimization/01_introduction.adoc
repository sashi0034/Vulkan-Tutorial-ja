= プロファイリング、バッチ化、最適化: GPU を絞り出す

== はじめに

おめでとうございます。**Synchronization 2** の核心的な仕組み、**タイムラインセマフォ** の単調増加の世界、**非同期コンピュート** と **アセットストリーミング** の複雑さを極めました。堅牢でモダンでバリデーション済みのレンダラを構築しました。

しかし高性能グラフィックスの世界では、「正しい」は始まりに過ぎません。最後の課題は、同期をできるだけ効率的にすることです。記録するバリアとシグナルするセマフォの一つひとつに——ドライバオーバーヘッドと潜在的なハードウェアストールの両面で——コストがあります。

この最終章では、同期の「どう」と「なぜ」を超え、「どれだけ速く」に焦点を当てます。プロのエンジン開発者が GPU から最後の一滴まで性能を絞り出すために使う高度な技法を探ります。

== 最適化のマインドセット

同期における最適化はバランスの取り合いです。一方では、不要なストールを避けるためできるだけ具体的でありたい。他方では、ドライバへの呼び出し回数を最小化したい。

鍵は **バッチ化** と **可視性** の観点で考えることです。各リソースを孤立して考える代わりに、フレーム全体として考えるべきです。どこで依存をグループ化できますか？より多くの作業がオーバーラップできるよう、どこへバリアを移せますか？フレームレートを抑えている「見えない」ボトルネックを見つけるために、どこでハードウェアプロファイリングツールを使えますか？

== 探ること

この最終章では、Vulkan 最適化の高度な世界に入ります。次を探ります。

1.  **バリアのバッチ化**: ドライバオーバーヘッドを減らすため、複数のグローバル・イメージ・バッファバリアを単一の `vkCmdPipelineBarrier2` 呼び出しにどうまとめるか。
2.  **ストールの可視化**: 第 6 章の「バブル」問題に戻りますが、今度はハードウェアプロファイラを使い、スケールでそれらを特定し排除することに焦点を当てます。
3.  **最終的な洗練**: エンジンが成長しても高性能を保つためのベストプラクティスとよくある落とし穴のチェックリストでシリーズを締めくくります。

この章の終わりには、レンダラを「バリデーション済み」から「最適化済み」へ進める知識と道具を手にし、同期コードが正しいだけでなく速いことも保証できるようになるはずです。
