= Host Image Copies とメモリマップ同期: 直接アクセス

== はじめに

Vulkan の歴史の大半で、データをイメージへ移したい場合、非常に特定の儀式に従う必要がありました。ステージングバッファを作成し、マップし、データを書き、`copyBufferToImage` コマンドを記録し、そのコマンドバッファをキューへサブミットする。大きな非同期アップロードには効率的ですが、小さな UI テクスチャや単一ミップレベルの更新のような単純で直接的な更新には、オーバーヘッドが大きすぎます。

**Vulkan 1.4** の到来により、強力な新しい道具があります。**Host Image Copies**（`VK_EXT_host_image_copy`）です。この機能により、コマンドバッファを一つも記録・サブミットせずに、CPU がデータを GPU 最適イメージへ直接コピーできます。CPU と GPU メモリ間でデータを移す最も直接的な方法です。

[NOTE]
====
Host Image Copies は Vulkan 1.4 でコアに昇格しましたが、この機能のサポートは依然として **任意** です。この直接パスをハードウェアとドライバがサポートしていることを確認するため、`VkPhysicalDeviceHostImageCopyFeaturesEXT`（または 1.4 相当）をチェックしなければなりません。
====

== 同期の課題

Host Image Copies はデータを移す「方法」を簡素化しますが、「いつ」からは免除されません。ホスト（CPU）上で直接データを動かしているため、書き込み中に GPU が同じイメージを使おうとしていないことを極めて慎重に保証しなければなりません。

これは別種の同期を導入します。二つの GPU キューを同期するだけでなく、**ホスト** と **デバイス** を同期します。ホストの書き込みが GPU から **見える（visible）** こと、およびホストコピーを始める前に先行する GPU 作業が **利用可能（available）** であることを保証する必要があります。

== 探ること

この章では、ホスト側同期の世界に入ります。次を探ります。

1.  **直接の CPU からイメージへのアクセス**: コマンドバッファオーバーヘッドなしでデータを効率的に移すため、新しい Vulkan 1.4 Host Image Copy 機能をどう活用するか。
2.  **可視性とフラッシュ**: ホストマップメモリ向けに `vk::MemoryBarrier2` を極める。バス越しのデータのコヒーレンシを保証し、CPU で書いたバイトが GPU が見るものと正確に一致するようにする方法を学びます。
3.  **ホスト–デバイスハンドシェイク**: ホスト側コピーがアクティブな GPU レンダリングと衝突しないよう、フェンスとタイムラインセマフォで協調する。

この章の終わりには、モダン Vulkan で直接メモリアクセスをどう管理するかの完全な理解が得られ、アセットを更新し続けるためのより速く柔軟な方法を手にしているはずです。
