= ダイナミックレンダリングにおける同期: パスレスの世界

== はじめに

Vulkan 同期の初期のかなりの期間は、**レンダーパス** と **サブパス** の概念に大きく結びついていました。モバイル GPU のオンタイルメモリ利用を最適化する助けになるよう設計されていましたが、開発者にはしばしば混乱を招き、過度に複雑なコードにつながりました。「サブパス依存」がレンダーパスの異なるステージ間でデータを同期する主な方法でしたが、モダンエンジンの働き方と必ずしも一致しないレガシー構造のように感じられました。

**ダイナミックレンダリング**（Vulkan 1.3 で導入）の登場により、API はこれらの硬直した構造から離れました。管理すべき `VkRenderPass` や `VkFramebuffer` オブジェクトはもうありません。代わりに、単に `beginRendering` と `endRendering` を呼びます。この変化により Vulkan は使いやすくなりましたが、同期の責任も完全に私たちへ移りました。

== 明示的な時代

サブパス依存のない世界では、すべての同期点は明示的でなければなりません。同じレンダリングブロック内で、あるドローコールの出力を別の入力として使いたい場合、遷移をレンダーパスに任せることはもうできません。第 3 章で学んだ **Synchronization 2** バリアを使わなければなりません。

この転換は実際には大きな利点です。はるかに明確な制御が得られます。自分で記録しているので、遷移がどこで起きているかが正確に分かります。また、レンダリングパスが流動的で実行時に決まることが多いモダンエンジンアーキテクチャとの統合もはるかに容易になります。

== 探ること

この章では、このモダンでパスレスな風景で同期がどう機能するかを掘り下げます。次を探ります。

1.  **サブパスの置き換え**: レガシーな `VkSubpassDependency` 構造体を置き換え、レンダリングアタッチメント間の同期を明示的なバリアでどう協調させるか。
2.  **ローカルリード同期**: **Vulkan 1.4** で最も興奮する機能の一つ、`VK_KHR_dynamic_rendering_local_read` を見ます。これにより、レガシーサブパスと同じ性能でオンタイル操作（フラグメントシェーダでのデプスバッファ読み取りなど）を、ダイナミックレンダリングのシンプルさで行えます。

この章の終わりには、最新の Vulkan 機能を使った高性能レンダラを自信を持って設計でき、同期がレンダリングコードと同じくスリムで効率的であることを保証できるようになるはずです。
