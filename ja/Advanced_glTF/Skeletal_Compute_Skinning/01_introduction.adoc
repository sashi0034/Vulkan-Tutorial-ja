= 高度なスケルタル＆コンピュートスキニング

== スキニングとは実際に何か

「シンプルなエンジンの構築」シリーズを終えていれば、スケルタルアニメーションの動作はすでに見ています。glTF ファイルを読み込み、アニメーションチャネルをパースし、毎フレームボーン行列を更新してモデルのジョイントを動かしました。しかしそのチュートリアルでは意図的に素直に保ちました。メッシュ変形—頂点をボーンに *追従* させる実際の処理—は、通常のラスタライゼーションパイプラインの一部として、頂点シェーダー内で頂点ごとに行われました。

そのアプローチは動きます。アニメーションするキャラクターが少数の単純なシーンでは十分許容できます。しかし規模が大きくなって初めて顕在化する隠れたコストがあります。変形後メッシュへのアクセスを必要とするすべてのシステムが、孤立して自ら数学を行わなければならないことです。

シーンに物理を追加すると何が起きるか考えてください。キャラクターのラグドールは、環境オブジェクトが元の T ポーズではなく *アニメーションした* 身体に衝突するよう、メッシュ表面の位置を知る必要があります。しかし物理エンジンは頂点シェーダーを知りません。変形後の位置は見えません。したがって、物理システム向けに CPU 上で第二のスキニング計算一式を走らせるか、物理コライダーが誤った場所にあることを受け入れるかです—アニメーションが劇的になるほど悪く見える近似です。

そこにレイトレーシングを加えると問題は複合します。Vulkan Ray Tracing Acceleration Structure—反射・アンビエントオクルージョン・レイトレース影などの効果を可能にするデータ構造—はメッシュの頂点位置から構築されます。スキニングがラスタライゼーションの頂点シェーダー内でのみ行われるために位置がまだ T ポーズなら、影と反射はアニメーションしたキャラクターではなく生気のないマネキンから投射されます。

これらすべての問題への解が、私たちが **"Skin Once, Use Everywhere"** と呼ぶ戦略です。描画の副作用としてメッシュをスキニングする代わりに、専用ステップとして *先に* スキニングし、結果の変形頂点位置を GPU バッファに格納します。ラスタライザ・レイトレーサ・物理システムはすべてその *同じ* バッファから読みます。すべてのシステムが正しいアニメーションメッシュを見て、高価な変形数学はフレームあたり一度だけです。

この戦略の実装には、スキニング作業を頂点シェーダーから **Vulkan コンピュートシェーダー** へ移す必要があります。本章は、そのコンピュートベースのスキニングパイプラインを一から構築することです。

== まず知っておくべきこと

実装に入る前に、本章がすでに理解していると仮定することと、ゼロから教えることを明示しておく価値があります。

「シンプルなエンジンの構築」シリーズからの glTF スケルタルアニメーションの基礎に慣れているべきです。具体的には、glTF ファイルにおける **ジョイント** と **スキン** とは何か、アニメーションチャネルが時間とともにボーン変換をどう駆動するか、`AnimationSampler` と `AnimationChannel` 構造がどう連携してポーズを生むかを理解しているべきです。これらの用語に不慣れなら、続ける前にその教材を見直してください。

本シリーズ前章で構築した **シーングラフ** システムもしっかり把握しているべきです。これから構築するスキニングシステムはその階層と深く統合されます。ジョイントのワールド行列を `Node` 配列から直接読むので、Dirty Flag パターンでそれらの行列がどう計算・キャッシュされるかの理解が不可欠です。

本章が *教える* のは、頂点のレスト位置とアニメーション位置の間の数学的関係—特に **Inverse Bind Matrix** の役割と、それがスキニング計算を元の T ポーズにどう固定するか—です。その後、その数学を Vulkan コンピュートシェーダーにどう符号化するか、入出力バッファをどう管理するか、出力を複数の下流システムが同時に消費できるようパイプラインをどう構造化するかを示します。

== 本章の計画

このシステムを層ごとに構築します。まず数学的基盤を確立します—各頂点に対して正確にどの計算が必要か、そしてなぜか。数学を理解しなければ、シェーダーコードは魔法を生む行列乗算の列にすぎません。理解すれば、シェーダーのすべての行が意図的で原理的なステップになります。

第二に、Vulkan コンピュートパイプラインを設計・実装します。入力頂点・ジョイント行列・出力頂点用のシェーダーストレージバッファの作成、Slang でのコンピュートシェーダー記述、毎フレームの正しいディスパッチを含みます。

第三に、出力バッファを共有リソースとして配線します。ラスタライゼーションパイプラインが元の頂点バッファの代わりにそれを読む方法、レイトレーシングの Bottom-Level Acceleration Structure（BLAS）のジオメトリソースとして使う方法、正確な衝突クエリのために物理システムへ公開する方法を示します。

最後に、アニメーションのより高度な側面に取り組みます。滑らかで自然な動きのための **三次スプライン補間** と、アニメーショクリップ間の遷移のための **クロスフェードブレンド**—スクリプトアニメーションから物理駆動ラグドールへのブレンドという重要なケースを含みます。
