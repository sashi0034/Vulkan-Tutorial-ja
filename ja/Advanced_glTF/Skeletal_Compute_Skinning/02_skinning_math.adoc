= スキニングの数学

== 数学が重要な理由

スキニングをブラックボックスとして扱う誘惑があります。GPU にボーンとメッシュを渡すと、頂点がどこへ行くべきかを計算してくれる、というわけです。多くの開発者が何年もこう働きます。しかし何かがうまくいかないとき—スケルタルアニメーションではいつか必ずうまくいかない—基盤の数学を理解している開発者は数分で診断でき、理解していない者は、ミキサーにかけられたような歪んだメッシュを見つめるだけで理由が分かりません。

本節は簡潔な概要ではありません。第一原理からスキニング方程式を構築し、終わりまでに、次節のコンピュートシェーダーコードを見て、すべての行列乗算がなぜそこにあるかを正確に理解できるようになるべきです。

== バインドポーズとその存在理由

すべてのスキニングされたキャラクターは、**バインドポーズ**（**T ポーズ** または **レストポーズ** とも呼ばれる）で人生を始めます。これはアーティストがメッシュをリグにアタッチしたときのスケルトンの特定構成です。スキニングウェイトがペイントされたとき、ボーンとメッシュが完璧に揃っていた「中立」状態です。

バインドポーズが極めて重要なのは、glTF ファイル内のすべてのアニメーションデータが *バインドポーズからのデルタ* として表現されるからです。ジョイントのアニメーションチャネルが「Z 軸まわりに 30 度回転」と言うとき、それは「バインドポーズ回転より 30 度多く」を意味します。これは、バインドポーズがアニメーションデータをメッシュ幾何に結びつける共有参照フレームであることを含意します。

glTF では、バインドポーズ情報は各ジョイントの **Inverse Bind Matrix** として格納されます。この行列は名前が示すとおりです。メッシュがスケルトンにバインドされた時点のジョイント変換行列の数学的逆です。バインドポーズでのジョイントのワールド行列が `B` なら、ファイルに格納される Inverse Bind Matrix は `B⁻¹` です。

なぜこの逆が必要かは、すぐに正確に見ます。

== 三段階のスキニング方程式

アニメーションした頂点がどこにあるべきかの計算は三段階のプロセスです。各ステップを個別に理解すると、合成された方程式は神秘ではなく自明になります。

=== ステップ 1: 頂点をボーンローカル空間へ変換

メッシュ内のすべての頂点は **モデル空間** に格納されます—位置はモデル原点に相対です。しかしジョイントの回転を適用するには、ジョイント自体が原点である **ボーンローカル空間** に頂点がある必要があります。

これがまさに Inverse Bind Matrix が行うことです。頂点位置にジョイントの Inverse Bind Matrix を掛けると、その頂点の位置を *バインドポーズに存在したそのジョイントに相対* で表現しています。本質的に、「バインドポーズのジョイントが自身の座標系の原点に座っていたら、この頂点はどこにあるか？」と問うています。

例えば、キャラクターの手首ジョイントがバインドポーズで位置 `(0.5, 1.2, 0.0)` にあったなら、手首の Inverse Bind Matrix はそのオフセットを符号化します。手近くの頂点にこの行列を掛けると、結果はワールド空間のどこかではなく—手首に近い—原点近くの位置になります。

=== ステップ 2: 現在のジョイント変換を適用

頂点がバインドポーズのジョイントに相対で表現された今、*現在の* ジョイント変換を適用できます。これはジョイントのアニメーションしたワールド行列—前章で Dirty Flag システムを使ってシーングラフが計算したもの—です。

ボーンローカルの頂点位置に現在のジョイントワールド行列を掛けるとき、こう言っています。「バインドポーズのジョイントに相対で位置づけられたこの頂点を、アニメーションしたジョイントが今ある場所へ動かせ。」手首がバインドポーズから現在 45 度回転しているなら、まず手首の原点に相対で表現したので、頂点はその回転を正しく追従します。

単一ジョイントの合成操作は次です。

`Animated Position = Joint World Matrix * Inverse Bind Matrix * Rest Position`

これを右から左へ読むと（行列乗算の働きどおり）: レスト位置を取り、ジョイントのローカル空間へ変換し、ジョイントのローカル空間から最終的なアニメーションワールド空間へ変換します。

=== ステップ 3: 複数ジョイント影響をブレンド

実キャラクターメッシュは単一ボーンに駆動されません。各頂点は複数ジョイントに同時に影響されます。これを **Linear Blend Skinning（LBS）**、または **Smooth Skinning** と呼びます。これがないと、キャラクターメッシュは実スケルトン上の皮膚のように滑らかに伸びるのではなく、段ボールのようにジョイントで折れ曲がりしわくちゃになります。

glTF では、各頂点は最大四つの **ジョイントインデックス** と対応する四つの **ウェイト** を格納します。ジョイントインデックスはこの頂点に影響するボーンを識別し、ウェイトは各ボーンがどれだけ寄与するかを指定します。頂点が誤ってスケールまたは平行移動されないよう、ウェイトは合計 1.0 でなければなりません。

単一頂点の完全なスキニング方程式は、ジョイントごとの変換の加重和です。

----
Skinned_Position = (w0 * J0 * IB0 + w1 * J1 * IB1 + w2 * J2 * IB2 + w3 * J3 * IB3) * Rest_Position
----

ここで `Jn` はジョイント `n` の現在のワールド行列、`IBn` はその Inverse Bind Matrix、`wn` はそのジョイントに対する頂点のウェイトです。

この加重和は頂点の **Skin Matrix** と呼ばれることもあります。ジョイント近くの異なる頂点は異なるウェイト分布を持ちます—肘ちょうど上の頂点は上腕 50%・前腕 50% かもしれないし、腕の中ほどは上腕 90%・前腕 10% かもしれません。アーティストはモデリングソフトのウェイトペイントツールでこれらのウェイトを制御します。

== ジョイント行列: 鍵操作の事前計算

上の方程式では、すべての頂点について `Jn * IBn` を掛けています。しかしこの積は、同じジョイントに影響されるすべての頂点で同じであることに注意してください。頂点 100 を計算していても頂点 100,000 を計算していても—両方ともジョイント `5` に影響されるなら、同じ `J5 * IB5` 行列を使います。

つまり、GPU へアップロードする前に、これらの積を CPU 上で事前計算でき、またそうすべきです。各シェーダー起動が二つの行列をルックアップして掛ける代わりに、事前乗算した **ジョイント行列**—ジョイントごとに一つ—の配列を作り、それだけを GPU へアップロードします。

[source,cpp]
----
// Called once per frame, after the animation update and scene graph update
void compute_joint_matrices(
    const Skin& skin,
    const std::vector<Node>& nodes,
    std::vector<glm::mat4>& joint_matrices_out)
{
    joint_matrices_out.resize(skin.joints.size());

    for (size_t i = 0; i < skin.joints.size(); ++i) {
        // Get the current world matrix of this joint's node
        // (already computed by the scene graph's dirty flag system)
        const Node& joint_node = nodes[skin.joints[i]];
        const glm::mat4& current_world = joint_node.world_matrix;

        // Multiply: Current Joint Transform * Inverse Bind Matrix
        // This pre-computes the per-joint portion of the skinning equation.
        // The shader only needs to multiply this by the vertex's rest position.
        joint_matrices_out[i] = current_world * skin.inverse_bind_matrices[i];
    }
}
----

ここでの `Skin` 構造は glTF ファイルからパースしたものと対応します。ジョイントノードインデックスのリストと対応する inverse bind matrices です。出力 `joint_matrices_out` は、**Shader Storage Buffer Object（SSBO）**—コンピュートシェーダーが自由にインデックスできる大きな GPU アクセス可能バッファ—として GPU へアップロードするものです。

== 法線の扱い

上のスキニング方程式は頂点位置を正しく変換しますが、完全なスキニング実装は **頂点法線** も変換しなければなりません。法線はメッシュ表面に垂直なベクトルで、すべてのライティング計算を駆動します。メッシュを変形しても法線を合わせなければ、キャラクターはまだ T ポーズにいるかのように照らされて見え—違和感のある視覚アーティファクトです。

法線は完全なジョイント行列（平行移動を含む）では変換できません。代わりに **Normal Matrix**—ジョイント行列の 3x3 回転・スケール部分の逆の転置—で変換しなければなりません。ほとんどの場合—そしてこれがキャラクターアニメーションの標準慣行ですが—ジョイント行列に非均一スケールが含まれないと仮定できます。そのシナリオでは、法線行列はジョイント行列の 3x3 部分そのものに簡略化されます。

[source,cpp]
----
// For joint matrices without non-uniform scale, the normal transform is simpler:
glm::mat3 normal_matrix = glm::mat3(joint_matrix);

// If non-uniform scale is present (e.g., squash-and-stretch animation):
// glm::mat3 normal_matrix = glm::transpose(glm::inverse(glm::mat3(joint_matrix)));
----

これは明示しておく価値のあるトレードオフです。完全な法線行列計算を省略するのは、実践ではほぼ常に見えない性能最適化です。キャラクターリグがスケルタルジョイントに非均一スケールを使うことは稀だからです。エンジンがいつかスカッシュアンドストレッチのボーンアニメーションをサポートする必要があれば、この仮定を再訪する必要があります。

== 接線と従法線

法線マップを使うキャラクター—AAA 制作ではほぼ普遍的—では、**接線ベクトル** もスキニングする必要があります。接線と従法線は、法線マップのテクスチャ空間を変形後メッシュのワールド空間へ写す頂点ごとの座標系を定義します。メッシュが変形しても接線フレームが追従しなければ、法線マップ付きライティングは不正になります。

接線は法線と同じアプローチ—平行移動なしのジョイント行列の 3x3 回転部分—で変換されます。従法線は通常直接格納されず、フラグメントシェーダーで法線と接線の外積として再構成され、接線の W 成分に格納された向き（handedness）符号を使います。

この数学的基盤が整った今、コンピュートシェーダーを書くのに必要なものはすべて揃いました。必要な入力（レスト位置、法線、接線、ジョイントインデックス、ウェイト、事前計算したジョイント行列）、実行すべき計算、生成すべき出力（他のすべてが読める新しいバッファ内のアニメーション位置・法線・接線）が分かっています。
