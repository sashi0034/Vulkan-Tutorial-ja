= スケルタル＆コンピュートスキニング: まとめ

== 構築したもの

本章では、アニメーションシステムを頂点シェーダーベースの近似から、専用の本番品質コンピュートパイプラインへ変えました。構築したものの全体像と、各片がなぜ重要かをひととき味わいましょう。

まず数学的基盤から始め、Inverse Bind Matrix がなぜ存在し、どの問題を解くかを正確に確立しました。三段階のスキニング方程式—ボーンローカル空間への変換、アニメーションしたジョイント変換の適用、複数影響のブレンド—はもはやブラックボックスではありません。壊れた、またはずれたスキニングメッシュを見て、方程式のどのステップが原因らしいかを推論できるようになりました。

そのスキニング数学を頂点シェーダーから Vulkan コンピュートシェーダーへ移し、その過程で、キャラクターメッシュの権威あるアニメーション状態を保持する持続的な GPU 常駐出力頂点バッファを作りました。このアーキテクチャ変更が、その後のすべてを可能にしました。

ラスタライザ・レイトレーシング BLAS・物理リードバックシステムを同じ出力バッファに向けることで、素朴な実装を苦しめる断片化を排除しました。T ポーズの影はもうありません。視覚メッシュから遅れる衝突ハルもありません。異なるレンダリング機能のためにスキニングコストを複数回払うこともありません。一度のディスパッチ、一つのバッファ、誰もが恩恵を受けます。

最後に、動きそのものの品質を改善しました。線形補間（正確だが生気がない）から三次 Hermite スプライン（アニメー夕の意図を保つ）へ移し、キャラクターが急にスナップするのではなく滑らかに状態遷移するようクロスフェードブレンドを追加しました。

== アニメーションからシーングラフへの橋渡し

初心者によくある混乱点は、アニメーションシステムの出力—`Pose`—が実際にワールド内でキャラクターをどう動かすかです。`Pose` はスケルトン内のすべてのジョイントのローカルトランスフォームを含みますが、それらのジョイントはシーングラフ内の `Node` オブジェクトでもあります。両者を橋渡しするには、ポーズデータをノードへ適用しなければなりません。

[source,cpp]
----
void apply_pose_to_scene_graph(
    std::vector<Node>& nodes,
    const Pose& pose,
    const std::vector<uint32_t>& joint_indices)
{
    // For each joint in the animation, update its corresponding node
    for (size_t i = 0; i < joint_indices.size(); ++i) {
        Node& node = nodes[joint_indices[i]];
        node.translation    = pose.translations[i];
        node.local_rotation = pose.rotations[i];
        node.scale          = pose.scales[i];

        // Mark as dirty so world matrices are recomputed!
        node.mark_dirty();
    }
}
----

このステップが、キャラクター固有のアニメーションシステムとエンジン全体のシーングラフの間の「接着剤」です。適用されると、次のシーングラフ更新がこれらのアニメーションしたトランスフォームを階層へ伝播し、キャラクターにアタッチされたもの（剣やヘルメットなど）が動きに完璧に追従することを保証します。

== 理解しておくべきトレードオフ

先へ進む前に、採ったアプローチの限界とトレードオフを明確にしておく価値があります。

**線形ブレンドスキニングは完璧ではない。** LBS は速く概ね「十分良い」ため業界標準ですが、よく知られたアーティファクト **"キャンディラッパー" または "つぶれる肘" 問題** があります。ジョイントが 180 度回転するとき（手首が手のひら上から下へ回転するなど）、LBS は体積ゼロを通って補間し—ジョイントでメッシュがつぶれてから再び膨らみます。代替の **Dual Quaternion Skinning（DQS）** は、行列ではなくクォータニオンベースの剛体変換をブレンドすることでこのアーティファクトを避けます。多くの本番ゲームは主要キャラクターに DQS、遠いまたは重要度の低いオブジェクトに LBS を使います。DQS の実装はここで構築した数学の素直な拡張です—コンピュートシェーダー構造は同一で、ブレンド式だけが変わります。

**BLAS リフィットには品質コストがある。** BLAS リフィットは完全リビルドより遥かに速いですが、BVH 木構造はレストポーズ幾何向けに最適化されており、メッシュがそのレストポーズから大きく変形するにつれ徐々に最適でなくなります。極端なケース（非常に大きく速い動作をするキャラクター）では、レイ走査性能の低下につながり得ます。一般的な本番技法は、間のフレームではリフィットを使いながら、時折（N フレームごと、またはスケルトンが閾値を超えて動いたとき）完全な BLAS リビルドを発行することです。

**物理のための GPU から CPU へのリードバックは最後の手段。** 完全性のためにこの選択肢を扱いましたが、キャラクター物理の正しいデフォルトは、CPU 側ジョイント行列から直接駆動するボーンプロキシコライダーです。リードバックアプローチは、布・ソフトボディ・メッシュ正確なクエリなど、実用的な代替がない特定ケースに予約すべきです。

== 次に来ること

構築したシステムはキャラクター変形の「どうやるか」を極めてよく扱います。しかしまだ扱っていないキャラクターパイプラインの層全体があります。キャラクター身体の物理プロパティです。

現時点では、正しく変形する視覚メッシュと、それを駆動するジョイントトランスフォームのセットがあります。しかし各ボーンが表す物理形状の形式的定義はありません。キャラクターの肘が誤った方向に曲がるのを防ぐジョイント制限もありません。ラグドールのジョイント同士がどう相互作用するかを定義する拘束もありません。

次章 **物理統合: コライダーとラグドール** で、この物理層を構築します。ボーンプロキシ衝突カプセルを自動生成し、適切な制限付きジョイント拘束を定義し（ラグドールでも腕が正しく曲がるように）、アニメーション制御と物理制御の間の遷移を司るステートマシンを実装します。前章で触れたトピック—衝突グループフィルタリング—もより深く扱い、自分自身と戦わずにワールドと正しく衝突するキャラクターの構築を探ります。

== 検証: 確認すべき点

コンピュートスキニング実装を検証するには:
1.  **Inverse Bind Matrices**: メッシュが「爆発」したり裏返ったりして見える場合、inverse bind matrices が誤った空間または座標系で適用されている可能性が高いです。
2.  **パイプラインバリア**: 「ジッター」やちらつくジオメトリが見える場合、コンピュートディスパッチとドローコールの間に `VkPipelineBarrier2` が正しく置かれているか確認します。
3.  **法線/接線の品質**: アニメーションするキャラクターのライティングがおかしい場合（例: ジョイントに継ぎ目が現れる）、Slang シェーダーでアニメーションした法線と接線を正しく正規化しているか確認します。
