= デバッグと視覚監査

== アセットは正しく、エンジンが間違っているとき

前章では、glTF アセットがエンジンに到達する前に正しいことを保証する方法を確立しました。本章は、アセットが検証を通過し、参照ビューアでは正しい結果を示しているのに、自分のエンジンではまだ何かがおかしく描画される、その瞬間から始まります。

この段階では、問題はエンジンコード側にあります。問いは「コードのどの部分か」であり、そこで視覚デバッグツールが不可欠になります。本シリーズで構築してきたキャラクターパイプラインには、それぞれ独立して複雑な段階がいくつかあります。シーングラフのトランスフォーム階層、コンピュートスキニングパス、物理シミュレーション、プロシージャルアニメーションレイヤーです。視覚的なアーティファクトは、そのいずれからも生じ得ます。各段階の状態を可視化するツールがなければ、当てずっぽうに戻ってしまいます。

本章の技法は、見えない状態を見えるようにすることです。物理シミュレーションはキャラクター身体のすべてのボーンについて完全な幾何表現を持っていますが、物理エンジンはデフォルトでは何も描画しないため見えません。スキニングシェーダーは最終的なアニメーション位置を含む頂点ごとの出力バッファを計算しますが、どの形式を見るべきかを知らなければ GPU デバッガで検査できません。頂点変形を駆動するボーン影響ウェイトは頂点あたり 4 つの float として格納されていますが、描画されたメッシュを見ただけでは正しいかどうか分かりません。

これらの死角それぞれに対処するツールを実装します。デバッグドロワーは、スケルトン・衝突形状・物理拘束の可視化をキャラクターに重ねるワイヤーフレーム幾何を描画します。スキニングヒートマップは、ボーン影響ウェイトから導いた頂点色でメッシュを描画し、不正または極端なウェイトペイントを即座に見えるようにします。RenderDoc のキャプチャと解析によりフレームを凍結し、スキニング後の出力バッファを直接調べ、GPU レベルで頂点位置と法線を検証できます。
