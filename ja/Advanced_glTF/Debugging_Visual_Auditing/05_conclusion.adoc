= デバッグ: まとめとシリーズの結論

== 本章で構築したもの

デバッグは単一のツールや技法ではなく、方法論です。可能な失敗原因の空間を体系的に絞り込む、順序立てて問う一連の質問です。

まずデバッグドロワーで、見えない物理世界を可視化しました。遅延ラインバッファ方式—フレームごとにセグメントを蓄積し、一度アップロードして単一コールで描画—により、開発中に有効にしたままでもフレームレートへの影響を抑えられます。スケルトンドロワーは、シーングラフが各ジョイントをワールド空間のどこに置くかを正確に示し、トランスフォーム階層の誤りを即座に見えるようにします。衝突形状ドロワー（キネマティックは緑、ダイナミックは赤で色分け）は、描画メッシュと並べて物理表現を示し、物理ボディがアニメーションを正しく追従しているかを一目で分かります。拘束ドロワーはラグドールの接続構造を示し、ラグドールが爆発したときに拘束が失敗したのか（ボディが離れている）、設定が間違っていたのか（アンカー点が誤り）を見分けられます。

スキニングヒートマップは診断の焦点を頂点スキニングデータ自体へ移しました。支配的ボーンヒートマップは、最も影響の大きいジョイントで各頂点を色付けし、逸脱したウェイト割り当てや不正なボーン領域を異常な色のパッチとして明らかにします。ウェイト分布ヒートマップはウェイト分布の複雑さで色付けし、ノイジーまたは過度に複雑なウェイトペイントの領域をフラグします。

RenderDoc 解析は GPU レベルの真値を提供しました。フレーム単位でスキニングコンピュート出力バッファの正確な内容を調べ、期待値に対する頂点位置を検証し、法線ベクトルの正しさを確認し、疑わしい特定頂点のシェーダー起動をステップ実行できます。`vkSetDebugUtilsObjectNameEXT` と `vkCmdBeginDebugUtilsLabelEXT` で GPU コマンドとバッファにラベルを付ける規律により、RenderDoc のリソースブラウザは圧倒的ではなくナビゲート可能なものになります。

== シリーズの振り返り

これら 8 章を通じて、Vulkan 上に本番品質のキャラクターパイプライン一式を構築しました。ファイルからフレームまでのデータの全経路をたどりましょう。

**glTF ファイル**はスケルトン階層、頂点スキニングデータ、アニメーション、物理 extras メタデータ、モーフターゲットを運びます。**第 2 章**では、その階層を Dirty Flag 伝播システム付きの再帰シーングラフに読み込み、ワールド行列を効率的に更新する方法を示しました。**第 3 章**ではスキニングデータを使い、GPU 上でメッシュを変形する Vulkan コンピュートパイプラインを構築し、結果をラスタライザ・レイトレーシングシステム・物理クエリが消費する共有バッファへ書き込みました。

**第 4 章**では物理シミュレーション層を確立しました。ボーン階層から導いたプロキシコライダー、人間の可動域を符号化した拘束、ANIMATED/BLENDED/RAGDOLL の三状態ハンドオフ、ビットマスクによる衝突フィルタリングです。**第 5 章**ではその上にプロシージャルアニメーションを追加しました。足置きとリーチのための CCD/FABRIK IK、頭部追従のルックアットコントローラ、脊椎回転として適用する物理由来のリーンです。

**第 6 章**ではコンピュートスキニングパイプラインをモーフターゲット対応に拡張し、Vulkan 1.4 のディスクリプタインデクシングで、ドローごとのディスクリプタ入れ替えなしに無制限の変位バッファ配列を扱いました。**第 7 章**では一歩下がって制作ワークフローを確立しました。命名規則、エクスポート設定、自動検証、参照ビューア監査です。**第 8 章**ではエンジン側デバッグツールでループを閉じました。

構築したシステムはデモではなく、本番フレームワークです。それぞれが他と合成できるように設計されています。IK は物理システムが駆動するのと同じシーングラフ上で動作し、モーフターゲットはラグドールが読むのと同じコンピュートシェーダーに統合され、デバッグドロワーはラグドールシステムが書き込むのと同じ物理ワールドデータを使います。この合成可能性こそ、各段階で慎重に行ったアーキテクチャ判断の成果です。

この基盤の上に次に作るキャラクターは、ここで学んだすべての教訓の恩恵を受けます。フレームワークは成長し、アセットパイプラインは進化します。しかし基本アーキテクチャ—シーングラフ、コンピュートスキニング、物理統合、プロシージャルレイヤー—は、他のすべてが掛かる骨格として残ります。

== 検証: 確認すべき点

デバッグツールを検証するには:

1.  **フラッシュ動作**: `DebugDrawer` が毎回の `flush()` 呼び出し後にラインバッファを正しくクリアすることを確認します。前フレームの線が残る場合、バッファがクリアされていません。
2.  **座標系**: デバッグ線がワールド空間に現れることを確認します。よくある誤りはローカル空間座標をドロワーに渡すことで、デバッグ幾何がキャラクターに正しく追従しなくなります。
3.  **RenderDoc 検査**: RenderDoc の **Buffer Viewer** でコンピュートスキニング出力バッファを検査します。バッファの値が期待どおりなのにキャラクターが正しく描画されない場合、問題は頂点入力記述またはグラフィックスパイプライン状態にある可能性が高いです。
