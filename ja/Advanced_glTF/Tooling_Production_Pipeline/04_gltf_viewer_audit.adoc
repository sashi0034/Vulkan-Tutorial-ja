= glTF ビューア監査: 真値の確立

== 真値の目的

アセットが検証を通過したあと、仕様準拠であることは分かります。見た目が正しいかどうかはまだ分かりません。glTF-Validator はファイルが整形式だと教えてくれますが、法線が正しい向きか、スケルトンがメッシュを正しく駆動するか、アニメーショクリップがアニメー夕の意図どおり再生されるかは教えてくれません。それが参照ビューアの役割です。

**真値** とは、アセットの既知の正しいレンダリング—glTF 仕様の成熟し徹底的にテストされた実装が生み出すもの—であり、エンジンの出力と比較できます。両者が一致すれば、その後の不一致はほぼ確実にエンジンコード側です。異なれば、不一致がアセット、または仕様に対するエンジンの解釈に由来し、神秘的なグラフィックス API の挙動ではないという証拠があります。

エンジンコードをデバッグする前に真値を確立する規律はてこが効きます。それがなければ、すべての視覚アーティファクトは曖昧です。シェーダーバグかもしれないし、Vulkan API の誤用、誤った行列、アセット問題かもしれません。真値参照があれば、アーティファクトを即座に分類できます。参照ビューアが同じ問題を示すならアセット問題—blend ファイルを直します。参照ビューアは正しいのにエンジンが違うならエンジン問題—コードをデバッグします。この分類だけで何時間も節約できます。

== 推奨ビューア

成熟した高品質な glTF ビューアがいくつか無料で利用できます。私たちの目的では二つを推奨します。

**Khronos glTF Sample Viewer**（https://github.khronos.org/glTF-Sample-Viewer-Release）。Khronos Group 自身—glTF 仕様を維持する同じ組織—による参照実装です。WebGL 上に構築され、モダンなブラウザであればインストールなしで即座に使えます。完全な glTF PBR マテリアルモデル、すべてのアニメーション補間モード（linear、step、cubic spline）、スキニング、モーフターゲットを実装しています。参照実装であるため、仕様準拠に関する問いでは最も権威あるビューアです。ここでアセットがおかしく見えるなら、問題は確実にアセット側です。

**Babylon.js Sandbox**（https://sandbox.babylonjs.com）。成熟した商用級実装で、すべての glTF 機能をよくサポートし、個々のアニメーション・ボーン・モーフターゲットを切り分けて検査しやすい UI がよく維持されています。ノードインスペクタでシーングラフ階層を調べ、各アニメーションフレームのボーントランスフォームを検査し、モーフターゲットウェイトが正しく駆動されているかを確認できます。スケルタルアニメーション問題のデバッグには、Khronos ビューアより有用なことが多いです。

どちらのビューアもドラッグ＆ドロップで `.glb` を受け付けるため、監査ワークフローは速いです。Blender からエクスポートし、ビューアへドラッグすれば、数秒で参照レンダリングが得られます。

== ビューアで見るべきこと

体系的な監査では、次を順に確認すべきです。

**レストポーズ。** ファイルを開き、アニメーションがあればすぐに一時停止します。キャラクターは Blender でバインドポーズとして確立された T ポーズまたは A ポーズであるべきです。レストポーズで変形している—肢が曲がり、メッシュが誤ったボーンへ引っ張られる—場合、スキニングウェイトまたは inverse bind matrices がソースで不正です。

**マテリアル。** すべてのマテリアルが正しい色・ラフネス・メタルネス・法線マップで描画されることを確認します。法線マップが正しい場所で凸または凹の結果を生むかに特に注意してください—法線マップの緑チャネルの反転はよくあるエクスポートエラーで、細かい表面詳細で反転したシェーディングとして現れます。

**アニメーション再生。** ビューアで各アニメーショクリップを再生し、アーティストの意図と一致することを確認します。誤ったメッシュ領域を駆動するボーン、誤った速度で再生されるアニメーション（タイムスタンプのスケール問題）、滑らかに補間せずスナップやジャンプするクリップを監視します。クリップが三次スプライン補間を使う場合、接線が滑らかな曲線を生むことを確認します—よくある誤りは、クリップ中盤に振動アーティファクトを生む誤ったスケールの接線をエクスポートすることです。

**モーフターゲット。** ビューアのマテリアルまたはアニメーションインスペクタで、各モーフターゲットウェイトを 0 から 1 へ手動で駆動し、顔が期待どおり変形することを確認します。モーフターゲットに効果がないように見える場合、変位アクセサがすべてゼロでエクスポートされたか、ターゲットが誤ったメッシュプリミティブに割り当てられた可能性があります。極端な変形を生む場合、変位ベクトルが誤った座標系にある（スケールまたは軸の慣習の問題）可能性があります。

**スケルトン階層。** ボーン検査をサポートするビューア（Babylon.js Sandbox など）でシーングラフを展開し、親子関係が Blender リグ階層と一致することを確認します。リーフボーン（誤ってエクスポートされた場合）が重要なジョイントとして扱われていないことを確認します。

== ビューアからエンジンへ: 異なるときの対処

エンジンのレンダリングがビューア参照と異なるとき、デバッグは構造化されます。

**マテリアル** が異なる場合: PBR シェーダーを glTF 仕様の BRDF 式と照合します。最もよくある差は、ラフネスから知覚ラフネスへの変換（glTF は知覚ラフネスを使い、一部の PBR 実装は線形ラフネスを使う）と、メタルネスワークフロー（glTF は常に metalness/roughness を使い、specular/glossiness は使わない）です。

**スケルトン** が間違っている場合: レストポーズでエンジンが計算するジョイントワールド行列を、ビューアが表示するものと比較します。エンジンが計算した行列をデバッグオーバーレイとしてエクスポートし、ジョイントごとに比較します。最もよくある原因は、誤った行列乗算順（`parent * local` 対 `local * parent`）と、inverse bind matrix の誤った適用（`joint_matrix = global_joint_transform * inverse_bind_matrix`。ここで global_joint_transform はルートからジョイントへシーングラフを走査して計算）です。

**モーフターゲット** が間違っている場合: 展開データを生の glTF JSON 値と比較し、expand_sparse_accessor 実装を検証します。スパースインデックスの散布ステップでオフバイワンが一つあるだけで、すべての変位が頂点一つずれ、顔が誤った領域で変形する特徴的な「泳ぐ」アーティファクトを生みます。

**アニメーション** が間違っている場合: glTF ファイルのサンプラー入力タイムスタンプを、エンジンが補間に使っているタイムスタンプと比較します。ファイルが秒で格納しエンジンがミリ秒として解釈する（またはその逆）場合、すべてのアニメーションが 1/1000 速または 1000 倍速で再生されます。三次スプライン補間式が glTF 仕様と正確に一致することも確認します—仕様は時間間隔で事前スケールされた接線ベクトル付きの Hermite 基底を使い、一部の教科書の提示とは異なります。
