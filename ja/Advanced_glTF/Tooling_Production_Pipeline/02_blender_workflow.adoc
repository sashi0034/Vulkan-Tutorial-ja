= Blender から Vulkan へのワークフロー

== 命名規則：暗黙の契約

アーティストとエンジンプログラマーの間で最も一貫して混乱を招くのが命名です。IK チェーンを接続するために `"Spine1"` という名前のボーンを探すエンジンローダーは、リグが `"spine_1"`、`"Spine_01"`、または `"spine.001"`（Blender でボーンを複製したときのデフォルト名）を使っている場合、静かに失敗します。これはローダーにもリグにもバグがあるわけではなく、両者の間の命名契約が崩れているのです。

最初のキャラクターを作る前に命名規則を定め、文書化し、一貫して守ることは、小規模チームができる最も効果の高い投資の一つです。規則は複雑である必要はありません—合意され、守られていれば十分です。

スケルトンのジョイントについては、実践でうまく機能する標準は次のとおりです：すべて小文字、単語はアンダースコアで区切る、左右はそれぞれ `_l` または `_r` の接尾辞で示す、そして一貫した階層命名パターンを用いる。この規則に従ったヒューマノイドの脊椎チェーンは `hips`、`spine`、`spine_upper`、`chest`、`neck`、`head` となります。脚のチェーンは `thigh_l`、`shin_l`、`foot_l`、`toes_l` です。これは Blender の組み込み Rigify システムや Mixamo が使う規則であり、既存のアニメーションコンテンツの多くと互換性があります。

エンジンの利便性のため、ローダーコードはジョイント名をハードコードしてはいけません—glTF ファイルから、または意味論的な名前を glTF 名にマッピングする小さな設定ファイルからそれらを発見すべきです。これにより、異なるアセットが異なる命名規則を使ってもエンジンコードの変更は不要になります。ただし、第 2 章で定義した extras ベースの物理・制約メタデータは `"parent_bone"` をジョイント名への文字列参照として使うため、それらはエクスポートされたジョイント名と完全に一致している必要があります。

== 頂点グループとスキニングウェイト

Blender では、スケルタルスキニングは **頂点グループ** によって定義されます。これは名前付きの頂点集合で、各頂点にはウェイト値（0 から 1 の間）があり、そのグループのボーンがどれだけ影響するかを示します。Blender は頂点グループのウェイトを glTF ファイルの `JOINTS` および `WEIGHTS` アクセサとしてエクスポートします。

Blender 側で次の問題が発生すると、エクスポート時にスキニングデータが破損する可能性があります：

**正規化されていないウェイト。** 頂点のボーンウェイトの合計が 1.0 でない場合、スキニング計算は誤った結果を生みます—頂点は移動に伴って縮んだり膨らんだりします。Blender のウェイトペイントモードには正規化オプションがあります。エクスポート前に使用してください。glTF 仕様は正規化されたウェイトを要求しませんが、スキニングシェーダーはほぼ確実に `sum(weight_i * joint_matrix_i)` としてブレンド行列を計算しており、これは正規化を前提としています。

**頂点あたり 4 つを超える影響。** glTF の標準スキニング（`JOINTS_0`/`WEIGHTS_0`）は、頂点あたりちょうど 4 つのボーン影響をサポートします。Blender のリグは、自動ウェイトペイントを使うと—特に脇の下や股のしわのようなジョイント交差部付近で—4 つを超える影響を頻繁に生成します。ウェイトペイントツールの Limit Total オプションは各頂点の影響を最大 N 個に減らします。エクスポート前に N を 4 に設定し、その後再正規化してください。

**ゼロウェイトの頂点グループ。** 頂点グループが存在するが頂点のウェイトがゼロの場合、Blender はそれを 4 つの影響の 1 つとして含めることがあり、意味のある影響に使えるスロットを無駄にします。Blender の Clean Weights オプションでこれを整理してください。

**レストポーズの不一致。** glTF エクスポートは、エクスポート時点のリグの現在のポーズをバインドポーズとしてキャプチャします。エクスポート時にリグがポーズされている（レストポーズでない）場合、すべての逆バインド行列が誤りになり、キャラクターは基本の T ポーズで変形して見えます。初回エクスポート前には必ずアーマチュアがレストポーズであることを確認してください（必要であれば `Pose mode → Pose → Apply Pose as Rest Pose` を実行してからエクスポート）。レストポーズからのみエクスポートするというワークフロールールにしてください。

== 物理 extras 用のカスタムプロパティ

第 2 章では、物理コライダーと制約の定義が glTF の "extras" JSON に保存され、エンジンがロード時に解析する方法を説明しました。Blender 側では、これらの extras は Blender の **カスタムプロパティ** パネルに由来します。ポーズモードのボーンで利用できます（Properties → Bone → Custom Properties）。

重要なのはデータ形式です。Blender はカスタムプロパティを extras フィールドの生 JSON 値としてエクスポートします。float プロパティは JSON 数値になり、string プロパティは JSON 文字列になります。最新の Blender glTF エクスポーターは辞書プロパティを直接処理し、手動の文字列エンコードなしで glTF の `extras` フィールドに複雑なネストされた JSON 構造を作成できます。

物理 extras に最もクリーンなアプローチは、Blender の Script Editor から実行する Python スクリプトです。スクリプトはアーマチュア内のすべてのボーンを反復し、設定辞書に基づいてプログラム的にカスタムプロパティを設定します：

[source,python]
----
import bpy, json

# Configuration: bone name -> physics settings
physics_config = {
    "shin_l": {
        "physics": {
            "collider": "capsule",
            "radius": 0.05,
            "half_height": 0.18,
            "mass": 3.0,
            "constraint": {
                "type": "hinge",
                "hinge_axis": [0, 0, 1],
                "limit_min_deg": -140,
                "limit_max_deg": 0,
                "parent_bone": "thigh_l"
            },
            "collision_group": "leg",
            "collision_mask": "world,props"
        }
    },
    # ... more bones
}

armature = bpy.data.objects["Armature"]
bpy.context.view_layer.objects.active = armature
bpy.ops.object.mode_set(mode='POSE')

for bone_name, config in physics_config.items():
    if bone_name in armature.pose.bones:
        bone = armature.pose.bones[bone_name]
        for key, value in config.items():
            # Modern Blender glTF exporters handle dicts directly,
            # ensuring 'extras' contains a proper JSON object.
            bone[key] = value

bpy.ops.object.mode_set(mode='OBJECT')
print("Physics properties applied.")
----

このスクリプトは、第 2 章の `parse_collider_extras` および `parse_constraint_def` 関数が期待するネストされた JSON 構造を生成します。最新の Blender は、これらの辞書プロパティが glTF の `extras` フィールドに適切な JSON オブジェクトとしてエクスポートされ、`tinygltf` を使って実装した C{pp} 解析ロジックと一致することを保証します。このスクリプトを blend ファイルと一緒に保存し、リグを変更するたびに実行してください。スクリプトを blend ファイルと一緒にバージョン管理することで、物理パラメータが再現可能で追跡可能になります。

== glTF エクスポートチェックリスト

Blender からエクスポートする際は、`Export glTF 2.0` ダイアログで次の設定を使用し、キャラクターデータがエンジンに正しく届くようにしてください：

*   **Format**: `glTF Binary (.glb)`
*   **Include**:
    *   [x] `Selected Objects`（メッシュとアーマチュアのみが選択されていることを確認）
    *   [x] `Custom Properties`（**重要**：物理メタデータはここに格納される）
*   **Transform**:
    *   [x] `+Y Up`（glTF の標準規約）
*   **Data -> Mesh**:
    *   [x] `UVs`, `Normals`, `Tangents`（PBR とスキニングに必要）
    *   [x] `Vertex Weights`（スケルタルアニメーションに必要）
    *   [x] `Shape Keys`（モーフターゲットに必要）
*   **Data -> Armature**:
    *   [x] `Use Rest Position Armature`
    *   [ ] `Export Deformation Bones Only`（IK が参照する非変形コントロールボーンがある場合は **オフ** のままにする）
    *   [ ] `Add Leaf Bones`（**オフ**：チェーン先端の余分なボーンを避ける）
*   **Animation**:
    *   [x] `Animation`（クリップをエクスポートするには有効化）
    *   [x] `Shape Key Animation`（表情がキーされている場合は必要）

== モーフターゲット（シェイプキー）ワークフロー

glTF のモーフターゲット（Blender では **シェイプキー** と呼ばれる）は、スケルタルアニメーションだけでは扱えない表情のような複雑な変形を可能にします。

1.  **作成**：`Mesh Data` プロパティパネルでシェイプキーを作成します。最初のキーは常に `Basis`（レストポーズ）です。追加のキーはその basis からの変位を定義します。
2.  **命名**：シェイプキーに意味論的な名前を付けます（例：`blink_l`、`smile`、`mouth_open`）。これらの名前は glTF の `extras.targetNames` フィールドにエクスポートされ、エンジンが名前でそれらを参照できるようになります。
3.  **範囲**：シェイプキーが加算的に設計されていることを確認してください。頂点が `smile` モーフと `blink` モーフの両方によって移動される場合、エンジンはそれらの変位を合算します。
4.  **エクスポート**：チェックリストで述べたとおり、glTF エクスポーターで **Shape Keys** と **Shape Key Animation** オプションが有効になっていることを確認してください。
5.  **検証**：エクスポート後、`.glb` を **glTF-Validator** で実行してください。シェイプキーが適切な場合に **Sparse Accessors** を使用しているかチェックします。これは複雑な顔リグにとって重要な最適化です。
