= ツーリングと制作パイプライン

== エンジンバグの隠れた原因

グラフィックスプログラマがやがて必ず遭遇し、デバッグツールが誤った方向を指すため深く苛立たしい問題のカテゴリがあります。アセットのバグがコードのバグのように見えるケースです。キャラクターの肢が後ろ向き、法線マップが完全に間違い、アニメーションが上腕を駆動すべきところで前腕を駆動する、ラグドールが有効化直後に崩れる、といった具合です。スキニングシェーダーを何時間もステップ実行し、クォータニオンの慣習を確認し、行列乗算の順序を検証した末に、結局 Blender のエクスポート設定が最初から間違っていたと分かる—エクスポートで法線が反転していた、あるいはジョイント命名規則がローダの期待と一致していなかった、といったことです。

本章は、その類の問題を体系的に防ぐためのものです。明確な制作パイプラインを確立します。アーティストのワークステーション上の Blender シーンを、検証済みでエンジン準備完了の glTF アセットへ運ぶ一連のステップで、各段階にチェックポイントを置き、エラーを可能な限り早く、診断しやすい文脈で捕捉します。

パイプラインには三段階あります。第一は **オーサリング**: アセットのエクスポート方法を決める Blender 側の選択です。リグ命名規則、頂点グループ割り当て、物理 extras 用のカスタムプロパティ定義、エクスポート設定がここに含まれます。これらを正しくすれば、結果の glTF ファイルは構造的に正しくなり、エンジンローダでの特別扱いが不要になります。第二は **検証**: エクスポートしたファイルを Khronos glTF-Validator に通し、ローダに届く前にフォーマット違反・欠落データ・仕様非準拠をチェックします。第三は **真値の確立**: プロ品質の glTF ビューアでアセットの見た目と挙動が正しいことを確認し、残る問題はエンジンコード側だと結論づける前の根拠を作ります。

華やかな作業ではありません。しかし、プロの制作パイプラインと、終わりなきデバッグループを分けるのは、まさにこの種の作業です。
