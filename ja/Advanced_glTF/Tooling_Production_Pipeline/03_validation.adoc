= 検証: Khronos glTF-Validator

== 読み込み前に検証する理由

エンジンの glTF ローダは汎用パーサではありません—正しく形成されたアセットを扱うよう書かれており、不正なデータに出会うと黙って誤動作したりクラッシュしたりし得ます。Khronos glTF-Validator は、glTF ファイルを完全な仕様と照合し、見つけたすべての違反・警告・ヒントを報告する参照実装です。エンジンで読み込む前にアセットをこのツールに通すコストは数秒で、何時間も節約できます。

バリデータは、エクスポート中にうっかり導入しやすい幅広い問題を捕捉します。

- コンポーネント型に対して正しくアラインされていないアクセサのバイトオフセット（カスタムエクスポータや、バイナリデータを手動で書く Python スクリプトでよくある問題）。
- シーン階層に列挙されていないノードを参照するスキンジョイント。
- 厳密に単調増加していない入力配列を持つアニメーションサンプラー（仕様違反であり、補間が意味をなさない結果を生み得ます）。
- バッファ長を超えて重なる、または延びるバッファビュー。
- 頂点数を誤ったモーフターゲットアクセサ（ベースプリミティブと一致しない）。
- 必須フィールドの欠落（例: `inverseBindMatrices` アクセサのないスキン）。

これらの多くは、tinygltf のような寛容なローダでは即クラッシュを起こさず—黙って誤ったデータを生むか無視されます。バリデータはアセットがまだ孤立しておりエラー源が明らかなうちに見つけます。

== バリデータのインストールと実行

Khronos glTF-Validator はコマンドラインツールと、Web ベースのドラッグ＆ドロップインタフェースとして利用できます。制作パイプラインへの統合には、コマンドラインツールが正しい選択です。Dart アプリケーションとして配布され、Dart パッケージマネージャ経由でインストールします。

[source,bash]
----
# Install Dart SDK (Linux, using apt)
sudo apt-get update && sudo apt-get install dart

# Install the validator
dart pub global activate gltf_validator

# Add Dart's global pub cache to your PATH (if not already done)
export PATH="$PATH:$HOME/.pub-cache/bin"
----

macOS で Homebrew を使う場合:

[source,bash]
----
brew install dart
dart pub global activate gltf_validator
----

インストール後、次でファイルを検証します。

[source,bash]
----
gltf_validator character.glb
----

CI 統合では、機械可読出力を得るために `--format json` フラグを使います。

[source,bash]
----
gltf_validator --format json character.glb > validation_report.json
----

エラーが見つかるとバリデータは非ゼロのステータスコードで終了するため、無効なアセットで CI ビルドを失敗させるのが単純です。

== バリデータ出力の読み方

バリデータは発見を四レベルに分類します。**Errors**、**Warnings**、**Infos**、**Hints** です。

**Error** は、確実に不正な挙動を生む仕様違反を示します。バリデータがエラーを報告したら、直るまでそのアセットは使うべきではありません。よくあるエラーには、バッファオーバーフロー（アクセサがバッファを超えてデータを読むと主張する）、無効なジョイントインデックス（スキンが存在しないノードインデックスを参照する）、不正なモーフターゲット数（プリミティブ間でモーフターゲットアクセサ数が一致しない）があります。

**Warning** は、仕様上は技術的に有効だが、おそらく間違っているか潜在的に問題のある状況を示します。正規化されていないスキニングウェイトは警告を生みます—仕様では許されますが、ほぼ常にアーティストワークフローの問題を示します。プリミティブ内の重複頂点位置も警告です—メッシュが正しく溶接されておらずトポロジ問題があることを示唆します。

**Info** メッセージは情報提供です。アセットは有効だが、意図的かもしれないしそうでないかもしれない珍しい特性を持ちます。非常に多数のモーフターゲット（例: 100 以上）は info メッセージを生成します—間違いだからではなく、注意を払うほど珍しいからです。

**Hint** は正しさに影響しないスタイルまたは最適化の問題を示唆します。例えば、品質損失なしに 16 ビットで十分なテクスチャ座標に 32 ビット float を使うことなどです。

キャラクターアセットでは特に、スキニングウェイトに関する警告と、アニメーションサンプラーに関するエラーに注意してください。アニメーションエラーは特に厄介で、タイムラインの特定点でのみ現れることがあります—フレーム 0 では正しく見えるクリップが、サンプラーの入力タイムスタンプが不正に形成されているとフレーム 120 で壊れたデータを生み得ます。

== ビルドパイプラインへの検証統合

チーム環境では、検証は手動ステップではなく、アセットビルドパイプラインの一部として自動実行すべきです。正確な統合はビルドシステムに依存しますが、パターンは常に同じです。glTF エクスポートステップの後にバリデータを実行し、エラーがあればビルドを失敗させます。

単純な Makefile ルール:

[source,makefile]
----
assets/%.glb: blender/%.blend
	blender -b $< --python scripts/export_gltf.py -- $@
	gltf_validator $@ || (echo "Validation failed for $@"; exit 1)
----

これにより、`assets/` ディレクトリ内の任意の `.glb` アセットが Blender エクスポートスクリプトで生成され、コミット前に検証を通過したことが保証されます。検証が失敗するとビルドが失敗し、エラーは即座に見えます—エンジニアが三日後にキャラクターを読み込んでクラッシュするのではありません。

Python ベースのパイプラインでは、バリデータをプログラムから呼び出すこともできます。`gltf_validator` パッケージは構造化結果オブジェクトを返すライブラリ API を公開し、カスタムポリシー（例: プロジェクトで特定の info レベルメッセージをエラーとして扱う）を実装できます。
