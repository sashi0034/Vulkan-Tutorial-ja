= ツーリング: まとめと次のステップ

== 確立したもの

本章は意図的にコードの話ではありませんでした。代わりにプロセス—キャラクターパイプラインを終わりなき謎の源泉ではなく、信頼でき保守可能なものにする規律—についてでした。

命名規則を、アーティストとエンジニアの間の暗黙の契約として確立しました。一貫したボーン命名—小文字、アンダースコア区切り、左右サフィックス—により、名前でジョイントを探すエンジン内のすべてのシステムが、デフォルトで信頼できるようになります。エクスポート時にスキニングデータを壊す Blender 側の具体的問題も議論しました。正規化されていないウェイト、頂点あたり 4 を超える影響、ゼロウェイトのゴースト影響、エクスポート時のレストポーズ不一致です。それぞれが特定クラスの視覚アーティファクトを生み、どのアーティファクトがどの原因に対応するかを知れば診断が速くなります。

物理 extras ワークフローを詳しく扱いました。Blender の Script Editor で Python スクリプトが書くカスタムプロパティが、どのように glTF ファイルの JSON "extras" になり、そのデータが第 2 章のコライダーと拘束のパースコードへ流れるかです。設定スクリプトを blend ファイルと並べ、バージョン管理に入れることで、物理パラメータは再現可能で監査可能になります。

glTF エクスポート設定の議論では、最も重要な選択を扱いました。バイナリ対分割フォーマット（本番ではバイナリ）、リーフボーンオプション（無効化）、Apply Modifiers（最終エクスポートではオン）、アニメーションの Force Sampling（デフォルトはオフ、プロシージャルドライバには注意）です。

Khronos glTF-Validator は仕様準拠の高速で権威あるチェックを提供します。ビルドパイプラインに組み込めば、無効なアセットは実行時ではなくソースで捕捉されます。四つの出力レベル—Error、Warning、Info、Hint—はアセット品質の段階的な見方を与えます。エラーは必ず直し、警告は調査すべきです。

最後に、参照ビューア（Khronos glTF Sample Viewer と Babylon.js Sandbox）は、エンジンデバッグを扱いやすくする真値を提供します。視覚問題がエンジンコード側だと結論づける前に参照ビューアでアセットを確認する規律は、グラフィックスプログラマが身につけられる最も有用な習慣の一つです。

== 次に来ること

第 8 章は **デバッグと視覚監査**—実行中のエンジン自体の問題を特定するツールと技法—でチュートリアルシリーズを締めくくります。アセットの問題はソースで捕捉すべきだと確立しました。第 8 章は、アセットが正しく問題がエンジン側にあるときに何をするかを扱います。

スケルトン・衝突形状・物理拘束向けのエンジン側デバッグドロワーを実装します。描画されたキャラクターと並べて物理表現を見せる視覚オーバーレイです。スキニングウェイトヒートマップも扱い、頂点のボーン影響を色として描画し、動きの中でのみ見える「つまみ」ジオメトリや不正なウェイトペイントを特定します。さらに RenderDoc でコンピュートスキニング出力バッファを検査し、ラスタライザが読む前にスキニング後の頂点位置と法線が正しいことを検証する手順を歩きます。

== 検証: 確認すべき点

制作パイプラインを検証するには:

1.  **バリデータがグリーン**: エクスポートしたすべての `.glb` が、エラーゼロ・警告ゼロで **glTF-Validator** を通過すること。
2.  **命名の一貫性**: 物理メタデータの `"parent_bone"` 文字列が、glTF JSON 内の実際のノード名と一致することを確認します。
3.  **エクスポート設定**: Babylon.js sandbox のような第三者ビューアでファイルを検査し、**Custom Properties**、**Shape Keys**、**Vertex Weights** がすべてエクスポートに含まれていることを確認します。
