= モーフターゲット: まとめと次のステップ

== 構築したもの

本章は第 3 章から始まった変形パイプラインを完成させました。スケルタルスキニングは身体を扱います。肢・脊椎・頭の大規模で回転駆動の変形です。モーフターゲットは顔を扱います。筋肉活性化に応じて滑り、しわ寄り、膨らむ皮膚の繊細で非剛体な変形です。

まず glTF ファイル形式のモーフターゲットへのアプローチから始めました。ベースメッシュと各表情の間で実際に変わる頂点だけを格納するスパースアクセサです。スパースアクセサを正しくパースし—密な変位配列全体へ展開する—ことは最初で最も誤りやすいステップです。特にインデックスのコンポーネント型がエクスポータ間で変わり得るからです。バイト・ショート・整数のインデックス幅を扱い、明示的に列挙されていない頂点にはゼロを返す堅牢な `expand_sparse_accessor` 関数を書きました。

変位データが CPU 上にある状態で、前章のステージングバッファパターンを使い、各モーフターゲットを自身の GPU バッファへアップロードしました。バインドレスアーキテクチャに備え、これらのバッファに `VK_BUFFER_USAGE_SHADER_DEVICE_ADDRESS_BIT` を付けました。

バインドレスディスクリプタ基盤は `VK_DESCRIPTOR_BINDING_VARIABLE_DESCRIPTOR_COUNT_BIT` と `VK_DESCRIPTOR_BINDING_PARTIALLY_BOUND_BIT` を使い、ランタイムサイズのストレージバッファディスクリプタ配列を持つディスクリプタセットを作成します。すべてのモーフターゲットバッファはロード時にこの配列へ登録されます。ディスパッチ時には *アクティブな* モーフターゲットのインデックスとウェイトだけを渡し、非アクティブバッファに触れるコストを避けます。

拡張したコンピュートシェーダーは、スケルタルスキニングの前にバインドポーズでモーフ変位を適用し、`NonUniformResourceIndex` でスレッドごとの値によりバインドレス配列を安全にインデックスします。出力はラスタライザ・レイトレーサ・物理システムがすでに消費しているのと同じ共有頂点バッファです。モーフターゲットサポートは下流パイプラインに新しい同期要件を追加しません。

== 心に留めるべきトレードオフ

モーフターゲットは、現代の GPU ハードウェアでは評価が速いです。恥ずかしいほど並列だからです。各頂点は独立で、頂点あたりの作業はアクティブターゲットごとに数回の乗加算だけです。ボトルネックはほぼ常にメモリ帯域—GPU メモリから変位配列をフェッチすること—であり、算術ではありません。だから「非アクティブターゲットを除外する」ステップが重要です。60 個のモーフターゲットがあり、このフレームで非ゼロが 8 個だけなら、60 ではなく 8 個のアクティブターゲットでディスパッチすると帯域が 7.5 倍減ります。

前節で挙げたプッシュ定数サイズの懸念にはフォローアップが必要です。同時にアクティブなモーフターゲット数が多い場合、ウェイトとインデックスデータを小さなユニフォームバッファ（`vkCmdUpdateBuffer` またはステージングコピーで更新）へ移すのが正しい選択です。この用途ではプッシュ定数と小さなユニフォームバッファフェッチの性能差は無視でき—プロファイルでは気付きません—が、最小値を超えるプッシュ定数サイズに依存しない堅牢性の利点は実在します。

法線モーフターゲット（表面法線の変位ベクトル）は言及しましたが、本章では完全には実装していません。皮膚の曲率を大きく変えるモーフターゲット—頬の膨らみ、眉のしわ—がある場合、位置モーフと並んで法線モーフを実装すべきです。シェーダー拡張は単純です。法線デルタ用の第二のバインドレス配列を追加し、同じ方法で蓄積します。法線モーフがなければ、モーフした顔のライティングはわずかに間違って見え、特に強いディレクショナルライトの下で顕著です。

== 次に来ること

第 7 章では、レンダリングと物理コードから一歩下がり、**制作パイプライン**—アーティストの Blender ファイルから検証済みでエンジン準備完了の glTF アセットへのワークフロー—を調べます。エンジン側コードを信頼できるものにする命名規則とエクスポート設定、ローダに届く前にフォーマットエラーを捕捉する Khronos glTF-Validator の使い方、エクスポート問題だと判明する何時間ものデバッグの前にマテリアルとアニメーションの「真値」を確立するプロ glTF ビューアの使い方を扱います。

制作パイプラインの理解はレンダリングコードの理解と同じくらい重要です。見かけのエンジンバグの多くは実はアセットバグであり、両者を区別する方法を知れば、デバッグ時間を大幅に節約できます。

== 検証: 確認すべき点

モーフターゲット実装を検証するには:

1.  **スパースアクセサ展開**: 特定のモーフターゲットで変位がない頂点が、正しくベース位置に留まることを確認します。
2.  **バインドレスインデキシング**: GPU デバッガ（RenderDoc など）で `morphTargets[]` 配列を検査します。`PushConstants` で渡すディスクリプタインデックスが正しいバッファスロットと一致することを確認します。
3.  **処理順序**: モーフ変位がスケルタルスキニングの **前** に適用されることを確認します。後に適用すると、頭が回転したときに顔の特徴が「浮く」または離れて見えます。
