= Advanced glTF: 高性能キャラクターパイプライン

== 静的モデルを超えて

「シンプルなエンジンの構築」シリーズは、Vulkan レンダリングの堅実な基盤を築きました。`tinygltf` による glTF 読み込みの核心、PBR マテリアルシェーダー、各 `Model` 内の基本的な階層シーングラフを導入しました。しかし、エンジンが静的なジオラマの描画から、複雑でインタラクティブな世界へと進化するにつれ、その初期アーキテクチャの限界に直面します。

本番品質のエンジンでは、キャラクターは単なるメッシュではなく、複数の高性能システムの交点です。キャラクターの足が凹凸のある地形に触れると、インバースキネマティクス（IK）システムが脚のジョイント回転を調整しなければなりません。同じキャラクターが飛翔体に当たると、アニメーションシステムはラグドールシミュレーションのために物理エンジンへ制御をシームレスに渡さなければなりません。この相互作用には、単純な再帰トラバーサルよりも遥かに堅牢で最適化されたアーキテクチャが必要です。

本チュートリアルシリーズでは、これらの高度なキャラクターパイプラインを扱えるようエンジンを進化させます。アニメーションや物理といった「機能」を足すだけでなく、モダンな 3D アセットの規模・性能・複雑さに耐えるよう、コアシステムを再設計します。

== 技術ロードマップ

=== 1. はじめに
本シリーズは、glTF ファイルのパースから世界との物理的相互作用のシミュレーションまで、モダンなキャラクターのライフサイクルに沿って構成されています。まず本導入で前提条件を整理し、開発環境を整えます。

=== 2. シーングラフとトランスフォーム階層
まずシーン表現を統一します。これまでのエンジンは、階層的モデルを含むエンティティのフラットなリストというハイブリッド方式でしたが、**統一されたグローバルシーングラフ**へ移行します。また、フレームごとの冗長な行列乗算を **Dirty Flag** システムと、よりデータ指向のレイアウト（DOD）に置き換えます。これにより、ワールド空間のトランスフォームは本当に必要なときだけ更新され、シーン複雑度が増しても重要な最適化になります。さらに glTF の **"extras" メタデータ**をパースしてコライダー設定を自動化し、物理プロキシが視覚メッシュと正確に揃うようにします。

=== 3. 高度なスケルタル＆コンピュートスキニング
スケルタルアニメーションはキャラクター動作の中核です。高性能な **コンピュートスキニング**を実装し、頂点変形の重い処理を頂点シェーダーから専用のコンピュートカーネルへ移します。この方式により「一度スキニングし、どこでも使う」が可能になり、結果をラスタライザ、レイトレーシング加速構造、さらには物理クエリからも参照できる共有バッファに格納します。

=== 4. 物理統合: コライダーとラグドール
物理とアニメーションは別物として扱われがちですが、高度なパイプラインでは深く結びついています。アニメーションが物理を駆動する（Kinematic）場合と、物理が骨格を駆動する（Ragdoll）場合の **双方向同期**を探ります。

=== 5. プロシージャルアニメーションと IK
キャラクターが生き生きと感じられるのは、環境に反応するときです。CCD（Cyclic Coordinate Descent）や FABRIK のようなアルゴリズムで **インバースキネマティクス（IK）**を実装し、足を地形に揃え、頭が関心点を追うようにします。この「プロシージャルレイヤー」は従来のアニメーショクリップの上に乗り、キャラクターが地面に根ざしているように感じさせる最終的な仕上げになります。

=== 6. モーフターゲットと表情アニメーション
表現豊かなキャラクターには、ボーンだけでは足りません。**モーフターゲット**（シェイプキー）のサポートを実装し、複雑な表情や局所的なメッシュ変形を可能にします。**Descriptor Indexing** のようなモダンな Vulkan 機能を活用し、ドローごとのディスクリプタ入れ替えのオーバーヘッドなしに、これらの変位バッファを扱います。

=== 7. ツーリングと制作パイプライン
コードは戦いの半分に過ぎません。「Blender-to-Vulkan」ワークフローを議論し、アセットがエンジンに届いたとき「そのまま動く」ための命名規則とエクスポート設定を確立します。また **Khronos glTF-Validator** をパイプラインに組み込み、不可解なエンジンバグの原因になる前にアセットの破損を検出します。

=== 8. デバッグと視覚監査
完璧なパイプラインでも問題は起きます。キャラクターシステムの隠れた状態を可視化する専用デバッグツールを構築します。スケルトンや衝突形状向けの遅延ラインドロワー、「つまみ」ジオメトリを見つけるスキニングウェイトのヒートマップ、コンピュートスキニング出力を監査するための RenderDoc 活用技法を含みます。

== 高度なツールキット

本シリーズ全体を通じて、開発ツールキットをモダンな標準へ引き上げます。

*   **Slang シェーダー**: コンピュートスキニングから PBR 拡張まで、新しいシェーダー作業はすべて **Slang** で書きます。Slang のモジュール性と自然なアラインメント機能により、複雑なレンダリングパイプラインでは生の GLSL より生産性が高くなります。
*   **Vulkan 1.3**: **Dynamic Rendering** や **Descriptor Indexing** などの機能を使い、パイプライン管理を簡素化し性能を向上させます。
*   **Jolt Physics**: シミュレーションエンジンとして link:https://jrouwe.github.io/JoltPhysics/[Jolt Physics]（link:https://github.com/jrouwe/JoltPhysics[GitHub のソース]）を選びました。性能優先の設計とクリーンな C++ API により、リアルタイムのキャラクターダイナミクスとラグドールに最適です。

== 開発環境のセットアップ

コードに入る前に、いくつかの新しいツールとライブラリで開発環境を準備する必要があります。

=== 1. Slang シェーダーコンパイラ
本シリーズのコンピュートおよびグラフィックスシェーダーにはすべて **Slang** を使います。

* **ダウンロード**: link:https://github.com/shader-slang/slang/releases[Slang GitHub Releases] から最新のバイナリリリースを入手します。
* **セットアップ**: `slangc` 実行ファイルがシステムの PATH にあることを確認します。
* **Vulkan 統合**: Slang は `.slang` ファイルを直接 SPIR-V（`.spv`）にコンパイルし、既存の Vulkan シェーダー読み込みコードでロードできます。

=== 2. Jolt Physics
Jolt は本シリーズで推奨する物理エンジンです。
* **統合**: CMake の `FetchContent` で Jolt をプロジェクトに直接取り込むことを推奨します。
[source,cmake]
----
include::FetchContent.cmake
FetchContent_Declare(
    Jolt
    GIT_REPOSITORY https://github.com/jrouwe/JoltPhysics.git
    GIT_TAG master
)
FetchContent_MakeAvailable(Jolt)
----
* **初期化**: Jolt には少量のボイラープレート（メモリ管理、ジョブシステム、レイヤインタフェース）が必要で、物理統合の章で扱います。

=== 3. Khronos glTF-Validator
カスタム glTF アセットが有効であることを保証するため、公式バリデータを使います。

* **インストール**: link:https://github.com/KhronosGroup/glTF-Validator[glTF-Validator リポジトリ] の手順に従います。スタンドアロン CLI ツール、または NPM 経由で利用できます。

== スタータープロジェクトの準備

本シリーズは、「シンプルなエンジンの構築」シリーズで動作するエンジンがあることを前提とします。Advanced glTF アーキテクチャへ移行するには、次を満たす「スタータープロジェクト」を用意してください。

1.  **Vulkan 1.3 をサポート**: インスタンスとデバイス作成で 1.3 API バージョンを要求していることを確認します。
2.  **Dynamic Rendering を有効化**: `VK_KHR_dynamic_rendering` が有効であることを確認します。Simple Engine シリーズの Advanced Topics 章でダイナミックレンダリングを導入していれば、すでに有効なはずです。
3.  **GLM を統合**: スケルタルアニメーションでのクォータニオン対応を含め、数学演算に GLM を広く使います。

ゼロから始める場合は、次の章に進む前に Simple Engine のコアをグローバルシーングラフ対応へリファクタすることを推奨します。
