= 推論の説明

== はじめに

前章では、ニューラルネットワークがどう学ぶか——勾配降下、逆伝播、反復的最適化を通じて数百万の重みを調整する詳細なプロセス——を探りました。第一原理から理解を築き、ネットワークが学習を通じてデータのパターンを発見する様子を見ました。

今度は焦点を完全に移します。本チュートリアルシリーズはニューラルネットワークの学習ではなく——*使う* ことについてです。具体的には、すでに学習済みのモデルを取り、Vulkan コンピュートシェーダを使って GPU 上で効率的に実行することです。このプロセスを *推論* と呼び、実装に深く関わる点で学習と根本的に異なります。

この区別の理解は学術的だけではありません——これから行うすべての設計判断を形作ります。書くコード、割り当てるメモリ、実装する同期はすべて、推論が何であり何でないかの理解から流れます。

== 二つのフェーズ、二つの世界

機械学習には、異なる時期、場所、目標、制約の下で起きる二つの異なるフェーズがあります。それぞれを明確に理解しましょう。

=== 学習: 学習フェーズ

学習を研究開発フェーズと考えてください。大学の研究室、テック企業、自分のワークステーション——どこかで、誰かがニューラルネットワークにタスクを教えています。

何も知らないネットワークから始めます。重みはランダム。猫の画像を与えれば無意味な出力——自信満々にトースターだと宣言するかもしれません。まだ何も学んでいません。

学習プロセスが始まります。数百万の例を見ます: 「cat」とラベル付けされた猫、「dog」の犬、「car」の車。各例で予測。間違うと（最初はほとんど常に）、学習アルゴリズムがどれだけ間違っていたか測り、次回のために重みをわずかに調整します。

これが数百万回起きます。ネットワークは徐々にパターンを学びます:「尖った耳とひげの画像は猫になりやすい」「車輪と窓の画像は車になりやすい」。前章で議論した数百万の数——重み——がランダム値から、この知識を符号化する意味のあるパターンへ徐々に変化します。

このプロセスは *高コスト* です。ImageNet 上の現代画像認識モデルの学習はハイエンド GPU で数日〜数週間。大規模言語モデルは計算時間で数百万ドル。大規模データセット、学習率などハイパーパラメータの慎重な調整、最適化アルゴリズムの深い専門知識が必要です。

良いニュース？ 通常、自分でやる必要はありません。研究者と企業がモデルを学習し公開します。物体認識、言語翻訳、画像生成が既にできる事前学習済みモデルをダウンロードできます。学習の困難な作業は既に完了しています。

=== 推論: デプロイフェーズ

推論は学習完了後に起きることです。学習済みモデル——数百万の学習済み重みを含むファイル——を持っています。新しいデータで予測するために *使いたい*。

完全に異なるシナリオです。重みは固定——もう変わりません。学習も最適化も逆伝播もありません。データをネットワークに通して予測を得るだけです。

制約も異なります。学習はオフラインで、しばしば強力なサーバーハードウェア上で、必要なだけ時間をかけられます。推論は本番で、しばしばより弱いデバイス上で、*速く* ある必要があります。ユーザーがアプリを開きカメラを物体に向けると、即座の認識を期待します。ゲームが AI 判断を下すとき、予測に数秒待てず——ミリ秒で答えが必要です。

ここで Vulkan が入ります。Vulkan レベルで推論を実装すると、GPU 実行に対するきめ細かい制御が得られます。推論の特定制約——固定重み、順伝播のみ、リアルタイム性能要件——向けに最適化できます。学習システムではなくデプロイシステムを構築しており、それは大きく異なるエンジニアリング課題です。

=== 実装にとってこの区別が重要な理由

学習-推論の分離を理解することは、Vulkan 推論エンジンの構築方法に実践的な含意を持ちます。

学習中は、逆伝播が勾配計算に各レイヤの中間活性化を必要とするため、すべてのレイヤの中間活性化を保存する必要があります。つまり順伝播全体をメモリに保持。推論では現在レイヤの活性化だけ必要——レイヤ 5 の出力を計算しレイヤ 6 に渡したら、レイヤ 5 の活性化は破棄してそのメモリを再利用できます。メモリ要件を劇的に削減します。

学習中は、より安定した勾配推定とより高い GPU 利用率のため、大きなバッチ（32、64、128 以上）で例をまとめて処理。推論では低レイテンシなら一度に一例、スループットならバッチ——アプリケーションのニーズに基づいて選択。

学習中は自動微分、勾配計算、オプティマイザ状態が必要。推論ではこれら一切なし。はるかに単純なシステム: 重み読み込み、順伝播実行、出力取得。この単純さは解放的——順伝播を可能な限り速くすることに完全に集中できます。

== 推論パイプライン: システムを通るデータの追跡

推論実行時に実際に何が起きるかたどりましょう。この流れを理解すると、Vulkan がどこに位置し何を実装すべきかが見えます。

=== ステージ 1: 入力データの到着

アプリケーションがモデルに処理させたいデータを持っています。カメラ画像、マイク音声バッファ、ユーザーテキスト、センサー読み取りなど。アプリケーションが自然に生成する形式——画像はバッファ内 RGB ピクセル、音声は PCM サンプル、テキストは文字列。

この生データはまだニューラルネットワーク向けではありません。モデルは入力形式について特定の期待を持ち、それを満たす必要があります。

=== ステージ 2: 前処理がデータを変換

前処理はアプリケーションのデータ形式とモデルの期待入力形式の橋です。複数の変換を含み得、正しく行うことが極めて重要——前処理がモデル期待と一致しなければ、モデルがどれだけ優れていても予測は無意味な結果になります。

==== 画像前処理: ピクセルからテンソルへ

グラフィックスアプリケーションで一般的な変換が直感的な画像から始めましょう。

*リサイズ*: モデルは特定の入力次元で学習。224×224 で学習したモデルはそのスケールでパターンを認識するよう学習。1920×1080 を与えるとパターンスケールが完全に異なり——顔が入力全体を占めたり、物体が小さな隅だけになったり。まずリサイズが必要。

リサイズは自明ではありません。クロップ？ どこから——中央クロップ、ランダムクロップ？ スケール？ アスペクト比が合わないとき、伸ばす（歪み）かレターボックス（黒帯）か？ 答えはモデルの学習方法次第。中央クロップで学習なら推論も中央クロップ。伸ばしで学習なら伸ばす。ミスマッチは精度低下。

*正規化*: 画像は通常 0〜255 の 8 ビット整数でピクセル保存。ニューラルネットワークは浮動小数点で動作し、入力値の範囲に敏感。多くのモデルは特定の範囲へ正規化された入力を期待。

一般的な正規化方式:

- [0, 1] へスケール: `pixel_float = pixel_int / 255.0`
- [-1, 1] へスケール: `pixel_float = (pixel_int / 127.5) - 1.0`
- 標準化: `pixel_float = (pixel_int - mean) / std`（mean/std は学習データセットから）

最後が特に重要。多くのモデル（特に ImageNet 学習）は ImageNet 統計で標準化された入力を期待: mean = [0.485, 0.456, 0.406]、std = [0.229, 0.224, 0.225]（RGB 各チャネル）。まったく同じ正規化を適用しないと、学習分布外の入力を見せ精度が低下。

*色空間変換*: 見落としやすい落とし穴。OpenCV は BGR 順で画像読み込み、多くの ML フレームワークは RGB 期待。一方で学習し他方で推論すると、赤と青が逆。一部のモデルでは予測が完全に破壊。

同様に、グレースケール（単一チャネル）期待のモデルと RGB（三チャネル）期待のモデルがあります。RGB からグレースケールはチャネル平均だけではない——人間知覚を考慮した重み付き和: `gray = 0.299*R + 0.587*G + 0.114*B`。

*チャネル順序*: RGB 内でもメモリレイアウトが異なります。「channels last」（高さ × 幅 × チャネル、RGBRGBRGB...）と「channels first」（チャネル × 高さ × 幅、赤全部→緑全部→青全部）。モデルが期待する方に合わせる必要があります。

==== テキスト前処理: 文字列からトークンへ

テキスト前処理は数値変換だけでなく言語構造を含むためより複雑。

*トークン化*: テキストを断片に分解が第一歩。最も単純は語レベル:「Hello world」→ ["Hello", "world"]。しかし句読点は？ 未見の語は？

現代モデルはサブワードトークン化。BPE や WordPiece が語をより小さな単位に分解。「unhappiness」→ ["un", "happiness"]。学習中に "unhappiness" を見なくても "un" と "happiness" を見ていれば理解可能。

各トークンは語彙から数値 ID に。 "Hello" が ID 5234、"world" が 8912。モデルは元テキストではなくこれら ID で動作。

*系列長処理*: 最大系列長——512、1024 トークンなど。短ければパディングトークンで埋める。長ければ切り詰める。どこで切るか——先頭、末尾、重要部分を賢く保持？

*特殊トークン*: 構造を示す特殊トークンを期待。先頭 [CLS] は系列開始、[SEP] はセグメント区切り、[PAD] は未使用位置埋め。語彙内の特定 ID、モデル期待位置に正確に挿入。

==== 音声前処理: 波形から特徴へ

音声前処理はしばしば生波形をニューラルネットワークが処理しやすい表現へ変換。

*リサンプリング*: 48kHz 録音、モデルは 16kHz 期待。エイリアシング回避フィルタと補間でサンプルレート変更。

*モノラル変換*: ステレオだがモデルはモノ期待ならチャネル合成。単純平均: `mono = (left + right) / 2`。より高度な方法はより多くの情報を保持。

*スペクトログラム*: 多くの音声モデルは生波形ではなく周波数領域のスペクトログラム。各時点の周波数成分。STFT を重なりウィンドウに適用して計算。

Mel スペクトログラムは周波数軸を mel スケールへ、音高知覚により適合。音声認識システムがしばしば期待。

==== 前処理の実行場所: CPU vs GPU

伝統的に前処理は CPU。ニューラルネットワーク実行と比べ比較的軽量、CPU はリサイズ、色変換、トークン化など多様な操作をうまく処理。

性能が重要なアプリでは GPU 前処理が差を生む。60fps 動画でフレームあたり 5ms CPU 前処理は積み上がる。リサイズと正規化をコンピュートシェーダとして GPU へ移すとレイテンシを大幅に削減。

トレードオフは複雑さ。CPU 前処理は既存ライブラリ（OpenCV、librosa、ML フレームワークのトークナイザ）。GPU 前処理はカスタムコンピュートシェーダと転送管理。多くのアプリでは CPU で十分。レイテンシが重要（リアルタイム動画、対話型 AR/VR）では GPU 前処理を検討。

==== 学習との一致の重要性

鍵: 推論前処理は学習前処理と *まったく同じ* である必要がある。双線形補間でリサイズ学習なら推論も双線形。ImageNet 正規化学習なら推論も ImageNet 正規化。

ここでのミスマッチは推論精度低下の一般的な原因。学習フレームワークでは優秀だがデプロイで失敗——前処理がわずかに異なる。モデルドキュメントまたは学習コードで正確な前処理を確認。

=== ステージ 3: ニューラルネットワーク実行

推論の核心——実際のニューラルネットワーク実行。前処理済み入力がレイヤごとにネットワークを流れ出力に至る。学習中のすべての学習がここで活用されます。

具体例。ResNet や MobileNet のような画像分類モデル。入力 224×224×3 テンソル（高さ × 幅 × 色チャネル）。150,528 浮動小数点数が画像を表す。

==== 第一レイヤ: 特徴抽出の開始

第一レイヤは 7×7、64 フィルタ、ストライド 2 の畳み込みかも。実務では何を意味？

畳み込みはスライディングウィンドウ演算。7×7 の拡大鏡を画像上で一歩ずつスライド。各位置で 7×7 パッチ（3 色チャネルで 7×7×3 = 147 値）。147 値それぞれに対応学習済み重みを掛け、合計し、バイアス加え、一つの出力値。

64 異なるフィルタ、各 147 重みセット。各位置で 64 出力——フィルタごとに一つ。各フィルタは異なるパターン検出: 水平エッジ、垂直エッジ、赤領域など。

ストライド 2 なら 1 ではなく 2 ピクセルずつ移動、各次元出力は半分。224×224×3 入力が 112×112×64 出力。空間ダウンサンプリング（224×224→112×112）だがチャネル増（3→64）。空間解像度と特徴の豊かさのトレードオフ。

この単一畳み込み層はおおよそ 224 × 224 × 64 × 7 × 7 × 3 = 1 億 4700 万回の乗算加算。大量計算だが GPU はまさにこの種の作業——異なるデータで同じ演算を数百万回——に優れる。

==== 活性化関数: 非線形性の導入

畳み込み後、活性化関数——ReLU とする——を通過。前章の理由: 非線形性なしではレイヤ積み上げも表現力増えない。活性化関数が複雑なパターン学習を可能に。

ReLU は非常に単純: `output = max(0, input)`。112×112×64 = 802,816 値それぞれ、正なら保持、負ならゼロ。計算は安価——比較と選択——だがネットワーク動作に不可欠。

GPU 視点では極めて並列。各出力値は他と独立。スレッド間同期不要で数千同時処理。

==== より深いレイヤ: 階層的特徴の構築

データがより深く流れるとパターン続くが特徴はより抽象的。第二畳み込み層は元画像ではなく第一層出力——既にエッジと基本的なパターン——を見る。第二層はエッジをコーナー、曲線、単純テクスチャなど複雑な形状に合成。

第三層は形状を物体部分、特徴的なテクスチャ、繰り返しモチーフへ。10〜20 層深くでは高レベル概念:「毛っぽい」「車輪っぽい」「顔っぽい」。

この階層的特徴学習が深いニューラルネットワークを強力にする鍵の洞察の一つ。早期層は一般的で再利用可能な特徴（エッジは任意画像）。中間層はドメイン固有（毛 vs 金属 vs ガラス）。後半層はタスク固有（猫 vs 犬 vs 車）。

==== バッチ正規化: 値を範囲内に

多くの現代ネットワークにバッチ正規化レイヤ。微妙な問題: 多層を通じて値分布がずれ広がる。0〜1 で始まった値がレイヤ 20 で平均 100、標準偏差 50 に。学習不安定、推論性能も損なう。

バッチ正規化は各レイヤで値を正規化。活性化の平均・標準偏差を計算し、平均 0・標準偏差 1（または学習済みパラメータで柔軟性）へシフト/スケール。

推論では簡略化: 平均・標準偏差は学習中に計算された固定値。オンザフライ統計計算なし——固定変換適用: `output = (input - mean) / sqrt(variance + epsilon) * scale + shift`。要素ごと演算だが各レイヤの固定統計（mean、variance、scale、shift）読み込み必要。

==== プーリング: 効率のためのダウンサンプリング

プーリングレイヤは重要情報を保ちながら空間次元縮小。最大プーリングが最も一般的——入力を小領域（通常 2×2）に分け各領域最大値のみ出力。

なぜ？ ネットワークを流れるデータ量削減、後続レイヤ高速化。平行移動不変性も——特徴が画像内でわずかに動いてもプーリング領域内なら捕捉。

実装視点では最大プーリングは単純: 各出力位置で 4 入力（2×2）読み最大見つけ書き出し。全出力位置で並列。

==== 全結合レイヤ: 最終判断

ネットワーク末端、多くの畳み込みで豊かな特徴抽出後、通常は全結合レイヤ。すべての特徴を合成して最終予測。

全結合レイヤは行列乗算。2048 入力特徴、1000 出力クラスなら 1000×2048 重み行列に 2048 次元入力ベクトル。2,048,000 回の乗算加算、すべて並列化可能。

最終レイヤは通常各クラスの生スコア（logits）出力。まだ確率ではない——各クラス存在の確信度を示す数。大きいほど高信頼。

==== 完全な順伝播

まとめると、ResNet-50 のようなネットワークの完全順伝播:

- 初期畳み込み 1 層
- 残差ブロック 16 個、各 3 畳み込み、3 バッチ正規化、3 ReLU、スキップ接続
- グローバル平均プーリング 1 層
- 全結合 1 層

おおよそ 50 層（「ResNet-50」）、数十億回の乗算加算。現代 GPU では全体がわずか数ミリ秒。

各レイヤが GPU メモリから入力テンソル読み、計算し、出力テンソル書き込み。一層出力が次層入力。データは管を流れる水のようにネットワークを通り各段階で変換され最終予測として現れる。

ここで Vulkan が輝く。各レイヤがコンピュートシェーダ。テンソルは GPU メモリのバッファ。コンピュート操作をディスパッチし、正しい順序を同期し、層間でデータを渡す。GPU の大規模な並列性で数千値を同時処理し、リアルタイムアプリケーションに十分速い推論を実現。

=== ステージ 4: 後処理が出力を解釈

ニューラルネットワークは生出力を生成するが、アプリケーションに有用にする前にしばしば解釈必要。ネットワークから出る生数値は——数値。後処理が実用的な情報へ変換。

==== 分類: ロジットからラベルへ

分類モデルでは出力は通常 *logits* と呼ばれるスコアベクトル——可能クラスごとに一数。ImageNet 1000 カテゴリなら 1000 数。

logits はまだ確率ではない。任意値——正、負、大、小。「cat」5.2、「dog」2.1 なら cat により確信だが、パーセントとして解釈できる確率ではない。

*Softmax 変換*: logits を確率へ:

[source]
----
probability[i] = exp(logit[i]) / sum(exp(logit[j]) for all j)
----

すべての確率 0〜1、合計 1。「モデルは 87% cat と確信」と言える。

*上位予測の検出*: 最も単純な後処理は最大確率とそのクラスラベル。出力ベクトルの線形走査——O(n)。

*上位 K 件の予測*: しばしば上位 1 件ではなく上位 5 件。モデルが不確かな時に有用——40% がゴールデン・レトリバー、35% がラブラドール、15% がイエロー・ラブ。三者とも妥当、三者表示でより多くの情報。

上位 K 件の取得は部分ソートまたはヒープ。K=5、n=1000 でも非常に高速。

*信頼度閾値*: 一定信頼度以上のみ返す場合も。上位が 30% だけなら「不確か」を返す方が良いかも。アプリ依存——医療 95%、写真アプリ 50% 許容。

==== 物体検出: ボックスから物体へ

物体検出はより複雑。ネットワークは複数検出出力、各:

- バウンディングボックス (x, y, width, height)
- クラスラベル（または各クラス確率）
- 信頼度スコア

典型的な検出器は 10,000 以上の候補ボックス、大半は誤検出または重複。

*信頼度フィルタ*: まず低信頼度除去。0.5 未満（閾値は選択）なら破棄。10,000 候補が 100〜200 へ。

*Non-Maximum Suppression (NMS)*: 複数ボックスが同一物体検出。同じ車を 5 ボックス——NMS が重複除去。

アルゴリズム:
1. 信頼度でソート（高い順）
2. 最高信頼度ボックスを出力に追加
3. このボックスと大きく重なる（通常 IoU > 0.5）ボックス除去
4. 残りで繰り返し

「重なり」は IoU（Intersection over Union）: 重なり面積 / 和集合面積。80% 重なりなら同一物体、高信頼度のみ保持。

NMS は最悪 O(n²) だが信頼度フィルタ後 n は小さく CPU 上で高速。多数検出では GPU 実装も。

*クラス別 NMS*: クラスごと NMS。「car」ボックスは重なっても「person」抑制しない。同一位置複数物体（車内の人）検出可能。

*座標変換*: 検出器はしばしば正規化座標（0〜1）またはアンカー相対。画像空間の絶対ピクセル座標へ。画像寸法に基づくスケール/オフセット。

==== セグメンテーション: マスクから領域へ

セグメンテーションモデルは各ピクセルにラベル。224×224、21 クラス（Pascal VOC）なら 224×224×21 テンソル——ピクセルごと 21 確率。

*Argmax でラベル*: 各ピクセルで最高確率クラス。224×224×21 → 224×224 ラベルマップ。

*マスク平滑化*: 生出力はノイズ——個別ピクセル誤分類。メディアンフィルタやモルフォロジー演算で整理。

*輪郭抽出*: ピクセルラベルより物体境界が必要ならマスクから輪郭。多角形/曲線で各物体の輪郭——ピクセルグリッドより有用かも。

*インスタンスセグメンテーション*: 同一クラス不同インスタンス（二台の別々の車）。後処理でピクセルをインスタンスへクラスタリング、連結成分など。

==== 生成モデル: ノイズからコンテンツへ

生成モデル（GAN、diffusion など）は画像、音声、テキスト出力。後処理は出力を使える形に。

*逆正規化*: [-1, 1] 出力なら画像用 [0, 255] へ: `pixel_int = (pixel_float + 1) * 127.5`。

*クリッピング*: 有効範囲内。ピクセル 0〜255、音声 [-1, 1] など。範囲外のアーティファクト防止。

*形式変換*: モデル出力形式からアプリ必要形式へ。プランラー RGB（赤全部→緑→青）からインターリーブ RGB（RGBRGB...）など。

==== 後処理の実行場所: CPU vs GPU

前処理と同様、後処理は伝統的に CPU。最大値検出、NMS、輪郭抽出は CPU 向き、成熟したライブラリあり。

高スループットでは GPU 後処理が助け。32 画像バッチ、各 200 検出なら 6400 ボックス。GPU NMS は CPU 転送・処理・戻しより速いかも。

トレードオフ同じ: CPU は単純・既存ライブラリ、GPU は大規模で速いがカスタム実装必要。

=== ステージ 5: 出力がアプリケーションへ戻る

最後に処理結果がアプリケーションへ。単純ラベル（「ゴールデン・レトリバー」）、検出物体リスト、生成文など。

アプリケーションは必要に応じて使用: ユーザー表示、ゲームロジック、アクションのトリガー、他システムへ渡す。推論パイプライン完了。

== コンピュート演算としてニューラルネットワークを考える

視点を移し、GPU プログラミング視点からニューラルネットワークを考えましょう。Vulkan 推論実装に不可欠なメンタルモデル。

=== テンソル: データ構造

ニューラルネットワーク内すべては *テンソル*——数値の多次元配列——で動作。グレースケール画像は 2D（高さ × 幅）。カラー画像 3D（高さ × 幅 × チャネル）。画像バッチ 4D（バッチ × 高さ × 幅 × チャネル）。

Vulkan 視点ではテンソルは GPU メモリのバッファ。224×224×3 画像テンソルは 150,528 浮動小数点のバッファ。テンソル「形状」(224, 224, 3) は線形バッファを多次元として解釈するメタデータ。

テンソル形状とレイヤを通じた変換理解が不可欠。畳み込みが 224×224×3 入力で 112×112×64 出力なら、112 × 112 × 64 = 802,816 個の浮動小数点数のバッファ割り当て必要。正しいメモリ割り当てに形状を知る必要。

=== レイヤ: 演算

各レイヤはテンソル上の特定の数学演算。

*Dense（全結合）レイヤ* は行列乗算。入力サイズ N、出力 M なら M×N 重み行列 × N 次元入力 + M 次元バイアス。古典的な行列-ベクトル乗算、GPU は非常に効率的に処理。大レイヤ（4096×4096）では 1600 万回の乗算加算——効果的な並列化が性能の鍵。

*畳み込みレイヤ* は小フィルタ（通常 3×3 または 5×5）を入力上でスライド、各位置で重み付き和。複数入力チャネル・出力フィルタで大量計算。3×3、64 入力、128 出力、112×112 ならおおよそ 11 億回の乗算加算。良い点: 高度に並列——各出力位置が独立。

*活性化関数* は要素ごと非線形。ReLU: `output = max(0, input)`。Sigmoid: 指数 `output = 1 / (1 + exp(-input))`。極めて並列——各要素独立——だが sigmoid などは計算コストが高い。

*プーリングレイヤ* は小領域の最大/平均でダウンサンプル。2×2 最大プーリングで 112×112 → 56×56。単純・並列だが正しい入力位置読み取りに注意深いインデックスが必要。

*正規化レイヤ*（バッチ正規化など）は次元にわたる統計（平均、分散）計算後正規化。統計計算はリダクションで同期必要。統計取得後の正規化適用は単純。

=== 実行モデル

Vulkan 視点では推論は適切同期付きコンピュートシェーダディスパッチの列。

==== 基本フロー

入力テンソルを GPU メモリへアップロード。ホストからデバイスへの転送。224×224×3（~600KB）は現代システムで速いがオーバーヘッド最小化したい。

第一レイヤコンピュートシェーダディスパッチ。入力テンソルと重み（起動時一度 GPU 常駐）読み、計算し、中間テンソルへ書き込み。中間テンソルは GPU メモリ——CPU へ戻さない。

メモリバリア挿入——書き込み完了と後続操作への可視性。正しさに不可欠——なければ次レイヤが前レイヤ書き込み完了前に読み始め競合状態。

第二レイヤシェーダディスパッチ、以降レイヤごとに。各レイヤがコンピュートシェーダディスパッチ、各中間結果が GPU バッファ、バリアが順序保証。

最後に出力テンソルを CPU へ。デバイスからホストへ。1000 クラス分類なら 1000 個の浮動小数点数（~4KB）——入力よりはるかに小さい。

==== メモリ管理: スケールの課題

問題: ニューラルネットワークは大量メモリ。ResNet-50 を追跡:

レイヤ 1: 入力 224×224×3、出力 112×112×64。150,528 + 802,816 個の浮動小数点数 + 重み 9,408。~3.8MB。

レイヤ 2: 112×112×64 入出力 + 重み。~3.2MB。

10 層深くで中間テンソル数十 MB。50 層で数百 MB。

素朴アプローチ——テンソルごと別バッファ——は GPU メモリをすぐに使い果たす。典型的なモバイル GPU 2〜4GB、グラフィックス・テクスチャ・他アプリと共有。

==== メモリ再利用: 解決策

鍵: 多くの中間テンソルは短命。レイヤ 5 完了、レイヤ 6 が出力消費後、レイヤ 5 出力は二度と不要。そのメモリをレイヤ 7 出力に再利用。

*メモリプール* または *メモリ再利用*——リソース制約デバイスで大モデル実行に不可欠。

アルゴリズム:
1. モデルグラフ分析で各テンソル寿命——生成/消費レイヤ
2. 寿命が重ならないテンソル同メモリ共有可能を特定
3. バッファプール割り当て、重なるテンソルが同バッファを共有しないよう割り当て

本質的にグラフ彩色。実務では貪欲法がうまく機能: レイヤ順処理、各出力テンソルで使用中でない最初の利用可能バッファ。

メモリ再利用で ResNet-50 フットプリント数百 MB → 数十 MB——5〜10 倍削減。「デバイスに載らない」と「快適に動く」の差。

==== 同期: 正しさの保証

GPU は大規模に並列だが推論は本質的に逐次——各レイヤは前レイヤ出力に依存。並列性を許しつつ順序を強制する同期必要。

*パイプラインバリア*: Vulkan のコンピュート向け主要同期プリミティブ。「この点より前のすべての操作が、この点より後の操作より前に完了。」レイヤ間に挿入。

バリアにはコスト——実行直列化。すべての操作後にバリアを入れると GPU がアイドル、並列性を浪費。

*粒度が重要*: すべての操作後にバリア不要——一操作出力が他操作入力になるとき必要。レイヤ 5 が A 書き、レイヤ 6 が A 読むならバリア。レイヤ 5 が A、レイヤ 7 が B（依存なし）なら並列可。

現代ネットワークはしばしば並列分岐——ResNet スキップ、Inception 複数パス。合流時同期すれば同時実行可。

*メモリバリア vs 実行バリア*: Vulkan は実行依存（A 完了後 B 開始）とメモリ依存（A 書き込みが B 読み込みに可視）を区別。両方必要。適切メモリバリア付きパイプラインバリアが実行順序とメモリ可視性を保証。

==== ワークグループサイズ: 並列性とリソースのバランス

コンピュートシェーダディスパッチ時、ワークグループ数と各グループ内スレッド（呼び出し）数を指定。性能に大きな影響。

*小さすぎ*: 8 スレッドなど——GPU 並列性活用不足。数千スレッド同時なのにごく一部のみ。

*大きすぎ*: 1024 スレッドなど——ハードウェア制限超過、共有メモリ使い果たし。各ワークグループにレジスタ/共有メモリ、リソースが許す分だけ実行。

*ちょうど良い*: 典型的 64、128、256 スレッド。並列性とリソース使用のバランス。

最適サイズはシェーダリソース使用と特定 GPU 依存。プロファイリング不可欠——サイズ変えて測定。

==== データレイアウト: メモリアクセスパターン

メモリ内データ配置が性能に大きく影響。GPU メモリ階層——グローバル（遅いが大）、共有（速いが小）、レジスタ（最速だが極小）。効率的アクセスが不可欠。

*メモリ結合アクセス*: 隣接スレッドが隣接アドレス読むと最良。スレッド 0 が 0、1 が 4、2 が 8… なら単一トランザクションに結合。ランダムアドレスは各別トランザクション、性能低下。

*テンソルレイアウト*: 要素格納順がアクセスパターンに影響。「channels last」(NHWC) vs 「channels first」(NCHW) で演算により 2〜3 倍差。

畳み込みは channels-first がしばしば有利——各空間位置ですべてチャネル読む。要素ごと演算は channels-last が有利かも。普遍の答えなし——演算とハードウェア依存。

*共有メモリ*: データ再利用演算（畳み込み——複数出力位置が同じ入力読む）では共有メモリへロードでグローバルトラフィック大幅削減。協調的にタイルを共有へ、同期後高速共有から読む。

== 学習 vs 推論: 詳細比較

詳細比較表で差を具体化。これらの区別理解が実装選択を導く。

[cols="1,2,2", options="header"]
|===
|側面 |学習 |推論

|*目的*
|重み調整でデータからパターンを学ぶ
|学習済みパターンを新データ予測に適用

|*計算*
|順伝播 + 逆伝播（backpropagation で勾配）
|順伝播のみ——データは一度ネットワークを流れる

|*重み*
|常に変化——各バッチ後損失減少のため調整
|固定——学習済みモデルファイルから読み込み、変更なし

|*メモリ要件*
|逆伝播用全中間活性化 + 勾配 + オプティマイザ状態
|現在レイヤ活性化のみ——進行に伴いメモリ再利用可

|*バッチサイズ*
|しばしば大（32、64、128+）——安定勾配、GPU 効率
|柔軟——低レイテンシなら 1、スループットなら大

|*ハードウェア*
|通常ハイエンド GPU/TPU、大メモリ
|モバイル GPU、組み込み、CPU でも可

|*性能焦点*
|長時間学習のスループット（例/秒）
|対話アプリのレイテンシ（ms/予測）またはバッチスループット

|*フレームワーク*
|PyTorch、TensorFlow、JAX——自動微分・最適化
|TFLite、ONNX Runtime、カスタム実装——順伝播最適化

|*典型的環境*
|強力なハードウェア、リアルタイム制約なし研究/開発
|多様なハードウェア、厳しいレイテンシ要件の本番
|===

Vulkan 実装の要点は単純さ: 順伝播のみ実装。勾配、逆伝播、オプティマイザ状態なし。タスクははるかに管理しやすいが良好な性能には注意深いエンジニアリング依然必要。

== 構築するもの

推論理解をもとに、本シリーズで実際に構築するものを明確に。

事前学習済みモデルを読み込み GPU 上で効率実行する Vulkan ベース推論エンジン:

*モデル読み込み*——標準形式（ONNX または TFLite）からアーキテクチャと学習済み重み抽出。

*GPU メモリ割り当て*——テンソル（入力、出力、中間活性化、重み）、スマートメモリ再利用でフットプリント最小化。

*ニューラルネットワーク演算実装*——畳み込み、行列乗算、活性化、プーリング、正規化などを Vulkan コンピュートシェーダとして。

*推論実行*——正しい順序と適切同期でコンピュートシェーダディスパッチ、データがネットワークを正しく流れる。

*性能最適化*——効率アルゴリズム、メモリ帯域最小化、GPU 固有機能活用。

学習システムは構築しません。逆伝播、オプティマイザ、データ拡張、学習率スケジュールも。それらは学習中に別途行われます。学習済みモデルを可能な限り速く実行することに純粋に焦点。

この焦点がタスクを実行可能に。推論は学習より単純だが注意深いエンジニアリング依然必要。性能要求は高い——リアルタイムは ms 単位——Vulkan がそれらの要求を満たす低レベル制御を提供。

== まとめ

推論は学習済みニューラルネットワークを使って新データで予測するプロセス。学習が逆伝播・最適化で重み調整するのに対し、推論は固定重みで順伝播のみ。

推論パイプライン五段階: 入力到着、前処理でモデル期待形式へ変換、ニューラルネットワークがレイヤ実行、後処理で生出力解釈、結果がアプリケーションへ。各段階は性能最適化可能、Vulkan は特にニューラルネットワーク実行段階に適している。

Vulkan 視点ではニューラルネットワークは GPU バッファ内テンソル上のコンピュート演算列。各レイヤが入力読み、数学演算、出力書き込みのコンピュートシェーダ。主要課題——メモリ管理、同期、性能最適化——はまさに Vulkan が扱うもの。

学習-推論区別が実装を形作る: 順伝播のみ、中間結果メモリ再利用、デプロイ制約向け最適化。Vulkan 推論エンジン構築は実行可能だがリアルタイム性能には注意深いエンジニアリング必要。

推論が何で学習とどう異なるかの理解ができたので、ゼロから学習せず既存モデルを新タスクに適応させる転移学習戦略を探る準備ができました。
