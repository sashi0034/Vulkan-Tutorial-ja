= モデル形式

機械学習で常に遭遇するシナリオ: PyTorch で見事なモデルを学習した。うまく動く。Android 端末へデプロイし、Vulkan ベースのゲームエンジンと統合し、サーバーでも動かしたい。環境ごとに制約、ハードウェア、最適化ニーズが異なる。

プラットフォームごとに別実装を維持する？ 別フレームワークでモデルを書き直す？ すべてが同じ結果を出すことをどう保証する？

これがモデル形式が解く問題です。学習とデプロイ、フレームワークとハードウェア、研究と本番の橋渡しです。実務的 ML 推論では理解が不可欠です。

== バベルの塔問題

深層学習の初期（驚くべきことに、わずか約十年前）には、各フレームワークが独自のモデル表現を持っていました。この分断は分野全体の進歩を遅らせかねない危機を生みました。

=== フレームワークの爆発

2015〜2017 年頃、深層学習が爆発的に普及しました。突然数十のフレームワークが現れ、それぞれ独自の哲学とアプローチを持ちました:

*TensorFlow*（Google, 2015）: 静的計算グラフ中心。モデル構造を事前定義してから実行。本番デプロイと最適化に優れるが、静的グラフはデバッグと実験を難しくした。

*PyTorch*（Facebook, 2016）: 動的計算グラフ（define-by-run）中心。実行しながらグラフを構築し、デバッグが直感的で実験が速い。研究者に愛されたが、当初デプロイは難しかった。

*Caffe*（Berkeley, 2014）: 初期の人気フレームワークの一つ、コンピュータビジョン向け。prototxt でモデル定義——人間可読だが冗長。CNN に優れるが他アーキテクチャには柔軟性が低い。

*MXNet*（Apache, 2015）: 効率とスケーラビリティ向け設計、複数言語サポート。独自 JSON ベース形式でシリアライズ。

*Theano*（Montreal, 2007）: 現代フレームワークの祖、自動微分を先駆。Python pickle でシリアライズ——脆弱で Python 専用。

各フレームワークに強みがあり、研究者はタスクごとに異なるツールへ移りました。しかしデプロイに深刻な問題を生みました。

=== ロックイン問題

世界中の企業で何千回も繰り返されたシナリオ:

研究チームが PyTorch で最先端モデルを学習。タスクで 95% 精度——これまで最高。デプロイに期待しています。

しかし本番インフラは TensorFlow 上に構築。デプロイパイプライン、監視ツール、最適化ワークフローはすべて TensorFlow モデルを前提。モバイルアプリは TensorFlow Lite。エッジデバイスは TensorFlow Lite for Microcontrollers。

どうする？

*選択肢 1: TensorFlow で再実装*。PyTorch から TensorFlow へすべてのレイヤ、演算、詳細を手動翻訳。数週間。バグ混入。TensorFlow 版は 95% ではなく 92%——翻訳で何か失われたが何？ さらに数週間デバッグ。完了頃には研究チームはより良いモデルへ移り、やり直し。

*選択肢 2: 本番で PyTorch をデプロイ*。デプロイインフラ全体の再構築。2017〜2018 年 PyTorch モバイルサポートは未成熟。エッジデバイスは PyTorch を全く動かせない。実行不可能。

*選択肢 3: 諦める*。モデルは本番に到達しない。研究努力は無駄。

これは仮想の問題ではなく、深層学習モデルをデプロイしようとしたほとんどの組織の現実でした。学習用に選んだフレームワークが、そのエコシステムへのデプロイもロックインしました。

=== 相互運用性の危機

個別企業を超え、分野全体が苦しみました:

*研究の再現性*: Caffe モデルで公開された論文は TensorFlow で容易に再現できなかった。研究者は一から再実装し、しばしば異なる結果。実装詳細の差か、元論文が間違いか？

*モデル共有*: 事前学習済みモデルはフレームワーク固有。PyTorch の優れた ResNet は TensorFlow ユーザーが使えない。コミュニティの知識が分断。

*ハードウェア最適化*: ハードウェアベンダー（NVIDIA、Intel、ARM など）は各フレームワークを個別サポート。TensorFlow 最適化は PyTorch ユーザーに役立たない。ハードウェア革新が遅れた。

*ツール開発*: プロファイラ、ビジュアライザ、オプティマイザなどすべてのツールを各フレームワーク向けに個別構築。エコシステムが分断・非効率。

=== 解決策: 相互運用形式

業界には共通言語——学習フレームワークに独立し、デプロイ向け最適化されたニューラルネットワーク表現——が必要でした。いくつかの相互運用形式が、 わずかに異なる問題を解く形で開発されました。

重要な洞察は関心の分離: 学習と推論は異なるタスクで異なる要件。学習には柔軟性、自動微分、最適化アルゴリズム。推論には効率、小サイズ、広いハードウェアサポート。

相互運用形式はこの二つの世界の間に位置します。好みのフレームワークで学習し、相互運用形式へエクスポートし、ターゲットプラットフォームに最適なランタイムでデプロイ。形式がロックインを破ります。

二つの主要形式が、この問題を異なる焦点で解決する形で登場しました:

*TensorFlow Lite*（2017）: Google のモバイルデプロイ向けソリューション。小サイズ、高速実行、積極的最適化に焦点。

*ONNX*（2017）: Microsoft、Facebook などの協力。フレームワーク横断互換性と広いエコシステムサポートに焦点。

それぞれ詳しく見ていきます。

== TensorFlow Lite: モバイルファースト最適化

Google は特定の課題に直面: 強力な TensorFlow はあったが、モバイルには大きすぎ複雑すぎた。フル TensorFlow ランタイムは数百 MB。モバイルアプリは小さく、速く、省電力である必要がある。

TensorFlow Lite はこの制約から生まれました。単なるファイル形式ではなく——モバイルと組み込みデプロイ向けの エコシステム全体です。

=== 設計思想

洞察は、推論には学習のすべての仕組みは不要、ということ。自動微分、オプティマイザ実装、任意の新演算定義の柔軟性は不要。

必要なのは:

- モデルのコンパクト表現
- 効率的に実行する小さなランタイム
- サイズと速度向けモデル最適化ツール

TFLite はモバイル制約に 重点 して三者すべてを提供します。

=== 形式: 効率のための FlatBuffers

形式自体は FlatBuffers ベース——Google が効率専用に開発したシリアライズライブラリ。JSON や Protocol Buffers と異なり、FlatBuffers はパース不要——シリアライズ形式から直接データアクセス。

モバイルで重要。50MB モデルファイルのパースは時間とメモリを要する。FlatBuffers ならファイルをメモリマップして直接アクセス。読み込みはほぼ瞬時、パース表現用の追加メモリ不要。

モデルはコンパクトバイナリ（.tflite）に保存。典型的 ResNet-50 は TensorFlow SavedModel で約 100MB、最適化後 TFLite では 25MB。4 倍削減は量子化から——次に議論。

=== 量子化: 決定的な機能

TFLite の真の力は最適化ツール、特に *量子化* から来ます。

ニューラルネットワークは通常重みと活性化に 32 ビット浮動小数点。各重み 4 バイト。2500 万パラメータモデルは 100MB。

量子化はこれを 8 ビット整数へ。各重み 1 バイト。同じ 2500 万パラメータモデルは 25MB——4 倍削減。

精度は下がらない？ 驚くほど、あまり下がりません。多くのモデルで量子化は 1% 未満の精度低下。一部はわずかに改善（精度低下が正則化として働く）。

サイズを超える利点:

*より速い推論*: 多くのモバイルプロセッサで整数演算は浮動小数点より速い。量子化モデルは 2〜4 倍速い。

*低消費電力*: 整数演算は消費電力が少ない。バッテリー寿命延長——モバイルに 不可欠。

*ハードウェア加速*: 多くのモバイルプロセッサに専用整数演算ユニット。量子化モデルはさらに高速化。

TFLite は複数の量子化方式をサポート:

*学習後量子化*: 最も単純。学習済み float32 モデルを int8 へ変換。再学習不要。多くのモデルで良好。

*量子化認識学習*: 量子化を念頭に学習。学習中に量子化効果をシミュレートし、量子化に頑健に学習。学習後量子化より精度良好だが再学習必要。

*動的範囲量子化*: 重みのみ int8、活性化は float32。完全量子化より適用しやすい中間案。

=== ランタイム: 小さく最適化

TFLite ランタイムはリソース制約デバイス向け:

*サイズ*: コアランタイム約 300KB。全演算込みでも 1MB 未満。TensorFlow の数百 MB と対比。

*依存関係*: 最小限。ランタイムはほぼ自己完結、モバイルアプリ統合が容易。

*ARM 最適化*: NEON SIMD で ARM プロセッサ向けに 重点的に最適化。モバイルは ほぼすべて ARM。

*Delegate サポート*: 専用ハードウェアへ計算をオフロードする「delegate」。GPU delegate は OpenGL ES または Vulkan。NPU delegate はニューラル処理ユニット。デバイスが提供する加速を活用。

=== エコシステム: ツールとモデル

TFLite には 成熟したエコシステム:

*TFLite Converter*: TensorFlow モデル（SavedModel、Keras、concrete functions）を TFLite へ。量子化、最適化、検証を処理。

*TFLite Model Maker*: 学習と変換の高レベル API。画像分類、物体検出、テキスト分類など 一般的なタスクを最小コードで。

*TFLite Support Library*: 前処理と後処理。画像リサイズ、正規化、結果解釈のユーティリティ。

*Model Zoo*: 数百の事前学習済みモデル。画像分類、物体検出、ポーズ推定、セグメンテーションなど。

*ベンチマークツール*: 実デバイスで推論時間とメモリ使用量を測定。最適化に不可欠。

=== トレードオフ

TFLite は完璧ではありません:

*TensorFlow ロックイン*: 他フレームワークから変換可能（PyTorch → ONNX → TensorFlow → TFLite）だが、常に スムーズではない。TFLite は TensorFlow モデル向け設計。

*演算カバレッジ*: TensorFlow 演算のサブセット。一般的なモデルは問題ないが、最先端アーキテクチャは未サポート演算を使うかも。

*量子化の限界*: すべてのモデルがうまく量子化されるわけではない。RNN やアテンションは量子化に敏感。量子化認識学習が必要かも。

*モバイル焦点*: モバイル/組み込み向け最適化。デスクトップ/サーバーでは他形式の方が良いかも。

=== TFLite が適する場合

TFLite は次の場合に 最適:

*モバイルへデプロイ*: Android/iOS アプリ。両プラットフォームで 第一級のサポート。

*サイズが重要*: アプリサイズ予算が 厳しい。TFLite の積極的最適化でモデルを小さく保てる。

*バッテリー寿命が重要*: バッテリー駆動モバイル。TFLite の効率が寿命を延ばす。

*TensorFlow エコシステム*: 学習に既に TensorFlow。変換パスがスムーズで十分に文書化されている。

*事前学習済みモデル*: TensorFlow Hub のモデルを使いたい。多くが TFLite 形式で利用可能。

モバイル向けゲーム開発では、TFLite がしばしば最良の選択。ツールが成熟し、ドキュメントが優れ、性能が匹敵しにくい。

== ONNX: 万能通訳

Google が TFLite でモバイル問題を解く一方、 より広い業界は相互運用性という異なる課題に直面。研究者は PyTorch で学習、本番は TensorFlow。またはその逆。企業はプロジェクトごとに複数フレームワーク。フレームワーク移行のたびに再実装と検証。

Open Neural Network Exchange（ONNX）は Microsoft、Facebook などの協力から生まれました。野心は大きい: 任意フレームワークがエクスポートでき、任意ランタイムが実行できる形式。

=== 設計: 共通言語

ONNX は単なるファイル形式以上——ニューラルネットワーク演算表現の仕様。ニューラルネットワーク向けプログラミング言語と考え、 明確に定義された構文と意味論。

中核は *演算セット*: 演算（畳み込み、行列乗算、活性化、プーリング、正規化など）の包括リストと 精密な定義。各演算は:

- 入力（形状と型のテンソル）
- 出力
- パラメータ（ストライド、パディングなど）
- 入力から出力を計算する 正確な方法

PyTorch が ONNX へエクスポートするとき、内部演算をこの共通言語へ翻訳。PyTorch `Conv2d` は特定パラメータの ONNX `Conv` 演算に。ONNX ランタイムがモデルを読み込むと、各演算の意味と実行方法を正確に知っている。

=== 形式: Protocol Buffers

ONNX は Protocol Buffers（protobuf）でシリアライズ。Google 開発のバイナリ形式、業界で 広く使用。

形式は次を保存:

- *グラフ構造*: ノード（演算）とエッジ（演算間データフロー）
- *演算タイプ*: 各ノードの ONNX 演算
- *属性*: 各演算のパラメータ（カーネルサイズ、ストライドなど）
- *初期化子*: 学習済み重みとバイアス
- *メタデータ*: モデル版、プロデューサ情報など

ONNX モデルはデフォルトで積極的な量子化を適用しないため TFLite よりしばしば大きい。ONNX の ResNet-50 は約 100MB（TensorFlow SavedModel と同程度）、量子化 TFLite は 25MB。

=== 拡張性: 分野と共に成長

ONNX の強みの一つは拡張性。深層学習は 急速に進化——新レイヤ型、アーキテクチャ、演算が 絶えず現れる。

ONNX は *演算バージョニング* で対応。演算セットは時間とともに進化:

- ONNX 1.0（2017）: 約 60 演算
- ONNX 1.6（2020）: 約 150 演算
- ONNX 1.13（2023）: 約 180 演算

新演算が 一般的になると標準に追加。モデルは使用 ONNX 版を指定し互換性を保証。

ONNX は *カスタム演算* もサポート。標準にない演算が必要なら独自定義可能。最先端研究で新演算が 定期的に発明されるため 不可欠。

=== エコシステム: ツールとランタイム

ONNX には 豊富なエコシステム:

*コンバータ*:

- PyTorch → ONNX: PyTorch 組み込み、優れた サポート
- TensorFlow → ONNX: tf2onnx 経由、ほとんどのモデルで良好
- Keras → ONNX: keras2onnx または tf2onnx
- Scikit-learn → ONNX: sklearn-onnx、従来 ML モデル向け

*オプティマイザ*:

- ONNX Runtime 最適化パス: 演算融合、冗長ノード除去、定数畳み込み
- ONNX Simplifier: 複雑グラフ簡略化、デプロイ容易化
- ONNX Optimizer: グラフレベル最適化

*ランタイム*:

- ONNX Runtime（Microsoft）: 高性能、CPU、WebGPU、DirectML、CUDA サポート
- TensorRT（NVIDIA）: NVIDIA GPU 向け最適化
- OpenVINO（Intel）: Intel ハードウェア向け最適化
- ONNX.js: Web ブラウザで ONNX モデル実行
- カスタムランタイム: 多くの企業が独自構築

*検証ツール*:

- ONNX Checker: モデル正しさ検証
- ONNX Shape Inference: グラフ全体のテンソル形状推論
- Netron: ONNX モデル可視化、デバッグに不可欠

=== 実務的影響

ONNX はフレームワークロックインを破ります。実世界のワークフロー:

1. 研究チームが PyTorch で学習（好みのフレームワーク）
2. 一行で ONNX へ: `torch.onnx.export(model, ...)`
3. エクスポート検証: PyTorch と ONNX Runtime 両方で推論、出力一致確認
4. ONNX Runtime でサーバーへデプロイ（本番向け最適化）
5. モバイル向け ONNX → TFLite 変換
6. NVIDIA GPU 向け ONNX → TensorRT 変換

一モデル、複数デプロイターゲット、手動再実装なし。

=== トレードオフ

ONNX は完璧ではありません:

*サイズ*: TFLite より大きい。デフォルトで積極的な量子化なし。モバイルでは ONNX → TFLite 変換が必要かも。

*変換問題*: すべての演算が 完全に変換されるわけではない。動的形状、制御フロー、カスタム演算が 問題のある。ONNX 互換のためモデル修正が必要かも。

*ランタイム性能のばらつき*: ONNX Runtime は速いが、実行プロバイダ（WebGPU、DirectML、CPU など）依存。最適化度はランタイムにより異なる。

*演算カバレッジ*: 多くの演算をサポートするが、最先端研究はまだ標準にない演算を使うかも。カスタム演算が必要で可搬性低下。

=== ONNX が適する場合

ONNX は次の場合に 最適:

*フレームワーク横断互換性*: 一フレームワークで学習、別でデプロイ。ONNX が橋。

*複数デプロイターゲット*: サーバー、エッジ、モバイルへ。ONNX からターゲット形式へ変換可能。

*デスクトップ/サーバーデプロイ*: モバイルほどサイズ critical ではない。ONNX の大きいモデルも許容。

*柔軟性*: すべて再実装せずフレームワーク切替。ONNX が柔軟性を提供。

*エコシステム*: 豊富なツール、オプティマイザ、ランタイムへアクセス。

Vulkan ベース推論エンジンでは、ONNX が しばしば最良。フレームワーク非依存、十分に文書化された、優れた ツール。本チュートリアルの原理は ONNX モデル向けだが、他形式にも適用可能。

== 形式間変換: 現実

実務ではめったに一形式だけ。典型的なワークフローは変換を含み、しばしば複数ステップ。一般的な変換パスと落とし穴を理解すると、この分野を成功裏に進められます。

=== 一般的な変換ワークフロー

*PyTorch → ONNX → TFLite（モバイルデプロイ）*:

1. PyTorch で学習（研究者向け、柔軟）
2. ONNX へ: `torch.onnx.export(model, dummy_input, "model.onnx")`
3. ONNX を TensorFlow へ: `onnx-tf` コンバータ
4. TensorFlow を TFLite へ: 量子化付き TFLite コンバータ
5. Android/iOS へデプロイ

PyTorch が研究で 人気、TFLite がモバイルに 最適なため 一般的なパス。

*TensorFlow → TFLite + ONNX（マルチプラットフォーム）*:

1. TensorFlow で学習
2. モバイル向け TFLite: TFLite コンバータで直接変換
3. 他プラットフォーム向け ONNX: `tf2onnx`
4. モバイルは TFLite、他は ONNX

TensorFlow の 幅広いサポートを活かしつつ柔軟性維持。

*ONNX を 汎用ハブ*:

1. 任意フレームワークで学習
2. ONNX へエクスポート（ほとんどのフレームワークがサポート）
3. 必要に応じ ONNX からターゲット形式へ:
   - モバイル: ONNX → TFLite
   - NVIDIA GPU: ONNX → TensorRT
   - Intel: ONNX → OpenVINO

ONNX が汎用中間表現となり、最大の柔軟性。

=== 検証: 重要ステップ

すべての変換はバグの 潜在的な原因。検証は 省略不可。

*数値検証*:

同じ入力を元モデルと変換後モデルで実行。出力比較:

[source,python]
----
# Original PyTorch model
pytorch_output = model(input_tensor)

# Converted ONNX model
onnx_session = onnxruntime.InferenceSession("model.onnx")
onnx_output = onnx_session.run(None, {"input": input_numpy})

# Compare
difference = np.abs(pytorch_output.numpy() - onnx_output)
max_diff = np.max(difference)
print(f"Maximum difference: {max_diff}")
----

許容範囲？ 浮動小数点演算は厳密ではないため小さな差は正常。最大差 1e-5 〜 1e-6 が典型的で許容範囲。1e-3 超は問題の示唆。

*統計検証*:

検証セット（数百〜数千例）で推論。比較:

- 正解率: 変換後モデルは同じ精度？
- Top-5 正解率: 分類では上位予測は一致？
- 平均絶対誤差: 回帰では平均誤差は同じ？

精度が 1% 以上下がれば調査。変換に問題 可能性が高い。

*視覚検証*:

ビジョンモデルでは出力を可視化:

- 分類: 予測は一致？
- 物体検出: バウンディングボックスは整合？
- セグメンテーション: マスクは同一に見える？

数値比較では 捕捉 できない問題を 目視検査が しばしば捉える。

=== 一般的な変換の落とし穴

*動的形状*:

一部フレームワークは動的入力形状（バッチサイズ可変）をサポート。他は固定形状必須。動的から固定への変換は 具体的な形状指定が必要。デプロイニーズと合わないかも。

対策: 本番で使う形状でエクスポート。複数形状が必要なら複数モデルをエクスポート。

*カスタム演算*:

カスタム演算（自分で定義）を使うモデルは変換されないかも。標準形式は標準演算のみ。

対策: 標準演算で再実装、またはターゲット形式にカスタム演算サポート追加（上級）。

*制御フロー*:

if 文やループ（動的制御フロー）を持つモデルは変換が 困難。すべての形式が制御フローをサポートするわけではない。

対策: 可能なら制御フロー回避。固定サイズ演算を使用。避けられないなら制御フローをサポートする形式（ONNX はサポート、TFLite は 限定的）を選択。

*量子化ミスマッチ*:

量子化付き TFLite 変換は 慎重でないと精度低下。量子化方式が重要。

対策: 精度低下が大きすぎれば量子化認識学習。量子化後 徹底的な検証。

*演算バージョンミスマッチ*:

ONNX 演算は時間とともに進化。ONNX 1.10 でエクスポートしたモデルは ONNX 1.6 未サポート演算を使うかも。

対策: 演算バージョン確認。ランタイム更新、または古い ONNX 版でエクスポート。

=== 変換問題のデバッグ

変換失敗または 誤った結果:

*演算サポート確認*:

ターゲット形式がモデルのすべての演算をサポート？ 形式固有ツール:

- ONNX: `onnx.checker.check_model(model)`
- TFLite: TFLite 演算互換性確認

*グラフ可視化*:

Netron（netron.app）で元と変換後を可視化。確認:

- 欠落演算
- 誤った接続
- 形状ミスマッチ

*モデル簡略化*:

複雑モデルが失敗するなら、より単純版を変換:

- カスタム演算除去
- 動的の代わりに固定形状
- 制御フロー簡略化

単純版が動いたら、徐々に複雑さを戻して 問題のある 成分を特定。

*フレームワークバージョン確認*:

変換ツールはフレームワーク版に 敏感。互換バージョン使用:

- PyTorch 1.10+ で 良好な ONNX エクスポート
- TensorFlow 2.x で tf2onnx
- 最新 onnx-tf で ONNX → TensorFlow

=== ベストプラクティス

*早く、頻繁にエクスポート*:

完全学習まで変換テストを待たない。開発早期にエクスポートし、障害になる前に問題を捕捉。

*前処理を文書化*:

モデルが期待する前処理を 正確に記録。この情報はモデルと共に 引き継ぎ必須。さもなければ推論失敗。

*すべてをバージョン管理*:

フレームワーク版、コンバータ版、モデル版を追跡。再現性には使用ツールの 正確な把握が必要。

*検証を自動化*:

変換モデル検証の自動テストを CI/CD に。性能退行を捕捉。

*元モデルを保持*:

常にネイティブ形式の元モデルを保持。変換モデルはデプロイ用、元は再学習・再変換の 正とするソース。

== Vulkan 推論向け形式の選択

Vulkan ベース推論実装では、どの形式を使うべき？

正直な答え: ソースと優先度次第。

*TensorFlow モデルから始め、モバイルがターゲット* なら TFLite が 最も抵抗の少ない経路。形式が 十分に文書化された、モデルが最適化済み、例が豊富。

*柔軟性とフレームワーク横断サポートが必要* なら ONNX が 最良の選択。最も汎用的、最も豊富なエコシステム。任意フレームワークのモデルと 優れた最適化ツール。

本チュートリアルシリーズでは主に ONNX を扱います。任意フレームワークのモデル、優れたツール、大きなモデル zoo の柔軟性。ただし扱う原理は任意形式に適用——すべて同じ根底概念（レイヤ、重み、演算）を、異なるシリアライズで表現しているだけ。

== 形式に実際に含まれるもの

選択形式に関わらず、本質的な情報は同じ:

*ネットワークアーキテクチャ*: モデル構造——どのレイヤがあり、どう接続され、各レイヤが何を行うか。推論の設計図。

*重みとバイアス*: 学習済みパラメータ——学習中に得た数百万の浮動小数点数。ファイルサイズの 大部分。

*メタデータ*: 入出力形状、データ型、前処理要件、その他正しい使用に必要な詳細。

*演算定義*: 各演算の仕様——畳み込みの働き、活性化関数の役割など。implicit（標準演算セット参照）または explicit（カスタム演算定義含む）。

Vulkan 推論を実装するとき、これら形式をパースして情報抽出、重みを GPU メモリへ読み込み、演算をコンピュートシェーダとして実装します。形式は 包装——本当の仕事は実行にあります。

== まとめ

モデル形式は、学習済みモデルを研究フレームワークから本番デプロイへ移す問題を解きます。二つの主要形式が支配: TFLite（モバイル最適化）と ONNX（フレームワーク非依存）。

TFLite は積極的最適化ツール、特に量子化でモバイルデプロイに優れます。ONNX は豊富なエコシステムでフレームワーク横断の汎用互換性を提供。

実務では形式変換が一般的で、典型的なワークフローは複数ステップ（モバイル向け PyTorch→ONNX→TFLite、または ONNX を汎用中間形式）。各形式は同じ本質的な情報——ネットワークアーキテクチャ、学習済み重み、メタデータ、演算定義——を含みます。

Vulkan 推論では ONNX が柔軟性とエコシステムの 最良のバランス を提供しますが、原理は任意形式に適用。これら形式の理解が、モデル読み込みと GPU 実行への写像の準備となります。

モデルのパッケージングと転送の理解ができたので、これらモデルが Vulkan で GPU 実行へどう写るかのメンタルモデルを探る準備ができました。
