= メンタルモデル: GPU 実行

機械学習とは何かから、ニューラルネットワークの働き、学習済みモデルをパッケージする形式まで、多くを扱ってきました。今度は、これらすべての概念を Vulkan による実際の GPU 実行につなぐメンタルモデルで全体像をつなぎます。

本章は一歩引いて全体像を見る章です。ディスク上の学習済みモデルから GPU 上で動く推論まで、完全な旅をたどり、途中の主要な課題と設計判断を特定します。

== モデルから GPU への旅

学習済みモデルがアプリケーション内でたどる完全なライフサイクルを追いましょう。この旅を理解すると、各要素がどこに位置し、なぜ重要かが見えます。

*ステップ 1: 学習（私たちの問題ではない）*

どこかで、誰かがニューラルネットワークを学習しました。あなた自身かもしれませんし、事前学習済みモデルを公開した研究者かもしれません。PyTorch や TensorFlow を使い、数百万の例を与え、数時間から数日の計算の後、学習済みモデルが得られました。重みとバイアスが学習され、テストデータで良好に動作し、デプロイの準備ができています。

物語はここ——本番で効率的に動かす必要のある学習済みモデル——から始まります。

*ステップ 2: 推論形式へエクスポート*

学習フレームワークのネイティブ形式はデプロイに最適ではありません。学習専用の仕組みを含み、特定フレームワーク版に縛られ、サイズや読み込み速度に最適化されていないことが多いです。そこでモデルは推論形式——TFLite または ONNX——へエクスポートされます。

このエクスポートは学習専用コンポーネントを取り除き、グラフを最適化（不要な演算の除去、可能なレイヤ融合）し、すべてをコンパクトな標準形式にパッケージします。得られるのは数 MB から数百 MB のファイルで、ネットワークアーキテクチャ、学習済み重みすべて、使用方法のメタデータを含みます。

*ステップ 3: 読み込みとパース（最初の課題）*

アプリケーションがこのモデルファイルを読み込む必要があります。これが最初の本当の課題です: 形式固有の構造をパースして必要な情報を抽出すること。

特定スキーマのバイナリファイルを読んでいます。ONNX なら Protocol Buffers、TFLite なら FlatBuffers。それぞれに癖がありますが、本質的な情報は同じです: どのレイヤがあるか、どう接続されているか、各レイヤがどの演算を行うか、各演算の重み。

パースしながら内部表現——モデルアーキテクチャを捉えるグラフ構造——を構築します。グラフの各ノードが演算（畳み込み、行列乗算、活性化関数など）を表し、エッジが演算間のデータフローを表します。

*ステップ 4: Vulkan へコンパイル（翻訳）*

モデルグラフが手元にあれば、Vulkan コンピュート演算へ翻訳する必要があります。ニューラルネットワークと Vulkan の両方の理解がここで交わります。

モデル内の各演算タイプに対応するコンピュートシェーダが必要です。畳み込みレイヤには畳み込みシェーダ、行列乗算には行列乗算シェーダ、活性化関数には活性化シェーダ。本質的にニューラルネットワーク演算を実装するコンピュートシェーダのライブラリを構築しています。

しかしシェーダだけではありません。次も必要です:

*GPU メモリの割り当て*——すべてのテンソル（入力、出力、中間活性化、重み）用。重要な設計判断です。すべてを事前に割り当てる（静的割り当て）か、必要に応じて（動的割り当て）か？ もう不要な中間結果のメモリを再利用するか？

*重みの GPU メモリへアップロード*。モデルを定義する数百万の学習済みパラメータ。推論中は定数なので、一度アップロードして GPU 上に常駐させます。

*各シェーダ用 Vulkan パイプラインの作成*。バッファアクセス用の適切なディスクリプタセット付き。標準的な Vulkan 作業ですが、規模は慣れより大きいかもしれません——複雑なモデルは数十種類の演算タイプを持ちます。

*実行順序の計画*。モデルグラフが依存関係——どの演算が完了してから次が始められるか——を教えます。これを適切な同期付きコンピュートディスパッチの列へ翻訳します。

*ステップ 5: 推論の実行（成果）*

推論を実行する準備ができました。各推論リクエストで起きること:

*入力データを GPU メモリへ転送*。画像、音声バッファ、その他の入力かもしれません。転送自体は単純ですが、レイテンシに敏感なアプリケーションではこのオーバーヘッド最小化が重要です。

*計画した順序でコンピュートシェーダをディスパッチ*。各シェーダが入力テンソルを読み、演算を行い、出力テンソルを書き込みます。データがネットワークをレイヤごとに流れ、最終出力に至ります。

*演算間の同期*。正しさに不可欠です。各演算が完了し、その書き込みが依存演算が始める前に可視であることを保証する必要があります。Vulkan はパイプラインバリア、メモリバリア、セマフォなどのツールを提供します。

*結果を CPU メモリへ読み戻し*。最終出力テンソルに予測——クラス確率、検出物体、生成テキストなど——が含まれます。

入力から出力まで——このプロセス全体が速く起きる必要があります。リアルタイムアプリケーションでは、わずか数ミリ秒で推論を完了する必要があるかもしれません。

== 主要な課題

各段階で直面する課題を深掘りしましょう。課題を理解すると、より良い設計判断ができます。

=== 課題 1: メモリ管理

ニューラルネットワークは大きな多次元配列（テンソル）で動作します。単一画像は 224×224×3 浮動小数点（約 600KB）。中間活性化ははるかに大きくなり得ます。典型的なモデルは数百 MB の GPU メモリを必要とします。

すべてのテンソルに新しいバッファを割り当てる素朴なアプローチは無駄です。多くの中間テンソルは短時間しか必要ありません。演算が完了し出力が次演算に消費されたら、そのメモリは再利用できます。

賢いアプローチはモデルグラフを分析し、どのテンソルがメモリを共有できるか特定すること。テンソル A が演算 1〜3 にのみ必要で、テンソル B が演算 5〜7 にのみ必要なら、同じメモリを使えます。これを *メモリ再利用* または *メモリプール* と呼び、メモリフットプリントを劇的に削減できます。

トレードオフは複雑さです。メモリ再利用付き静的割り当ては、グラフ全体を事前分析しメモリ使用を計画する必要があります。動的割り当ては単純ですが遅く、より多くのメモリを使います。多くのアプリケーションでは、静的割り当ての性能上の利点が実装の複雑さに見合います。

=== 課題 2: 同期

GPU は大規模並列ですが、ニューラルネットワーク推論は本質的に逐次的——各レイヤは前レイヤの出力に依存します。レイヤ 4 が完了するまでレイヤ 5 は始められません。並列性と依存関係の間に緊張があります。

Vulkan は同期に対する明示的制御を与え、強力かつ危険です。同期が少なすぎるとレース条件——書き込み前にデータを読む演算——が起き、不正な結果になります。同期が多すぎると実行が直列化され、並列性と性能を失います。

鍵は依存関係を正確に理解すること。必要な順序を強制しつつ、独立演算は重ねられるようパイプラインバリアを使います。例えばモデルに並列分岐（現代アーキテクチャで一般的）があれば、合流時に同期すれば分岐は同時実行できます。

=== 課題 3: 性能最適化

推論を動かすことと、速く動かすことは別です。最適化の機会は多くあります:

*演算融合*: 複数演算を単一シェーダにまとめオーバーヘッドを削減。例: 畳み込みの後の活性化を、両方を行う一つのシェーダに融合し、中間バッファとディスパッチを排除。

*メモリレイアウト*: メモリ内のデータ配置がキャッシュ効率に影響。一部レイアウトは特定演算に有利。適切なレイアウト——またはレイアウト間変換——が性能に大きく影響。

*バッチ処理*: 複数入力をまとめて処理すると GPU 利用率が上がりスループット改善。一度に一画像ではなく 4、8、16 まとめて推論。レイテンシは増えますがスループットは劇的に改善。

*ハードウェア固有機能*: 現代 GPU には特定演算向け専用ハードウェアがあります。例: NVIDIA の Tensor コア、AMD の WMMA、Intel の XMX が行列乗算を加速。ターゲットハードウェアを理解し、その能力（WebGPU や Vulkan などベンダー横断 API 経由で）を活用すると大幅な高速化が得られます。

課題は、どの最適化が特定モデルとハードウェアで重要か知ること。プロファイリングが不可欠——時間が実際にどこで使われているか測定し、そのホットスポットを最適化します。

== レイヤと演算として考える

推論を実装するとき、ニューラルネットワーク概念と GPU コンピュート演算の対応を考えると役立ちます。具体例で明確にしましょう。

*Dense（全結合）レイヤ*: 本質的に行列乗算の後に活性化関数。入力はベクトル（またはベクトルのバッチ）、重みは行列、`output = activation(weights × input + bias)` を計算。GPU 上では行列乗算シェーダの後に活性化シェーダ。行列乗算が高コスト——次元によって O(n²) または O(n³)——なので最適化が重要。

*畳み込みレイヤ*: スライディングウィンドウ演算。小さなフィルタ（3×3 や 5×5）を入力上でスライドし、各位置で重み付き和を計算。複数入力チャネルと複数出力フィルタで計算量は大きい。GPU では出力位置とチャネル間で並列化。各ワークグループが一つの出力タイルを計算し、グループ内スレッドが異なるチャネルや位置を担当できます。

*プーリングレイヤ*: より単純——小領域の最大または平均でダウンサンプリング。完全に並列: 各出力位置は独立。最小限の同期で多くの出力位置を並列処理するシェーダ。

*正規化レイヤ*: 次元にわたる統計（平均、分散）を計算してから正規化。統計計算はリダクション演算で慎重な同期が必要。統計が得られれば正規化の適用は単純。

これらの対応を理解すると、効率的なシェーダ設計と性能の理由付けができます。畳み込みの多いモデルは計算量制約。小さな演算の多いモデルはオーバーヘッド制約。これが最適化戦略を導きます。

== 全体像をつなぐ

具体例で全体をたどりましょう。画像分類モデル——ResNet-50 のような——で推論を実行していると想像してください。

224×224×3 ピクセルの画像から始めます。GPU メモリへ転送——約 600KB。最初のレイヤは畳み込み: 7×7 フィルタ、64 出力チャネル。畳み込みシェーダをディスパッチし、入力画像と重みテンソルを読み、畳み込みを計算し、出力を書き込み。出力は 112×112×64（ストライドによるダウンサンプリング）。

次にバッチ正規化と ReLU 活性化。これらの演算用シェーダをディスパッチし、畳み込み出力を読み、正規化・活性化結果を書き込み。

その後 ResNet の核心である一連の残差ブロックへ。各ブロックに複数の畳み込み、正規化、活性化と、より早い活性化を後のものに加えるスキップ接続。各演算用シェーダをディスパッチし、スキップ接続が正しいデータを読むよう慎重に同期。

多くのレイヤの後、グローバル平均プーリング——各チャネルを単一値へ。最後にクラス確率を出す全結合レイヤ。最終シェーダをディスパッチし、出力は 1000 クラス（ImageNet）の確率ベクトル。

最後にこの出力を CPU メモリへ読み戻し。最大確率を見つけ、それが予測です:「ゴールデンレトリバー」や「コーヒーマグ」など。

このプロセス全体——数十のシェーダディスパッチ、GPU を流れる数百 MB のデータ——がミリ秒で完了します。それがニューラルネットワーク推論向け GPU 加速の力です。

== まとめ

本章は Vulkan による ML 推論の完全なメンタルモデルを提示し、学習済みモデルから GPU 実行までの旅をたどりました。パイプラインは推論形式へのエクスポート、モデルファイルの読み込みとパース、Vulkan コンピュートシェーダへのコンパイル、GPU 上での推論実行から成ります。

主要課題はメモリ管理（バッファの効率的割り当てと再利用）、同期（過度な直列化なく依存関係を強制）、性能最適化（融合、レイアウト、バッチ処理、ハードウェア機能）です。ニューラルネットワークレイヤと GPU コンピュート演算の対応を理解すると、効率的な実装を設計できます。

ML 推論パイプライン章シリーズ全体を通じて、包括的な基盤を築きました: 機械学習とは何か、ニューラルネットワークの働き、学習と推論の区別、転移学習戦略、モデル形式、完全な実行パイプライン。この概念的理解が、次章の Vulkan コンピュートシェーダによる実装実践の準備となります。

== 振り返りと先へ

章シリーズ全体で学んだことを振り返りましょう:

*機械学習* がコンピュータに明示的なルールではなくデータから学ばせることを理解しました。この根本的転換——ルールのプログラミングからパターンの学習へ——が、明示的にプログラム不可能なタスクを可能にします。

*ニューラルネットワーク* を生物学的ニューロンに触発された数学的関数として探りました。*線形関数だけでは不十分*——より単純な形に崩れる——理由と、*非線形活性化関数* が複雑な関数近似に不可欠な理由を見ました。

*学習* と *推論* を区別し、推論は学習プロセス自体ではなく学習済みモデルを効率的に使うことだと理解しました。*推論パイプライン*——前処理、ニューラルネットワーク実行、後処理——を探りました。

*転移学習* により、ゼロから学習する代わりに事前学習済みモデルを活用できることを学びました。学習フレームワークとデプロイ環境をつなぐ *モデル形式*（TFLite と ONNX）を扱いました。

最後に、学習済みモデルから GPU 実行までの *完全なパイプライン* をたどり、メモリ管理、同期、性能最適化の主要課題を特定しました。

数学的基礎から実装パイプラインまで、このメンタルモデルが整いました。次章では ML ワークロード向け Vulkan コンピュートパイプラインを探り、基本演算を実装し、完全な推論エンジンへ向かいます。
