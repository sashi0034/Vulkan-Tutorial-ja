= CI/CD レンダー検証
:stem: latexmath

Vulkan レンダラーをようやく完成させました。高性能開発マシンでは美しく見えます。照明も完璧、テクスチャも鮮明、フレームレートも滑らか。しかしコードを push すると CI パイプラインが失敗します。あるいはさらに悪いことに、通過するが同僚がブランチを pull すると黒画面だけ——考慮していなかった微妙なドライバ差のせいで。

異なるハードウェアとドライバでレンダラーが正しく動いていることを*どう知る*か？ 従来のユニットテストは画面を見られません。ピクセル単位比較は notorious に fragile——1 ピクセルずれやドライバのわずかな色変化で false positive になり得ます。

本章では、より賢い検証パイプラインを構築します。「このピクセルは同一か？」を超え、学んだ ML 技法で*何を見ているか*を理解するシステムを作ります。

== 標準テストがグラフィックスで失敗する理由

レンダラーのテストを書いたことがあれば、「Gold Image」問題に直面したでしょう。「完璧な」スクリーンショットを保存し、将来のレンダーを毎回比較。うまくいくまで:

*   **解像度 drift**: UI スケールを変えると、すべてのピクセルがわずかに異なる。
*   **ドライバ jitter**: GPU ドライバを更新すると、アンチエイリアシング実装のサブピクセルサンプリングパターンが変わる。
*   **ヘッドレス不一致**: CI でソフトウェアレンダラー（SwiftShader など）を使うと、NVIDIA や AMD ハードウェアのラスタライザより浮動小数点丸めが異なる場合がある。

 suddenly、人間の目には同一に見える数千の「失敗」テスト。これが**Oracle 問題**——画像が間違っているのは見やすいが、すべてのピクセルに対する「正しさ」を定義する数学的ルールを書くのは非常に難しい。

これを解くには、適切な仕事に適切なツールを使う検証階層が必要です。

== 人間中心のテスト階層

人間テスターがアプリを検証する方法を考えてください。ピクセルを数えません。次を尋ねます:

1.  **画面に何かあるか？**（サニティチェック）
2.  **メニューは読めるか？**（機能検証）
3.  **シーンは「正しく」見えるか？**（意味的検証）

CI パイプラインでもこのアプローチを鏡写しし、高速・低コストチェックからより洗練された ML 解析へ進みます。

.グラフィックス向け検証階層
image::images/ML_Inference/Desktop_Applications/validation_hierarchy.svg[Pyramid showing Sanity, Functional, and Semantic validation levels,600]

*   **レベル 1: 「パニック不要」サニティチェック。** 高許容ピクセル比較と **LPIPS** で、ミリ秒で壊滅的失敗（黒画面、白画面、静止）を捕捉。
*   **レベル 2: OCR で UI を読む。** 光学文字認識でテキストがレンダリングされているだけでなく、実際に判読可能でボックス内に収まっていることを確認。
*   **レベル 3: ML による意味的検証。** MNIST と ImageNet 向けに構築したものに似た軽量分類器で、Z-fighting、欠落テクスチャ、壊れた照明など特定レンダリングアーティファクトを検出。

本章の終わりまでに、コードがコンパイルするかだけでなく、アプリケーションが正しく動いていることを実際に「見る」パイプラインが得られます。

== 舞台設定: CI でのヘッドレス Vulkan

レンダーを検証する前に、モニターなしでレンダリングできる必要があります。GitHub Actions など CI ランナーは本質的に「ヘッドレス」サーバー。GPU もディスプレイもありません。

これに対処するため**ソフトウェアレンダラー**を使います。Vulkan API の CPU ベース実装。ゲームには遅すぎますが CI には最適——決定的で、すべてのマシンで同じ結果を出すべきです。

=== ツールの入手
Linux ベース CI ランナーでは通常 **SwiftShader**（Google 開発）または **LavaPipe**（Mesa プロジェクト）を使います。制限環境での安定性から SwiftShader が好まれることが多い。

[source,yaml]
----
# Example GitHub Actions snippet

    echo "VK_ICD_FILENAMES=$PWD/swiftshader/vk_swiftshader_icd.json" >> $GITHUB_ENV
----

=== ヘッドレス世界向けコード
通常アプリではウィンドウ接続のため `VkSurfaceKHR` を作成。CI では surface を完全にスキップし、「オフスクリーンフレームバッファ」に直接レンダリング。

`HeadlessVulkanApp` クラスをウィンドウなしで動くよう適応する方法:

[source,cpp]
----
// Simplified HeadlessVulkanApp focusing on the offscreen setup
class HeadlessVulkanApp {
public:
    HeadlessVulkanApp(uint32_t width, uint32_t height) : width_(width), height_(height) {
        // Standard Vulkan setup (Instance, Device, etc.) omitted for brevity
        setupVulkanHeadless();
        createFramebuffer();
    }

    void createFramebuffer() {
        // Create offscreen image (no swapchain needed)
        vk::ImageCreateInfo imageInfo{
            .imageType = vk::ImageType::e2D,
            .format = vk::Format::eR8G8B8A8Unorm,
            .extent = {width_, height_, 1},
            .tiling = vk::ImageTiling::eLinear,  // Linear for easy CPU readback
            .usage = vk::ImageUsageFlagBits::eColorAttachment | vk::ImageUsageFlagBits::eTransferSrc,
            // ...
        };
        framebufferImage_ = device_.createImage(imageInfo);

        // Allocate and bind host-visible memory for direct CPU access
        // (In a real app, you'd use a more efficient staging buffer approach)
        auto memReqs = framebufferImage_.getMemoryRequirements();
        allocateAndBindMemory(framebufferImage_, vk::MemoryPropertyFlagBits::eHostVisible);
    }

    std::vector<uint8_t> readFramebuffer() {
        void* mapped = framebufferMemory_.mapMemory(0, imageSize_);
        std::vector<uint8_t> pixels(imageSize_);
        std::memcpy(pixels.data(), mapped, imageSize_);
        framebufferMemory_.unmapMemory();
        return pixels;
    }

    // ... rendering and boilerplate omitted ...
};
----

ウィンドウシステムなしの Vulkan コンテキストを作り、オフスクリーンイメージにレンダリングし、検証用 readback を提供します。完全実装（標準 instance/device 作成含む）は `ci_render_validation.cpp` 添付にあります。

== レベル 1: LPIPS で全体像を見る

特定 UI 要素やテクスチャ検証の詳細に入る前に、シンプルな問いに答える方法が必要:「これは同じアプリケーションに見えるか？」

グラフィックス経験があれば、単純な平均二乗誤差（MSE）比較が悪夢だと分かります。キューブがタイミングの fluke でわずかに異なる回転、またはドライバがわずかに異なる AA カーネルを使うと、画像は同一に見えても MSE が skyrocket します。

=== 知覚的差異

従来のピクセル比較は画像を数値のスプレッドシートのように扱います。1 数値が変わればテスト失敗。

**LPIPS（Learned Perceptual Image Patch Similarity）**は人間の目のように画像を扱います。事前学習ニューラルネットワーク（VGG や AlexNet など）で特徴を抽出。「犬らしさ」「エッジらしさ」「テクスチャらしさ」を探します。

高レベル特徴が一致すれば、ピクセルが数学的に同一でなくても知覚的に類似。

.LPIPS 知覚的類似度メカニズム
image::images/ML_Inference/Desktop_Applications/lpips_diagram.svg[LPIPS diagram showing how reference and test images are compared via feature maps,800]

=== フィーチャマップがピクセルに勝つ理由

画像が CNN（第 2 章で学んだ）を通ると、各レイヤが**フィーチャマップ**を生成。

1.  **初期レイヤ**: エッジと色を捕捉。
2.  **中間レイヤ**: テクスチャと形状。
3.  **深いレイヤ**: 意味的物体。

LPIPS は「Golden」画像と「Test」画像の両方について、事前学習ネットワークの複数レイヤのフィーチャマップを取り、これらのマップ間距離を計算。

[latexmath]
++++
d(x, x_0) = \sum_l \frac{1}{H_l W_l} \sum_{h,w} || w_l \odot (\hat{y}^l_{h,w} - \hat{y}^l_{0,h,w}) ||^2_2
++++

この距離 latexmath:[d] が類似度スコア。

*   **latexmath:[d \approx 0]**: 知覚的に同一。
*   **latexmath:[d < 0.1]**: 軽微な差（例: 異なる AA）。
*   **latexmath:[d > 0.3]**: 何かがおかしい（例: 誤テクスチャ、欠落物体）。

=== LPIPS の実装
画像分類器と同じ ONNX Runtime 設定で LPIPS を実行できます。2 画像——現在レンダーと「golden」参照——を渡すと距離スコアを返します。0 付近なら同じに見える; 0.3 超は通常深刻な問題。

[source,cpp]
----
class LPIPSValidator {
public:
    // ... setup and ONNX session initialization ...

    float computeSimilarity(const std::vector<uint8_t>& img1, const std::vector<uint8_t>& img2) {
        // Preprocess: convert RGBA to RGB and normalize to [-1, 1]
        auto tensor1 = preprocessImage(img1);
        auto tensor2 = preprocessImage(img2);

        // Run LPIPS ONNX model
        auto outputs = session_.Run(Ort::RunOptions{nullptr}, inputNames, inputs.data(), 2, outputNames, 1);

        // Return the perceptual distance (lower is more similar)
        return outputs[0].GetTensorMutableData<float>()[0];
    }
};

bool testSanityCheck() {
    HeadlessVulkanApp app(800, 600);
    auto pixels = app.readFramebuffer();
    auto reference = loadReferenceImage("references/main_menu.png");

    LPIPSValidator validator("models/lpips_vgg.onnx");
    float lpips = validator.computeSimilarity(pixels, reference);

    if (lpips > 0.15f) {
        std::cerr << "Sanity check FAILED: LPIPS distance " << lpips << " > 0.15\n";
        return false;
    }
    return true;
}
----

# Export to ONNX

dummy_img2 = torch.randn(1, 3, 224, 224)

=== 単純ピクセル比較より LPIPS を選ぶ理由

**従来 MSE は失敗:**

[source]
----
Reference: Cube rotated 0°
Test:      Cube rotated 1° (due to timing differences)
MSE:       High - thousands of pixels differ
Human:     Looks identical
Result:    False positive failure
----

**LPIPS は成功:**

[source]
----
Reference: Cube rotated 0°
Test:      Cube rotated 1°
LPIPS:     0.03 - features are nearly identical (shape, lighting, texture)
Human:     Looks identical
Result:    Pass
----

**両方とも本当のバグを捕捉:**

[source]
----
Reference: Cube with brick texture
Test:      Cube with checkerboard texture (wrong texture loaded)
MSE:       High
LPIPS:     0.45 - features differ significantly (texture pattern changed)
Human:     Obviously wrong
Result:    Correctly fails
----

=== 性能上の考慮

LPIPS はピクセル比較より遅い（20–30ms vs 1ms）が CI には十分速い:

- 単純ピクセルチェック: 1–2ms（黒画面など壊滅的失敗用）
- LPIPS チェック: 20–30ms（知覚的類似度用）
- OCR テキストチェック: 100–200ms（テキスト検証用）
- 完全 ML 検証: 50–100ms（複雑な意味的チェック用）

最適速度のため両方を組み合わせ可能:

[source,cpp]
----
bool testSanityCheckOptimized() {
    HeadlessVulkanApp app(800, 600);
    auto pixels = app.readFramebuffer();
    auto reference = loadReferenceImage("references/main_menu.png");

    // Level 1a: Quick pixel check (< 1ms)
    // Catches black screens or catastrophic corruption instantly
    if (countDrasticDifferences(pixels, reference) > threshold) {
        return false;
    }

    // Level 1b: LPIPS for perceptual similarity (~30ms)
    LPIPSValidator validator("models/lpips_vgg.onnx");
    return validator.computeSimilarity(pixels, reference) < 0.15f;
}
----

1ms で明らかな失敗を捕捉し、30ms で知覚的類似度を検証——合計 31ms、CI には許容範囲。

== レベル 2: UI を実際に読む

LPIPS チェックでシーンが「おおむね正しい」と分かった。しかし「Submit」ボタンが空白？ フォントレンダリングが壊れテキストが tofu（欠落文字の四角）の列だけ？ 全体色・形状が似ていれば LPIPS は捕捉しないかもしれません。

これを解くため**ML ベース光学文字認識（OCR）**を使います。

=== OCR パイプライン: 検出 + 認識

単一「ブラックボックス」ではなく、現代 OCR は 2 段階プロセス。

==== 1. テキスト検出（「どこ」）
**EAST**（Efficient and Accurate Scene Text detector）や **CRAFT** のようなモデルが 1080p フレーム全体を見て、テキストがあると「思う」境界ボックス一覧を出力。まだ内容は不明; 「ブロブ」だけ見つける。

==== 2. テキスト認識（「何」）
境界ボックスをメイン画像からクロップし、**CRNN**（Convolutional Recurrent Neural Network）など第 2 モデルに渡す。

*   **CNN 部分**: 文字の視覚特徴を抽出。
*   **RNN 部分**: 系列を理解（例: 「S」の次「u」、次「b」...）。

=== ボックスの発見
Vulkan アプリケーションでは UI レイアウトコードからボタン*があるべき*位置は通常分かります。しかしテキストが実際にそのボタン*内*にレンダリングされたかを検証するのが目標。検出モデルにレンダー内のすべてのテキスト「ブロブ」を見つけさせます。

[source,cpp]
----
struct TextMatch {
    std::string text;
    Rect bounds;
};

class OCRValidator {
public:
    std::vector<TextMatch> detectAndRead(const std::vector<uint8_t>& pixels) {
        // 1. Detect: Find all bounding boxes containing text
        // (Using an ONNX model like EAST)
        std::vector<Rect> boxes = textDetector_.findBoxes(pixels);
        std::vector<TextMatch> results;

        for (const auto& box : boxes) {
            // 2. Recognize: Convert the pixels in this box to a string
            // (Using an ONNX model like CRNN)
            auto cropped = cropPixels(pixels, box);
            std::string text = textRecognizer_.read(cropped);
            results.push_back({text, box});
        }
        return results;
    }
};
----

=== レイアウト検証（テキストフィット）
重要チェック: テキストは収まるか？ UI 開発の一般的バグは、特に長い言語（ドイツ語）や異なるフォントでテキストがコンテナから「はみ出す」こと。

**検出境界ボックス**（ピクセルが実際に現れた位置）と**期待 UI ボックス**（レイアウトの指定領域）を比較して検証。

[source,cpp]
----
bool validateTextFitting(const TextMatch& detected, const Rect& expectedUIBox) {
    // Check if the detected text is entirely contained within the expected box
    bool isInside = expectedUIBox.contains(detected.bounds);

    // Check for "legibility margin" - text shouldn't be touching the very edge.
    // We expect at least a 2-pixel gap on all sides.
    bool hasMargin = expectedUIBox.width > (detected.bounds.width + 4) &&
                     expectedUIBox.height > (detected.bounds.height + 4);

    return isInside && hasMargin;
}
----

ボタン存在確認より強力。検証内容:

*   **意味的内容**: テキストが「Submit」で「Error」や「???」ではない
*   **レイアウト整合性**: テキストが中央・内包（ボタン端でクリップされない）
*   **判読性**: ML モデルが読めなければユーザーも読めない可能性が高い

自前 ONNX 推論で CI パイプラインを自己完結し ML 原則に集中できます。

== レベル 3: 意味的検証（「視覚」）

最後にパイプラインの最も高度なレベル。ピクセルとテキスト的には「正しい」が、意味的には「間違い」のレンダーがある。

例: シェーダバグで**Z-fighting**——2 面が同じ深度で flicker。ピクセル比較器にはノイズに過ぎない。OCR には無関係。人間には明らかなバグ。

欠落テクスチャも。形状は正しいが真っ赤マゼンタ。

これを捕捉するため、カスタム軽量 CNN を学習。MobileNet のような巨大 1000 クラス分類器ではない。専門家。自アプリケーションの「良い」レンダー数百と「悪い」レンダー数百で学習。特定ゲームやアプリ*がどう見えるべきか*を学ぶ。

=== 専門家の構築
このモデルは*自アプリ*だけ知ればよいので tiny——数レイヤだけ。CI 実行が非常に速い。

[source,python]
----
# Simpler, faster model than before - just needs to detect "broken" vs "correct"
import torch
import torch.nn as nn

class FastRenderValidator(nn.Module):
    def __init__(self):
        super().__init__()

        # Lightweight architecture - only 3 conv layers
        self.features = nn.Sequential(
            nn.Conv2d(3, 32, 5, stride=2, padding=2),  # 800x600 -> 400x300
            nn.ReLU(),
            nn.MaxPool2d(2),  # 400x300 -> 200x150

            nn.Conv2d(32, 64, 3, padding=1),
            nn.ReLU(),
            nn.MaxPool2d(2),  # 200x150 -> 100x75

            nn.Conv2d(64, 64, 3, padding=1),
            nn.ReLU(),
            nn.AdaptiveAvgPool2d((8, 8)),  # 100x75 -> 8x8
        )

        self.classifier = nn.Sequential(
            nn.Flatten(),
            nn.Linear(64 * 8 * 8, 128),
            nn.ReLU(),
            nn.Dropout(0.3),
            nn.Linear(128, 2)  # Binary: correct=0, broken=1
        )

    def forward(self, x):
        x = self.features(x)
        x = self.classifier(x)
        return x
----

このモデルははるかに小さい（500KB vs 50MB）、20–30ms で実行、自アプリケーションのレンダリングバグ検出に特化学習。

=== 生きたデータセットの構築

ML エンジニアリングの一般的誤りは、始める前に「完璧」データセットが必要と考えること。実際、最も堅牢な検証器は反復的に構築。**Bug Harvesting** と呼ぶプロセスを推奨。

何が壊れるか推測するだけでなく、持っているデータで学習。新バグ発見時——人間開発者でも自動テストでも——その特定「Broken」状態を捕捉、学習セットに追加、再学習。時間とともに AI が防ぎたい正確な失敗のライブラリを自然に集める。

==== 段階 1: 「コールドスタート」（Synthetic Induction）
最初は real バグライブラリがないかも。モデルを立ち上げるため**Synthetic Induction** を使う。Vulkan レンダラーをスクリプトで意図的に「悪い」データを生成:

*   **Z-Fighting**: 頂点シェーダで 2 重な平面の `z` 座標を完全同一に強制。
*   **Missing Textures**: 実アセットの代わりに 1×1 真っ赤マゼンタテクスチャをバインド。
*   **Shader Corruption**: `.slang` や `.glsl` シェーダに色をランダム反転したり光寄与をゼロにする小片を注入。
*   **Culling Issues**: 有効であるべき背面カリングを無効化し「透け」ジオメトリを作成。

モデルに「壊れ」の baseline 理解を与える。ジオメトリは技術的に正しいので単純ピクセルテストは見逃すかもしれないが、CNN は Z-fighting の高周波パターンやマゼンタエラーテクスチャの特定 Hue をすぐ学ぶ。

=== 過学習について

CI 検証の一般的罠は **Gold Image への過学習**。1 カメラ角度だけで学習すると、他角度はすべて「壊れ」と学ぶ。

堅牢な検証器を構築するには:

1.  **カメラを変える**: データ収穫時に view-projection 行列をわずかにランダム化。
2.  **照明を変える**: 太陽位置や光強度を変更。
3.  **データ拡張**: 第 2 章で学んだように、学習サンプルにランダムクロップとノイズを適用。

特定スクリーンショットを暗記するモデルではなく、自アプリケーションの「動作レンダー」の**概念**を理解するモデルを学習させる。

=== 学習とエクスポート

# Simple training loop snippet
optimizer = torch.optim.Adam(model.parameters(), lr=0.001)

for epoch in range(10):
    for images, labels in dataloader:
        optimizer.zero_grad()
        outputs = model(images)
        loss = criterion(outputs, labels)
        loss.backward()
        optimizer.step()

エクスポート後、ONNX 推論エンジンで意味的チェックを実行。CI パイプラインでは高速サニティと OCR チェック通過後に呼び出す。

[source,cpp]
----
class SemanticValidator {
public:
    SemanticValidator(const std::string& modelPath) : session_(env_, modelPath.c_str(), options_) {}

    bool isRenderCorrect(const std::vector<uint8_t>& pixels) {
        // 1. Preprocess: resize to 224x224 and normalize
        std::vector<float> input = preprocess(pixels);

        // 2. Run inference
        auto outputTensors = session_.Run(Ort::RunOptions{nullptr}, ...);

        // 3. Interpret results: Argmax of [correct_score, broken_score]
        float* results = outputTensors[0].GetTensorMutableData<float>();
        return results[0] > results[1]; // Index 0 is 'Correct'
    }
};

== まとめ

実践的 CI/CD レンダー検証は 3 つの補完アプローチを使います:

**高速ピクセル比較**: 軽微なドライバ差を許容しミリ秒で壊滅的失敗を捕捉。

**OCR ベーステキスト検証**: 専門 ML モデル（検出 + 認識）で UI テキストを実際に読み、期待内容一致とボックス内フィット（クリップなし）を検証。

**ML 意味的検証**: ピクセル比較が fragile すぎるとき、軽量分類器が特定レンダリング問題を検出。

**ヘッドレス Vulkan 設定**: SwiftShader や LavaPipe ソフトウェアレンダラーで GitHub Actions 動作、GPU 不要。

この組み合わせで堅牢テスト: 高速チェックは fail fast、テキスト検証で判読可能 UI、ML 検証で環境横断の意味的レンダリング問題を捕捉。
