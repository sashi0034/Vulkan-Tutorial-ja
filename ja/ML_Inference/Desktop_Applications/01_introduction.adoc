= デスクトップアプリケーション

== 理論から実践へ

推論エンジンを構築しました。ONNX モデルを読み込み、GPU 上で実行しました。結果が浮動小数点精度の範囲内で PyTorch の出力と一致することを確認しました。素晴らしい——基礎を理解しています。

さあ、実際に使えるものを作りましょう。

本章は教育的演習から実践的なデスクトップアプリケーションへと焦点を移します。リアルタイムで画像を分類する完全なプログラムを構築し、ML 推論をグラフィックスレンダリングと統合し、Vulkan の明示的制御が高レベルフレームワークより実際に有利になる場面を示します。

MNIST の例との違いはここです。推論が動くことを確認するだけでなく、実際のユーザーインタフェースを持つアプリケーションを構築し、任意の入力を扱い、対話的な性能向けに最適化します。目標は畳み込みを理解したことを証明することではなく——グラフィックスパイプラインが結果をレンダリングしながら、ライブカメラ入力で 60 FPS を達成することです。

== 何を構築するか

本章では、Vulkan との実践的な ML 推論統合を示す完全なデスクトップ画像分類アプリケーションを構築します。静的画像分類から始め、リアルタイムカメラ入力へ拡張し、応答性の高い対話アプリケーションの作り方を学びます。

アプリケーションは MobileNetV2 を使い、ImageNet の 1000 カテゴリのいずれかに画像を分類します。完全なワークフローを学びます: 事前学習済みモデルの読み込み、入力の前処理（リサイズ、正規化、色変換）、効率的な推論の実行、出力の後処理（softmax、top-k 選択）、結果の可視化。

さらに重要なのは、性能を犠牲にせず ML 推論を Vulkan グラフィックスアプリケーションに統合する方法です。転送と計算のオーバーラップのための非同期コンピュート、効率的な CPU-GPU 同期、ライブカメラ入力で 30 FPS 以上を維持しながら結果をレンダリングするのに必要なパイプライン設計を扱います。

これは MNIST で確立した教育パターンに従います: 問題から始め、段階的に解決策を進め、必要になったときに概念を説明します。理論を最初に一括投入しません——必要なときに必要なことを教えます。

== 章構成

本セクションは焦点を絞った実践的な章に整理されています。

**xref:ML_Inference/Desktop_Applications/02_image_classification.adoc[画像分類: MNIST から CNN へ]**: 全結合ネットワークが画像で失敗する理由と CNN が問題をどう解決するかを理解します。畳み込み演算、畳み込みレイヤの構築、プーリング、現代的なアーキテクチャパターンを学びます。MNIST の知識を土台に、適切な CNN アーキテクチャで実世界の画像分類を扱います。

**xref:ML_Inference/Desktop_Applications/03_practical_implementation.adoc[CPU ベースライン実装]**: ONNX Runtime を使った動作する画像分類器を構築します。モデルを読み込み、正しく前処理し、推論を実行し、PyTorch と照合して検証します。これが参照実装です。

**xref:ML_Inference/Desktop_Applications/04_ci_render_validation.adoc[CI/CD レンダー検証]**: ML を使って CI で Vulkan レンダーを検証します。レンダリングバグを検出する分類器を学習し、GitHub Actions と統合し、意味的類似度を測定し、UI レイアウトを検証します——エンジニアリング問題向けの実践的 ML です。

**xref:ML_Inference/Desktop_Applications/05_real_time_camera.adoc[リアルタイムカメラ統合]**: 分類器を 30 FPS 以上でウェブカメラ入力を処理するよう拡張します。GPU 加速前処理、非同期コンピュートパイプライン、応答性のためのスレッド、対話アプリケーション向けの性能チューニング。

**xref:ML_Inference/Desktop_Applications/06_rl_automated_exploration.adoc[RL ベースの自動探索]**: 強化学習を使って Vulkan アプリケーションを自動探索・テストします。Random Network Distillation を備えた好奇心駆動エージェント、自動バグ発見、ベンチマーク、CI 以降向けの GUI 自動化。

**xref:ML_Inference/Desktop_Applications/07_ray_tracing_optimization.adoc[レイトレーシング最適化]**: AI を使って高性能レイトレーシングを最適化します。AI 駆動デノイジング、レイ予算を最大化する適応サンプリング、時間的更新が必要な領域を ML で検出する方法を学びます。

**xref:ML_Inference/Desktop_Applications/08_future_directions.adoc[将来の方向性: RAG、GAN など]**: デスクトップアプリケーション向けの高度な改良を探ります。スタイル転送に GAN を、インテリジェントなエンジニアリング診断に RAG を使う方法を学びます。

各章は 1 つの実践トピックに焦点を当て、静的画像からリアルタイム対話アプリケーション、自動テスト、インテリジェント探索へ段階的に構築します。

== 三経路アプローチ

導入で述べた三経路の教え方を覚えていますか？ 本章全体に現れます。

**経路 1 - ONNX Runtime**（実務的経路）: ニューラルネットワーク推論には ONNX Runtime を使い、前処理と可視化は Vulkan で扱います。本番で実際に行うことです。ONNX Runtime は数百の演算子の実戦検証済み実装、自動最適化、広いハードウェアサポートを提供します。Vulkan コードは得意分野——画像処理とグラフィックス——に集中できます。

**経路 2 - カスタム Vulkan**（教育的経路）: ニューラルネットワークレイヤを含め、パイプライン全体を Vulkan コンピュートシェーダで実装します。この経路は内部で何が起きているかを理解し、ONNX Runtime が自動的に処理する複雑さへの理解を深めます。カスタム演算を実装する場合やレンダリングパイプラインと密に統合する場合に使う技法も示します。

**経路 3 - ML コンパイラ**（コンパイラ駆動経路）: ML コンパイラ（IREE、Apache TVM、OpenXLA、MAX/Mojo）を使い、モデルを最適化カーネルに lower し、最小ランタイムで、しばしば SPIR‑V 経由で Vulkan をターゲットにします。汎用ライブラリより移植性が高くランタイムフットプリントが小さい一方、手書きに近い性能を、シェーダを書かずに得られます。

実務では多くのアプリケーションがハイブリッドアプローチを使います。メインネットワークに ONNX Runtime、性能とフットプリントのバランスが良いモデルは ML コンパイラでコンパイル、密統合のためのカスタム前後処理は Vulkan で実装、という組み合わせもあります。3 経路すべてを理解すると、情報に基づいたアーキテクチャ判断の柔軟性が得られます。

まだインストールしていなければ ONNX Runtime も必要です。「サードパーティライブラリ」章でインストールを扱いますが、Linux 向けの簡易版は次のとおりです。

== 性能期待について

現実的な期待を設定しましょう。「リアルタイム」性能と言うとき、アプリケーションによって意味が異なります。

**画像分類**: 224×224 の単一画像の処理は、中級 GPU で 20ms 未満であるべきです。オンデマンドで画像を分類する対話アプリケーションには十分速いです。

**カメラ分類**: 30 FPS 以上を目標にします。フレーム予算は 33ms です。適切なパイプライン化があれば、現代ハードウェアでは 60 FPS（16ms 予算）も達成可能です。

**バッチ処理**: オフラインで数千枚を分類する場合、レイテンシよりスループットが重要です。適切な GPU 利用のバッチ推論で、1 秒に数百枚を処理できます。

これらの数値は、最適化モデル（MobileNetV2 など）と妥当な入力サイズを前提とします。統合グラフィックスで ResNet-152 を 1024×1024 で走らせようとすれば失望するでしょう。ML 推論のデプロイの一部は、ハードウェアと性能要件に合うモデルを選ぶことです。

各アプリケーション構築時に実際の性能数値を測定し、期待に届かないときは最適化戦略を議論します。ただし覚えておいてください: 目標は概念を示す動作アプリケーションの構築であり、最先端ベンチマーク数値の達成ではありません。パイプラインを理解すれば、最適化は謎ではなくエンジニアリング演習になります。
