= 高度なトピック: 本番への道
:pp: ++
:stem: latexmath

推論エンジンを構築し、レイヤごとに正しさを検証し、最初のプロファイルと最適化のラウンドさえ適用しました。Vulkan ML の「やり方」を理解しています。次は **「プロのやり方」** を習得する番です。

本セクションでは、概念実証デモを超え、**本番グレードのエンジニアリング** の世界に入ります。ここが「チュートリアル」と「製品」を分ける場所です。ハイエンド RTX 4090 からバッテリ制約のあるモバイルまで、現代の ML アプリケーションを動かす洗練された技法を見ていきます。

== 成熟の弧: デモからデプロイへ

学習中の目標は **正しさ** です。数値が合ってほしい。デプロイ時の目標は **制約下の効率** です。

プロの Vulkan 開発者として、常に **最適化トレードオフ空間** を航行します。もはや「最良」の実装ではなく、要件への「最良の適合」を探します。これには根本的な心構えの転換が必要です: 「コーダー」から「アーキテクト」へ。

=== プロモデルのライフサイクル
本番環境では、モデルは 3 つの異なる「人生」を経ます。

1.  **研究の人生（PyTorch/TensorFlow）**: 精度、学習速度、データ多様性に焦点。モデルは「たるんで」おり、デバッグノードと未最適化レイヤで満ちている。
2.  **変換の人生（ONNX／最適化）**: 本章の技法を適用する場所。量子化し、刈り込み、融合する。「研究」グラフを「本番」グラフに書き換える。
3.  **推論の人生（Vulkan／ランタイム）**: モデルは今や引き締まった、ベンダーを意識したエンジン。エンドユーザーのデバイスで動き、バッテリ寿命とレイテンシだけが重要な指標。

=== トレードオフの 3 次元
1.  **精度対速度（精度トレードオフ）**:
    医療アプリケーション（例: 腫瘍検出）では、0.1% の精度低下は許容できません。おそらく FP32 を使い、レイテンシを受け入れます。一方、ゲームキャラクターの AI やリアルタイムカメラフィルタでは、分類の信頼度が 95% ではなく 92% でもユーザーは気づきませんが、フレームレートが 60 から 10 FPS に落ちれば *気づきます*。このような場合、量子化による latexmath:[10\times] の高速化のために 5% の精度低下を受け入れるかもしれません。

2.  **メモリ対レイテンシ（帯域トレードオフ）**:
    大規模言語モデル（LLM）や高解像度生成モデルは、しばしば利用可能な VRAM を超えます。選択に直面します: すべてを VRAM に置くか（低レイテンシだがモデルサイズを制限）、オンデマンドでシステム RAM から重みを引く **重みストリーミング** を実装するか（メモリを節約するが、latexmath:[50\text{ms}+] の PCIe バスレイテンシを追加）？

3.  **汎用性対特化（移植性トレードオフ）**:
    2018 以降のすべての GPU で動く単一のきれいな Vulkan シェーダを書くか（高い保守性）、NVIDIA Tensor Core、AMD WMMA、Intel XMX、ARM Dot-Product 命令向けに調整した 4 本の別々の「高速経路」を保守するか（高い性能）？

== 非効率という静かな失敗

従来のソフトウェアでは、「バグ」はクラッシュです。高性能 ML では、「バグ」は **非効率** です。

*   シェーダがメモリバウンドなのに **共有メモリタイリング** を使っていなければ、GPU の潜在能力の 80% を「漏らして」いる。
*   専用 INT8 ハードウェアを持つチップでモデルが FP32 で動いていれば、電力予算の 75% を浪費している。
*   統合メモリ SoC で毎フレーム CPU から GPU へデータをコピーしていれば、理由なくバッテリを燃やしている。

これらは単なる「逃した最適化」ではありません——製品が市場で生き残るか、「重い」としてアンインストールされるかを決めるアーキテクチャ上の失敗です。

== ML のパレートフロンティア

高度なエンジニアリングでは、しばしば **パレートフロンティア** の話をします。ある指標（速度など）を、別の指標（精度やメモリなど）を悪化させずに改善できない数学的限界です。

X 軸が「推論時間」、Y 軸が「精度」のグラフを想像してください。素朴な実装はフロンティアから遠く——遅く、中程度の精度です。最適化は実装を端へ「押し」ます。*フロンティア上* に乗ったら、さらなる高速化は *必ず* 精度の代償を伴います（例: さらに量子化する）。

これを **多目的コスト関数** latexmath:[J] で形式化します。

[latexmath]
++++
J = \alpha \cdot \text{Error} + \beta \cdot \text{Latency} + \gamma \cdot \text{Memory Consumption} + \delta \cdot \text{Power Draw}
++++

高度な ML エンジニアとしての仕事は、製品のミッションを満たす重み latexmath:[\{\alpha, \beta, \gamma, \delta\}] を見つけることです。ドローン技術者は latexmath:[\delta]（電力）を最も気にし、クラウド AI プロバイダはスループット（latexmath:[1/\beta]）を最も気にします。

== 「メモリウォール」と「コンピュートルーフ」

モデルが複雑になるにつれ、単純な「コーディング」では直せない 2 つの物理的障壁にぶつかります。

1.  **メモリウォール**: GPU は毎秒兆回の数学を実行できますが、メモリからの読み取りは毎秒十億回程度です。現代の多くのレイヤ（LayerNorm や Depthwise Convolution など）は、時間の 90% を VRAM からのデータ到着待ちに費やします。これを破るには **カーネル融合** と **レジスタタイリング** を学ぶ必要があります。
2.  **コンピュートルーフ**: モデルが巨大（トランスフォーマーなど）なら、最終的に GPU の算術ユニットを飽和させます。この時点で「より良いコード」は効きません——「より良い数学」が必要です。ここで **Winograd 変換** のような技法が入り、巧妙な代数で畳み込みに必要な乗算回数を最大 latexmath:[2.25\times] 減らします。

== 扱う内容: 高度なツールキット

本セクションでは、それらの境界を押し、壁を破る道具を提供します。

*   **量子化戦略**: 32 ビット浮動小数から 8 ビット（さらには 4 ビット）整数へ。**学習後量子化（PTQ）** と **量子化Aware学習（QAT）** の数学、および GLSL での固定小数点演算の実装を見ます。
*   **ベンダー固有最適化**: `VK_KHR_cooperative_matrix` 拡張を使い、**NVIDIA Tensor Core**、**AMD Matrix Core（WMMA）**、**Intel XMX** ユニットから最後の一滴の性能を絞る。
*   **高度なメモリ管理**: 物理 VRAM の 2 倍の大きさのモデルを動かせる **リソースエイリアシング** と **非同期重みストリーミング** などの技法。
*   **モデル変換（グラフ最適化）**: コード最適化を超え **グラフ書き換え** へ。演算子融合、構造化刈り込み（「死んだ」ニューロンの除去）、数学的 **Winograd 変換** を学びます。

== 本番の哲学: シェーダを超えて

プロの推論エンジンは、速いシェーダの集まりだけではありません。現実世界の「野生」を生き延びる堅牢なシステムです。本番グレードのエンジンは次でなければなりません。

1.  **堅牢**: GPU の「Lost Device」エラーを優雅に扱う。推論が中断されても「ゼロリーク」保証でメモリプールを管理する。
2.  **決定的**: ユーザーのドライバ版や OS にかかわらず、実行するたびにまったく同じ結果を出す。安全重要アプリケーションに不可欠。
3.  **ハードウェア適応的**: 実行時にハードウェアを検出する。古い GPU で単に「失敗」するのではなく、遅いが機能する「フォールバック経路」を選び、現代ハードウェアでは自動的に「超高速」経路を有効にする。

== 達成すること

本セクションの終わりまでに、「チュートリアル向けに速い」だけでなく **App Store 向けに速い** 推論エンジンを構築するスキルを持てます。任意の ONNX モデルを、高度に最適化され、ベンダーを意識し、メモリ効率の良い Vulkan パイプラインに変換できるようになります。

もはやシェーダを書いているだけではありません。現代 AI インフラの基盤を構築しています。
