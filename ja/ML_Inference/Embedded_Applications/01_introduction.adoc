= 組み込みアプリケーション: エッジでの AI
:pp: {plus}{plus}
:stem: latexmath

高性能な推論エンジンを構築しました。量子化、ベンダー固有の行列ハードウェア、メモリエイリアシングアロケータを習得しました。開発ワークステーションではモデルが 500 FPS で動き、GPU ファンはほとんど回りません。しかし今、ミッションが変わります。同じモデルを **ドローン**、**VR ヘッドセット**、工場の天井に取り付けられた **産業用スマートカメラ** へデプロイする任務が与えられました。

**組み込みシステム** の世界へようこそ。

本セクションでは、「無限のリソース」の世界から **有限の制約** の世界へ移ります。デスクトップではシェーダが遅ければユーザーがより良い GPU を買うか設定を下げます。組み込みでは「より良い GPU」はありません——基板に半田付けされたシリコンのみです。コードが非効率なら、デバイスは過熱し、バッテリは尽き、ロボットは物理的にクラッシュします。

これが Vulkan ML 旅の最終フロンティアです。「リーンで、強く、決定的」である方法を学びます。

== エッジの容赦ない現実

組み込み開発は「小さなコンピュータ向けのコーディング」だけではありません。エンジニアリング優先度の根本的転換です。サーバー環境では **スループット**（秒あたりリクエスト数）を最適化します。モバイル環境では **レイテンシ**（ユーザーが応答を見るまでの速さ）を最適化します。ミッションクリティカルな組み込み環境では **レジリエンス** と **リソース効率** を最適化します。

プロとして、エッジ製品の成否を決める 4 つの相互連鎖する柱を航行する必要があります。

=== 1. 電力: エネルギー予算

デスクトップでは成功を **フレームレート（FPS）** で測ります。バッテリ駆動のドローンやウェアラブル医療機器では **推論あたりのジュール** で測ります。トランジスタの反転ごとにエネルギーがかかります。カメラと GPU の間の冗長なメモリコピーごとに、飛行時間のミリ秒が失われます。

[latexmath]
++++
\text{Intelligence Density} = \frac{\text{Successful Inferences}}{\text{Total Energy Consumed (Joules)}}
++++

*（注: SI 標準単位ではありませんが、Intelligence Density はエッジ AI エンジニアが異なるモデルとハードウェア経路の効率を比較するための重要な概念指標です。）*

Vulkan ML エンジンが 15W を消費すれば、ドローンは 10 分で墜落します。本セクションの技法で 3W に最適化できれば、1 時間のミッションが可能です。電力最適化は「節約」だけではありません——**不可能なミッション** を可能にするものです。

考慮すべき点:
* **静的電力（リーク）**: デバイスをオンにしているだけで消費されるエネルギー。
* **動的電力**: 演算中のトランジスタ切り替えで消費されるエネルギー。Vulkan では冗長な状態変更を減らし、シェーダ命令を最適化することで最小化できます。
* **「Race to Sleep」戦略**: 作業を可能な限り速く完了させ、GPU を低電力状態に入れる。ゆっくりと作業を垂れ流し、レールを「ホット」に保つのではなく。

=== 2. サーマル: シリコンの上限

組み込みデバイスはファンなしの防水 IP67 ケースに密封されることが多いです。受動冷却に依存します。ML モデルが GPU を 100% デューティサイクルで動かすと、SoC 温度は **サーマルトリップポイント**（SoC により通常 85°C〜105°C）に達するまで上昇します。その瞬間、ハードウェアは「ハードスロットル」——警告なしにクロック速度を 50% 以上下げます。

リアルタイム VR アプリケーションでは、このスロットルは 90 FPS から 15 FPS への突然の急落のように感じられ、即座に酔いを引き起こします。**サーマルAware推論** を実装する必要があります。これには次が含まれます:
* **動的解像度スケーリング（DRS）**: デバイスが熱いとき ML モデルの入力サイズを縮小。
* **モデル切り替え**: サーマルヘッドルームが低いとき、重い高精度モデルを「ライト」版に差し替え。
* **デューティサイクル管理**: 推論の間に意図的に「休息」期間を挿入し、熱を散逸させる。

=== 3. 決定性: 99.9 パーセンタイル

自動運転車やロボティクスプラットフォームでは、「平均レイテンシ」は危険な指標です。平均 10ms でも、OS がファイルシステムのインデックスを走らせたり、ドライバが「ストップ・ザ・ワールド」GC をしたりして 1 分に 1 回 200ms かかれば、システムは根本的に安全ではありません。

**テールレイテンシ**（99.9 パーセンタイル）を最適化します。Vulkan では **明示的同期** を習得します。ドライバが正しいことをしてくれることを期待せず、次を使います:
* **フェンスとセマフォ**: GPU の開始・完了を CPU との関係で正確に制御。
* **タイムラインセマフォ**: パイプラインの「バブル」を避ける高度なマルチキュー同期。
* **リアルタイムスケジューリング**: ML スレッドがバックグラウンドタスクにプリエンプトされないよう、Linux link:https://man7.org/linux/man-pages/man7/sched.7.html[`SCHED_FIFO` または `SCHED_RR`] と連携。

=== 4. 統合メモリ: UMA の利点

PCIe バス越しの独立 VRAM を持つデスクトップとは異なり、多くの組み込み SoC（ARM Mali、Qualcomm Adreno、Apple Silicon、NVIDIA Jetson）は **統合メモリアーキテクチャ（UMA）** を使用します。CPU と GPU は同じ物理 RAM を共有します。

これが最大の競争優位です。`dmabuf` でメモリを正しく管理すれば、カメラセンサーから ML エンジンへ **ゼロコピー** でデータを移せます。多くの開発者は冗長転送で性能の 30% を浪費します。完全にバイパスする方法を学びます。

[cols="1,1"]
|===
| 独立 GPU（デスクトップ） | 統合メモリ（組み込み）

| CPU メモリ latexmath:[\to] PCIe latexmath:[\to] GPU VRAM | 共有 RAM（CPU と GPU が同じ物理ページにアクセス）
| 高レイテンシ転送 | ゼロレイテンシ（ポインタ受け渡し）
| 電力を食う PCIe バス | エネルギー効率の良い内部インターコネクト
| 別々のメモリプール | 共有プール（慎重な管理が必要）
|===

== 組み込みソフトウェアエコシステム

組み込み向けビルドには Vulkan 以上の理解が必要です。「垂直スタック」のソフトウェアと向き合います。

1.  **シリコン（ハードウェア）**: 物理 ARM コア、GPU クラスタ、専用 NPU（Neural Processing Unit）。
2.  **カーネル（Linux/Android/QNX）**: メモリ、スケジューリング、デバイスドライバを管理。link:https://docs.kernel.org/driver-api/dma-buf.html[`dmabuf`] と link:https://docs.kernel.org/userspace-api/media/v4l/v4l2.html[V4L2] がここに存在。
3.  **ドライバ（Vulkan 実装）**: SoC ベンダー（ARM、Qualcomm、NVIDIA、Broadcom）提供。多くはクローズドソースで固有の癖あり。
4.  **ローダー（`libvulkan.so`）**: アプリケーションとドライバの橋。
5.  **あなたのアプリケーション**: 本書を通じて構築してきた ML エンジン。

本セクションではスタックの「地下室」——カーネルとドライバと対話し、ハードウェアから最後の一滴を絞り出す——に多くの時間を費やします。

== ハードウェアアーキタイプ

すべての組み込みシステムが同じではありません。通常 3 つの性能「ティア」に分類します:

*   **ティア 1: シングルボードコンピュータ（SBC）**: **Raspberry Pi 5** など。汎用 Linux マシン。そこそこの GPU はあるが専用 AI アクセラレータはない。Vulkan がこれらのデバイスで高性能 ML を得る *唯一* の方法。
*   **ティア 2: エッジ AI モジュール**: **NVIDIA Jetson Orin** など。巨大な GPU パワーと専用 Tensor Core。Vulkan は CUDA と競合するが、ベンダー横断の移植性を提供。OpenCL と SyCL も選択肢だが、Jetson プロジェクトでよく望まれるグラフィックス能力を標準装備していない。
*   **ティア 3: モバイル/ウェアラブル SoC**: **Qualcomm Snapdragon** や **MediaTek Dimensity** など。何よりも電力最適化。厳しいサーマル制限と多くの拡張を持つ複雑な Vulkan ドライバが多い。

== なぜ Vulkan がエッジの秘密兵器か

TensorFlow Lite や ONNX Runtime のような高レベルフレームワークは迅速なプロトタイピングに優れますが、GPU を「ブラックボックス」として扱うことが多いです。Vulkan を直接使うと、高レベルユーザーが存在すら知らない「ハードウェア超能力」を得られます:

*   **直接 DMA アクセス**: 専用 Linux カーネルハンドルで生カメラセンサーをコンピュートシェーダに直接供給。
*   **明示的ワークグループチューニング**: 特定チップの **実行ユニット（EU）** や **コンピュートユニット（CU）** 数にぴったり合わせてシェーダワークグループを調整。
*   **非同期パイプライン**: ML の「思考」時間をグラフィックスパイプラインの「表示」時間内に隠し、AI コストをフレーム予算 0 ミリ秒に。
*   **マルチキュー実行**: 「グラフィックスキュー」が UI を処理している間、専用「コンピュートキュー」で ML を実行し、重い推論中の UI ラグを防止。

== セクション構成: ハードニングへの道

本セクションは、デスクトップ開発者から本番対応のエッジ AI エンジニアへ導きます。章は実製品構築の論理フローに沿って構成されています:

1.  **xref:ML_Inference/Embedded_Applications/02_cross_compilation.adoc[クロスコンパイル: 鍛冶場]**: マイクロコンピュータでは複雑なエンジンをコンパイルできません。**Sysroot**、**Docker**、**CMake Toolchain** で液体冷却ワークステーションから ARM ターゲット向けにビルドする方法を学びます。
2.  **xref:ML_Inference/Embedded_Applications/03_camera_integration.adoc[カメラ統合: ゼロコピーパイプライン]**: ビジョンの「聖杯」。**V4L2** と **dmabuf** を習得し、CPU メモリコピー 1 回もなしでピクセルを Vulkan に供給します。
3.  **xref:ML_Inference/Embedded_Applications/04_power_optimization.adoc[電力とサーマル最適化]**: 物理限界の管理。**DVFS**、**Race to Sleep** 戦略、温度管理のソフトウェア **PID コントローラ** を探ります。
4.  **xref:ML_Inference/Embedded_Applications/05_complete_example.adoc[ケーススタディ: スマート野生動物カメラ]**: すべてを統合。フィールドで数日間生き残り、電力と信頼性を自ら管理するハードニングされた自己修復システムを構築します。
5.  **xref:ML_Inference/Embedded_Applications/06_scene_understanding_openxr.adoc[シーン理解: XR での ML]**: 組み込み ML のボーナス章。**OpenXR** で空間センサーを統合し、90 FPS で 3D 空間の床と壁を見つける方法を学びます。
