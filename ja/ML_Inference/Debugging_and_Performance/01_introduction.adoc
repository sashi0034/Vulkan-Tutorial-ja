= デバッグと性能: 検証と速度の技法
:pp: ++
:stem: latexmath

ML 推論アプリケーションを構築しました。ディスクリプタセットを整え、コンピュートシェーダを書き、Vulkan リソース管理の複雑な舞を扱いました。コンパイルでき、クラッシュせずに動き、テンソルの数値さえ出します。しかし百万ドルの問いはこれです: **本当に正しく動いているのか？**

従来のグラフィックスでは、バグがあれば通常はっきりします。画面がマゼンタにちらつく、影が地面から離れる、キャラクターがギザギザの三角形の塊になる、などです。機械学習では、バグはしばしば **静か** です。正規化定数の設定ミスやわずかに誤ったメモリバリアはアプリをクラッシュさせず、「ほぼ正しい」が数学的には誤った出力を出すだけです。

自動運転システムを構築していると想像してください。推論エンジンにわずか 1% の数値ドリフトがあれば、道を完全に見失うわけではないかもしれませんが、「停止」標識を「速度制限」標識と誤認するかもしれません。それが賭けの大きさです。

本セクションはクラッシュを見つけることだけではありません——**真実** を見つけることです。「動いているように見える」から「地上真実に対して小数点以下まで検証した」へ移ります。正しさが取れたら、エンジニアリングの第二の柱である **性能** に目を向けます。現実世界では、2 FPS で動く完璧なモデルは、60 FPS で動く壊れたモデルと同じくらい役に立たないからです。

== 「静かな失敗」問題

ML 向け Vulkan コンピュートは綱渡りです。明示的 GPU 制御の力をすべて持ちますが、責任もすべて負います。TensorFlow や PyTorch のような高レベルフレームワーク——同期とメモリレイアウトの「配管」を代行してくれる——と違い、Vulkan はすべてのバイトの主人であることを求めます。

深いニューラルネットワークでは、誤差は **累積** します。最初の畳み込みレイヤにわずかな精度誤差があれば、その誤差は次の 50 レイヤで乗算・変換されます。Softmax 出力に到達するころには、そのわずかな 0.001 のずれが、信頼度 99% と 51% の差になり得ます。

=== 静かなバグの解剖

Vulkan ML で最もよくある「静かな殺し屋」には次があります。

*   **同期の隙間**: ReLU シェーダが、GPU が書き込みをフラッシュし終わる前に畳み込みの出力バッファから読みます。速いデスクトップ GPU では 99% 動くかもしれませんが、キャッシュ挙動が異なるモバイル SoC ではゴミを出します。
*   **数値ドリフト**: 適切なスケーリングなしに高ダイナミックレンジデータに `float16` を使う。活性化が半精度形式の限界を超えると、「NaN」（Not a Number）がネットワーク全体に伝播し、最終的に出力全体をゼロにします。
*   **レイアウト不一致**: シェーダは平面（NCHW）テンソルを期待するのに、前処理はまだインターリーブ（HWC）データを出している。ネットワークはスクランブルされた混乱を「見て」も分類しようとし、ランダムで低信頼度の予測になります。
*   **量子化ノイズ**: 適切なキャリブレーションなしに 32 ビット重みを 8 ビット整数へ変換する。浮動小数範囲と整数範囲の厳密な写像がなければ、ネットワークは存在しないパターンを「幻覚」します。

== 専門家の心構え: 検証、プロファイル、最適化

これらの問題を解くため、プロのエンジン（TensorRT や ONNX Runtime など）が構築されるのと同じ三段階のエンジニアリング規律を採用します。

=== 1. 正しさの検証（「地上真実」フェーズ）
Vulkan の結果を **ゴールデン参照**——通常は CPU 上の PyTorch または ONNX Runtime——と比較します。最終出力だけを見るのではなく、**すべてのレイヤ** を確認します。このバグの「二分探索」により、数学的真実からずれている正確なコンピュートシェーダを切り分けられます。

=== 2. ボトルネックのプロファイル（「探偵」フェーズ）
コードが正しいと分かったら問います: 「なぜ遅いのか？」 **RenderDoc**、**NVIDIA Nsight**、**Radeon GPU Profiler** などのツールで内部を見ます。カーネルの **演算強度（Arithmetic Intensity）** を測り、数学の速度（コンピュートバウンド）に縛られているか、メモリバスの速度（メモリバウンド）に縛られているかを見ます。

=== 3. クリティカルパスの最適化（「フォーミュラ 1」フェーズ）
ボトルネックを知り、外科的最適化を適用します。メモリバウンドなら **共有メモリタイリング** を使います。レイテンシバウンドなら **カーネル融合** で VRAM 往復をなくします。与えられたアーキテクチャで物理的に可能な最大性能である **ルーフライン限界** に到達することを目指します。

== 非効率のコスト

組み込みやモバイル環境では、性能は「なめらかさ」だけではありません——**バッテリ寿命** と **熱** です。

[latexmath]
++++
\text{Energy} = \text{Power} \times \text{Time}
++++

50ms のあいだ GPU 使用率 100% で動く未最適化の Vulkan シェーダは、10ms で終わり GPU をスリープ状態に入れる融合・最適化シェーダよりバッテリを 5 倍速く消費します。本セクションでは、エネルギー効率の良い開発者になる方法を教えます。

== 習得すること

本章の終わりまでに、「Vulkan を知っている」だけでなく、性能エンジニアになります。次を学びます。

*   **数値検証**: latexmath:[MAE]（平均絶対誤差）と latexmath:[MSE]（平均二乗誤差）を使い、実装が正確であることを数学的に証明する方法。
*   **Vulkan デバッグ**: RenderDoc を使い「時間を止め」、GPU バッファ内の生ビットを検査する。
*   **GPU アーキテクチャ**: **Warp**、**Wavefront**、**メモリ階層** の隠れた世界を理解する。
*   **最適化レシピ**: 共有メモリタイリングと NC/4HW4 ベクトル化レイアウトの実務的でコピー可能なパターン。
*   **非同期コンピュート**: 複数の Vulkan キューを使い、グラフィックスレンダリングパスの「内側」に ML 推論を隠す方法。

デバッグと最適化は「追加」のステップではなく、開発プロセスの中核です。コードがコンパイルできたときではなく、結果が正しく性能が最適になったときに終わります。
