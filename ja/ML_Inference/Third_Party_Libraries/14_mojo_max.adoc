= Mojo / MAX: Modular によるコンパイラ駆動推論

== 概要

Modular の Mojo（プログラミング言語）と MAX（推論エンジン）は、Python のような体験とコンパイラレベルの性能を開発者に提供することに焦点を当てています。ワークフローは、一般的なモデル形式のインポート、グラフとカーネルの最適化、コンパクトで本番向けの成果物のデプロイを強調します。スタックは急速に進化するため、本章は固定仕様ではなく実務的な指針として扱ってください。ここでの目標は、最初の動作するビルドを実行し適合性を評価できるよう、メンタルモデルと統合の形を教えることです。

== 必要なツール

まず Python で検証する計画を立ててください。エクスポート用の学習フレームワーク、ONNX に標準化するなら ONNX ツールチェーン、プラットフォーム向けの MAX/Mojo SDK を用意します。正確なインストーラと CLI は頻繁に変わるので、OS に合ったバージョンを得るには Modular の最新ドキュメントに従ってください。コンパイラや精度を変えるたびに正しさを定量できるよう、CPU のゴールデン参照（例: ONNX Runtime）を手元に置いてください。

== IREE/TVM との位置づけ

今日 Vulkan AOT を高度に制御できるオープンコンパイラが必要なら、xref:ML_Inference/Third_Party_Libraries/12_iree.adoc[IREE] と xref:ML_Inference/Third_Party_Libraries/13_apache_tvm.adoc[TVM] が Vulkan 向けとして最も成熟しています。MAX/Mojo は開発者体験と統合された体験を優先します。プラットフォーム要件を満たすなら、強い性能を伴う非常に快適なワークフローが得られます。

バックエンド要件は明示してください。MAX は一般に CPU とベンダー GPU バックエンド（例: CUDA/Metal）を強調します。Vulkan の可用性はプラットフォーム依存で、一部の環境では間接的（SPIR‑V/WebGPU 経路経由）に届く場合があります。ターゲット全体で Vulkan が必須要件なら、デバイス上で現行サポートを確認してください。そうでなければ、本チュートリアルの Vulkan 優先経路では IREE/TVM を推奨します。

== コマンドより先に考え方

MAX はモデルを一度コンパイルし、パッケージ化された成果物と、アプリから呼び出す小さなランタイムを与えます。前処理／後処理は Vulkan 側に置き、本チュートリアルの他の経路とコードを共有できるようにしてください。MAX をライブラリ境界として扱い、テンソルを入れてテンソルを出します。

== 実務ワークフロー（高レベル）

変換／インポート、コンパイル／パッケージ、実行／デプロイの順です。多くのチームは ONNX を引き渡しとして標準化し、PyTorch→ONNX または TensorFlow→SavedModel/ONNX を使います。ターゲット向けにコンパイルし、ランタイムと成果物を同梱します。アプリではパッケージ化されたモデルをライブラリのように呼び出します。Vulkan の前処理／後処理は外に置き、コンパイラを変えても残りのコードを同じに保ちます。

固定形状のモデル 1 つと、再現可能な入力テンソル 1 つでプロセスを固定してください。CPU のゴールデン参照を一度実行し、期待出力を保存します。コンパイラオプションやバックエンドを変えるたびに、単純な誤差指標（MAE/RMSE）で同じ参照と比較します。これで統合が当て推量ではなく制御された実験になります。

CLI と API の詳細は時間とともに変わるため、正確な呼び出しは現行ドキュメントを参照してください。重要なアーキテクチャの考え方は本チュートリアルの他の部分と一貫しています: モデル／設定ごとに一度コンパイルし、小さなランタイムを同梱し、明確さと性能のために Vulkan の前処理／後処理は分離します。

== ターゲットバックエンドと Vulkan

バックエンドの対応範囲はリリースに応じて変わります。デスクトップでは強力な CPU とベンダー GPU バックエンド（CUDA/Metal）が期待できます。Vulkan サポートはプラットフォームとリリースにより間接的（SPIR‑V または WebGPU 経路）に届く場合があるので、ターゲットハードウェアで確認してください。モバイル／組み込みでは、サポートされるバックエンドを再確認し、フットプリントにも注意してください。

Vulkan の可用性が必須で今すぐ出荷する必要があるなら、IREE/TVM を優先してください。MAX のサポートするバックエンドがハードウェアを満たすなら、統合ツールチェーンを重視するチームには最も快適な経路になり得ます。

== 統合パターン

コンパイラの選択にかかわらず、Vulkan アプリとの統合パターンは同じです。

1) Vulkan で前処理: CPU オーバーヘッドを避けるため、リサイズ／正規化してテンソルバッファへ。
2) エンジンランタイム経由でコンパイル済みモデルを呼び出す。
3) Vulkan/CPU で後処理して提示する。

これによりコピーを最小限にし、非同期コンピュートで作業を重ねられます。

== 正しさの検証（飛ばさないこと）

コンパイル済み経路の隣に参照経路（例: CPU 上の ONNX Runtime）を置いてください。同一入力を与え、誤差指標を計算し、結果がずれたらレイヤごとに二分探索します。ビルドごとに走らせる小さなスクリプトに固定してください。ドライバ更新や精度変更で静かな回帰が出る最初の場面で元が取れます。

== Android と組み込みの注意

ランタイムを ABI 向けにクロスコンパイルし、パッケージ化されたモデルをアセットと一緒に同梱し、デバイス上で測定してください。熱のためのデューティサイクル制御が必要なら、組み込み章と同じ枠組みを適用します: 推論を断続的に実行し、視覚的連続性が重要なときは Vulkan で結果間を補間します。

== トラブルシューティング

モデルのインポートに失敗したら、エクスポート時にグラフを単純化し、形状を確認し、別の交換形式（ONNX 対 SavedModel）を試してください。ランタイム出力がおかしく見える場合は、ゴールデン参照（CPU）と比較し、性能を追う前に最初にずれたレイヤを直してください。
