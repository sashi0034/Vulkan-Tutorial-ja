= サードパーティ ML ライブラリの統合: はじめに

== 導入

率直に言いましょう: ほとんどのアプリケーションでは、ML 推論エンジンをゼロから自作すべきではありません。前章ではニューラルネットワークの仕組み、推論の意味、GPU 実行への写し方といった基礎を学びました。その知識は価値があります——情報に基づいた判断、問題のデバッグ、性能の最適化に役立ちます。しかし本番環境で ML モデルを実際にデプロイするとき、賢明な選択は通常、既存の実戦で鍛えられたライブラリを活用することです。

Vulkan で ML 推論を扱うチュートリアルなのに、なぜサードパーティライブラリに時間を使うのか——矛盾しているように見えるかもしれません。シリーズの残りは独自推論エンジンの構築を教えるのに。答えは、両方のアプローチを理解し、いつどちらを使うかを知ることが、効果的なエンジニアになる鍵だからです。

本章では、サードパーティ ML 推論ライブラリを Vulkan アプリケーションと統合する方法を探ります。ML フレームワークと Vulkan レンダリングパイプラインの橋渡し、GPU リソースの効率的な共有、推論とレンダリングの同期、用途に合ったライブラリの選び方を学びます。

== 3 つの経路という考え方

本チュートリアルは独自の教育アプローチを取ります: シリーズ全体で可能な限り、ML 推論目標を達成する *3 つの方法* を提示します。

*経路 1: サードパーティライブラリ* — ほとんどの本番アプリケーションに推奨されるアプローチです。成熟した実戦向け ML フレームワークを Vulkan アプリケーションと統合する方法を学びます。

*経路 2: カスタム Vulkan 実装* — Vulkan コンピュートシェーダで独自推論エンジンを構築します。GPU レベルで ML 推論が実際にどう動くかを学ぶ経路です。本チュートリアルでは主に学術的演習ですが、高度な最適化とデバッグの理解には不可欠です。AI はコンピュータサイエンスの非常に専門的な分野であり、本チュートリアルが教えられるのは基礎に限られます。シェーダ開発で完全な推論が必要な場合、 decent な出発点になるよう努めましたが、AI の包括的ガイドだと思わないでください。

*経路 3: ML コンパイラ* — ML コンパイラでモデルを最適化カーネルと最小ランタイム（多くは SPIR‑V 経由で Vulkan をターゲット）に下げます。汎用ランタイムを超える性能と移植性を、Vulkan シェーダを手書きせずに得られます。

なぜ 3 つすべてを教えるのか。内部の核心概念に慣れた人ほど、本番でライブラリとコンパイラを効果的に使えるからです。畳み込みが GPU 操作にどう写るか、メモリがどう管理され、同期がどこで重要かを理解すると、デバッグ、最適化、意思決定、アーキテクチャ設計が向上します。

目標は単一の経路を選ばせることではなく——3 つすべての核心理解を与え、状況ごとに適切なツールを選び、その理解を効果的に適用できるようにすることです。

== なぜサードパーティライブラリを使うのか

本番 ML ライブラリ（TensorFlow Lite、ONNX Runtime、PyTorch Mobile、DirectML）は説得力のある利点を提供します:

*実戦で鍛えられた実装*: 何千人年のエンジニアリング、数百万アプリへのデプロイ。エッジケース、数値安定性、プラットフォームの癖、セキュリティ問題に対応しています。

*広範なオペレータサポート*: 数百の操作が実装済み（畳み込み、プーリング、正規化、注意機構、埋め込みなど）。各操作は入力サイズ、データ型、ハードウェア向けに最適化されています。

*ハードウェア加速*: Apple Neural Engine、Qualcomm Hexagon DSP、NVIDIA Tensor Cores、AMD Matrix Cores、Intel AMX、各種 NPU など専用 AI ハードウェアを自動利用。本番ライブラリは Vulkan コンピュートだけでは使えないハードウェアを活用します。

*継続的メンテナンス*: 新操作、性能改善、バグ修正、ハードウェアサポート、セキュリティパッチの継続更新。ライブラリ版を更新するだけで Vulkan アプリケーションも恩恵を受けます。

*開発時間の短縮*: 数ヶ月の実装・デバッグ・最適化の代わりに、数日または数時間で ML 推論が動きます。

== カスタム Vulkan 実装が意味を持つとき

これらの利点にもかかわらず、カスタム実装が正しい選択となるシナリオがあります:

*グラフィックスとの密結合*: ML 機能（アップスケーリング、デノイズ、手続き生成）が既存 Vulkan レンダリングパイプライン内で毎フレーム動くとき。カスタム実装は別 GPU コンテキスト間のデータ転送とコンテキスト切り替えを排除し、フレームあたりミリ秒を節約します。これらの機能は通常かなり小さく単純なネットワークなので、コンピュートシェーダでの実装が現実的な規模です。

*レンダリング AI 手法*: ガウシアンスプラッティング、ニューラルレンダリング、グラフィックス向け AI は限られた操作セットの軽量アーキテクチャを使うことが多い。カスタム実装はフレームレートが重要な手法向けに積極的な最適化を提供します。

*組み込み・リソース制約プラットフォーム*: Vulkan はサポートするが標準 ML フレームワークはサポートしない専用ハードウェア。カスタム実装はメモリ使用量、バイナリサイズ、実行の精密制御を提供——産業機器、自動車、航空宇宙、IoT に不可欠です。ただし OpenCL や SyCL など他の解決策が同等以上に埋め合わせる場合もある、と言わざるを得ません。組み込みでグラフィックスが必要で、すでに Vulkan コンテキストがある場合もあります。

*新しいモデルアーキテクチャ*: 標準フレームワークがまだサポートしない操作を使う最先端研究モデル。カスタム実装は最新論文やドメイン固有操作への柔軟性を提供します。ニューラルレンダリングや手続き生成のグラフィックス研究も、操作の独自性からカスタム実装を要することが多いです。

*学習と理解*: 最終的に本番ライブラリを使う場合でも、Vulkan レベルでの ML 推論の理解はデバッグ、最適化、アーキテクチャ判断を向上させます。

== ハイブリッドアプローチ

実務では多くのアプリがハイブリッドを取ります: 大半の推論ワークロードは本番ライブラリ、特定の性能クリティカルまたは密結合コンポーネントはカスタム Vulkan 実装です。

例: ゲームが ONNX Runtime で複雑な AI 行動（NPC 意思決定、戦略計画、対話生成）、カスタム Vulkan シェーダでリアルタイムアップスケールとデノイズ、TensorFlow Lite でより単純なモデルのモバイルデプロイ、といった組み合わせです。

両方の良さを得られます: 意味のある場所では本番ライブラリの信頼性と広いサポート、最も重要な場所ではカスタム実装の性能と統合の利点です。

== 本章で扱う内容

本章は Vulkan アプリケーションへのサードパーティ ML ライブラリ統合を包括的に扱います:

=== ライブラリ別の章

*xref:ML_Inference/Third_Party_Libraries/02_onnx_runtime.adoc[ONNX Runtime]* — Microsoft のフレームワーク非依存・クロスプラットフォーム推論エンジン。PyTorch、TensorFlow などからのモデルをサポート。汎用クロスプラットフォーム加速には **WebGPU** 実行プロバイダを推奨（**Vulkan** または **DirectML** をネイティブ利用）、**CUDA** など専用プロバイダでハードウェア固有最適化も可能。

*xref:ML_Inference/Third_Party_Libraries/03_tensorflow_lite.adoc[TensorFlow Lite]* — Google のモバイル向けライブラリ。サイズ、効率、バッテリー寿命に最適化。モデル圧縮向けの優れた量子化サポート。Android、iOS などモバイル向けの第一選択。

*xref:ML_Inference/Third_Party_Libraries/04_pytorch_mobile.adoc[PyTorch Mobile]* — Meta のモバイル推論ソリューション。PyTorch 学習ワークフローとの密結合。学習に PyTorch を使い TorchScript で学習からデプロイまでシームレスに行きたい場合に最適。

*xref:ML_Inference/Third_Party_Libraries/05_directml.adoc[DirectML]* — Microsoft の DirectX 12 ベース ML ライブラリ。Windows でベンダー非依存の GPU 加速。NVIDIA、AMD、Intel GPU で動作。ONNX Runtime の DirectML 実行プロバイダ経由での利用が一般的。

=== 統合と最適化の章

*xref:ML_Inference/Third_Party_Libraries/06_integration_patterns.adoc[統合パターン]* — Vulkan と ML ライブラリを橋渡しする 3 つの基本アプローチ: 別コンテキスト（シンプル、CPU 経由）、共有リソース（GPU メモリ共有）、インターリーブ実行（統一コマンドストリーム）。

*xref:ML_Inference/Third_Party_Libraries/07_gpu_resource_sharing.adoc[GPU リソース共有]* — Vulkan 外部メモリ拡張、プラットフォーム固有ハンドル型、クロスコンテキスト同期プリミティブによるゼロコピーデータ転送の技術実装。

*xref:ML_Inference/Third_Party_Libraries/08_data_transfer_optimization.adoc[データ転送の最適化]* — 転送オーバーヘッドの最小化、非同期転送キュー、メモリマッピング戦略、バッチングによるスループット改善。

*xref:ML_Inference/Third_Party_Libraries/09_practical_examples.adoc[実践例]* — デスクトップ、モバイル、リアルタイム向けの完全な動作例（ソース、ビルド手順、性能分析付き）。

*xref:ML_Inference/Third_Party_Libraries/10_choosing_library.adoc[適切なライブラリの選択]* — プラットフォーム、性能要件、統合の複雑さを考慮した体系的決定フレームワークとユースケース別推奨マトリクス。

=== ML コンパイラの章

*xref:ML_Inference/Third_Party_Libraries/11_ML_Compilers.adoc[ML コンパイラ概観]* — ML コンパイラとは何か、いつ使うか。
*xref:ML_Inference/Third_Party_Libraries/12_iree.adoc[IREE]* — Vulkan 中心のコンパイラ/ランタイム。AOT/JIT とモバイル/デスクトップ向け。
*xref:ML_Inference/Third_Party_Libraries/13_apache_tvm.adoc[Apache TVM]* — 自動チューニングと Vulkan/OpenCL バックエンドを備えたエンドツーエンドコンパイラスタック。

*xref:ML_Inference/Third_Party_Libraries/15_openxla.adoc[OpenXLA]* — IREE 経由の Vulkan を含む複数バックエンドの XLA/StableHLO エコシステム。

== 本章の読み方

本章はチュートリアルとリファレンスの両方として設計されています。効果的な使い方:

*ML ライブラリ統合が初めて*: ライブラリ別章（02–05）を順に読んで選択肢を理解し、統合パターン（06–08）で実装技法を学びます。

*使うライブラリが決まっている*: 該当章へ直接（例: モバイルなら TensorFlow Lite）。統合章（06–08）で実装ガイドを読みます。

*既存統合の最適化*: 章 07–08（GPU リソース共有とデータ転送最適化）に集中します。

*ライブラリ間の決定*: 章 10（適切なライブラリの選択）から体系的フレームワークを始め、該当ライブラリ章を読みます。

*単一プラットフォームのみ*: 非対応ライブラリはスキップ可能。例: Windows デスクトップのみなら PyTorch Mobile を飛ばし ONNX Runtime + DirectML に集中。

== 前提条件

本章は次に慣れていることを前提とします:

*Vulkan の基礎*: メモリ管理、コマンドバッファ、同期プリミティブ（バリア、セマフォ、フェンス）、コンピュートパイプライン。復習が必要ならメイン Vulkan チュートリアルへ。

*ML 推論の基礎*: ニューラルネットワーク推論とは何か、データがネットワークをどう流れるか、テンソル、レイヤ、操作の基本概念。前章（ML 推論パイプライン）が基盤を提供します。

*C/C++ プログラミング*: 例は C++17/20 のモダン機能を使用。RAII、スマートポインタ、基本的なテンプレートに慣れていること。

*ビルドシステム*: 例は CMake。CMake プロジェクトの設定とビルドができること。

扱う ML ライブラリの事前経験は不要——各ライブラリをゼロから紹介します。

== 実用性について

本章はシリーズ全体で最も即座に実用的です。シリーズの残りは低レベルで ML 推論の仕組みを教えますが、本章は今日の本番アプリで ML モデルを実際にデプロイする方法を示します。

多くの読者にとって本章はプロジェクトに直接当てはまります。すぐ使えるパターンと技法を学べます。カスタム推論エンジン構築の後章は理解と特殊用途に価値がありますが、大多数のアプリでは本章が理論と実践の接点です。

それでも本番ライブラリを使う予定でも後章を飛ばさないでください。内部の推論の仕組み——どの操作が高コストか、メモリ管理、同期が必要な場所——を理解すると、ライブラリベースの利用と最適化が格段に効果的になります。

主要な ML 推論ライブラリを見て、それぞれが何を提供するか理解することから始めましょう。
