= 適切な ML コンパイラの選択

== 導入

「最良」の ML コンパイラは一つではありません——制約に対する最良の適合があります。入力（ONNX、StableHLO、TFLite、NNEF）、ターゲットプラットフォーム（デスクトップ、モバイル、組み込み）、必要なバックエンド（Vulkan、Metal、CUDA）、開発者ワークフロー、ランタイムフットプリント、チューニングにどれだけ投資するかに合わせて最適化してください。

== 譲れない条件から始める

絶対に必要なものを決めてください。プラットフォーム全体で Vulkan が必須なら、最初の候補は xref:ML_Inference/Third_Party_Libraries/12_iree.adoc[IREE] と xref:ML_Inference/Third_Party_Libraries/13_apache_tvm.adoc[TVM] です。小さなランタイムと ahead‑of‑time パッケージングが不可欠なら、IREE または TVM の AOT/CRT 経路を優先します。特定のスマホや iGPU 向けのデバイス固有オートチューニングが必要なら、TVM が輝きます。学習パイプラインがすでに StableHLO/XLA を話すなら、StableHLO をエクスポートして Vulkan 向けに IREE でコンパイルすることを優先し、非 Vulkan ターゲットでは XLA はそのレーンに留まります。チームが Modular のツールに標準化され、そのバックエンドがハードウェアをカバーするなら、MAX/Mojo が最も快適な適合になり得ます。

このフローで初期選択に素早く到達できます。そこから概念実証を作り、検証してください。

== シナリオ別の指針

デスクトップと Android で Vulkan を出荷し、小さく単純なランタイムが欲しければ、IREE から始めてください。1 台のスマホや 1 つの iGPU でミリ秒がすべてなら、Vulkan 上の TVM を使いチューニングしてください。組織がすでに TensorFlow/JAX から StableHLO をエクスポートしているなら、そのエクスポートを保ち、Vulkan 向けに IREE でコンパイルしてください。Modular のエコシステム内で快適でデバイスがサポートされているなら、MAX/Mojo が最も摩擦が少ないかもしれません。

考え方の錨になる具体例を 2 つ:

1) 組み込みカメラアプリ（バッテリ／熱が重要）: 熱スロットリング下での予測可能なレイテンシと小さなフットプリントが欲しい。固定形状モデルをエクスポートし、Vulkan 向けに IREE でコンパイルし、CPU ゴールデン参照に対して数値を検証して出荷する。デバイス上で p95 レイテンシがまだ高すぎるなら、TVM とオンデバイスオートチューンを試し、同じ検証ハーネスを保つ。レイテンシ／分散目標を満たし、統合コストが最小の方を選ぶ。

2) デスクトップバッチツール（スループットが支配）: 画像フォルダや短い動画を処理する。IREE も TVM も動く。単純な経路が勝つ。多様なハードウェア群にわたる GPU ごとのチューニングを見込むなら TVM から。ポータブルな 1 つの成果物と最小限のランタイム荷物が欲しければ IREE から。いずれの場合も、マシンに課金するならワットあたり・ドルあたりのスループットを測る。

== NNEF の考慮

真実のソースが NNEF なら、まず ONNX または StableHLO に変換します。StableHLO からは IREE で Vulkan に到達し、ONNX からは TVM（Relay）にインポートして Vulkan をターゲットにします。非 Vulkan ターゲットでは、ネイティブの下流（MAX/Metal、XLA/CUDA）に従います。変換をスクリプト化し、小さなサンプルでテストして、驚きが早く出るようにしてください。

== 概念実証チェックリスト

実モデルを実デバイスで使います。買い物リストではなく物語として保ってください。

まず、固定入力テンソルを固定し、CPU ゴールデン出力（ONNX Runtime）を保存します。次に、IREE と TVM で Vulkan 向けに、既定の保守的オプションでモデルをコンパイルします。ターゲットで数分間両方を実行します: N 回ウォームアップし、クロック／熱が落ち着いたところで p50/p95 レイテンシとメモリを測ります。両方の経路が正しく安定してから初めて fp16 や INT8 を試します。特定のスマホ／iGPU で最後の 20% が必要なら、TVM のオートスケジューラを回して再測定し、そうでなければ単純な統合を優先します。数字と成果物／ランタイムサイズを手元に置いて決定してください。

避けるべきよくある罠:

- 自分のモデルとデバイスではなく、他人のベンチマークに基づいて選ぶ。
- ゴールデン参照比較を飛ばし、静かな数値ドリフトを「許容」する。
- 前処理とコンパイル済みグラフでレイアウトを混ぜる。一度決めて守り通す。
- 正しい前にチューニングする、またはカーネル時間が支配的なのにコピーを最適化する（あるいはその逆）。
- 平均レイテンシだけを報告する。ユーザーが感じるのは p95（対話アプリでは p99）です。
