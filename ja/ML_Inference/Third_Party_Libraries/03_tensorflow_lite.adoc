= TensorFlow Lite

== 導入

Google が機械学習をモバイルデバイスへ持ち込もうとしたとき、根本的な課題に直面しました。旗艦 ML フレームワークである TensorFlow は、スマートフォンや組み込み機器に載せるには大きすぎ、複雑すぎたのです。フル機能の TensorFlow ランタイムは数百 MB にもなり、相当なバッテリーを消費し、モバイルハードウェアには存在しないリソースを要求します。

TensorFlow Lite はこの制約から生まれました。単に小型化した TensorFlow ではなく——リソースの限られたデバイス向けの推論ランタイムがどうあるべきかを、根本から見直したものです。すべての設計判断が、柔軟性や開発のしやすさよりもサイズ、効率、バッテリー寿命を優先しています。

本章では TensorFlow Lite を深く探ります。フル機能の TensorFlow との違い、モバイル ML デプロイの標準的な選択肢である理由、Vulkan アプリとの統合方法、そしていつ適切なツールになるのかを見ていきます。

== モバイルファーストの哲学

TensorFlow Lite を理解するには、それが設計された制約を理解する必要があります。モバイルデバイスは単に小さい PC ではありません——ソフトウェア設計のあらゆる面を形作る、根本的に異なる制約の下で動作しています。

=== バッテリーの制約

デスクトップはコンセントに接続されています。サーバーは電力が無限に供給されるデータセンターにあります。モバイルは 1 日持つことが期待されるバッテリーで動きます。1 ミリワットたりとも重要です。

機械学習は計算集約的で、計算は電力を消費します。素朴な ML の実装は、1 時間でスマートフォンのバッテリーを使い切ってしまうかもしれません。TensorFlow Lite の設計全体は、正確で高速な推論を維持しながら消費電力を最小化するという必要性に形作られています。

これはすべてに影響します。ファイル形式（小さいモデルは読み込みが速く消費電力も少ない)、ランタイム（柔軟性より効率を優先した最適化)、量子化サポート（整数演算は浮動小数点より消費電力が少ない)、ハードウェア加速（利用可能なら専用の低消費電力 AI アクセラレータへ処理を委譲する)などです。

=== サイズの制約

モバイルアプリは限られたストレージを奪い合っています。大きすぎるアプリはユーザーに削除されてしまいます。アプリサイズの 1 MB ごとが、インストールと維持を妨げる障壁になります。

TensorFlow Lite のランタイムは、コア機能だけならおよそ 300KB です。すべてのオペレータを含めても 1MB 未満です。フル機能の TensorFlow の数百 MB と比較してみてください。この劇的なサイズ削減は徹底的な最適化から生まれています。学習用コードの削除、不要なオペレータの排除、コンパクトなバイナリ形式、依存関係の最小化です。

モデル自体もサイズ向けに最適化できます。TensorFlow Lite の量子化ツールを使えば、精度の低下を最小限に抑えながらモデルサイズを 4 倍削減できます（32 ビット浮動小数点から 8 ビット整数へ)。モバイルデプロイでは、速度の改善よりもこのサイズ削減の方が価値を持つことが多いのです。

=== 性能の制約

モバイルのプロセッサはデスクトップの CPU やサーバーの GPU より非力です。生の性能ではなく効率を重視した設計になっています。それでもユーザーは ML 機能が即座に反応することを期待します——カメラアプリがシーン認識に何秒も待たせるのは論外です。

TensorFlow Lite はいくつかの戦略でこれを実現しています。一般的な操作の徹底的な最適化、デリゲート（GPU、NPU、DSP)によるハードウェア加速、量子化(多くのモバイル CPU では整数演算が浮動小数点より速い)です。結果として、ミッドレンジのスマートフォンでもリアルタイムアプリに十分な速さの推論が得られます。

=== 多様性の制約

ハードウェアを自分で管理できるサーバーデプロイとは異なり、モバイルは数千種類の異なるデバイスをサポートする必要があります。Android だけでも数百のメーカーがあり、それぞれ異なる CPU、GPU、AI アクセラレータを積んでいます。iOS はより均一ですが、世代によって能力が異なります。

TensorFlow Lite はデリゲートという仕組みでこの多様性に対応します。同じモデルが CPU（どこでも動く)、OpenGL ES または Vulkan 経由の GPU（多くのモダンなデバイス)、NPU（高級機に搭載されるニューラル処理ユニット)、専用アクセラレータ（Google の Edge TPU や Qualcomm の Hexagon DSP など)で動作します。ランタイムが利用可能な最良のオプションを自動的に選び、専用ハードウェアがなければ穏やかにフォールバックします。

== アーキテクチャ: 効率を重視した設計

TensorFlow Lite のアーキテクチャはモバイルファーストの哲学を反映しています。各コンポーネントが、これまで述べた制約に対して最適化されています。

=== FlatBuffers 形式

モデルの形式そのものが重要な革新です。TensorFlow Lite は、Google が効率を重視して開発したシリアル化ライブラリである FlatBuffers を使用します。

JSON や Protocol Buffers といった従来の形式はパースが必要です。ファイルを読み、メモリ上の表現を構築し、そこからデータへアクセスします。パースには時間もメモリもかかります。50MB のモデルなら、読み込み中に 100MB（ファイル本体とパース後の表現)ものメモリが必要になり、読み込みに数秒かかることもあります。

FlatBuffers はパースそのものをなくします。ファイル形式は、メモリマップしてそのまま直接アクセスできるように設計されています。読み込みはほぼ瞬時です——ファイルをメモリにマップするだけです。メモリ使用量も最小限です——ファイルサイズ分だけで、追加のパース済み表現は不要です。

モバイルではメモリが貴重で、ユーザーは即座の起動を期待するため、これは非常に重要です。TensorFlow Lite のモデルは秒単位ではなくミリ秒単位で読み込め、同等の形式に比べて半分のメモリで済みます。

=== オペレータセット: 焦点を絞った最適化

TensorFlow は数百の操作をサポートしています。TensorFlow Lite は、本番のモバイルモデルに実際に現れる操作だけを厳選したサブセットです。

これは制限ではなく機能です。小さな操作セットに集中することで、各操作を徹底的に最適化できます。各操作には、ARM NEON の SIMD（モバイル CPU 向け)、最適化された GPU カーネル、NPU 向けの実装など、アーキテクチャ別に手で調整された実装が用意されています。

このセットには、一般的なモバイル ML タスクに必要なものがすべて含まれています。畳み込み（2D、depthwise、grouped)、プーリング、正規化、活性化、全結合などです。モデルがこれらだけを使うなら（多くの本番モデルはそうです)、高度に最適化された推論が得られます。エキゾチックな操作を使うなら、フル機能の TensorFlow やカスタム操作が必要になるかもしれません。

=== インタープリタ: 軽量で高速

TensorFlow Lite のインタープリタはモデルを実行するランタイムです。小さく、速く、予測可能であるよう設計されています。

フル機能の TensorFlow の複雑なグラフ実行エンジンとは異なり、TFLite のインタープリタは単純明快です。モデルの操作を順に歩いて各操作を実行するだけです。動的なグラフ構築、自動微分、学習の仕組みは一切ありません。可能な限り速い順方向推論だけを行います。

このシンプルさには、サイズと速度を超えた利点があります。動作が予測可能で、挙動を推論しやすいのです。メモリ使用量も決定的で、実行時間も一貫しています。モバイルアプリでは、予測不能な動作が悪いユーザー体験に直結するため、これは重要な特性です。

=== デリゲートシステム: ハードウェア加速

デリゲートシステムは、ハードウェアの多様性に対する TFLite の答えです。デリゲートとは、特定のハードウェアで推論を加速するプラグインです。

GPU デリゲートは OpenGL ES または Vulkan を使ってモバイル GPU 上で操作を実行します。NNAPI デリゲート（Android Neural Networks API)は、デバイスが提供する AI アクセラレータ（DSP、NPU、専用の AI チップ)を使用します。Hexagon デリゲートは Qualcomm の Hexagon DSP をターゲットにします。Core ML デリゲートは iOS で Apple の Core ML を使用します。

アプリの視点ではシンプルです。デリゲートを作成してインタープリタに付けるだけで、推論が自動的に加速用ハードウェアを使うようになります。デリゲートがある操作を扱えない場合は CPU にフォールバックします。この穏やかな劣化により、すべてのデバイスで動作し、専用ハードウェアがある場合はより速く動きます。

== 量子化: 一番の強み

TensorFlow Lite を特徴づける機能を一つ挙げるなら量子化でしょう。モバイルデプロイにおける TFLite の価値の中心にあります。

=== 量子化とは何か

ニューラルネットワークは通常、重みと活性化に 32 ビットの浮動小数点数を使います。重み 1 つあたり 4 バイトです。2500 万パラメータのモデルなら 100MB になります。

量子化はこれを 8 ビットの整数に変換します。重み 1 つあたり 1 バイトです。同じ 2500 万パラメータのモデルが 25MB になります——4 倍の削減です。

精度は損なわれないのでしょうか。驚くほど損なわれません。多くのモデルで、量子化による精度低下は 1% 未満です。一部では精度がわずかに改善することもあります(精度の低下が正則化のように働き、過学習を防ぐためです)。

サイズを超えた利点もあります。

*推論の高速化*: 多くのモバイル CPU では整数演算が浮動小数点演算より速く、量子化モデルは 2〜4 倍速くなります。

*低消費電力*: 整数演算は消費電力が少なく、バッテリー寿命が延びます——モバイルにとって非常に重要な特性です。

*ハードウェア加速*: 多くのモバイル AI アクセラレータは整数演算向けに最適化されており、量子化モデルは専用ハードウェアを活用できます。

=== 量子化の仕組み

基本的な考え方はシンプルです。浮動小数点値の範囲を 8 ビット整数の範囲(0〜255 または -128〜127)へマッピングします。

例えば、重みが -2.0 から 2.0 の範囲であれば、-2.0 を -128 に、2.0 を 127 にマッピングします。その間の値は比例して配置されます。0.0 は 0 に、1.0 は 64 になる、といった具合です。

このマッピングは *scale*（尺度)と *zero-point*（ゼロ点)という 2 つのパラメータで定義されます。scale は各整数のステップが浮動小数点空間で何を表すかを示し、zero-point はどの整数が 0.0 を表すかを示します。

推論中、操作は整数のまま行われます。2 つの量子化された値を乗算するときは整数どうしを乗算し、結果を scale で調整します。整数演算はコストが低いため、浮動小数点の乗算より依然として速く済みます。

=== 量子化の種類

TensorFlow Lite は複数の量子化方式をサポートしており、それぞれ異なるトレードオフがあります。

*動的レンジ量子化*: 最もシンプルな方式です。重みは int8 にしますが、活性化は float32 のままにします。モデルサイズを 4 倍削減し、ある程度の高速化も得られます(重みの読み込みが速くなるため)が、整数のみの推論による性能の全利益は得られません。

使うべき場面: 最小限の手間で小さなモデルが欲しく、適度な高速化で十分な場合です。

*完全整数量子化*: 重みと活性化の両方を int8 にします。最大のサイズ削減と高速化が得られますが、代表データセットによるキャリブレーション(活性化の範囲を決定するため)が必要です。

使うべき場面: 最大の性能が必要で、代表的なデータセットを用意できる場合です。

*Float16 量子化*: 重みを 16 ビットの浮動小数点数にします。2 倍のサイズ削減が得られ、精度の低下はごくわずかです。完全精度と int8 の中間に位置します。

使うべき場面: サイズを削減したいが、モデルが int8 化に敏感な場合です。

*量子化認識学習*: 学習後に量子化するのではなく、学習中に量子化をシミュレートします。モデルが量子化に対して頑健になるよう学習されるため、学習後量子化よりも良い精度が得られます。

使うべき場面: 学習後量子化では精度の低下が許容できない場合です。

=== 実践的な量子化ワークフロー

TensorFlow Lite 向けにモデルを量子化する典型的な流れは次のとおりです。

* TensorFlow または Keras で学習する（フル精度の float32）
* 量子化を適用しながら TensorFlow Lite 形式へ変換する。

[source,python]
----
converter = tf.lite.TFLiteConverter.from_keras_model(model)
converter.optimizations = [tf.lite.Optimize.DEFAULT]
converter.representative_dataset = representative_dataset_gen
converter.target_spec.supported_ops = [tf.lite.OpsSet.TFLITE_BUILTINS_INT8]
converter.inference_input_type = tf.int8
converter.inference_output_type = tf.int8
tflite_model = converter.convert()
----

* 量子化したモデルを検証する（テストデータで推論を行い、元のモデルと精度を比較する）
* 許容できるならデプロイする。そうでなければ量子化認識学習を試す。

`representative_dataset_gen` はサンプル入力を生成する関数です。コンバータがこれをモデルに通して活性化の範囲を決定し、量子化パラメータの設定に利用します。データセットは大きくなくても構いません——通常は数百例で十分です。

== Vulkan アプリケーションとの統合

実践的な問いです。TensorFlow Lite を Vulkan ベースのアプリとどのように統合すればよいでしょうか。

=== 基本的なアプローチ: 別コンテキスト

最もシンプルな統合方法は、TensorFlow Lite と Vulkan を CPU メモリを介して通信する別々のシステムとして扱うことです。

Vulkan アプリはフレームのレンダリング、入力処理、ゲーム状態の管理を行います。ML 推論が必要になったときは次の流れになります。

1. 関連データを GPU メモリから CPU メモリへコピーする（例: レンダリング済みのフレーム）
2. 必要に応じて前処理を行う（リサイズ、正規化など）
3. TensorFlow Lite で推論を行う（GPU デリゲート経由で内部的に GPU を使う可能性もある）
4. CPU で推論結果を処理する
5. レンダリングに必要なら結果を GPU メモリへ戻す

これは単純明快で、多くのアプリでうまく機能します。推論が毎フレーム発生しない場合や、推論時間が転送時間を上回るような場合には、CPU-GPU 間の転送オーバーヘッドは許容できる範囲です。

=== GPU デリゲートの活用

TensorFlow Lite の GPU デリゲートはモバイル GPU 上で推論を加速します。Android では内部的に OpenGL ES または Vulkan を使用します。ここで興味深い問いが生まれます。Vulkan によるレンダリングと TFLite の GPU 推論で、GPU リソースを共有できるのでしょうか。

短い答えは、直接は共有できないが、オーバーヘッドは最小化できるということです。

TFLite の GPU デリゲートは独自の Vulkan（または OpenGL ES）コンテキストを作成します。アプリの Vulkan コンテキストとメモリや同期プリミティブを共有することはありません。ただし両方のコンテキストは同じ物理 GPU を使うため、GPU のリソースが十分であれば並行して実行できます。

オーバーヘッドを最小化する方法を挙げます。

*推論のバッチ化*: 毎フレームではなく、複数フレームをまとめて処理します。コンテキスト切り替えのオーバーヘッドを分散できます。

*非同期実行*: 別スレッドで推論を行います。GPU がレンダリングコマンドを処理している間に、CPU が次の推論リクエストを準備します。計算を重ね合わせることで、感じられるレイテンシを削減できます。

*転送の最適化*: CPU-GPU 間の転送を最小限にします。可能であれば CPU 上でデータを保持し（CPU ベースの前処理を行うなど)、往復を避けます。

=== 上級編: 共有メモリ（プラットフォーム固有）

一部のプラットフォームでは、外部メモリ拡張を使うことで Vulkan コンテキストと TFLite の GPU デリゲートの間でメモリを共有できます。上級者向けでプラットフォームに依存しますが、コピーを排除できます。

一般的なアプローチは次のとおりです。

1. Vulkan の外部メモリ拡張(VK_KHR_external_memory)でメモリを割り当てる
2. メモリハンドルをエクスポートする（Linux/Android では fd、Windows では handle）
3. TFLite の GPU デリゲートへインポートする（カスタムデリゲートの実装や改変が必要な場合があります）
4. 両方のコンテキストが同じメモリにアクセスし、コピーを排除する

これは複雑で、TFLite の標準の GPU デリゲートでは公式にサポートされていません。Vulkan と TFLite 内部の深い理解が必要です。多くのアプリでは、よりシンプルな別コンテキストの方式で十分です。

=== モバイル固有の考慮点

モバイルでは追加の考慮点があります。

*Android*: TFLite の Android サポートは優秀です。加速には GPU デリゲートを使い、デバイス固有の AI アクセラレータには NNAPI デリゲートを使います。Android のメモリ制限には注意してください——大きなモデルは低スペックの端末で OOM（メモリ不足)を起こす可能性があります。

*iOS*: 最高の性能を得るには Core ML デリゲートを使います。Neural Engine を搭載したデバイス(iPhone 8 以降、2018 年以降の iPad Pro)で活用されます。デリゲートを有効にするだけで、あとは Core ML が処理します。

*バッテリー寿命*: 開発中は消費電力を監視してください。連続的な ML 推論はバッテリーを急速に消耗させます。推論の頻度を減らす、小さなモデルを使う、必要なときだけ推論する(ユーザーが機能を実際に使っているときだけなど)といった戦略を検討してください。

== 実践例: 画像分類

完全な例として、Vulkan ベースのモバイルゲームに TensorFlow Lite の画像分類を統合してみましょう。

=== シナリオ

カメラを使った機能を持つモバイルゲームを構築します。プレイヤーがカメラを実世界のオブジェクトに向けると、ゲームがそれを識別します。ミッドレンジの Android デバイスでリアルタイム推論(少なくとも 10 FPS)が求められます。

=== モデルの選択

モバイル向けに設計された MobileNetV2 を選びます。小さく(14MB)、速く(ミッドレンジのスマートフォンで 30ms の推論)、正確です(ImageNet で top-1 精度 71%)。

TensorFlow Hub から事前学習済みモデルをダウンロードし、量子化を適用しながら TensorFlow Lite 形式へ変換します。

[source,python]
----
import tensorflow as tf

# モデルを読み込み
model = tf.keras.applications.MobileNetV2(weights='imagenet')

# 量子化を適用して TensorFlow Lite に変換
converter = tf.lite.TFLiteConverter.from_keras_model(model)
converter.optimizations = [tf.lite.Optimize.DEFAULT]
tflite_model = converter.convert()

# モデルを保存
with open('mobilenet_v2_quantized.tflite', 'wb') as f:
    f.write(tflite_model)
----

量子化したモデルは 3.5MB です——ダウンロードサイズを大きく増やすことなくアプリに同梱できます。

=== Android への統合

Android アプリで TensorFlow Lite を統合します。

[source,java]
----
// モデルを読み込み
Interpreter tflite = new Interpreter(loadModelFile());

// 加速のため GPU デリゲートを有効化
GpuDelegate delegate = new GpuDelegate();
Interpreter.Options options = new Interpreter.Options();
options.addDelegate(delegate);
tflite = new Interpreter(loadModelFile(), options);

// 入力バッファを準備（224x224x3 画像、[0,1] に正規化）
ByteBuffer inputBuffer = ByteBuffer.allocateDirect(224 * 224 * 3 * 4);
inputBuffer.order(ByteOrder.nativeOrder());

// 出力バッファを準備（1000 クラス）
float[][] output = new float[1][1000];

// 推論を実行
tflite.run(inputBuffer, output);

// 結果を処理
int predictedClass = argmax(output[0]);
String label = labels[predictedClass];
----

=== Vulkan との統合

Vulkan のレンダリングパイプラインがカメラフレームをキャプチャします。推論の統合手順は次のとおりです。

1. *フレームキャプチャ*: Vulkan のパイプラインがカメラフィードをテクスチャへレンダリングします
2. *読み戻し*: `vkCmdCopyImageToBuffer` でテクスチャをステージングバッファへコピーします
3. *メモリマップ*: ステージングバッファを CPU メモリにマップします
4. *前処理*: 224×224 にリサイズし、[0,1] に正規化し、TFLite が期待する形式へ変換します
5. *推論*: バックグラウンドスレッドで TensorFlow Lite の推論を実行します
6. *結果表示*: 推論が完了したら UI を予測ラベルで更新します

鍵となるのは非同期推論です。GPU が次のフレームをレンダリングしている間に、CPU が前のフレームの推論を処理します。計算を重ね合わせることで滑らかなフレームレートを維持できます。

=== 性能の最適化

最初のテストでは推論に 40ms かかり、25 FPS と許容範囲ではあるものの最高ではありませんでした。以下の最適化を行いました。

*GPU デリゲートの利用*: 推論が 20ms(50 FPS)になりました。畳み込みにモバイル GPU を活用しています。

*入力サイズの削減*: MobileNetV2 はより小さな入力も受け付けられます。224×224 の代わりに 192×192 を使うと 15ms(66 FPS)になり、精度の低下はごくわずかでした。

*量子化*: モデルは既に量子化済みでしたが、GPU デリゲートが整数演算を使っていることを確認しました。さらに 20% の高速化が得られ、推論は 12ms(83 FPS)になりました。

*推論のバッチ化*: 毎フレームではなく 3 フレームごとに推論を行うようにしました。平均のオーバーヘッドが削減され、60 FPS のレンダリングを維持しつつ、応答性の良い物体認識を実現できました。

最終的な実装は、ユーザーには即座に反応するように感じられ、ミッドレンジデバイスでも滑らかで、バッテリーへの影響も最小限でした。

== TensorFlow Lite が適している場面

TensorFlow Lite が常に正しい選択というわけではありません。輝く場面と代替案を検討すべき場面を見ていきましょう。

=== 理想的なユースケース

*モバイルアプリケーション*: Android または iOS へのデプロイであれば TFLite が既定の選択です。エコシステム、ツール、最適化のどれも群を抜いています。

*サイズ制約のある環境*: バイナリサイズが重要な場合（モバイルアプリ、組み込み)には、TFLite の小さなランタイムが大きな利点になります。

*バッテリー駆動のデバイス*: 効率を重視した最適化により、TFLite はバッテリー駆動デバイスに理想的です。

*量子化に適したモデル*: モデルが量子化にうまく対応する場合(多くの vision モデルがそうです)、TFLite の量子化サポートは大きな利益をもたらします。

*標準的なアーキテクチャ*: 一般的な操作(畳み込み、プーリング、標準的な活性化)だけを使うなら、TFLite の最適化されたオペレータセットが最適です。

=== 代替案を検討すべき場面

*デスクトップ/サーバーへのデプロイ*: デスクトップやサーバーでは、TFLite のモバイル向け最適化の意義は薄れます。ONNX Runtime やフル機能の TensorFlow の方が適しています。

*カスタム操作*: TFLite のオペレータセットにないエキゾチックな操作を使うなら、カスタム操作の実装や別のランタイムが必要になります。

*動的なモデル*: 動的な形状や制御フローを持つモデルは TFLite でのデプロイが難しくなります。ONNX Runtime の方が適しています。

*クロスフレームワークの互換性*: PyTorch で学習したなら、TFLite への変換には余分なステップがかかります(PyTorch → ONNX → TensorFlow → TFLite)。ONNX Runtime の方がシンプルかもしれません。

*密な Vulkan 統合*: ML 推論と Vulkan のレンダリングの間でメモリや同期プリミティブを共有する必要があるなら、TFLite よりもカスタムの Vulkan コンピュートシェーダの方が適しているかもしれません。

== まとめ

TensorFlow Lite は、リソースの限られたデバイス向けにゼロから設計された、Google のモバイル ML デプロイ向けソリューションです。モバイルファーストの哲学がすべての面を形作っています。即座に読み込める FlatBuffers、徹底的な最適化を可能にする焦点を絞ったオペレータセット、予測可能な性能を実現する軽量なインタープリタ、ハードウェア加速のためのデリゲートシステムです。

量子化は TFLite の一番の強みで、最小限の精度低下で 4 倍のサイズ削減と 2〜4 倍の高速化を実現します。ツールが量子化を扱いやすくし、ランタイムが量子化された推論を効率的に処理します。

Vulkan アプリとの統合は通常、CPU メモリを介したデータ転送を伴う別コンテキストの方式になります。カスタムの Vulkan コンピュートほど密には結合していませんが、シンプルで信頼性が高く、多くのアプリで十分な性能を発揮します。

TensorFlow Lite はモバイルデプロイ、サイズ制約のある環境、バッテリー駆動デバイスで最も輝きます。デスクトップへのデプロイや密な Vulkan 統合が必要な場合は、ONNX Runtime やカスタム実装を検討してください。
