= ML コンパイラ: 概要と使うタイミング

== 導入

ML コンパイラは、学習済みモデルを特定デバイス向けのプログラムに変えます。実行時にグラフを演算ごとに解釈する代わりに、コンパイラは中間表現（IR）を通じてモデルを lower し、グラフとカーネルの最適化（融合、タイリング、レイアウト特化、精度変更）を適用し、ハードウェア向けに調整されたコード（GPU、CPU、DSP/NPU。われわれにとっては SPIR‑V 経由の Vulkan）を出力します。

本セクションは、推論への三経路アプローチにおける ML コンパイラ経路を紹介し、実際に何かを出荷できるよう考え方を教えます。

*経路 1: サードパーティランタイム*（ONNX Runtime、TensorFlow Lite など）— 統合が速く、広くサポートされる。

*経路 2: カスタム Vulkan* — 教育的で、密なグラフィックス統合には時に必要。

*経路 3: ML コンパイラ* — 高レベルなワークフローを保ちつつ、効率的でポータブルなバイナリやコマンドストリームを生成する中間地帯で、しばしば Vulkan/WebGPU/Metal バックエンドを持つ。アプリは引き続き「自分のもの」。コンパイラはコンパクトな成果物とそれを実行する小さなランタイムを与える。

== メンタルモデル: コンパイル対解釈、そして境界契約

最も有用な転換は、コンパイル済みモデルを狭い API 面を持つプログラムとして扱うことです。

- アプリは引き続きカメラ、UI、ファイル I/O、ログ、電力／熱ポリシー、および任意の Vulkan 前処理／後処理シェーダを所有する。
- コンパイラのランタイムはモデル実行を所有する: SPIR‑V モジュール、ディスクリプタレイアウト、ディスパッチスケジューリング、モデル内の同期。

交換は退屈に保つ: 「テンソルを入れて、テンソルを出す。」境界ではまず単純なホスト可視コピーから始める。正しさが退屈になったら、重要なところでだけデータ移動（ゼロコピー、外部メモリ、またはランタイム相互運用フック）を最適化する。

=== 60 秒入門: MLIR、StableHLO、XLA、IREE

- MLIR は IR と変換を構築するためのコンパイラ基盤です。演算と型を「ダイアレクト」に整理します。IREE のようなプロジェクトは MLIR を使い、ML グラフを Vulkan（SPIR‑V 経由）のようなターゲットへ段階的に lower します。xref:ML_Inference/glossary.adoc[用語集] を参照。
- StableHLO は標準化された ML IR（かつ MLIR ダイアレクト）です。フレームワークからのエクスポート後の、ニューラルネットワーク向けポータブル「アセンブリ」として扱ってください。
- XLA は OpenXLA エコシステム内のコンパイラで、TensorFlow/JAX が StableHLO/HLO を CPU/GPU に lower するのに使います。
- IREE は別の OpenXLA プロジェクトで、StableHLO/MLIR を Vulkan（SPIR‑V 経由）、Metal、CUDA、CPU などのターゲット向けデプロイ成果物にコンパイルし、小さなランタイムを持ちます。

言及されがちな一般的な MLIR ダイアレクト: StableHLO、TOSA、Linalg、MHLO。詳細を心配しないでください——各章が具体的なコマンドとコードを示し、用語集に簡潔な定義があります。

Vulkan では、本セクションで使う経路は典型的に次です: StableHLO → IREE → SPIR‑V（Vulkan）、または ONNX → TVM → SPIR‑V（Vulkan）。MLIR/StableHLO に不慣れなら、xref:ML_Inference/Third_Party_Libraries/15_openxla.adoc[OpenXLA] をざっと読み、xref:ML_Inference/glossary.adoc[用語集] の短い定義を見てください。

== ML コンパイラが行うこと

1. モデルをインポートする（ONNX、TFLite、StableHLO/XLA、TorchScript など）。
2. 1 つ以上の中間表現（IR）に lower する。
3. グラフレベルとカーネルレベルの最適化を適用する（融合、タイリング、量子化、レイアウト変換）。
4. 選択したバックエンド向けにコードを生成する（SPIR‑V、LLVM、CUDA、Metal など）。
5. デプロイ用の成果物と最小限のランタイムスタブをパッケージする。

== モデル形式との関係（NNEF を含む）

コンパイラはしばしば ONNX および／または StableHLO/XLA を受け付けます。NNEF（Neural Network Exchange Format）をサポートするものや、変換ツールを提供するものもあります。実務上、今日最も一般的なのは ONNX と StableHLO です。ただしツールチェーンやベンダーが NNEF を使うなら、ML コンパイラは SPIR‑V 経由、または WebGPU 経路経由で NNEF モデルを Vulkan に載せる実務的な手段になり得ます。グラフ、テンソル、レイアウトの考え方については xref:ML_Inference/Building_the_Inference_Engine/06_nnef_introduction.adoc[NNEF 入門] を参照——同じ考え方がコンパイラ出力の検証にも役立ちます。

== ML コンパイラを選ぶタイミング

- 正確なデバイス向けの ahead‑of‑time（AOT）または just‑in‑time（JIT）特化が欲しい。
- 汎用ランタイムを超える性能が必要だが、シェーダを手書きしたくない。
- 1 つのソースモデルから複数バックエンド（Vulkan、Metal、CUDA）が必要。
- 量子化や混合精度を使い、コンパイラ駆動の検証が欲しい。

続く章ではいくつかの選択肢を扱います: IREE（OpenXLA プロジェクト）、Apache TVM、Modular の MAX/Mojo ツールチェーン、OpenXLA の XLA/StableHLO エコシステム——最後に短い「コンパイラの選択」ガイドで締めます。

== 最初の動作するビルド（概念であり買い物リストではない）

網羅的なフラグより、退屈な再現性を目指してください。実務で効く弧は次です。

1. 学習から固定形状のモデル変種を 1 つ選ぶ。ソースが NNEF なら、まず ONNX または StableHLO に変換し、コンパイラの幸福な経路に乗る。
2. 選んだツールで Vulkan 向けにコンパイルする:
   - IREE: StableHLO/MLIR 入力に `--iree-hal-target-backends=vulkan` → `.vmfb` 成果物。
   - TVM: ONNX を Relay にインポートし `target="vulkan"` → デプロイ可能なモジュール（共有ライブラリまたは AOT）。
3. まず開発マシンで実行し（必要ならヘッドレス Vulkan）、CPU ゴールデン参照（ONNX Runtime）と出力を比較し、それからデバイスへデプロイする。

Vulkan の前処理／後処理はコンパイラランタイムの外に置き、経路（ランタイム対コンパイラ対カスタム）を変えてもアプリの残りを同一に保ちます。これにより A/B 検証と将来の入れ替えが低リスクになります。

=== 「SPIR‑V だけくれ」についての注記

IREE も TVM も最終的に SPIR‑V を生成しますが、共有 ABI を持つランタイム経由で実行します。カーネルを取り出し、自前の Vulkan エンジン内でホストスケジューラを再実装する経路は、現状は安定でもサポートされてもいません。コンパイラを ONNX Runtime/TFLite と同様に扱ってください: ランタイムをリンクし、成果物を読み込み、入力を設定し、実行し、出力を取得する。より密な相互運用が必要なら、特定ランタイムのサポートされるフックを探してください。
